# 审计事件持久化设计

## 背景

当前系统已有审计事件发布契约和基础表结构，但还缺少持久化监听、签名链推进和真实合规查询。这个缺口会阻塞规则发布、知识包发布、导出、回滚和合规证据链。

## 目标

- 在业务事务成功提交后持久化审计事件。
- 每个租户维护独立防篡改签名链。
- 合规审计 API 返回真实持久化数据。
- 审计写入失败可观测，但不破坏已经成功的业务事务。

## 关键设计

### 事务边界

`AuditPersistenceSink` 在 `AFTER_COMMIT` 监听事件。写入使用 `REQUIRES_NEW`，避免审计失败回滚业务事务。没有外层事务的路径通过普通事件监听补齐。

### 签名链

每个租户一条链。写入前锁定 `audit_chain_head`，读取上一事件和上一签名，生成规范化载荷，再用 `SmCryptoService.sm3Hex` 生成签名。写入事件和更新链头在同一事务内完成。

缺失租户时写入 `__SYSTEM__` 链，用于系统级事件，但业务控制器仍应尽量提供真实租户上下文。

### 查询边界

`AuditQueryService` 强制要求当前租户上下文，医院和科室只作为可选过滤条件。当前不提供跨租户“看全部”接口，避免审计数据误暴露。

### 控制器

`/api/v1/compliance/audit/events` 返回游标分页的真实审计记录。`/snapshot` 发布真实 `EXPORT` 事件，并回读刚持久化的审计视图，形成端到端证据。

## 风险与缓解

| 风险 | 缓解 |
|---|---|
| 审计写入失败被忽略后难以及时发现 | 记录结构化错误、递增失败指标，并保留 traceId |
| 全平台单链导致写入串行瓶颈 | 使用租户级链头，降低锁竞争 |
| 跨租户审计查询误用 | 服务层强制租户上下文，控制器只暴露租户范围查询 |
| 签名字段不稳定导致无法校验 | 使用规范化载荷生成签名，测试覆盖篡改检测 |

## 非目标

- 不实现 TSA 时间戳服务。
- 不实现异步导出任务。
- 不给所有业务控制器一次性补审计调用；各业务任务按需接入。
