# 02 场景剧本图（Storyboards）

> 版本：V2.0 · 2026-05-18  
> 这是 6 大客户验收剧本的精确 storyboard。每个剧本是 GA 必须通过的端到端测试边界。  
> 任何前端页面、后端 API、演示数据、Playwright 脚本都应当能从本文映射到 [04_页面规格书.md](04_页面规格书.md)。

---

## 0. 剧本通用结构

每个剧本固定 7 段：

1. **角色与背景** — 谁在做、做之前的处境
2. **触发事件** — 时间、空间、系统状态
3. **逐帧 Storyboard** — 4-8 帧屏幕变化（含旁白）
4. **背后系统时序** — 用户感知背后的 API 调用、Provider 调用、数据库写入
5. **关键设计抉择** — 为什么这帧这么做、备选方案被否定的原因
6. **验收点（DoD）** — 必须通过的 N 个条件
7. **演示数据与跑通脚本** — 样例数据路径、Playwright 脚本骨架

## S1 AMI/STEMI 路径推荐与入径

> 归属产品：**B 临床嵌入器** + **A 配置工厂（路径已发布的前提）**  
> 主角：R07 主治医师 + R06 主任医师（审核已发布的路径版本）  
> 演示数据：`ai-dev-input/06_samples/sample_patient_context_ami.json`

### 1.1 角色与背景

李医生（R07）在心内科门诊接诊一位 58 岁男性患者，主诉持续性胸痛 2 小时。心电图刚回报，平台后台已收到 ECG_12_LEAD 含 ST_ELEVATION_CONTIGUOUS_LEADS 三导联抬高。李医生希望快速判断"这位患者要不要按 AMI/STEMI 路径走？"

**前置：** 主任医师陈主任（R06）昨天已审核发布了 AMI_STEMI 路径 v2.1。

### 1.2 触发事件

- **时间**：09:23（门诊高峰）
- **空间**：李医生坐在 HIS 工作站前，刚保存完患者主诉
- **系统状态**：临床事件总线收到 `ECG_RESULT` 事件，含组织上下文 `tenant=TENANT_DEMO, hospital=HOSPITAL_DEMO, dept=DEPT_CARDIOLOGY`

### 1.5 关键设计抉择

| 抉择 | 选择 | 否定的备选 | 理由 |
|---|---|---|---|
| 嵌入条 vs 弹窗 | 嵌入条 | 弹窗 | 弹窗打断医生当前操作，会被快速关掉 |
| 480px 抽屉 vs 全屏 | 抽屉 | 全屏 | 抽屉允许医生对比 HIS 内容，全屏切断上下文 |
| 一键入径 vs 多步确认 | 一键入径 | 多步 | AMI 是时间窗敏感场景，"金标准"是 D2B < 90min，多步确认会被骂 |
| 显示置信度 92% | 显示 | 不显示 | 医生有权利知道 AI 把握有多大，"看不到置信度的 AI 是黑盒" |
| 显示来源条 | 必须 | 可选 | 不变量 #8，医学内容必须有来源 |
| 入径后 toast 自动消失 | 自动消失 | 持久占用 | 医生要继续问诊，不能被通知占住屏幕 |

### 1.6 验收点（DoD）

| # | 条件 | 验证方式 |
|---|---|---|
| 1 | 嵌入条在 ECG 事件后 1.5s 内显示 | Playwright + 计时 |
| 2 | 嵌入条显示 ≥ 4 个元素：规则名、依据、来源、操作 | DOM 断言 |
| 3 | 点"查看完整证据"抽屉宽度 480px、从右滑入 < 200ms | Playwright 测量 |
| 4 | 抽屉内必须显示规则版本、来源审核人、置信度 | DOM 断言 |
| 5 | 点"确认入径"必须 P95 < 500ms 返回 | 性能测试 |
| 6 | 入径成功后 `patient_instance` 写入 Oracle + LOCAL_H2 | 数据库断言 |
| 7 | 入径触发 ≥ 4 条 `ENGINE_AUDIT_LOG`（事件接收、规则执行、路径召回、实例创建） | 数据库断言 |
| 8 | toast 3 秒自动消失，"任务清单"入口持久 | DOM + 计时 |
| 9 | 整个流程组织上下文 `tenant=TENANT_DEMO, hospital=HOSPITAL_DEMO, dept=DEPT_CARDIOLOGY` 全程透传 | 日志断言 |
| 10 | 患者事实与规则版本可追溯（点审计日志可回放） | 后台查询 |

## S2 病历内涵质控

> 归属产品：**C 质控驾驶舱**（质控员视角） + **B 临床嵌入器**（医生视角）  
> 主角：R04 医务质控 + R07 主治医师  
> 演示数据：`ai-dev-input/06_samples/sample_emr_qc_rules.json`

### 2.1 角色与背景

王质控（R04）每天早上 8:30 习惯打开"病历质控看板"巡检前一日新增的不合格病历。她希望系统能在 5 分钟内让她看到：

- 总体合格率趋势
- 今日新增不合格病历 TOP 10
- 哪些医生集中产生质控问题
- 一键派单给医生整改

李医生（R07）昨晚写的"胸闷待查"病历漏写了既往史，今早一登录工作站，应该收到一条整改提示。

### 2.2 触发事件

- **时间**：王质控 08:32 打开驾驶舱
- **空间**：办公室独立屏幕（非 HIS）
- **系统状态**：昨日 23:59 已完成全量病历质控扫描，含 18 条新问题

### 2.4 背后系统时序

```
[Cron 23:59] 触发病历质控扫描
  └─► /api/quality/scan-emr  
        ├─ 拉取昨日全部住院病历（适配器）
        ├─ 加载 EMR 质控规则版本 v1.3
        ├─ 字典映射（ICD 既往史标识）
        ├─ 规则评估 × N 条
        └─ 写入 qc_problems 表 + 推送驾驶舱 KPI

[08:32 王质控打开驾驶舱]
  ├─► GET /api/quality/dashboard?scope=DEPT_CARDIOLOGY&date=today
  │     返回 4 KPI + 分类分布 + 明细
  └─ 前端用 React Query 缓存 60 秒

  ├─► POST /api/quality/problems/{id}/assign
  │     { assignee: 李医生 user_id, deadline, note }
  ├─ 创建 WF_TASK（统一待办）
  ├─ 通过 NotifyProvider 推给李医生工作站（嵌入条）
  └─ 写 ENGINE_AUDIT_LOG

  └─► GET /api/wf/tasks?assignee=me&status=PENDING

        ├─ 通过 → POST 内部 /api/quality/problems/{id}/resolve
        └─ 推送王质控驾驶舱 KPI 实时刷新
```

### 2.5 关键设计抉择

| 抉择 | 选择 | 否定的备选 | 理由 |
|---|---|---|---|
| 驾驶舱信息密度 | 4 KPI + 1 图 + 1 表（高密度） | 单一图表 | 质控员每天巡检 200+ 条，必须信息密度高 |
| 派单期限默认值 | 当日 18:00 | 让用户必填 | 80% 场景就是当日 18:00，默认值节省时间 |
| 备注必填 vs 可选 | 可选 | 必填 | 必填会让质控员烦躁、写"无"，等于没写 |
| 整改后再质控 | 自动 | 手动触发 | 自动闭环才能形成真正的"质控-整改-验证"环 |
| 整改通过通知质控员 | 无打扰（驾驶舱刷新） | 弹窗/邮件 | 已闭环是低优先级信号，不能打断质控员 |

### 2.6 验收点（DoD）

| # | 条件 | 验证方式 |
|---|---|---|
| 1 | 驾驶舱 P95 加载 < 2s（含 4 KPI + 图 + 表） | 性能测试 |
| 2 | 4 个 KPI 同周期对比箭头方向正确（↑/↓） | 业务断言 |
| 3 | 派单弹窗预填医生准确（来自病历主治医师字段） | 数据断言 |
| 4 | 派单触发 WF_TASK + 推送嵌入条 + 写审计 | 端到端测试 |
| 5 | 医生补写后自动重质控 < 60 秒 | 计时测试 |
| 6 | 整改闭环驾驶舱 KPI 实时刷新 | 端到端测试 |
| 7 | 来源条显示规则版本、规则来源、审核人 | DOM 断言 |
| 8 | 派单操作记录 ENGINE_AUDIT_LOG（含派单人、被派人、期限、原因） | 数据库断言 |

### 3.4 背后系统时序

```
[周医生提交医嘱]

        │   ├─ AdapterProvider 拉取病历当前诊断
        │   ├─ AdapterProvider 拉取 LIS 历史报告
        │   └─ 匹配："无 EGFR 突变" 缺失
        ├─ 返回 matched + warning + reason
        └─ 嵌入条显示

[周医生选"EGFR 检测报告"]
  └─► GET /api/lis/reports?patient=刘XX&item=EGFR
        返回 LIS 报告列表

  └─► POST /api/rule-engine/evidence
        { rule_id, order_id, evidence_type, evidence_ref }

        ├─ 写 ENGINE_AUDIT_LOG（补证据动作 + 补证据人 + 来源审核人）
        └─ 嵌入条更新"已通过"

[徐医保打开看板]
  └─► GET /api/quality/insurance-dashboard?month=...
        返回 4 KPI + 高风险明细
```

### 3.5 关键设计抉择

| 抉择 | 选择 | 否定的备选 | 理由 |
|---|---|---|---|
| 警告色 vs 危险色 | 警告色（黄） | 危险色（红） | 这是合规风险不是医疗安全，红色滥用会脱敏 |
| 强制拦截 vs 软性提醒 | 软性提醒（医生可选自费/取消） | 强制 | 不变量 #7，未授权不得自动阻断 |
| 补证据系统辅助 | 系统自动检索 LIS | 手动填写 | 减少医生负担，提升补证率 |
| 拒绝检测知情同意 | 支持上传 | 不支持 | 现实存在患者拒绝检测的合规场景 |
| 看板核心指标"可能退费金额" | 是 | 仅条数 | 金额对医保办更敏感、更易触发整改动作 |

### 3.6 验收点（DoD）

| # | 条件 |
|---|---|
| 1 | 医嘱提交触发规则 P95 < 800ms |
| 2 | 嵌入条用 warning 色（不是 danger）+ 4 元素（要求/缺失/来源/操作） |
| 3 | "补充证据"流程支持 3 种证据类型 |
| 4 | LIS 报告检索 < 1s 返回 |
| 5 | 补证据后审计含：原始命中事实、证据 ref、证据审核状态 |
| 6 | 看板"可能退费金额"= 待补 + 待审条数对应的医嘱金额合计 |
| 7 | 看板支持按科室、按月、按医生筛选 |

| 7 | 拦截类规则必须在规则定义里显式标记 `action_mode=BLOCK`，否则系统拒绝发布 |

### 5.3 关键设计抉择

| 抉择 | 选择 | 理由 |
|---|---|---|
| 向导式 vs 一页式 | 向导式 5 步 | 跨环境发布是高风险操作，分步降低出错率 |
| 来源缺失阻断 vs 警告 | 阻断（不可绕过） | 不变量 #8，医学内容缺来源不得发布 |
| 现有患者保留旧版本 | 是 | 避免治疗方案中途切换造成混乱 |
| 推荐定时发布 | 是 | 业务低峰期减少影响 |
| 三级危险确认 | 是 | 不变量"危险操作三级防护" |

### 5.4 验收点（DoD）

| # | 条件 |
|---|---|
| 1 | 5 步向导可保存草稿，刷新回到同一步 |
| 2 | 数字签名/SHA256 校验必须前置 |
| 3 | 缺来源资产必须用 `MISSING_SOURCE` 错误码阻断 |
| 4 | 影响分析必须显示现有患者数（基于 patient_instance 表实时查询）|
| 5 | 必填发布原因 + 必输入包名确认 |
| 6 | 发布操作记录 ENGINE_AUDIT_LOG（含操作人、源环境签名、目标环境、定时） |
| 7 | 支持回滚（PKG-005 实现后），回滚同样需向导 |
