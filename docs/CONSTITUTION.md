## §0 产品定位与命名

> MedKernel = **集团医疗智能中枢**：把指南、路径、规则和院内数据接起来，在临床现场提醒医生，在质控侧形成整改闭环，并留下合规证据。

| 类型 | 标准名称 | 使用范围 |
|---|---|---|
| 中文产品名 | **集团医疗智能中枢** | 对外介绍、页面标题、登录页、文档标题、客户汇报 |
| 英文 / 代码名 | **MedKernel** | 仓库、接口、部署包、技术文档 |
| 内部能力名 | 引擎核心 / 医疗智能引擎核心 | 架构说明、服务边界、模块设计 |

→ **不得把含"引擎中心"的名称作为中文产品名**。面向客户只讲四件事：试点准备 / 临床运行 / 质控改进 / 合规运维。

**工程节奏 = 按业务域纵向推进、两波**：

| 波次 | 口径 |
|---|---|
| 第一波 · B0 真实纵向 | 沿真实用户旅程逐域交付（登录 D0 → 工作台 D1 → 试点准备 D2 → 临床运行 D3 → 质控改进 D4 → 合规运维 D5 → 高级工具 D6）；每域"无模型也真实可用、可验证"，**域级验收通过才走下一域** |
| 第二波 · AI 加深 | D0~D6 的 B0 全跑通后，建 AI 工厂、模型网关、首发知识资产等跨域共享深引擎，回灌相关域做模型增强（铁律 #4：B0 先于模型）|

## §1 不可妥协的硬约束（违反 = PR 拒）

| 6 | 任何客户可见页面默认 1 主按钮 / 1 主目标 / ≤ 3 默认筛选 | 首屏 5 个同权按钮；横向滚动列表 |
| 7 | 任何 API 输入用 Record DTO + Bean Validation | `@RequestBody Map<String,Object>` |
| 8 | 任何敏感字段必须脱敏 + 字段级加密 + 审计 | 直接返回身份证号 / 手机号明文 |
| 9 | 任何 AI 生成内容必须明显标识（生成式 AI 管理办法 §7）| CDSS 提醒不标"AI 生成" |
| 10 | 任何医疗决策必须有医师确认才进病历 | CDSS 自动写入医嘱不要求确认 |
| 11 | 任何 PR base 必须 → `main`（trunk-based）| 长期 feat 分支（>3 天）；直推 main |
| 12 | 任何文档改动与代码改动同 PR | 文档 PR 与代码 PR 分开 |
| 13 | 任何适合大模型赋能的能力必须经统一模型能力网关接入，并具备无模型可运行基线 | 业务代码直接绑定模型厂商；关闭模型后链路不可用 |
| 14 | 任何 AI 知识生成先识别新旧与重复；待审新版只供审核和替换提醒，现行已审版继续执行；新版审核生效后旧版退出新临床决策 | 未审新版参与临床；新版上线后实时推荐仍混用被替代知识 |
| 15 | 引擎核心必须覆盖护理、医技报告、床旁权威知识和中医药等全医疗专业领域；每个领域有**施工卡**、审核、降级和验收计划 | 只做医生推荐；把专业领域留成空名称 |
| 16 | 全系统产品与交互体验必须遵守 [体验契约](EXPERIENCE_CONTRACT.md) 与本核心：一页一目标、低打扰、服务端分页、可信解释、六态、证据和降级门禁 | 大列表前端全量加载；默认筛选 >3；临床提醒遮挡主流程 |
| 17 | 当前有效文档必须使用简体中文；功能完成后必须 PR 合并到 GitHub 远程 `main` | 英文主体文档；只开 PR 不合并；本地 main 合并但远程未更新 |
| 18 | 引擎真实性与开发严谨性：绝对真实、去 Mock、杜绝假闭环，禁伪造任何哈希/调用结果/RTT/随机数投递；算法与测试物理化，严格通过 ESLint 和 CI 真实门禁 | 随机数伪造 Ping/死信重试；catch 块伪造成功；Evidence 导出返回假哈希；绕 `no-page-mock` |
| 19 | **配置外置**：除启动必需（DB 连接/端口/profile/迁移位置/启动密钥）外，业务与运营配置不得写死 yml；必须经配置中心管理、可审计、高危项有护栏（审计持久化等不可从 UI 关闭）| 功能开关/JWT TTL/备份策略写死 yml；高危项可从 UI 随意关 |
| 20 | **内置超级管理员**：系统强制内置、启动自动授满五维 + 系统配置中心、不可降权/删除/移出超管组、**不旁路**（走 RBAC 引擎）、全程独立高亮审计、强制 MFA | 维护全部功能依赖手配漏配；存在旁路黑洞；超管可被降权顶替 |

## §2 菜单 IA 锁（5+1 一级 + 27+5 二级，产品 IA 锁定）

### §2.1 一级菜单（永远 5 组 + 1 隐藏式）

| 顺序 | 一级 | 客户能理解的话 | 谁主要在用 |
|---|---|---|---|
| 0 | **工作台** | 今天系统是否健康、试点是否可演示、我该做什么 | 全角色 |
| 1 | **试点准备** | 把医院、系统、路径、规则准备好 | 实施工程师、信息科 |
| 2 | **临床运行** | 患者进路径、医生收到提醒、任务有人处理 | 医生、医务处 |
| 3 | **质控改进** | 看问题、派整改、看效果 | 院长、质控办、医保办 |
| 4 | **合规运维** | 身份、审计、安全、监控、交付证据 | 信息科主任、合规审计 |
| ⊕ | **高级工具**（SideMenu 底部小图标）| 来源追溯、图谱、AI 工作流、国产化自检、开发者控制台 | 架构师、实施、运维 |

### §2.2 二级菜单（27 项 + 5 高级；不净增）

| 一级 | 二级（顺序固定） |
|---|---|
| 工作台 | 工作台 · 演示与校验（合并为 Tab）|
| 试点准备 | 客户实施向导 · 租户开通 · 配置包中心 · 路径配置 · 规则库 · 字典映射 · 适配器中心 |
| 临床运行 | 患者主索引 · 患者路径 · 临床提醒治理 · 规则校验 · 待办中心 · 通知中心 · 智能随访 |

| 资产类 | 状态机 | 默认动作 |

## §4 7 步极简配置流（全平台一套）

```
选模板/导入 → 自动校验 → 看影响 → 提交审核 → 灰度发布 → 全量 → 留证据/可回滚
```

→ 配置包、规则、路径、图谱、字典、适配器、评估指标 全部复用同一个 7 步组件。

## §6 临床医疗安全红线（视角：临床医疗安全）

- **医师确认才进病历**：AI/CDSS 不自动开医嘱、上报、支付、签署、输血、手术、设备控制、病历归档。
- **AI 内容明显标识**："AI 候选 / AI 生成 / AI 建议"。
- **高风险强制审核 + 双签**：DDI、危急值、剂量上限、抗菌限制、特殊人群禁忌。
- **高危近似映射**（钾/钠、肌钙蛋白 T/I、左/右、剂量量级）强制 HIGH + 禁批量 + 禁自动确认。
- **旧版隔离**：新版生效后旧版退出新临床判断，仅历史重放。
- **临床知情同意**：涉及患者的高风险决策与 AI 辅助须可见同意/责任边界；页面文案不得暗示系统自动诊断/治疗。
- **静默试运行**通过后才允许打扰医生；嵌入提醒按打扰等级（信息/弱打断/红线强打断）。

## §7 知识与数据治理不变量（视角：知识与数据治理）

- **唯一权威知识**：同一适用域（knowledge_identity + organization_scope + applicable_population + effective_time）同时只有一个 `ACTIVE_AUTHORITATIVE` 版本；待审新版只审不执行；新版激活时原子失效旧版。
- **字典语义映射**：基于医学语义（同义词典 + 编码交叉表 + 模型嵌入），**禁字符 LCS**；高危近似负样本判别器强制 HIGH。
- **来源可追溯**：来源类型/标题/版本/引用位置/hash 100% 可回溯；证据分级 A 法规 / B 国家指南 / C 共识文献 / D 院内 / E 反馈（GRADE 兼容）。
- **关系库权威**：业务事实唯一权威源为院内关系数据库；图数据库（Neo4j）、Dify、模型、缓存只能是**投影或执行器**，必须可重建。
- **AI 工厂只产候选不产事实**：候选须经安全门禁 + 人工审核才能成为知识。

## §8 安全合规与监管基线（视角：安全合规与监管）

- 等保 2.0 三级；国密 SM2/SM3/SM4；个保法（PIPL）；数据出境合规评估与备案。
- 敏感字段脱敏 + 字段级加密 + 审计；最小数据外调（字段白名单 + 审批）。
- **审计链**全留证：来源 / 生成 / 审核 / 发布 / 运行 / 反馈 / 整改 / 回滚。
- 互联互通测评与电子病历评级支撑；**伦理与（数据）知情同意**（科研/RWD 须伦理授权 + 脱敏队列）。

## §9 集团化与多租户治理（视角：集团化与多租户）

- **多层继承覆盖**：平台 → 集团 → 医院 → 院区 → 社区服务点 → 科室 → 专病；支持继承、局部覆盖、冲突仲裁、影响范围。
- **租户隔离**：跨租户数据不串；数据范围五维之一。
- **6 阶段生命周期**：0 准备 / 1 试运行 / 2 验收 / 3 推广 / 4 正式运行 / 5 续约升级；按规则自动推进，每次切换自动生成阶段总结报告。
- **多维治理切片**：组织层级 / 临床对象 / 服务能力 / 生命周期 / 外部接入 / 风险证据；默认视图最多 3 维，更多进专家模式。
- 灰度默认 10% 床位，仅院级管理员可"直接全量"。

## §10 集成与互操作边界（视角：集成与互操作）

- 外部系统（HIS/EMR/LIS/PACS/手麻/输血/护理/医保/公卫/区域平台/Provider）**只经统一对接链路**：适配器 + 标准上下文 + 临床事件 + 嵌入 + 回调 + 包发布同步 + 审计证据。
- FHIR / CDS Hooks 风格门面；字段映射 / 健康检查 / 重试 / 死信 / 回放。
- **不绕引擎**直写医疗结论、医嘱、病历、法定上报、支付或设备控制。
- 同步超时不阻断医生主流程；断连诚实标 `NOT_CONNECTED`，不伪造连接。

## §11 AI / 模型治理与可降级（视角：AI / 模型治理）

- **B0 先于模型**（铁律 #4）：每个含 AI 增强的能力先交付"无模型确定性 + 人工"路径并验收，再叠加模型。
- 所有 AI 能力统一经**模型能力网关**：能力代码 / 路由 / 脱敏 / 结构化输出 / 审计 / B0 降级；**模型只产候选不产事实**。
- 提示词、工具、模型版本治理：可重放 / 可回滚 / 可导出 / 可审计（数据策略、模型模式、来源、版本、输出 hash、审核动作、降级情况）。
- 幻觉拦截与引用真实性；不把未核验内容标为"最新"。
- **降级诚实**：模型 / Dify / 图投影 / 外部系统 全部关闭时，主链路仍真实通过验收（返回 `B0`/`NOT_CONNECTED`/`NOT_SYNCED`/`MODEL_DISABLED`，绝不伪造）。Dify 仅可选运行器；图数据库仅投影。

## §12 运维 / SRE / 国产化交付（视角：运维 / SRE / 国产化）

- **内外网双形态**：内网（医院本地机房，国产化栈，业务权威源在院内关系库，不出网）/ 外网（SaaS，MySQL/PostgreSQL + Dify Cloud/OpenAI Compatible，数据出境合规评估）。任何功能都要回答"内网怎么用？外网怎么用？"
- **国产化栈**：麒麟 / 统信 / openEuler · 达梦 / 人大金仓 · KAE-JDK 21 / BiSheng JDK 21 · Ollama 本地大模型；国密算法 smoke。
- **5 方言迁移**：h2 / postgres / oracle / dm / kingbase 一致性 + 中文注释 + 索引约束。
- 监控 + 健康检查；备份恢复 RPO-RTO；离线许可；SLA ≥ 99.9%；灾备。
- **降级链端到端**：模型 / Dify / 图 / 外部 四类逐项关闭后主链路可运行。

## §13 验收铁律 + 门禁 + 真实性（视角：质量与真实性审计）

**11 条验收铁律**（所有任务通用，详见 [质量基线](audit/质量基线.md)）：

1. 真实性 2. 诚实降级 3. 医疗安全 4. 无模型可运行 5. 关系库权威 6. 唯一权威知识 7. 五维权限（菜单/动作/数据/资产/环境，到 27 二级+5 高级粒度）8. 六态完整 9. 中文优先 10. 文档同步 **11. 配置外置**（呼应硬约束 #19）。

**通用拒绝项**（PR 出现任一项直接 reject）：新增一级菜单 / 多个同权主按钮 / 默认筛选 >3 / 大列表前端全量加载 / AI 无标识或无来源或无人工确认 / 高风险发布无影响分析审核回滚证据 / 六态缺失 / 暴露 JSON·DSL·trace·模型提示词 / 嵌入阻断主流程且无降级 / 驾驶舱不可下钻 / `eslint-disable medkernel/no-page-mock` / `.module.css` 含 hex 字面量 / 写死医学常量（病种·药品·编码·剂量）/ 业务直接绑定模型厂商 / 配置写死 yml（违 #19）。

**真实性门禁**：前端 `no-page-mock` + stylelint、后端 CI（禁 Math.random 造数 / catch 吞错伪造 / UUID 充哈希 / 占位 Javadoc 于生产路径）= T-GATE，前后端全绿才可 done。
