## 0. 一句话产品定位

> MedKernel = **集团医疗智能中枢**
> 把指南、路径、规则和院内数据接起来，在临床现场提醒医生，在质控侧形成整改闭环，并留下合规证据。

命名口径：

| 类型 | 标准名称 | 使用范围 |
|---|---|---|
| 中文产品名 | **集团医疗智能中枢** | 对外介绍、页面标题、登录页、文档标题、客户汇报 |
| 英文 / 代码名 | **MedKernel** | 仓库、接口、部署包、技术文档 |
| 内部能力名 | 引擎核心 / 医疗智能引擎核心 | 架构说明、服务边界、模块设计 |

→ **不得把含“引擎中心”的名称作为中文产品名使用**；“引擎”只描述内部核心能力。

面向客户只讲四件事：

| 顺序 | 客户问题 | 产品回答 |
|---|---|---|
| 1 | 怎么开始试点？ | **试点准备**：建租户、接系统、导路径和规则 |
| 2 | 医生怎么用起来？ | **临床运行**：患者入径、节点流转、医嘱提醒、待办处理 |
| 3 | 管理层怎么看效果？ | **质控改进**：预警、医保审核、评估结果、整改追踪 |
| 4 | 能不能放心上线？ | **合规运维**：身份、审计、安全、监控、备份、SLA |

当前工程执行只讲一个节奏：

| 阶段 | 口径 |
|---|---|
| 0 业务引擎全能力上线 | 先打通知识、字典、规则、路径、推荐、评估、随访、包发布、嵌入、模型网关、审计证据和降级链路 |
| 业务服务包装 | 引擎全能力验收后，再按唯一详细规范包装试点、临床、质控、合规、第三方业务接口和专业领域服务 |

## 1. 不可妥协的硬约束（违反 = PR 拒）

| # | 约束 | 反例（自动 fail） |
|---|---|---|
| 1 | 国情合规：等保 2.0 三级 + 国密 + 个保法 + 数据出境 + 备案 | 默认调用境外大模型；明文存敏感字段；无审计链 |
| 2 | 左侧 SideMenu 永远是主菜单 | 把主菜单改成顶部横向导航 |
| 3 | 客户主路径 ≤ 5 个一级菜单 | 7 个一级菜单；高级工具放一级 |
| 4 | 任何配置功能必须走 7 步极简流 | 自创"草稿→提交"两步流；跳过"看影响"|
| 5 | 任何资产用 4 套统一状态机之一 | 自创"已就绪 / 已确认 / 已激活"|
| 6 | 任何客户可见页面默认 1 主按钮 / 1 主目标 / ≤ 3 默认筛选 | 首屏 5 个同权按钮；横向滚动列表 |
| 7 | 任何 API 输入用 Record DTO + Bean Validation | `@RequestBody Map<String, Object>` |
| 8 | 任何敏感字段必须脱敏 + 字段级加密 + 审计 | 直接返回身份证号 / 手机号明文 |
| 9 | 任何 AI 生成内容必须明显标识（生成式 AI 管理办法 §7）| CDSS 提醒不标"AI 生成" |
| 10 | 任何医疗决策必须有医师确认才进病历 | CDSS 自动写入医嘱不要求确认 |
| 11 | 任何 PR base 必须 → `main`（trunk-based） | 长期 feat 分支（> 3 天）；直推 main |
| 12 | 任何文档改动与代码改动同 PR | 文档 PR 与代码 PR 分开 |
| 13 | 任何适合大模型赋能的能力必须经统一适配层接入，并具备无模型可运行基线 | 业务代码直接绑定模型厂商；关闭模型后知识、规则、路径、质控或发布链路不可用 |
| 14 | 任何 AI 知识生成先识别新旧与重复；待审新版只供审核和替换提醒，现行已审版继续执行；新版审核生效后旧版退出新临床决策 | 反复审核旧知识；未审新版参与临床；新版上线后实时推荐仍混用被替代知识 |
| 15 | 引擎核心必须覆盖护理、医技报告、床旁权威知识和中医药等全医疗专业领域；每个领域有实现卡、审核、降级和验收计划 | 只做医生推荐；把专业领域留成空名称；遗漏护理、报告或说明书查阅闭环 |
| 16 | 全系统产品与交互体验必须遵守 [产品体验固定规范](MEDKERNEL_PRODUCT_EXPERIENCE_RULES.md)：一页一目标、低打扰、服务端分页、可信解释、六态、证据和降级门禁 | 大列表前端全量加载；默认筛选超过 3 个；临床提醒遮挡主流程；驾驶舱指标不能下钻 |
| 17 | 当前有效文档必须使用简体中文；功能完成后必须 PR 合并到 GitHub 远程 `main` | 英文主体文档；只开 PR 不合并；本地 main 合并但远程 main 未更新 |
| 18 | 引擎真实性与开发严谨性：任何功能实现必须绝对真实、去 Mock、杜绝假闭环，禁止伪造任何哈希、调用结果、RTT 或随机数投递；算法与测试必须物理化，且严格通过 ESLint 和 CI 真实门禁 | 使用随机数或掷骰子伪造 Ping 握手及死信重试；在 catch 块伪造成功；在 Evidence 导出时返回假哈希或写死结果；绕过 ESLint `no-page-mock` 规则 |

## 3. 4 套统一状态机（全平台一套，禁止自创）

| 资产类 | 状态机 | 默认动作 |
|---|---|---|
| **配置类**（包/规则/路径/图谱/字典/适配器/评估指标）| 草稿 → 待审核 → 已发布 → 生效中 → 已下线 →（已归档）| 列表按"待我处理 / 生效中"分组 |
| **变更类** | 待发布 → 灰度（默认 10% 床位）→ 全量 → 回滚 | 默认走灰度，仅院级管理员可"直接全量" |
| **待办类** | 未读 → 处理中 → 已完成 →（已升级）| 按 SLA 倒序 |
| **告警类** | 新建 → 已派单 → 整改中 → 已闭环 →（已豁免）| 按责任科室分组 |

→ **任何 PR 提交新页面，缺这 4 套之一即拒**。

## 6. 租户生命周期（**v1.0 GA 新约束**）

### 6.1 6 阶段

| 阶段 | 时长 | 谁主导 | 试点准备组在做 |
|---|---|---|---|
| **0 准备期** | 合同签后 ~2-4 周 | 实施工程师 | 走完 7 步流，第 1 个科室上线 |
| **1 试运行期** | 1-3 个月 | 实施 + 试点科室 | 不动配置，只观察 + 微调规则 |
| **2 验收期** | 1-2 周 | 院方 + 乙方联合 | 导出审计快照、对比 KPI、签验收单 |
| **3 推广期** | 3-6 个月 | 信息科 | 新增科室 / 新增病种 / 新增院区（复用配置包） |
| **4 正式运行期** | ≥ 1 年 | 信息科 + 医务处 | 周期维护：规则月更 / 路径季更 / 字典年更 |
| **5 续约/升级期** | 合同末 ~1 个月 | 销售 + 信息科 | 跨版本升级 + 配置打包导出 |

### 6.2 阶段自动推进规则

| 触发 | 自动 |
|---|---|
| 试点科室运行 ≥ 30 天 + 提醒采纳率 ≥ X% + 0 P0 事故 | 1 → 2 |
| 院方 + 乙方双签验收单（系统内电子签）| 2 → 3 |
| 推广覆盖率 ≥ 80% 科室 + 稳定运行 ≥ 90 天 | 3 → 4 |
| 合同剩余 < 90 天 | 4 → 5（不停服）|

每次阶段切换**自动生成阶段总结报告**，作为验收 / 商务 / 续约材料的出处。

### 6.3 多维度并行

医院实际是 **多维度并行生命周期** — 不是整个医院在同一个阶段：

| 维度组 | 例子 |
|---|---|
| 组织层级 | 集团、医院、院区、社区服务点、科室、病区、医生团队 |
| 临床对象 | 患者、就诊、病种、专病包、路径模板、患者路径 |
| 服务能力 | 接入、知识、字典、规则、路径、推荐、评估、随访、包同步、嵌入、模型网关 |
| 生命周期 | 0 准备、1 试运行、2 验收、3 推广、4 运行、5 续约 |
| 资产版本 | 知识包、配置包、规则集、路径包、指标集、随访计划、模型策略 |
| 外部接入 | HIS、EMR、LIS、PACS、医保、区域平台、Provider、Webhook |
| 风险证据 | P0-P3 风险、数据质量、降级状态、审计事件、证据快照、导出任务 |

→ 工作台面板支持 **多维治理切片**。它不是固定三维表，而是按角色默认展示最少维度，并在专家模式中下钻组织、临床对象、服务能力、生命周期、接入、风险和证据。

## 10. 内外网双形态（不可妥协）

| 形态 | 部署位置 | 关键约束 |
|---|---|---|
| **内网** | 医院本地机房 | 国产化栈（人大金仓 / 达梦 / 麒麟 / Ollama 本地大模型）；业务权威源在院内关系库；不出网 |
| **外网** | SaaS | MySQL / PostgreSQL + Dify Cloud / OpenAI Compatible；数据出境合规评估；客户业务数据以平台关系库为权威源 |

→ **任何功能设计都要回答"内网怎么用？外网怎么用？"**

→ **图数据库只能是投影**：图谱节点、边、引用、版本的业务权威源必须在关系数据库；Neo4j 等图数据库只做可重建查询投影。

→ **Dify 只能是可选运行器**：AI 工作流定义、版本、绑定、审核结论、运行审计必须保存在 MedKernel 数据库；Dify 有则同步，没有则降级到内置编排 / 规则执行，不影响主链路验收。

→ **大模型只能是可插拔增强层**：知识探索、抽取、映射、规则/路径/推荐/质控/随访生成和解释等能力统一经模型能力网关调用；模型有则增强，没有或故障时以已发布知识、确定性引擎和人工治理完成业务与验收。

→ **知识有效版本必须唯一**：AI 自动采集生成阶段即对照当前权威知识识别新版本、重复、旧版、冲突和撤回；常规新版待审期间仅提供新版差异审核和替换提醒，现行已审版继续服务；已核验的禁用、召回或重大安全警示可按授权紧急限制旧版；审核激活新版时原子失效旧版，旧版此后仅能用于历史审计和重放，不得用于新的治疗推荐或临床判断。

→ **专业领域必须可实现**：护理分级/评估/计划、报告解读、床旁权威知识和中医药/中西医结合都使用同一资产治理、安全和发布链；专科路径必须支持独立专病包、分型、关键时钟、质控和随访接续；全医疗领域必须按实现卡明确 API 归类、AI 与无模型路径、审核责任、安全边界和验收证据。

→ **外部系统必须走统一对接链路**：HIS/EMR/LIS/PACS/手麻/输血/护理/医保/公卫/区域平台/模型 Provider 等第三方系统只能通过适配器、标准上下文、临床事件、嵌入、回调、包发布同步和审计证据链接入；不得绕过引擎直接写医疗结论、医嘱、病历、法定上报、支付或设备控制。
