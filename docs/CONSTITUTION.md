> 版本：1.0 · 2026-05-23
> 状态：v1.0 GA 必须遵守的最高准则
> 适用对象：所有 AI 团队成员、产品、架构、实施、客户

## 0. 一句话产品定位

> MedKernel = **集团医疗智能中枢**
> 把指南、路径、规则和院内数据接起来，在临床现场提醒医生，在质控侧形成整改闭环，并留下合规证据。

面向客户只讲四件事：

| 顺序 | 客户问题 | 产品回答 |
|---|---|---|
| 1 | 怎么开始试点？ | **试点准备**：建租户、接系统、导路径和规则 |
| 2 | 医生怎么用起来？ | **临床运行**：患者入径、节点流转、医嘱提醒、待办处理 |
| 3 | 管理层怎么看效果？ | **质控改进**：预警、医保审核、评估结果、整改追踪 |
| 4 | 能不能放心上线？ | **合规运维**：身份、审计、安全、监控、备份、SLA |

## 1. 不可妥协的硬约束（违反 = PR 拒）

| # | 约束 | 反例（自动 fail） |
|---|---|---|
| 1 | 国情合规：等保 2.0 三级 + 国密 + 个保法 + 数据出境 + 备案 | 默认调用境外大模型；明文存敏感字段；无审计链 |
| 2 | 左侧 SideMenu 永远是主菜单 | 把主菜单改成顶部横向导航 |
| 3 | 客户主路径 ≤ 5 个一级菜单 | 7 个一级菜单；高级工具放一级 |
| 4 | 任何配置功能必须走 7 步极简流 | 自创"草稿→提交"两步流；跳过"看影响"|
| 5 | 任何资产用 4 套统一状态机之一 | 自创"已就绪 / 已确认 / 已激活"|
| 6 | 任何客户可见页面默认 1 主按钮 / 1 主目标 / ≤ 3 默认筛选 | 首屏 5 个同权按钮；横向滚动列表 |
| 7 | 任何 API 输入用 Record DTO + Bean Validation | `@RequestBody Map<String, Object>` |
| 8 | 任何敏感字段必须脱敏 + 字段级加密 + 审计 | 直接返回身份证号 / 手机号明文 |
| 9 | 任何 AI 生成内容必须明显标识（生成式 AI 管理办法 §7）| CDSS 提醒不标"AI 生成" |
| 10 | 任何医疗决策必须有医师确认才进病历 | CDSS 自动写入医嘱不要求确认 |
| 11 | 任何 PR base 必须 → `main`（trunk-based） | 长期 feat 分支（> 3 天）；直推 main |
| 12 | 任何文档改动与代码改动同 PR | 文档 PR 与代码 PR 分开 |

## 2. 5 组菜单 + 30 项二级菜单（产品 IA 锁定）

### 2.1 一级菜单（永远 5 组 + 1 隐藏式）

| 顺序 | 一级 | 客户能理解的话 | 谁主要在用 |
|---|---|---|---|
| 0 | **工作台** | 今天系统是否健康、试点是否可演示、我该做什么 | 全角色 |
| 1 | **试点准备** | 把医院、系统、路径、规则准备好 | 实施工程师、信息科 |
| 2 | **临床运行** | 患者进路径、医生收到提醒、任务有人处理 | 医生、医务处 |
| 3 | **质控改进** | 看问题、派整改、看效果 | 院长、质控办、医保办 |
| 4 | **合规运维** | 身份、审计、安全、监控、交付证据 | 信息科主任、合规审计 |
| ⊕ | **高级工具**（SideMenu 底部小图标，专家点开） | 来源追溯、图谱、AI 工作流、国产化自检、开发者控制台 | 架构师、实施、运维 |

### 2.2 二级菜单（30 项 + 5 高级）

| 一级 | 二级（顺序固定） |
|---|---|
| 工作台 | 工作台 · 演示与校验（合并为 Tab）|
| 试点准备 | 客户实施向导 · 租户开通 · 配置包中心 · 路径配置 · 规则库 · 字典映射 · 适配器中心 |
| 临床运行 | 患者主索引 · 患者路径 · 临床提醒治理 · 规则校验 · 待办中心 · 通知中心 |
| 质控改进 | 院级质控驾驶舱 · 质控预警 · 医保智能审核 · 评估指标库 · 评估结果 · AI 知识审核（合并去重）|
| 合规运维 | 用户管理 · 身份绑定 · 审计日志 · 安全基线 · Provider 状态 · 通知设置 |
| 高级工具 | 来源追溯 · 图谱查询 · AI 工作流 · 国产化自检 · 开发者控制台 |

## 3. 4 套统一状态机（全平台一套，禁止自创）

| 资产类 | 状态机 | 默认动作 |
|---|---|---|
| **配置类**（包/规则/路径/图谱/字典/适配器/评估指标）| 草稿 → 待审核 → 已发布 → 生效中 → 已下线 →（已归档）| 列表按"待我处理 / 生效中"分组 |
| **变更类** | 待发布 → 灰度（默认 10% 床位）→ 全量 → 回滚 | 默认走灰度，仅院级管理员可"直接全量" |
| **待办类** | 未读 → 处理中 → 已完成 →（已升级）| 按 SLA 倒序 |
| **告警类** | 新建 → 已派单 → 整改中 → 已闭环 →（已豁免）| 按责任科室分组 |

→ **任何 PR 提交新页面，缺这 4 套之一即拒**。

## 4. 7 步极简配置流（全平台一套）

```
选模板/导入 → 自动校验 → 看影响 → 提交审核 → 灰度发布 → 全量 → 留证据/可回滚
```

→ **配置包、规则、路径、图谱、字典、适配器、评估指标 7 处全部复用同一个 7 步组件**。

## 5. 7 角色硬指标（GA 验收用）

| 角色 | 任务 | 验收标准 |
|---|---|---|
| 院长 | 看懂今天系统状态 | 首屏 ≤ 10 秒，0 技术名词 |
| 医务处 | 30 分钟跑完 6 大剧本 | 不打开任何 JSON / DSL |
| 临床医生 | 0 培训用 CDSS 提醒 | "采纳 / 不采纳并说明" 两个主按钮 |
| 信息科主任 | 1 小时完成新医院接入 | 客户实施向导完成度 ≥ 80% 不打电话 |
| 路径专家 | 画 X6 节点 | 必须切到专家模式 |
| 实施工程师 | 一键导入专病配置包 | 不进图谱 / 规则两个模块 |
| 合规审计 | 任意页面导出审计快照 | 右上角全平台一致 |

## 6. 租户生命周期（**v1.0 GA 新约束**）

### 6.1 6 阶段

| 阶段 | 时长 | 谁主导 | 试点准备组在做 |
|---|---|---|---|
| **0 准备期** | 合同签后 ~2-4 周 | 实施工程师 | 走完 7 步流，第 1 个科室上线 |
| **1 试运行期** | 1-3 个月 | 实施 + 试点科室 | 不动配置，只观察 + 微调规则 |
| **2 验收期** | 1-2 周 | 院方 + 乙方联合 | 导出审计快照、对比 KPI、签验收单 |
| **3 推广期** | 3-6 个月 | 信息科 | 新增科室 / 新增病种 / 新增院区（复用配置包） |
| **4 正式运行期** | ≥ 1 年 | 信息科 + 医务处 | 周期维护：规则月更 / 路径季更 / 字典年更 |
| **5 续约/升级期** | 合同末 ~1 个月 | 销售 + 信息科 | 跨版本升级 + 配置打包导出 |

### 6.2 阶段自动推进规则

| 触发 | 自动 |
|---|---|
| 试点科室运行 ≥ 30 天 + 提醒采纳率 ≥ X% + 0 P0 事故 | 1 → 2 |
| 院方 + 乙方双签验收单（系统内电子签）| 2 → 3 |
| 推广覆盖率 ≥ 80% 科室 + 稳定运行 ≥ 90 天 | 3 → 4 |
| 合同剩余 < 90 天 | 4 → 5（不停服）|

每次阶段切换**自动生成阶段总结报告**，作为验收 / 商务 / 续约材料的出处。

### 6.3 多维度并行

医院实际是 **多维度并行生命周期** — 不是整个医院在同一个阶段：

| 维度 | 例子 |
|---|---|
| 院区 | 总院 = 阶段 4 / 东区分院 = 阶段 3 / 南区分院 = 阶段 0 |
| 病种 | 胸痛 = 阶段 4 / 卒中 = 阶段 1 / 高血压 = 阶段 2 |
| 模块 | CDSS = 阶段 4 / 医保审核 = 阶段 3 / 质控驾驶舱 = 阶段 0 |

→ 工作台面板支持 **(院区 × 病种 × 模块) 三维切片**。

### 6.4 工作台租户生命周期面板（必装）

```
┌── 当前阶段：3 推广期（第 65 天，距阶段结束约 95 天）────────────┐
│                                                                      │
│  ✅ 0 准备   ✅ 1 试运行   ✅ 2 验收   🔵 3 推广   ⬜ 4 运行   ⬜ 5 续约 │
│                                                                      │
│  本周建议动作（3 项）：                                              │
│  • 神经内科卒中路径还差 2 节点 — [继续配置 →]                       │
│  • 第 2 院区 HIS 适配器待联通 — [处理 →]                            │
│  • DRG 8 月规则未导入 — [更新 →]                                    │
└──────────────────────────────────────────────────────────────────────┘
```

## 7. 术语表（医院听得懂的话 vs 技术名词）

| 客户面（推荐）| 不要这么说 | 含义 |
|---|---|---|
| 试点准备 | 系统初始化 / 实施配置 | 医生用之前实施 + 信息科要做的全部配置 |
| 临床运行 | 临床嵌入器 / 临床应用层 | 医生日常用的提醒、待办、路径执行 |
| 质控改进 | 数据中台 / 分析平台 | 院长 / 医务处 / 质控办看效果 |
| 合规运维 | DevSecOps / SRE | 信息科主任 + 合规审计的工具 |
| 配置包 | Asset Bundle / 资产包 | 路径 + 规则 + 字典 + 适配器配置打包 |
| 提醒治理 | CDSS 疲劳治理 | 治理"医生收到太多无用提醒"的问题 |
| 灰度发布 | Canary Deploy | 先发 10% 床位 / 一个科室验证 |
| 配置包中心 | Asset Registry | 配置包的管理与发布 |
| 适配器中心 | Adapter Hub | HIS / EMR / LIS / PACS 等接入点的管理 |
| 院级驾驶舱 | Executive Dashboard | 院长 / 医务处看的全院总览 |
| 来源追溯 | Provenance | 任何提醒可追溯到指南、文献、知识库哪一条 |
| 国密 | SM2/SM3/SM4 | 国家商用密码标准算法 |
| 国产化自检 | Domestication Check | 检查当前 OS/JDK/DB/中间件的国产化程度 |
| MPI | Master Patient Index | 患者主索引 |
| DRG / DIP | Diagnosis Related Groups / Diagnosis Intervention Packet | 医保付费分组 |

→ **客户面菜单 / 按钮 / 列名严禁出现技术缩写**（除非已经是医院术语，如 DRG / MPI / ICD）。

## 10. 内外网双形态（不可妥协）

| 形态 | 部署位置 | 关键约束 |
|---|---|---|
| **内网** | 医院本地机房 | 国产化栈（人大金仓 / 达梦 / 麒麟 / Ollama 本地大模型）；不出网 |
| **外网** | SaaS | MySQL / PostgreSQL + Dify Cloud / OpenAI Compatible；数据出境合规评估 |

→ **任何功能设计都要回答"内网怎么用？外网怎么用？"**
