# BASE-01 · 组织与租户上下文

> 读卡前置：[核心 CONSTITUTION](../../CONSTITUTION.md) + [D0 域简报](_brief.md)。
> 迁移来源（覆盖矩阵锚点）：落地规划 §4 集团化与组织复用模型（L189–236）· 详规 §7.3 领域模块与所有权 · §7.6 核心表族 · §7.7 版本继承发布（组织维）。

## 身份
- 卡 ID：BASE-01（= backlog 任务 ID）
- 域：D0 登录域 / 平台脊柱
- 关联场景：S1 集团与租户开通（提供其组织上下文底座）

- 工作量：3d
- owner / reviewer：待派单（owner ≠ reviewer）

## 目标

提供贯穿全链路的**八层组织与租户上下文**：登录解析租户 → 注入请求级不可变 `OrgContext` → 下沉到数据访问与审计，使任何业务调用都携带正确的组织作用域与包版本快照，且唯一权威源是院内关系库。

## 功能要求（原子可测条目）

- [ ] **FR-1 组织层级实体**：建 tenant(租户根) / group(集团) / hospital(医院) / campus(院区) / site(社区服务点) / department(科室) + 专病维度 七层（核心 §9），父子链**单根、无环**；提供 `orgPath` 闭包查询（给定节点返回其祖先链与后代集）。
- [ ] **FR-2 上下文解析**：登录态 → 解析出 `{tenantId, orgPath, userId, roleSet, packageVersionId, env}`，封装为**请求级不可变** `OrgContext`；解析失败不 fallback 默认租户。
- [ ] **FR-3 全链路注入**：每个 REST 入口经统一拦截器注入 `OrgContext`，service / repository 层可取；跨异步与批量任务可传播（与 [OBS-01](OBS-01.md) 的 TraceId 传播同源机制，不另造）。
- [ ] **FR-4 数据作用域下沉**：数据访问层按 `OrgContext.orgPath` 自动加**行级组织过滤**；跨租户/越权组织查询默认返回 0 行并写审计，不抛裸异常泄露存在性。
- [ ] **FR-5 包版本快照**：每请求绑定生效 `packageVersionId` 快照，同一请求内多次读配置/知识返回**同一版本**（呼应核心 §7 唯一权威知识）。
- [ ] **FR-6 缺上下文即拒**：无法解析租户/组织的业务请求返回 `ProblemDetail` 401/403（诚实拒绝，不静默放行、不落默认租户）。

## 接口契约 / 页面契约

### 接口契约（脊柱 / 内部契约）
- 端点：本卡不新增客户面业务端点；组织 CRUD 在 S1「租户开通」卡（[BASE-02](BASE-02.md) 五维权限 + D2 租户开通页）承接。本卡交付**内部上下文契约**：
  - `OrgContext`（不可变 Record）+ `OrgContextHolder`（请求级 / 异步可传播）
  - `OrgScopePort`：数据访问层组织过滤切入点（仓储基类或 MyBatis/JPA 拦截器）
- DTO：`OrgContext` 为 Record；对外暴露的组织查询用 Record DTO + Bean Validation（核心 #7）。
- 响应信封：`ApiResult` / 越权与缺上下文用 `ProblemDetail`（核心 #7 / §13）。
- 状态机：N·A —— 组织节点非「配置/变更/待办/告警」四类资产，是结构主数据（其**开通生命周期**走核心 §9 六阶段，由 S1 租户开通卡承载，本卡只提供层级与上下文）。
- 幂等 / 错误码 / traceId：组织写操作按 `(tenantId, code)` 幂等；越权=`ORG_SCOPE_DENIED`，缺上下文=`ORG_CONTEXT_MISSING`；全链路带 traceId（与 OBS-01 同源）。

### 页面契约（页面卡）
N·A —— 本卡无页面。租户切换器/组织树展示在 [BASE-06](BASE-06.md) 前端骨架 + 工作台（D1）呈现；租户开通页在 D2。

## 数据与迁移
- 表族：`org_tenant` / `org_group` / `org_hospital` / `org_campus` / `org_site` / `org_department` / `org_specialty`；`org_closure`（祖先-后代闭包，支撑层级查询）；`sys_user` / `sys_role` / `pkg_version`（绑定实体，权限细节在 BASE-02、包细节在 PKG 卡，本卡只立组织归属字段）。
- 主键：雪花/ULID 字符串主键；唯一约束：`(tenant_id, code)` 每层唯一；外键/索引：`parent_id`、`org_path`（前缀索引支撑闭包查询）、`tenant_id`。
- 组织字段：**所有业务表**统一带 `tenant_id` + `org_path`（本卡确立规约，门禁在 BASE-09 校验）。
- 状态 / 版本 / 审计字段：节点 `status`（启用/停用）；`version`（乐观锁）；`created_by/at` `updated_by/at`（审计字段规约，留痕骨干在 BASE-04）。
- 5 方言迁移：h2 / postgres / oracle / dm / kingbase 一致性 + 中文注释 + 索引约束（核心 §12；Oracle 下注意 flyway_schema_history 小写排查见运维手册）。

## 视角清单（11 视角逐条）
1. **产品架构**：八层组织 = 单一源脊柱；所有域的组织归属只有这一个家，消除多源漂移。
2. **产品体验**：N·A —— 本卡无页面；上下文驱动的「默认角色视图 / 租户切换器」在 BASE-06 + D1 呈现，本卡只供数据。
3. **系统与数据架构**：`OrgContext` 不可变 + 请求级 + 异步传播；行级数据作用域；组织节点 10 万级，组织树缓存（关系库权威、缓存投影、可重建），`orgPath` 闭包解析 P95 < 5ms；写操作按 `(tenant,code)` 幂等。
4. **临床医疗安全**：N·A（脊柱不直接触临床）—— 但组织作用域错误会致**跨患者/跨院数据串**，故 FR-4 行级隔离按高危对待，越权默认 0 行 + 审计。
5. **知识与数据治理**：`packageVersionId` 请求快照保证同一请求内知识/配置版本一致，承接核心 §7 唯一权威知识在请求层的落点。
6. **安全合规与监管**：跨租户隔离（个保法数据范围，核心 §8）；`OrgContext` 的 {who, which-org, env} 写入审计链（核心 §8 审计链）。
7. **集团化与多租户治理**：★本卡主战场 —— 七层继承覆盖 + 租户隔离 + 影响范围（核心 §9）；继承/局部覆盖/冲突仲裁的**数据底座**在此，规则求值在 BASE-02 / 配置卡。
8. **集成与互操作**：外部系统院区/科室编码 → 本组织层级的映射锚点（具体适配在 D2 INTEG 卡）；本卡只定义可被映射的组织主键口径。
9. **运维 / SRE / 国产化**：5 方言迁移；组织树缓存可重建（关系库权威）；内外网双形态下组织树均在院内关系库。
10. **质量与真实性审计**：无写死租户/组织常量；缺上下文**诚实拒绝**不 fallback（铁律#1 真实性 / #2 诚实降级）；门禁校验"业务表必带 tenant_id+org_path"。
11. **AI / 模型治理与可降级**：N·A —— 脊柱层无模型；本卡天然 B0（纯确定性）。

## 适用不变量
- 命中核心约束：**#7 五维权限（数据维）** · **§9 集团多租户继承** · **§7 关系库权威 / 铁律#5** · **#19 配置外置**（组织树不写死 yml）。
- 本卡落点：以 `OrgContext` 注入 + 行级数据作用域 + `packageVersion` 请求快照，把"组织/租户/版本"三件事在请求入口一次性钉死，下游零散落。
