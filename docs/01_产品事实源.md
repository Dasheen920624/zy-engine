## 1. 一句话

> **让医院能像产品经理管理软件一样管理临床路径、规则和医学知识：版本化、可灰度、可追溯、可下线，每一次诊疗决策都能回到来源。**

## 2. 产品定位三问

### 2.1 解决什么问题

医院当前的临床路径管理普遍卡在三个口子：

| 卡口 | 现状 | 后果 |
|---|---|---|
| **临床路径"画在 Word 里"** | 路径定义是文档，无法执行、无法版本化、无法度量 | 路径形同虚设，质控只能事后审单 |
| **规则"埋在医生脑子里"** | 医嘱合理性靠经验，新人靠老师傅带 | 高风险医嘱漏报，医保超付追责 |
| **知识"散在指南/文献里"** | 最新指南更新慢、查阅难、关联弱 | 临床决策落后于循证更新 |

平台要回答的核心问题：

> 一个医院如何把"专家经验 + 行业指南 + 监管要求"持续地、可审计地、可灰度地注入到日常诊疗流程？

### 2.2 不解决什么问题

明确不做：

- ❌ 替代 HIS / EMR / LIS / PACS 系统本体
- ❌ 做 AI 聊天机器人入口（AI 是后台能力，不是前台形态）
- ❌ 替代医生判断（所有临床建议必须由医生最终确认）
- ❌ 做大数据 BI 报表平台（只提供与质控/路径直接相关的看板）
- ❌ 做手术室排程、床位管理、药品库存等运营调度系统

### 2.3 谁为之付费

- **付费方**：三级医院 / 区域医联体 / 集团化民营医疗
- **决策链**：院长（战略）→ 医务处主任（业务）→ 信息科主任（实施）→ 临床科室主任（采纳）
- **付费触发点**：JCI/HIMSS 评审、医保 DRG/DIP 改革、单病种质控考核、专科联盟建设

### 2.4 部署形态与外网可用性

MedKernel 支持两类部署形态，但用途、权限和数据边界必须分清：

| 形态 | 可用范围 | 当前 MVP 结论 |
|---|---|---|
| 院内 / 专网部署 | HIS/EMR/CDSS 适配、真实患者数据、临床嵌入、规则执行、质控运营 | 当前优先交付形态，可在医院内网、专线、VPN 或零信任网关后运行 |
| 外网 / SaaS 基准服务 | 租户开通、标准知识包运营、配置包生成、脱敏治理包、客户演示和实施协作 | 产品方向支持，但不得接收院内真实患者隐私；公网开放前必须完成 `SEC-011`、强认证、MFA、租户隔离、审计、网关/WAF 和合规评估 |

因此，前端统一称为「管理工作台」（v0.3-final 主题描述同步更新），不要写死为"内网管理台"或旧称"治理控制台"。这套系统不是只能内网使用；但当前代码基线尚未实现完整外网账号体系和生产级权限链路，**不能直接裸露到公网作为生产系统**。外网使用应先限制在脱敏演示、基准知识包运营或受控客户协作场景。

## 3. 三产品分层（V2 核心决策）

### 3.1 为什么要分层

平台**本质上是三个不同节奏的产品共用一套底座**。试图用一套设计规范覆盖三种用户是 V1 的最大错误。

```
A 配置工厂     B 临床嵌入器     C 质控驾驶舱
───────────  ───────────────  ───────────────
创作型         事件触发型        巡检型
周/月使用      秒/分钟使用       日/周使用
中等密度       极低密度          极高密度
Notion 范      Linear 范         Grafana 范
错可回滚       错伤患者          错偏统计
```

### 3.2 A 配置工厂（A-Factory）

**定位：** 让医学专家像产品经理一样创作和发布"临床产品"。

| 维度 | 设计取向 |
|---|---|
| 灵感参照 | Notion（块编辑） + Figma（版本对比） + Airtable（结构化） |
| 主导界面 | 大画布、抽屉式属性面板、左侧树、顶部版本控制条 |
| 节奏 | 一次创作几分钟到几小时；保存可异步 |
| 关键操作 | 拖拽、撤销/重做、版本对比、发布门禁 |
| 不能犯的错 | 误覆盖已发布版本、缺来源直接发布 |

**核心场景：**
1. 临床路径专家拖拽创建 AMI/STEMI 路径
2. 规则工程师写"血红蛋白 < 70 + 年龄 ≥ 65 → 输血指征评估"DSL
3. 知识工程师审核 AI 从指南里抽取的候选知识
4. 信息科把配置包从测试环境同步到生产

**对应模块（已实现/待实现）：**
- 路径模板编辑器（待）
- 规则 DSL 编辑器（部分）
- 图谱编辑器（待）
- 字典映射工作台（部分）
- 知识审核台（待）
- 配置包发布向导（部分）

### 3.3 B 临床嵌入器（B-Embed）

**定位：** 在医生护士已有工作流（HIS/EMR）的"边上"，在关键时刻显现，其余时间隐身。

| 维度 | 设计取向 |
|---|---|
| 灵感参照 | Linear（通知极简） + Apple Health（健康提醒） + Slack 桌面通知 |
| 主导界面 | 浮层、通知条、迷你抽屉、嵌入式卡片 |
| 节奏 | 单次交互 < 30 秒；用户在做其它事，必须"快进快出" |
| 关键操作 | 一键确认、一键忽略（要审计原因）、查看证据、转人工 |
| 不能犯的错 | 噪声多致医生开发出"忽略一切"的肌肉记忆；漏报关键预警 |

**核心场景：**
1. 心内科医生在 HIS 开"低分子肝素"医嘱，平台浮出"该患者血小板 < 50，出血风险评估提醒"
2. 病房护士执行护理任务时，看到节点流转提示"该患者下一步：肝功能检查"
3. 药师审方时收到"该患者近 24h 已用 3 种 PPI，重复用药警示"

**对应模块（基本都待实现）：**
- HIS/EMR 嵌入式预警条
- 医生工作站浮层（独立 Web 或 iframe 嵌入）
- 节点流转提示卡
- 一键确认/忽略 + 原因记录
- 患者路径状态迷你视图

### 3.4 C 质控驾驶舱（C-Cockpit）

**定位：** 让管理层能看见全院临床质量、规则命中、变异分布，并能下钻到具体患者/医生。

| 维度 | 设计取向 |
|---|---|
| 灵感参照 | Datadog（指标钻取） + Grafana（图表组合） + Linear Cycles（趋势对比） |
| 主导界面 | 看板卡片矩阵、时间序列图、热力图、漏斗图、表格 |
| 节奏 | 每日早晨 5-10 分钟巡检；每周深度分析 1 次 |
| 关键操作 | 筛选、对比、下钻、导出、推送给责任人 |
| 不能犯的错 | 数据口径前后不一致、看板加载慢、无法导出 |

**核心场景：**
1. 医务处主任早晨打开"全院质控驾驶舱"，看到今日 127 条预警、43 待处理
2. 主任医师查看本科室"本月路径变异 TOP10"，下钻到具体患者
3. 医保审核员对比"本月医保规则命中分布"，定位高发科室

**对应模块（基本都待实现）：**
- 院级驾驶舱（FE-008）
- 科室质控看板
- 规则命中分析
- 路径变异分析
- 整改任务跟踪

### 3.5 产品归属判断规则

任何新功能、新页面、新 API 必须明确归属 A/B/C：

```
是"创作 / 编辑 / 配置 / 版本 / 发布 / 审核"？ → A 配置工厂
是"事件 / 预警 / 实时 / 嵌入 / 工作流嵌"？   → B 临床嵌入器
是"统计 / 看板 / 报表 / 钻取 / 趋势"？       → C 质控驾驶舱
是基础设施（身份/审计/组织/字典）？         → 共用底座（无产品归属）
```

兼具多重身份的对象（如"规则"）→ **拆成 A 视角和 B 视角两个页面/接口**，不要一锅炖。

## 7. 22 个不变量（任何 AI 不得违反）

不变量是"任何任务都不能破坏的硬约束"。22 项分为 4 类：

### 7.1 数据完整性（5 项）

1. 每个业务对象必须有 `tenant_id`，写入前必须从组织上下文获取
2. 每个人类操作必须有 `platform_user_id`，外部账号（HIS 工号/LDAP DN）只作为身份绑定
3. 每个服务调用必须有 `client_id` + 签名 + nonce + 时间戳（±5 分钟）
4. 每条配置变更必须有 `version`，同 `code + version` 内容不同必须返回错误（不允许静默覆盖）
5. 每个患者标识必须可追溯到 MPI（患者主索引），允许"暂未注册"但不允许"无标识"

### 7.2 医学安全（5 项）

6. 临床建议必须由医生最终确认（系统给建议，医生做决定）
7. 阻断类规则必须有 `action_mode + doctor_decision` 字段，未明确授权不得自动阻断
8. 医学/医保/质控配置必须有来源（`source_review_id`），缺来源必须返回 `MISSING_SOURCE` 阻断发布
9. AI 生成内容必须有 🤖 标识，必须经过人工审核才能进入正式版本
10. 患者隐私数据日志必须脱敏，禁止输出完整身份信息/密码/API Key

### 7.3 可审计可追溯（5 项）

11. 所有写操作必须写 `ENGINE_AUDIT_LOG`
12. 所有 API 必须带 `trace_id`，前端→网关→业务→数据库→外部系统全链路串联
13. 审计表禁止物理删除，禁止 UPDATE（仅 INSERT），防篡改触发器必须开启
14. 发布、回滚、导入、导出、审核、权限变更必须有操作人、时间、原因、上下文
15. 每个规则命中必须能回溯到规则版本、命中事实、命中时间

### 7.4 架构与运维（7 项）

16. 业务服务只调 Provider 接口，不得直接依赖 Oracle / Neo4j / Dify / HIS 等具体实现
17. Neo4j / Dify / 大模型不可用时，DB-only 核心流程必须仍可执行
18. DDL 必须同步 Oracle、达梦、PostgreSQL/Kingbase、LOCAL_H2_FILE 四套
19. 不允许硬编码单医院/单院区/单系统厂商逻辑（差异通过配置包/字典映射/适配器绑定实现）
20. API 必须返回 `ApiResult { success, code, message, data, trace_id }` 统一结构
21. 所有 API 必须接收组织上下文（Header 全集 + Body 优先级最高）
22. 前端必须实现六态：加载、空、错误、无权限、处理中、成功反馈

| 术语 | 定义 | 不要混用 |
|---|---|---|
| 路径（Pathway） | 临床路径的可执行配置，含阶段-节点-任务三级 | ❌ 路径文档 / ❌ 工作流 |
| 路径模板（PathwayTemplate） | 路径的元配置（可以被发布为多个版本） | ❌ 路径定义 |
| 路径版本（PathwayVersion） | 模板的某个发布版本 | ❌ 路径快照 |
| 患者路径实例（PatientPathInstance） | 某位患者按某个路径版本执行的运行实例 | ❌ 患者路径 |
| 规则（Rule） | 单条 if-then 规则的 DSL 定义 | ❌ 业务规则 / ❌ 校验规则 |
| 规则组合（RuleGroup） | 多条规则的逻辑组合 | ❌ 规则集 |
| 配置包（ConfigPackage） | 一组可发布资产的快照 + 版本 + 来源审核 | ❌ 配置文件 / ❌ 部署包 |
| 资产（Asset） | 配置包内的单个对象（路径/规则/字典/图谱/Dify 模板等） | ❌ 文件 / ❌ 配置项 |
| 来源（Source） | 医学/医保/质控内容的依据（指南/标准/法规/院内文件） | ❌ 数据源 |
| 引用片段（Citation） | 来源文档中的具体段落（含定位信息） | ❌ 摘录 |
| 字典（Terminology） | 标准编码体系（ICD/SNOMED/LOINC + 院内标准 + 系统内部码） | ❌ 数据字典 |
| 映射（Mapping） | 院内编码 ↔ 标准编码的转换规则 | ❌ 字典转换 / ❌ 翻译 |
| 适配器（Adapter） | 与外部系统（HIS/EMR/LIS/PACS）交互的防腐层 | ❌ 接口 / ❌ 网关 |
| 组织上下文（OrgContext） | 租户+集团+医院+院区+站点+科室的 5 段层级 | ❌ 组织信息 |
| 平台用户（PlatformUser） | 本系统独立用户身份，所有审计的主体 | ❌ HIS 用户 / ❌ SSO 用户 |
| 服务账号（ServiceAccount） | 系统对系统调用的客户端身份（API Key + 签名） | ❌ API 账号 |
| 预警（Alert） | 规则命中后向用户推送的实时信号 | ❌ 告警 / ❌ 通知 |
| 变异（Deviation） | 患者路径执行与模板不符的情况 | ❌ 例外 / ❌ 偏离 |
| 整改（Rectification） | 质控问题的处理与闭环 | ❌ 修复 / ❌ 解决 |
| 不变量（Invariant） | 任何任务都不得违反的硬约束 | ❌ 规范 / ❌ 约束 |

## 9. 状态机统一定义

### 9.1 配置类对象通用状态机

适用于：路径模板版本、规则版本、图谱版本、字典映射、配置包、知识候选

```
        ┌────────┐  提交审核   ┌─────────┐  审核通过   ┌──────────┐
        │ DRAFT  │ ─────────► │REVIEWED │ ─────────► │PUBLISHED │
        │ 草稿   │            │ 待发布  │            │ 已发布   │
        └────────┘            └─────────┘            └────┬─────┘
            ▲                       │                     │
            │ 驳回                  │ 驳回                │ 同步至目标环境
            │                       ▼                     ▼
            └─────────────── REJECTED                  ┌──────┐
                            已拒绝                    │SYNCED│
                                                      │已同步│
                                                      └──┬───┘
                                                         │ 激活
                                                         ▼
                                                      ┌──────┐
                                                      │ACTIVE│
                                                      │运行中│
                                                      └──┬───┘
                                                         │ 下线
                                                         ▼
                                                      ┌───────┐
                                                      │RETIRED│
                                                      │已下线 │
                                                      └───────┘
```

### 9.2 患者路径实例状态机

```
PENDING（待入径） → ACTIVE（执行中） → COMPLETED（完成）
                       │
                       └─► DEVIATED（变异） → ACTIVE（继续）
                       │                  └─► EXITED（退出路径）
                       │
                       └─► SUSPENDED（暂停） → ACTIVE
```

### 9.3 预警处理状态机

```
NEW（新预警） → ACKNOWLEDGED（已查看） → PROCESSING（处理中） → RESOLVED（已处理）
                                                              ─► IGNORED（已忽略，必须记原因）
                                                              ─► ESCALATED（已升级）
```
