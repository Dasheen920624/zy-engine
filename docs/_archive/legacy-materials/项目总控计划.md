# 专科诊疗路径智能管理平台项目总控计划

> 说明：本文是早期总控计划，用于保留项目背景。后续 AI 开发、产品生命周期、集团化医院支持、跨环境一致性和验收流程，以 `zy-engine-mvp/docs/AI接手执行手册.md` 与 `zy-engine-mvp/docs/产品化方案与AI开发编排.md` 为准。

## 1. 项目定位

本项目建设目标是形成一套独立能力型的专科诊疗路径管理平台，可被医生工作站、质控平台、随访平台、数据中台和 Dify 编排能力调用。

平台核心不是单一页面或单一 AI 问答，而是三类可复用引擎能力：

- 路径引擎：管理患者从识别、入径、治疗、出院、随访到质控闭环的状态流转。
- 规则引擎：承载时限质控、内涵质控、安全拦截、路径准入和节点完成判断。
- 图谱引擎：承载权威医学知识、路径证据、疾病关系、症状指标关系和推荐解释。

现有院内能力：

- 已部署 Dify，可作为流程编排与大模型协作层。
- 已使用 Neo4j，可作为医学知识图谱与路径关系层。
- Oracle 数据库已可访问，`ZYENGINE` 用户已创建核心表。
- 本项目后端 MVP 已按 JDK 1.8 兼容方式搭建。

## 3. 近期四周计划

### 第 1 周：工程地基稳定

目标：

- 稳定 GitHub 版本管理。
- 建立 CI 自动化验证。
- 固化编码、构建、启动、建表、测试脚本。
- 梳理 P1 后端模块边界和任务拆分。

交付：

- GitHub 仓库 main 分支。
- GitHub Actions CI。
- 项目总控计划和研发路线图。
- P1 开发任务清单。

### 第 2 周：规则引擎 P1

目标：

- 实现规则定义导入、保存、发布。
- 实现最小规则 DSL 解析器。
- 实现规则模拟执行接口。
- 实现规则执行日志。

验收：

- `sample_ami_rules.json` 可导入。
- AMI/STEMI 候选规则可模拟命中。
- 心电图时限规则可判断合格/超时。
- 执行日志可查询。

### 第 3 周：路径引擎 P1

目标：

- 实现路径配置导入、校验、发布。
- 实现路径节点、任务、流转条件解析。
- 实现患者路径实例幂等入径。
- 实现节点任务状态与路径变异记录。

验收：

- `sample_ami_pathway.json` 可导入发布。
- 患者入径后可按配置流转。
- 节点任务可记录完成状态。
- 偏离路径可记录变异原因。

### 第 4 周：图谱/Dify/适配器闭环

目标：

- 实现 Neo4j 查询适配接口。
- 实现图谱候选疾病召回和证据查询。
- 实现 Dify 工作流调用、超时和降级。
- 实现第三方适配器 Mock 与字典映射。

验收：

- AMI 样例从患者上下文到推荐解释形成完整闭环。
- Dify 不可用时不影响路径核心状态。
- 字典映射可处理院内编码与平台标准编码不一致场景。

## 4. 产品工作流

产品设计优先顺序：

1. 医生工作站入口：风险提示、候选路径、入径确认、推荐解释。
2. 路径配置中心：路径阶段、节点、任务、规则绑定、Dify 绑定、版本发布。
3. 规则配置中心：规则类型、条件、动作、模拟、发布、回滚。
4. 图谱管理中心：概念、关系、证据、版本、发布。
5. 质控看板：时限质控、内涵质控、变异统计、路径执行率、医生响应率。
6. 随访管理：随访计划、任务生成、患者触达、异常回流。

## 5. 技术工作流

后端服务拆分建议：

- `pathway-engine-service`：路径定义、版本、患者实例、节点流转、变异。
- `rule-engine-service`：规则定义、DSL 解析、执行、日志、模拟。
- `graph-engine-service`：Neo4j 查询、证据查询、图谱版本。
- `adapter-hub-service`：HIS/EMR/LIS/PACS/ECG 等第三方数据适配。
- `terminology-service`：院内字典到平台标准字典映射。
- `dify-adapter-service`：Dify 工作流调用、超时、降级、审计。
- `portal-ui`：医生工作站嵌入页、配置后台、质控看板。

当前 MVP 先采用单体工程组织，模块边界按包拆分，P2 后根据部署和性能需要再拆服务。

## 6. 验证体系

每个阶段必须通过四类验证：

- 功能验证：接口、页面、流程是否按预期工作。
- 临床逻辑验证：规则、路径、证据是否符合院内专家共识。
- 数据验证：字典映射、患者上下文、落库、审计是否正确。
- 工程验证：JDK8 编译、编码校验、脚本执行、CI、数据库兼容是否稳定。

## 7. 风险与控制

| 风险 | 控制方式 |
|---|---|
| 医院系统字典不一致 | 建设 `TM_STANDARD_CONCEPT` 和 `TM_CONCEPT_MAPPING`，所有规则优先使用标准编码 |
| 大模型输出不稳定 | LLM 仅做解释和辅助排序，安全拦截和核心状态由规则/路径引擎负责 |
| Dify 不可用 | 设置超时、降级、重试和审计，不影响路径核心状态 |
| 中文乱码 | 统一 UTF-8，Oracle 备注使用 `UNISTR` 写入，脚本使用 PowerShell 7 优先 |
| JDK 版本限制 | 后端保持 JDK 1.8 编译目标，CI 使用 JDK 8 验证 |
| 临床责任边界 | 推荐卡片明确“医生确认”，关键动作保留人工确认和审计 |

## 8. 我后续的执行方式

我会按以下节奏持续推进：

1. 每轮先确认当前仓库状态、服务状态和测试状态。
2. 对一个明确阶段目标进行设计、编码、验证。
3. 每次完成后提交 Git，并推送到 GitHub。
4. 每个阶段输出可交付文档、接口、样例和验收结果。
5. 对影响临床安全、数据安全、部署稳定性的事项优先处理。

近期下一步建议直接进入 P1：

- 先做规则引擎 DSL 和规则发布闭环。
- 同时完善路径引擎配置导入和发布。
- 用 AMI 样例作为第一条自动化验收主线。
