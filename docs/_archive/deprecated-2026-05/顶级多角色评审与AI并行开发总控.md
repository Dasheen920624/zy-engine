## 2. 总体结论

现有方案方向正确，已经具备并行开发条件：

- Oracle/关系型数据库作为配置、版本、审计和运行记录主数据源是正确选择。
- Neo4j 和 Dify 定位为投影与执行目标，不作为 DB-only 验收强依赖，便于内网交付。
- 组织上下文已经开始织入规则、路径、质控和审计，方向正确。
- 规则、路径、图谱、Dify、字典、适配器等配置资产都需要版本、发布、回滚和来源追溯。
- 客户提出的“规则和知识要有来源查阅”应升级为正式来源追溯平台，而不仅是规则表上的几个字段。
- 后续最大风险不是单个接口能否写出来，而是多个 AI 并行时边界不清、验收不一致、Oracle 未验证、前端不可演示、医学安全缺审查。

本次评审后的核心优化策略：

```text
以配置包为交付单元
+ 以组织范围为隔离单元
+ 以来源追溯为可信单元
+ 以 Oracle 落库和 smoke 为验收单元
+ 以前端演示工作台为客户验收单元
```

## 3. 顶级医疗产品经理评审

### 3.1 当前优点

- 产品已经从“后端引擎接口”升级为“集团化医疗智能引擎平台”。
- 功能覆盖路径、规则、图谱、Dify、字典、适配器、质控、审计、前端配置平台，方向完整。
- 已经意识到非开发人员需要前端演示与配置，不应只靠 Postman。
- 已经把来源追溯作为客户可信任系统的关键能力。

### 3.2 主要缺口

- 用户旅程还不够产品化：医生、临床专家、质控科、医保办、信息科、院领导的入口和日常任务需要进一步分层。
- 功能清单有模块，但还缺“客户现场验收故事线”：如何从导入配置、审核来源、发布规则、模拟患者、生成质控结果、追溯来源、导出报告完成闭环。
- 来源追溯不能只满足“能查到来源”，还要满足“来源是否适用当前组织、是否过期、谁审核过、哪些规则受影响”。
- 医保质控、病历质控和医嘱安全要有清晰业务价值指标，便于院长和科室主任判断投入产出。

### 3.3 产品优化决策

1. 所有演示必须围绕四条主线建设：
   - AMI 专病路径推荐与入径。
   - 病历质控缺项与时限提醒。
   - 医保质控风险提示与申诉材料。
   - 医嘱安全、用药适应症和检查合理性。
2. 前端首屏不做营销页，必须是可操作工作台。
3. 每个质控或规则结果必须展示：
   - 命中原因。
   - 患者事实。
   - 规则版本。
   - 规则包版本。
   - 来源标题、机构、版本、章节或摘要。
   - 审核人和审核时间。
   - traceId。
4. 配置发布前必须有 review 页面：
   - 配置 hash。
   - 版本 diff。
   - 来源完整性。
   - 组织影响范围。
   - 高风险变更提示。
   - 回滚方案。
5. 客户验收手册必须按角色写，而不是按接口写。

## 4. 顶级架构师评审

### 4.1 当前优点

- Oracle 主数据、Neo4j 投影、Dify 执行目标的分层正确。
- Provider 化和 DB-only 验收是正确的交付架构。
- 配置包生命周期和组织上下文已经形成基础。
- 审计、traceId、降级和来源追溯已经进入架构约束。

### 4.2 架构缺口

- 缺少统一“资产注册表”：规则、路径、图谱、Dify、字典、适配器、质控指标都应作为配置资产注册，便于 review、来源绑定、影响分析和包发布。
- 缺少正式“发布指针表”：ACTIVE 版本不能散落在各模块内存或各自表里，应有统一 active pointer 或模块内等价结构。
- 缺少“同步任务模型”：Oracle 到 Neo4j、Dify binding、配置导入导出、前端发布都需要异步任务、重试、状态和审计。
- 缺少“运行证据链模型”：一次规则命中、路径推荐、Dify 解释、质控结论，需要串起患者事实、规则、来源、动作和审计。
- 缺少“安全与权限基座”：用户、角色、组织范围、第三方调用签名、审计脱敏要尽快立项。

### 4.3 架构优化决策

统一引入五类底座：

```text
CFG_*    配置包、资产注册、发布指针、同步任务
ORG_*    组织目录、组织上下文、组织继承与覆盖
SRC_*    来源文档、引用片段、资产绑定、来源审核、运行证据
RUN_*    运行实例、规则结果、质控结果、Dify 调用、任务状态
SEC_*    用户、角色、权限、API key、签名、防重放、脱敏策略
```

统一运行链路：

```text
请求进入
-> 解析组织上下文
-> 标准化患者事实
-> 选择 ACTIVE 配置包和组织覆盖
-> 执行规则/路径/图谱/Dify
-> 生成结果和运行证据链
-> 写审计与运行日志
-> 返回可解释、可追溯结果
```

统一 Provider 边界：

```text
DatabaseProvider      Oracle/达梦访问、迁移、健康检查
GraphProvider         DB 图谱查询 / Neo4j 查询
WorkflowProvider      Local fallback / Dify
AdapterProvider       HIS/EMR/LIS/PACS/医保
TerminologyProvider   标准码和映射
SourceProvider        来源文档、引用和资产绑定
AuditProvider         审计、trace 和脱敏
```

## 5. 顶级开发工程师评审

### 5.1 当前优点

- Java 后端已经有契约测试、smoke 脚本、API 示例和 Oracle 脚本。
- 任务编号体系清楚，便于多人并行。
- 已有脚本能在真实 Oracle 上验证组织上下文落库。

### 5.2 开发缺口

- 部分能力仍是内存环形缓冲，缺少 Oracle 持久化和跨实例聚合。
- 配置包、来源追溯、发布指针和同步任务还没有统一表结构。
- API 错误码、分页、排序、过滤字段需要持续统一。
- 前端工程尚未落地，客户演示仍依赖后端脚本。
- 领域模型应更明确区分配置态、发布态、运行态、审计态。

### 5.3 开发优化决策

后续开发必须坚持：

- 新增接口先写请求/响应样例，再实现代码。
- 新增表先写生产库 DDL（Oracle/达梦/PostgreSQL-Kingbase）、开发库 DDL（LOCAL_H2_FILE）、迁移脚本、中文备注，再接应用。
- 新增持久化必须有 LOCAL_H2_FILE 开发库验证；有生产库环境时必须有对应生产库 smoke，没环境时必须写明补验计划。
- 新增配置资产必须进入配置包 review。
- 新增医学/医保/质控配置必须接来源追溯检查。
- 新增前端页面必须接真实 API 或明确 mock 契约，不能只画静态页面。
- 每个任务只改自己的模块，不顺手重构其它模块。

## 6. 顶级测试人员评审

### 6.1 当前优点

- 已有 `run-tests.ps1`、`build.ps1`、`verify-encoding.ps1`、多组 smoke 脚本。
- 已要求 Oracle 真实验证，避免只在内存里“假通过”。
- 已覆盖 DB-only、Dify 降级、Neo4j 降级的方向。

### 6.2 测试缺口

- 缺少统一测试矩阵，把模块、组织范围、来源状态、发布状态、数据库模式串起来。
- 缺少客户 UAT 剧本，尤其是医生、质控、医保、信息科、院领导视角。
- 前端尚未有 Playwright/E2E、可访问性和截图回归。
- 性能与稳定性口径不够清楚，例如批量规则评估、配置包导入、质控报表聚合。
- 安全测试尚未成为 DoD，如越权访问、签名错误、重放请求、日志脱敏。

### 6.3 测试优化决策

每个能力至少要有四层验证：

```text
契约测试：接口结构、错误码、traceId、兼容性
单元/服务测试：领域逻辑、边界条件、降级路径
smoke 测试：真实运行链路，必要时 Oracle 真实落库
UAT 剧本：非开发人员按页面或脚本完成验收
```

测试矩阵必须覆盖：

- 组织范围：默认、集团、医院、院区、站点、科室。
- 来源状态：缺来源、未审核、已审核、过期、被替换。
- 发布状态：DRAFT、REVIEWED、PUBLISHED、SYNCED、ACTIVE、RETIRED。
- Provider 状态：DB-only、Neo4j 可用/不可用、Dify 可用/不可用、适配器超时。
- 数据库：内存/JUnit、LOCAL_H2_FILE 开发库、Oracle/达梦/PostgreSQL-Kingbase 生产库 DDL 兼容。
- 安全：匿名、低权限、越权组织、错误签名、重复 nonce。

## 8. 顶级医生评审

### 8.1 当前优点

- 系统强调医生确认，未把 AI 结果直接变成医疗处置。
- 规则、路径、图谱证据和解释都要求来源追溯。
- 规则结果已经有命中原因、证据、动作、严重级别的方向。

### 8.2 医疗安全缺口

- 临床建议需要区分：提示、强提醒、阻断、需上级确认、仅记录。
- 规则命中必须显示“适用条件”和“不适用条件”，避免医生误解。
- 必须支持医生覆盖/忽略原因，并形成质控复盘数据。
- 需要避免提醒疲劳：低价值提醒应可统计、降级、合并或按科室配置。
- 医学来源不能只记录标题，还要记录推荐等级、证据等级、适用人群、禁忌条件。

### 8.3 医生优化决策

所有临床相关结果必须包含：

```text
clinical_severity      INFO / WARNING / CRITICAL / BLOCKING
action_mode            REMIND / STRONG_REMIND / BLOCK / REQUIRE_CONFIRM / RECORD_ONLY
applicable_population  适用人群
exclusion_criteria     排除条件
evidence_summary       来源摘要
evidence_level         证据等级
recommendation_grade   推荐强度
doctor_decision        ACCEPTED / OVERRIDDEN / DEFERRED
override_reason        覆盖原因
```

临床安全原则：

- 系统给建议，医生做决定。
- 阻断类规则必须经过医学审核和院内授权。
- AI/Dify 解释不能生成未绑定来源的治疗建议。
- 所有高风险建议必须能追溯到来源、规则版本和患者事实。

## 11. 并行开发总原则

### 11.1 任务粒度

一个 AI 一次只领取一个明确任务。每个任务必须能在 1 到 3 个工作单元内完成，且有清晰文件范围。

任务必须包含：

- 任务编号。
- 业务目标。
- 输入资料。
- 修改范围。
- 不允许修改范围。
- 接口契约。
- 数据库影响。
- 来源追溯影响。
- 前端影响。
- 测试和 smoke。
- Oracle 验证要求。
- 文档要求。

### 11.2 并行写入边界

多 AI 并行时按文件和模块分工：

```text
数据库 AI：db/oracle、ai-dev-input/04_database、scripts/run-oracle-*.ps1
后端 AI：src/main/java/com/zyengine、src/test/java/com/zyengine
前端 AI：frontend 或后续前端工程目录、前端规划文档
测试 AI：scripts/run-*-smoke.ps1、api-examples.http、测试矩阵
文档 AI：docs、ai-dev-input/09_ai_task_cards
工程 AI：scripts、部署说明、环境检查、日志和运维文档
```

禁止两个 AI 同时无协调地修改同一服务类、同一 DDL 文件或同一前端路由文件。必须先拆分任务，避免互相覆盖。

### 11.3 自主运行和任务选择

当用户授权 AI 自主开发时，AI 不能随机选择任务，必须按以下优先级推进：

```text
先修复阻断 review
-> 接管 stale/handoff
-> 执行当前最高优先级任务池
-> 补测试、smoke、文档和 DDL 一致性缺口
-> 选择低冲突小切片
```

自主运行必须创建：

```text
ai-dev-input/12_autonomous_runs/active/<run_id>.md
```

运行记录必须写清：

- 为什么选择当前任务。
- 当前 claim 和 review。
- 验证结果。
- 停机条件是否触发。
- 下一步交接。

详细流程以 `zy-engine-mvp/docs/AI自主开发运行守则.md` 为准。

任务选择还必须对齐 `zy-engine-mvp/docs/产品功能业务核查与开工清单.md`，明确本任务服务的用户角色、业务闭环和客户验收故事线。

### 11.4 任务认领和锁定

多 AI 并行开发必须使用 `ai-dev-input/10_task_claims` 作为任务锁目录。任何 AI 未成功推送 active claim 前，不得修改业务代码、DDL、脚本、样例或核心文档。

认领文件：

```text
ai-dev-input/10_task_claims/active/<claim_id>.md
```

认领要求：

- `claim_id` 必须是任务切片，例如 `PROV-001-S03`，不能只写 `PROV-001`。
- claim 必须声明 `task_id`、`owner`、`status`、`write_scope`、`forbidden_scope`、`verification`。
- `write_scope` 与其它 active claim 重叠时，后认领者必须停止并拆分任务。
- claim 推送失败时，必须重新 `git pull --ff-only` 并检查是否已被其它 AI 抢先认领。
- 超过 60 分钟的任务必须更新 `last_heartbeat`。
- 超过 4 小时无心跳视为可能 stale；超过 24 小时且无相关提交，可由新 AI 创建接管 claim。
- 完成后必须把 claim 移入 `archive/YYYYMMDD/` 并写明验证结果、提交 hash 和风险。

冲突优先级：

```text
远端已推送 claim > 更小写入范围 > 明确用户接管指令 > 已测试并推送的实现 > 集成任务统一处理
```

共享文件必须在 claim 中写明具体修改区块，包括 README、API 示例、总控文档、契约测试、DDL 和核心服务类。

详细流程以 `zy-engine-mvp/docs/AI任务认领与并行开发机制.md` 为准。

### 11.5 质量门禁和整改闭环

任务认领只能防止多个 AI 撞车，不能证明实现符合要求。所有任务完成后必须进入 `ai-dev-input/11_ai_reviews` 质量门禁。

评审状态流：

```text
SELF_CHECK
-> REVIEW_REQUESTED
-> CHANGES_REQUESTED / FIXING
-> APPROVED
-> READY_TO_SUBMIT
-> DONE
```

执行规则：

- Builder AI 完成实现后，先自检并创建 `pending/<review_id>.md`。
- Reviewer AI 按需求、架构、医疗安全、数据库一致性、测试、安全、前端体验和运维维度审查。
- 有 P0/P1/P2 开放问题时，review 必须进入 `changes_requested/`，claim 必须进入 `CHANGES_REQUESTED` 或 `FIXING`。
- 原 Builder AI 仍在线时，必须由原 Builder AI 自行整改；不可用时，由新的 AI 创建修复 claim 接管。
- `review_status=APPROVED` 且 `open_findings=0` 后，业务代码才允许正式提交或合入主版本。
- 高风险任务不得自审批准；医学、医保、质控、安全、数据库、持久化和发布链路必须独立评审。
- claim、heartbeat、review 元数据可以同步到 `main`；未通过质量门禁的业务代码不能进入主版本。

详细流程以 `zy-engine-mvp/docs/AI开发质量门禁与评审整改机制.md` 为准。

## 14. 不同能力 AI 的执行规则

### 14.1 初级 AI

允许做：

- 单个接口的小字段补充。
- 单个文档或样例更新。
- 单个 smoke 脚本补充。
- 单个测试用例补充。

禁止做：

- 跨模块重构。
- 修改核心发布状态机。
- 修改多张 DDL 表。
- 同时改前端和后端。
- 删除或回滚别人改动。

必须照抄任务卡执行，不自行扩大范围。

### 14.2 中级 AI

允许做：

- 一个模块内的完整小功能。
- 一个表和对应 API 的持久化。
- 一个前端页面和 API client 对接。
- 一个 smoke 脚本和测试矩阵。

要求：

- 必须先读相关代码和测试。
- 必须补 API 示例。
- 必须跑完整验证。
- 涉及 Oracle 必须真实验证。

### 14.3 高级 AI

允许做：

- 跨后端、数据库、测试、文档的完整任务。
- 架构边界内的模块抽象。
- 性能和稳定性优化。
- 来源追溯、组织继承、配置包发布等底座能力。

要求：

- 必须维护向后兼容。
- 必须控制写入范围。
- 必须更新总纲或总控文档中的状态。
- 必须给后续 AI 留清楚的下一步。

## 15. 产品级验收剧本

### 15.1 医生演示

输入：AMI 患者上下文。

流程：

1. 系统推荐 AMI_STEMI 路径。
2. 医生查看命中规则、检查事实和来源摘要。
3. 医生确认入径。
4. 系统生成任务和节点状态。
5. 医生完成或跳过任务。
6. 若偏离路径，记录变异原因。

验收：

- 医生能看懂为什么推荐。
- 医生能看到来源但不会被冗余信息淹没。
- 医生可以覆盖建议并说明原因。

### 15.2 质控演示

输入：病历质控样例。

流程：

1. 选择病历质控场景。
2. 执行 dry-run。
3. 展示缺项、时限、严重级别、整改建议。
4. 打开来源卡片。
5. 导出质控报告。

验收：

- 缺来源规则不能发布。
- 质控结果能追溯规则和来源。
- 报告可用于科室整改。

### 15.3 医保演示

输入：医保质控样例。

流程：

1. 执行医保规则评估。
2. 展示限定支付、费用风险和申诉材料建议。
3. 查看政策来源。
4. 标记复核状态。

验收：

- 医保政策版本可见。
- 风险金额和规则依据可解释。
- 支持复核闭环。

### 15.4 信息科演示

输入：配置包。

流程：

1. 导入配置包。
2. 执行 review。
3. 查看 hash、diff、来源完整性、组织影响。
4. 发布。
5. 回滚。
6. 查看审计和 trace。

验收：

- 发布过程可审计。
- 失败有错误码和修复建议。
- 回滚后 ACTIVE 指针正确。

### 15.5 院领导演示

输入：多组织运行数据。

流程：

1. 查看全院/集团指标。
2. 按医院、院区、科室下钻。
3. 查看路径变异率、规则命中率、医保风险、质控缺陷。
4. 查看规则忽略率和来源影响分析。

验收：

- 能看到管理价值。
- 能定位问题科室和规则。
- 能追溯配置责任链。

## 17. 当前最高优先级执行顺序

如果用户没有指定下一步，后续 AI 按以下顺序领取任务：

1. `PROV-001` 来源文档、引用片段、资产绑定 DDL 与 API 草案。
2. `PKG-004` 配置包 Oracle/达梦正式表和持久化。
3. `ORG-003` 组织目录 Oracle 持久化。
4. `RULE-005` 规则医学来源字段、发布阻断和结果展示。
5. `RULE-008` 第三方规则评估结果 Oracle 持久化。
6. `TEST-001` 组织、来源、发布、Oracle 落库回归矩阵。
7. `FE-002` 前端工程脚手架。
8. `FE-003` 演示与规则校验工作台。
9. `FE-004` 配置包中心。
10. `OPS-001` 初始化检查和环境诊断。

排序理由：

- 来源追溯是客户明示需求，也是医学可信底座。
- 配置包 Oracle 持久化是发布治理底座。
- 组织持久化是集团化交付底座。
- 规则来源和评估结果持久化直接影响客户可验收性。
- 前端演示工作台决定客户是否能直观看到价值。

## 18. 最终判断

本项目下一阶段不是继续堆接口，而是形成“可信配置治理 + 可解释临床决策支持 + 可视化验收 + 可追溯运行证据”的平台能力。

最重要的交付标准：

- 医生相信：结果有依据、可确认、可覆盖。
- 质控相信：问题可复核、可整改、可追责。
- 医保相信：政策版本清楚、申诉材料有据。
- 信息科相信：可部署、可回滚、可审计、可运维。
- 院领导相信：能看到管理价值和风险变化。
- 开发团队相信：不同 AI 能并行，不互相踩代码。

只要后续 AI 严格按本文、任务卡和 DoD 执行，就可以把项目稳定推进到可客户验收、可生产试点的状态。
