# 九大产品能力详细设计与落地实施方案

> 本文是后续 AI 开发 9 项产品能力的实施说明书。先读 `00_总入口与AI接手导航.md` 和 `02_任务台账.md`，再按任务编号阅读本文对应章节。

## 0. 通用落地规则

所有章节都必须遵守以下规则：

- 任务状态以 `02_任务台账.md` 为准，开发前必须先创建并推送 active claim。
- 新增表、字段、索引、约束必须同步 Oracle、达梦、PostgreSQL/Kingbase、LOCAL_H2_FILE。
- 生产库表和关键字段必须有中文注释；LOCAL_H2_FILE 只做开发库等价验证。
- 客户可见功能必须创建 `ai-dev-input/13_feature_acceptance/` 验收记录。
- 医学、医保、质控、临床建议、AI 生成内容必须有来源证据和人工审核路径。
- 前端必须按 `07_前端开发规范.md` 和 `前端产品交互与视觉规范.md` 落地，用户可见文案默认标准中文。
- 所有实现必须支持 N 个医院、集团、院区、科室和外网通用服务，不允许硬编码单医院逻辑。

## 1. 用户体系、院内同步和单点登录

详细身份方案见 `用户体系与身份联邦详细设计方案.md`。本节给开发拆解口径。

### 1.1 产品目标

建立本系统独立用户体系，院内 HIS/EMR/OA/统一身份平台/LDAP/AD/CAS/OIDC/SAML 只是外部身份源。外网通用服务、院内部署、混合交付都必须依赖平台内部 `user_id` 做权限和审计。

### 1.2 核心流程

```text
平台租户开通
-> 创建客户管理员 / 平台运营账号 / 服务账号
-> 配置组织目录和身份源
-> 院内用户同步或本地邀请
-> 绑定外部身份
-> 登录会话
-> 加载角色、权限、组织范围、数据范围
-> 写审计责任链
```

### 1.3 数据模型

| 模型 | 说明 | 实现任务 |
|---|---|---|
| `SEC_TENANT` | 客户租户、外网租户、院内实例租户 | `SEC-001`、`SEC-011` |
| `SEC_USER` | 平台内部用户主表，唯一审计主体 | `SEC-001` |
| `SEC_LOCAL_CREDENTIAL` | 本地密码、MFA、密码策略 | `SEC-001`、`SEC-011` |
| `SEC_IDENTITY_PROVIDER` | 院内/外部身份源配置 | `SEC-006`、`SEC-007` |
| `SEC_IDENTITY_BINDING` | 外部 subject 与平台用户绑定 | `SEC-006`、`SEC-007`、`SEC-012` |
| `SEC_ROLE` / `SEC_PERMISSION` | 角色和权限点 | `SEC-001`、`SEC-008` |
| `SEC_USER_ORG_SCOPE` | 用户组织范围 | `SEC-001`、`SEC-002` |
| `SEC_SESSION` | 登录会话、失效、踢出 | `SEC-001`、`SEC-007` |
| `SEC_API_CLIENT` | 服务账号、API Key、签名 | `SEC-003`、`SEC-011` |

### 1.4 后端实现

| 服务/API | 说明 |
|---|---|
| `TenantService` | 租户开通、暂停、到期、扩容、状态校验 |
| `UserService` | 用户创建、停用、锁定、资料更新、组织范围维护 |
| `RolePermissionService` | 角色、权限点、菜单按钮、接口动作绑定 |
| `IdentityProviderService` | CAS/OIDC/SAML/LDAP/AD/HIS/OA 身份源配置 |
| `IdentityBindingService` | 外部账号绑定、合并、解绑、冲突处理 |
| `AuthSessionService` | 登录、登出、刷新、会话失效、MFA |
| `ApiClientService` | 服务账号、密钥 hash、签名、nonce、防重放 |
| `UserSyncService` | 院内全量/增量/手动同步、差异报告、停用处理 |

必须提供 API：

- `POST /api/security/login`
- `POST /api/security/logout`
- `GET /api/security/me`
- `GET/POST /api/security/users`
- `GET/POST /api/security/roles`
- `GET/POST /api/security/identity-providers`
- `POST /api/security/user-sync/jobs`
- `POST /api/security/sso/{providerCode}/callback`
- `GET/POST /api/security/api-clients`

### 1.5 前端实现

| 页面 | 关键能力 |
|---|---|
| 登录页 | 本地登录、SSO 入口、租户识别、错误中文化 |
| 用户管理 | 用户列表、状态、角色、组织范围、身份绑定 |
| 角色权限 | 菜单、按钮、接口动作、数据范围配置 |
| 身份源配置 | CAS/OIDC/SAML/LDAP/AD/院内同步源配置和测试连接 |
| 同步任务 | 全量/增量/手动同步、差异预览、失败重试 |
| 外网租户开通 | 客户租户、管理员邀请、服务账号开通 |

### 1.6 验收标准

- 无院内身份源时，外网租户可开通、管理员可登录、服务账号可调用 API。
- 院内 SSO 登录后，权限来自本系统角色和组织范围，不直接信任外部角色文本。
- 离职用户不能登录，但历史审核、发布、回滚审计仍可追溯。
- 同一用户可绑定多个身份源；同一外部 subject 重复绑定必须阻断。

## 2. 界面功能交互和完整方案

### 2.1 产品目标

把系统从“功能堆叠”变成按角色工作的配置平台。医生、质控、医保、信息科、实施、院领导、医学审核员、平台运营看到不同工作台和待办。

### 2.2 信息架构

| 一级入口 | 面向角色 | 功能范围 | 实现任务 |
|---|---|---|---|
| 工作台 | 全角色 | 待办、风险、同步状态、最近发布 | `FE-011` |
| 演示与校验 | 实施、客户验收 | 规则、路径、知识包 dry-run | `FE-003`、`TEST-004` |
| 配置中心 | 信息科、实施 | 配置包、版本、发布、回滚 | `FE-004`、`PKG-*` |
| 标准化中心 | 医学审核、信息科 | 字典、映射、未映射治理 | `TERM-*` |
| 来源追溯 | 医学审核、质控 | 来源库、证据、影响分析 | `PROV-*`、`FE-008` |
| AI 审核台 | 医学审核、质控 | AI 候选、diff、证据、驳回 | `FE-AI-001` |
| 质控运营 | 质控、医保、院领导 | 指标、评分、整改、报告 | `EVAL-*`、`FE-007` |
| 审计运维 | 信息科、运维 | 同步、日志、告警、审计 | `OPS-*` |

### 2.3 统一交互规范

- 所有列表必须有筛选、排序、空态、错误态、加载态、traceId。
- 所有审核动作必须有 diff、来源证据、风险提示、确认二次弹窗。
- 所有 AI 产物必须显示 `AI候选`、模型/工作流、提示词版本、来源、置信度、审核状态。
- 所有危险动作必须写审计：发布、回滚、导入、导出、删除、权限变更、身份绑定。
- 所有页面必须有无权限态，前端隐藏只做体验，后端仍需强校验。

### 2.5 验收标准

- 9 个核心角色至少各有一个可操作工作台入口。
- 页面无中英文混杂；按钮、空态、错误、状态为标准中文。
- 浏览器验证覆盖桌面和常见院内低分辨率终端。
- 关键客户路径可由 AI 验收机器人自动截图和回放。

## 3. AI 医疗知识生成、订阅、审核和知识包同步

### 3.1 产品目标

外网基准服务持续接入权威医疗知识，AI 生成可审核的标准概念、同义词、规则、图谱、路径候选；院内系统按需订阅、同步、审核、打包、导入、发布和回滚。

### 3.2 知识生产流程

```text
用户订阅知识主题
-> 来源注册
-> 定时/手动同步
-> 文档解析和切片
-> AI 抽取候选知识
-> 自动质检
-> 前台人工审核
-> 生成知识包
-> 院内导入 dry-run
-> 发布 / 回滚
```

### 3.3 数据模型

| 模型 | 说明 | 任务 |
|---|---|---|
| `AI_KNOWLEDGE_SUBSCRIPTION` | 用户订阅主题、科室、疾病、指南、医保、药品 | `AIK-001` |
| `AI_KNOWLEDGE_SOURCE` | 来源注册、授权、语言、地区、证据等级 | `AIK-001`、`PROV-001` |
| `AI_KNOWLEDGE_JOB` | 同步、抽取、审核、打包任务 | `AIK-002`、`AIK-003` |
| `AI_MODEL_CALL_LOG` | 模型、提示词、输入输出 hash、耗时、错误 | `AIK-002`、`AI-GOV-001` |
| `AI_CANDIDATE_ASSET` | 候选字典、规则、路径、图谱、知识片段 | `AIK-004` |
| `AI_REVIEW_DECISION` | 前台审核、驳回、修订、发布意见 | `FE-AI-001` |
| `MEDICAL_KNOWLEDGE_PACKAGE` | 知识包 manifest、签名、hash、版本 | `PKG-AI-001` |

### 3.4 后端实现

| 服务 | 职责 |
|---|---|
| `KnowledgeSubscriptionService` | 订阅主题、频率、范围、通知 |
| `KnowledgeSourceService` | 来源注册、授权、过期、语言、证据等级 |
| `KnowledgeSyncJobService` | 定时同步、手动同步、重试、任务状态 |
| `AiExtractionService` | 调用模型/工作流生成候选资产 |
| `AiCandidateQualityService` | 来源缺失、冲突、过期、低置信度扫描 |
| `AiReviewService` | 审核、驳回、修订、锁定、发布前检查 |
| `KnowledgePackageService` | manifest、hash、签名、导出、导入、dry-run |

API 必须覆盖：

- `POST /api/knowledge/subscriptions`
- `POST /api/knowledge/sources`
- `POST /api/knowledge/sync-jobs`
- `GET /api/knowledge/candidates`
- `POST /api/knowledge/candidates/{id}/review`
- `POST /api/knowledge/packages/export`
- `POST /api/knowledge/packages/import/dry-run`
- `POST /api/knowledge/packages/{id}/publish`

### 3.5 前端实现

| 页面 | 能力 |
|---|---|
| 知识订阅 | 按疾病、科室、指南、医保、药品、质控主题订阅 |
| 来源管理 | 来源授权、语言、地区、证据等级、过期提醒 |
| 同步任务 | 定时/手动同步、状态、失败原因、重试 |
| AI 审核台 | 候选资产、证据、diff、置信度、审核/驳回 |
| 知识包中心 | 导出、导入、dry-run、发布、回滚 |

### 3.6 验收标准

- 用户可创建订阅并触发手动同步。
- AI 候选没有来源或未审核时不得进入发布包。
- 知识包有 manifest、hash、签名、版本和回滚路径。
- 无 Dify/外部模型时，院内可使用外网基准包和本地规则完成核心流程。

## 4. AI 全功能验证和项目验收

### 4.1 产品目标

建立 AI 验收机器人，让后续每个客户可见功能都能被自动打开页面、执行业务动作、跑 API/DB/审计检查、截图并归档证据。

### 4.2 验收对象

| 对象 | 验收内容 | 任务 |
|---|---|---|
| 页面 | 路由、渲染、空态、错误态、权限态、截图 | `TEST-004` |
| API | 请求、响应、错误码、traceId、越权 | `TEST-004` |
| 数据库 | 生产 DDL、开发库验证、表注释、字段注释 | `TEST-002`、`TEST-007` |
| 审计 | 发布、回滚、审核、导入导出、权限变更 | `TEST-005` |
| 医学安全 | 来源、适用人群、排除条件、医生确认 | `CDSS-004` |
| 前端文案 | 中文统一、资源键、多语言缺失 | `TEST-006` |

### 4.3 证据模型

| 模型 | 说明 |
|---|---|
| `ACCEPTANCE_FEATURE` | 功能编号、名称、角色、页面、风险等级 |
| `ACCEPTANCE_RUN` | 验收批次、环境、commit、执行人或 AI |
| `ACCEPTANCE_EVIDENCE` | 截图、API 响应、日志、DB 查询、审计记录 |
| `ACCEPTANCE_FINDING` | 缺陷、等级、责任任务、修复状态 |

### 4.4 实现要求

- 前端 E2E 使用 Playwright；关键页面必须截图。
- 后端 smoke 脚本覆盖 API、数据库、审计和权限。
- 每个验收结果分为 `GOLD/SILVER/BRONZE/REJECTED`。
- P0/P1/P2 finding 必须进入 `needs_optimization/` 并回写任务台账。

### 4.5 验收标准

- 至少覆盖 9 大能力的主流程验收剧本。
- 验收失败时能定位到任务、页面、接口、数据库或文案。
- 无验收证据的客户可见功能不得标记 DONE。

## 5. 标准中文统一和中英文混杂治理

### 5.1 产品目标

用户可见文案统一为标准中文；技术标识、API 字段、枚举代码保留稳定英文。系统不得出现菜单、按钮、错误、状态中英文随意混杂。

### 5.2 词表分类

| 类别 | 规则 | 示例 |
|---|---|---|
| 菜单/按钮 | 标准中文 | 发布、回滚、审核、驳回 |
| 状态 | 标准中文 + 稳定枚举 | 待审核 `PENDING_REVIEW` |
| 错误消息 | 标准中文，附 traceId | 当前用户无权限访问该组织数据 |
| 技术字段 | 保留英文 | `tenant_id`、`traceId` |
| AI 标识 | 标准中文 | AI 候选、模型调用、提示词版本 |

### 5.3 实现要求

- 建立 `中文术语与文案词表`，覆盖菜单、状态、按钮、空态、错误、风险提示。
- 前端所有用户文案进入资源文件或统一常量。
- 后端异常返回 `message_key + zh-CN message + traceId`。
- 扫描脚本检查未登记英文、乱码、硬编码错误消息、缺资源键。

### 5.4 任务映射

| 任务 | 交付 |
|---|---|
| `I18N-001` | 中文术语和界面文案统一 |
| `DOC-007` | 标准中文术语和文案词表 |
| `TEST-006` | 中文文案和多语言扫描 |

### 5.5 验收标准

- 客户可见页面和错误消息默认中文。
- 扫描脚本无未登记英文文案。
- 技术英文有白名单，不被误替换。

## 6. 临床决策支持系统底座

### 6.1 产品目标

系统具备 CDSS 底座能力：在医嘱、病历、检查、入径、医保结算前等业务触发点给出有证据、有风险分级、可人工覆盖、可审计的临床建议。

### 6.2 触发点

| 触发点 | 输入 | 输出 | 任务 |
|---|---|---|---|
| 开立医嘱 | 诊断、药品、检验、患者事实 | 禁忌、重复、剂量、适应症建议 | `CDSS-002` |
| 病历书写 | 病历段落、诊断、路径节点 | 缺项、时限、风险提示 | `CDSS-002` |
| 入径/出径 | 病种、诊断、手术、住院日 | 路径推荐、变异提示 | `CDSS-002` |
| 医保结算前 | 费用、项目、诊断、医保规则 | 限定支付、违规风险 | `CDSS-002` |
| 发布前扫描 | 规则、知识、来源、适用人群 | 高风险阻断 | `CDSS-004` |

### 6.3 数据模型

| 模型 | 说明 |
|---|---|
| `CDSS_TRIGGER_EVENT` | 触发点、来源系统、患者上下文 hash |
| `CDSS_ADVICE` | 建议内容、风险等级、动作模式、来源证据 |
| `CDSS_DOCTOR_DECISION` | 医生采纳、忽略、覆盖、原因 |
| `CDSS_FATIGUE_METRIC` | 提醒频率、忽略率、误报率 |

### 6.4 后端实现

- `CdssTriggerService`：接收触发点事件。
- `CdssContextBuilder`：构造患者事实和组织上下文，禁止外部任意 SQL/Cypher。
- `CdssRuleEvaluator`：调用规则、路径、知识、图谱 Provider。
- `CdssAdviceService`：生成建议、风险等级、来源证据。
- `DoctorDecisionService`：记录医生确认、覆盖原因、审计。
- `AlertFatigueService`：统计提醒疲劳并支持阈值调整。

### 6.5 接口形态

- API：院内系统调用 CDSS 触发接口。
- iframe：院内系统嵌入建议面板。
- CDS Hooks：用于标准化决策支持触发。
- 消息队列：异步场景，如医保结算前批量扫描。

### 6.6 验收标准

- 高风险建议必须有来源、适用人群、排除条件和医生确认。
- 阻断类规则必须由院方授权，不允许 AI 自动阻断临床行为。
- 医生覆盖必须记录原因并进入审计。
- 缺来源、未审核、过期来源的 CDSS 建议不得发布。

## 7. 智能评估系统底座

### 7.1 产品目标

建立统一评估底座，对病历质量、医保风险、临床路径执行、规则/模型效果、科室配置质量进行评分、解释、复核和整改闭环。

### 7.2 评估流程

```text
定义评估对象
-> 选择指标集
-> 绑定来源和规则
-> 执行评分
-> 生成解释和风险等级
-> 人工复核
-> 整改任务
-> 再评估
-> 报告归档
```

### 7.3 数据模型

| 模型 | 说明 | 任务 |
|---|---|---|
| `EVAL_OBJECT` | 病历、科室、路径、规则、知识包等评估对象 | `EVAL-001` |
| `EVAL_INDICATOR_SET` | 指标集、版本、适用范围 | `EVAL-001` |
| `EVAL_INDICATOR` | 指标、权重、阈值、来源 | `EVAL-001` |
| `EVAL_RUN` | 执行批次、环境、数据范围 | `EVAL-002` |
| `EVAL_RESULT` | 分数、风险等级、解释、证据 | `EVAL-002` |
| `EVAL_REVIEW` | 人工复核、确认、驳回 | `EVAL-003` |
| `EVAL_REMEDIATION` | 整改任务、负责人、截止时间 | `EVAL-003` |

### 7.4 后端实现

- `EvaluationIndicatorService`：指标集、指标、权重、阈值管理。
- `EvaluationRunService`：按对象和范围执行评分。
- `EvaluationExplainService`：解释分数来源、缺失事实、风险原因。
- `EvaluationReviewService`：复核、整改、再评估。
- `EvaluationReportService`：导出报告和归档。

### 7.5 前端实现

| 页面 | 能力 |
|---|---|
| 指标模型 | 指标集、权重、阈值、来源绑定 |
| 评分运行 | 选择对象、范围、执行、查看进度 |
| 评估报告 | 分数、风险、解释、证据、导出 |
| 整改闭环 | 负责人、截止时间、状态、再评估 |

### 7.6 验收标准

- 评分结果可解释，不能只显示分数。
- 指标必须绑定来源或规则版本。
- 人工复核和整改闭环可追溯。
- 报告导出内容为中文，并保留 traceId 和证据摘要。

## 8. 多语言系统要求

### 8.1 产品目标

默认 `zh-CN`，架构上支持未来多语言。多语言不仅是前端文案，还包括后端消息、配置包、知识来源、报告模板、审计事件和导出内容。

### 8.2 资源范围

| 范围 | 说明 | 任务 |
|---|---|---|
| 前端资源 | 菜单、按钮、状态、空态、错误 | `I18N-002` |
| 后端消息 | 异常、校验、审计、同步结果 | `I18N-002` |
| 知识来源 | 原始语言、中文译名、翻译审核 | `I18N-003` |
| 知识包 | package locale、翻译状态、地区 | `I18N-003` |
| 报表导出 | 报告模板、日期、数字、单位 | `I18N-004` |

### 8.3 数据模型

| 模型 | 说明 |
|---|---|
| `I18N_MESSAGE` | 后端消息资源 |
| `I18N_FRONTEND_RESOURCE` | 前端资源构建来源或导入表 |
| `KNOWLEDGE_TRANSLATION` | 知识来源翻译和审核 |
| `REPORT_TEMPLATE_LOCALE` | 多语言报表模板 |

### 8.4 实现要求

- API 支持 `Accept-Language` 和默认租户语言。
- 后端错误返回 `message_key`，前端按 locale 渲染。
- 知识包 manifest 包含 `locale`、`source_language`、`translation_review_status`。
- 默认中文必须完整，其他语言可分阶段扩展。

### 8.5 验收标准

- 切换 locale 后，菜单、按钮、状态、错误、报表标题可切换。
- 未翻译资源有 fallback，不出现空白或技术 key。
- 医疗知识翻译必须有审核状态，不能未经审核直接发布。

## 9. 产品文档同步、去重和精简

### 9.1 产品目标

后续 AI 读最少文档获取最全面信息：入口清晰、事实源唯一、历史文档降级、任务状态不冲突。

### 9.2 文档分层

| 层级 | 文档 | 规则 |
|---|---|---|
| 唯一入口 | `00_总入口与AI接手导航.md` | AI 首读 |
| 状态事实源 | `02_任务台账.md` | 任务状态唯一权威 |
| 实施详设 | 本文和专项详细方案 | 开发按章节读 |
| 专项规范 | 架构、后端、前端、数据库、部署 | 触发时补读 |
| 历史参考 | 长文档和早期计划 | 不作为默认必读 |

### 9.3 清理动作

- 将重复任务、旧优先级、过期入口标记为历史参考。
- 在 `00_总入口` 中保留“触发条件 -> 补读 1 份文档”。
- 每个新任务必须有唯一编号、依赖、独占文件、核心交付。
- 每次提交后同步 claim、review、feature acceptance 和任务台账。

### 9.4 验收标准

- 新 AI 10 分钟内能确定该读哪几份文档、领哪个任务、改哪些文件。
- 任务状态冲突时只看 `02_任务台账.md`。
- 历史文档不再误导后续开发优先级。
