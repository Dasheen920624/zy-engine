# 系统补充能力纳入范围

> 本文用于记录“还应纳入考虑范围”的产品成熟度能力。它不替代任务台账，所有实施仍以 `02_任务台账.md` 为准。

## 1. 本次补充结论

现有详设已经覆盖新增 9 项、原有产品能力和已实现能力统一改造，但从医院级产品、外网通用服务、N 医院交付、长期运营角度，还需要把以下能力纳入范围：

| 能力 | 是否立即实现 | 纳入原因 |
|---|---|---|
| 统一待办和审批工作流 | P0 | 审核、发布、整改、知识包、合规、实施都需要统一流程，不应各模块自建待办 |
| 临床安全风险管理 | P0 | CDSS 和智能评估必须有 hazard log、安全案例和上线风险接受机制 |
| 患者主索引和就诊标识治理 | P0 | 医嘱、路径、病历、医保、评估都依赖患者和就诊标识一致 |
| 医疗标准码体系目录 | P1 | ICD、手术、药品、检验、医保、院内码版本必须可治理 |
| 通知和消息中心 | P1 | 审核、同步失败、来源过期、告警、试运行问题需要统一触达 |
| 开放 API 门户和开发者中心 | P1 | N 医院接入、第三方系统调用、外网通用服务需要标准 OpenAPI、SDK、限流和示例 |
| 数据迁移和初始化工具 | P1 | 既有医院上线、升级和多库部署需要可重复迁移、回滚和校验 |
| 报表模板和导出中心 | P1 | 客户验收、质控、医保、评估、合规都需要统一报表口径 |
| 客户环境资产和配置漂移管理 | P1 | 多医院、多版本、多部署环境需要知道客户现场真实状态 |
| 安全事件响应和应急预案 | P1 | 等保、医院安全和外网服务需要告警、处置、复盘、证据链 |
| 客户服务台和 SLA | P2 | 商业化交付后需要工单、服务级别、问题闭环和客户成功 |
| 全局资产搜索和知识检索 | P2 | 配置、规则、来源、知识包、日志、审计多了以后需要统一检索 |
| 帮助中心和内嵌操作指引 | P2 | 降低实施和医院人员培训成本 |
| 用量计量、订阅和商务合同 | P2 | 外网通用服务、SaaS、版本套餐和授权需要可计量、可对账 |

## 2. P0 纳入范围

### 2.1 统一待办和审批工作流

目标：把审核、发布、回滚、整改、知识包导入、合规检查、用户同步异常、来源过期、AI 候选复核等动作统一进入任务/待办中心。

核心模型：

- `WF_PROCESS_DEF`：流程定义。
- `WF_TASK`：待办任务。
- `WF_APPROVAL`：审批动作。
- `WF_TASK_LINK`：关联业务对象。
- `WF_DELEGATION`：委托和转办。

必须支持：

- 角色、组织、权限决定待办可见范围。
- 支持通过、驳回、退回修改、转办、加签、超时提醒。
- 所有审批动作写审计。
- 前端提供统一待办入口，并允许业务页面嵌入待办状态。

任务：`WF-001`。

### 2.2 临床安全风险管理

目标：对 CDSS、智能评估、医嘱安全、AI 候选和规则发布建立临床安全风险管理机制。

核心模型：

- `CLINICAL_HAZARD`：危害项。
- `CLINICAL_RISK_CONTROL`：风险控制措施。
- `CLINICAL_SAFETY_CASE`：安全案例。
- `CLINICAL_RISK_ACCEPTANCE`：上线风险接受记录。

必须支持：

- 每个高风险临床建议绑定危害、控制措施、来源和医生确认策略。
- 阻断类能力必须有院方授权和风险接受记录。
- 安全红线扫描发现问题时阻断发布。
- 验收报告输出临床安全结论。

任务：`RISK-001`。

### 2.3 患者主索引和就诊标识治理

目标：在不替代医院主索引的前提下，建立系统内患者、就诊、住院、门诊、医保结算标识映射，避免多系统数据错配。

核心模型：

- `MPI_PATIENT_REF`：患者引用和脱敏标识。
- `MPI_ENCOUNTER_REF`：就诊引用。
- `MPI_IDENTITY_LINK`：HIS/EMR/LIS/PACS/医保标识映射。
- `MPI_MATCH_RESULT`：匹配结果和冲突。

必须支持：

- 本系统不保存不必要患者明文隐私。
- 只保存业务必需的脱敏标识、hash、后四位或院内引用。
- 同一就诊在不同系统标识可归并。
- 冲突进入人工处理，不能静默合并。

任务：`MPI-001`。

## 3. P1 纳入范围

| 任务 | 目标 | 关键交付 |
|---|---|---|
| `STD-001` 医疗标准码体系目录和版本治理 | 统一 ICD、手术、药品、检验、医保、院内码版本 | 标准目录、版本、适用地区、更新记录、影响分析 |
| `NOTIFY-001` 通知和消息中心 | 统一审核、同步、告警、来源过期、任务超时触达 | 站内信、邮件/短信/企微适配、已读、订阅、模板 |
| `API-001` 开放 API 门户和开发者中心 | 支撑院内系统和第三方开发者接入 | OpenAPI、示例、SDK、沙箱、限流、签名说明 |
| `MIG-001` 数据迁移和初始化工具 | 支撑客户上线、升级、回滚、多数据库迁移 | 迁移计划、dry-run、差异报告、回滚脚本、校验报告 |
| `RPT-001` 报表模板和导出中心 | 统一质控、医保、评估、合规、审计报表 | 模板、参数、导出、签名、归档、多语言 |
| `ENV-001` 客户环境资产和配置漂移管理 | 管理 N 医院部署、版本、依赖、配置差异 | 环境资产、版本指纹、配置漂移、巡检报告 |
| `IR-001` 安全事件响应和应急预案 | 支撑安全告警、事件处置、复盘和监管证据 | 事件分级、处置流程、证据、复盘、改进任务 |

## 4. P2 纳入范围

| 任务 | 目标 | 关键交付 |
|---|---|---|
| `SUPPORT-001` 客户服务台和 SLA | 支撑商业化后的工单、服务级别和客户成功 | 工单、SLA、问题分类、处理时限、客户反馈 |
| `SEARCH-001` 全局资产搜索和知识检索 | 统一检索规则、路径、来源、知识包、审计、日志 | 全局搜索、权限过滤、结果分类、跳转 |
| `HELP-001` 帮助中心和内嵌操作指引 | 降低培训成本，提升医院用户自助能力 | 操作指引、FAQ、页面级帮助、版本更新说明 |
| `COMM-001` 用量计量、订阅和商务合同 | 支撑外网通用服务、SaaS 和商业对账 | 用量指标、套餐、合同、账单、对账导出 |

## 5. 与现有方案的关系

- `WF-001` 是所有审核、发布、整改和合规闭环的统一流程底座。
- `RISK-001` 是 `CDSS-*`、`EVAL-*`、`AI-GOV-*` 的临床安全补强。
- `MPI-001` 是 `CDSS-*`、`PATH-*`、`EVAL-*`、`INTEROP-*` 的患者和就诊标识基础。
- `STD-001` 是 `TERM-*`、`DATA-*`、`AIK-*` 的标准码目录补强。
- `API-001` 与 `INTEROP-*`、`SEC-003`、`ADAPT-*` 共同支撑对外接入。
- `MIG-001`、`ENV-001` 与 `REL-001`、`OPS-*` 共同支撑交付和升级。
- `RPT-001` 与 `EVAL-*`、`COMP-*`、`AUDIT-*` 共同支撑验收和报告。

## 6. 执行建议

短期优先把 `WF-001`、`RISK-001`、`MPI-001` 纳入 P0。它们会影响后续大量功能是否能形成统一闭环。

中期按客户交付节奏推进 `STD-001`、`API-001`、`MIG-001`、`RPT-001`、`ENV-001`、`IR-001`。

长期商业化阶段推进 `SUPPORT-001`、`SEARCH-001`、`HELP-001`、`COMM-001`。
