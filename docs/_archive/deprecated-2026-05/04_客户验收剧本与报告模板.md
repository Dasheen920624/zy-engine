# 04 客户验收剧本与报告模板

## 0. 用途

本文是医院侧 UAT、信息科联调、医学审核、医保审核、院领导走查时使用的 **标准剧本** 和 **可签字版报告模板**。

任何 AI 在开发 FE-003 演示工作台、RULE-006 规则包样例、QC 看板时，必须使产出与本文剧本对齐。

剧本覆盖：

1. AMI/STEMI 路径推荐与入径（临床医生）
2. 病历质控（质控办 + 临床医生）
3. 医保质控（医保办）
4. 医嘱安全（临床医生 + 药师）
5. 配置发布（信息科）
6. 院级驾驶舱（院领导）

每个剧本都给出：**前置 → 输入 → 操作步骤 → 期望结果（屏幕字段级）→ 验收口径 → 签字栏**。

## 1. 通用前置条件

| 项 | 要求 |
|---|---|
| 后端 | `zy-engine-mvp` 启动（内存或 Oracle 模式均可），`/api/health` 返回 200 |
| 前端 | （FE-002 完成后）`frontend` 启动，能访问演示工作台 |
| 演示组织 | `tenant_id=TENANT_DEMO`、`hospital_code=HOSPITAL_DEMO`、`department_code=DEPT_CARDIOLOGY` 已导入 |
| 演示数据 | `ai-dev-input/06_samples/` 下样例已导入 |
| 演示账号 | 至少 4 个：医生、质控员、医保员、信息科管理员 |

> 在 FE 前端未完成前，所有剧本可用 `api-examples.http` 文件中对应请求 + curl/Postman 完成；交付报告时需注明"接口验证"或"UI 验证"。

## 2. 剧本一：AMI/STEMI 路径推荐与入径

**目标角色**：临床医生（急诊心内科）
**业务目标**：3 分钟内识别 STEMI 患者并推荐再灌注路径，医生 1 键确认入径
**演示数据**：[sample_patient_context_ami.json](../../ai-dev-input/06_samples/sample_patient_context_ami.json)

### 2.1 输入

```json
{
  "patient": { "patient_id": "P_AMI_001", "gender": "M", "age": 62 },
  "encounter": { "encounter_id": "E_AMI_001", "visit_type": "EMERGENCY", "department_code": "ER" },
  "facts": {
    "chief_complaints": [{ "text": "胸痛2小时，伴大汗", "time": "2026-05-15T10:04:00+08:00" }],
    "exams": [{ "code": "ECG_12_LEAD", "finding_codes": ["ST_ELEVATION_CONTIGUOUS_LEADS"] }]
  }
}
```

### 2.2 操作步骤

| # | 操作 | 期望屏幕字段 |
|---|---|---|
| 1 | 在"演示与校验 → 患者上下文构建器"加载 AMI 样例 | 表单回填胸痛、心电图字段 |
| 2 | 点击"识别候选路径" | 返回 `AMI_STEMI`，score `90.45`，confidence `HIGH` |
| 3 | 查看命中规则卡片 | `R_AMI_STEMI_CANDIDATE` 命中，证据 `ST_ELEVATION_CONTIGUOUS_LEADS`，来源摘要 + 推荐强度 + 审核人 |
| 4 | 点击"确认入径" | 创建实例 `INSTANCE-xxx`，当前节点 `AMI_CHEST_PAIN_IDENTIFY` |
| 5 | 完成 ECG 任务 | 任务状态 COMPLETED，自动拉取 ECG_ADAPTER 数据 |
| 6 | 完成节点 | 流转到 `AMI_REPERFUSION_EVAL` |
| 7 | 记录变异（可选） | 变异记录写入 + 审计 |

### 2.3 验收口径

- [ ] 候选返回 `AMI_STEMI`，score ≥ 85，confidence `HIGH`。
- [ ] 命中规则卡片显示：命中条件、患者事实、规则版本、来源摘要、traceId。
- [ ] 入径后实例状态 `ACTIVE`，当前节点 `AMI_CHEST_PAIN_IDENTIFY`。
- [ ] 完成首节点后当前节点 `AMI_REPERFUSION_EVAL`。
- [ ] 整个过程产生 ≥ 4 条审计日志（候选/入径/任务完成/节点完成）。
- [ ] 传入 `Header X-Hospital-Code=HOSPITAL_DEMO` 后实例 hospital_code 字段被正确写入。

## 3. 剧本二：病历质控

**目标角色**：质控办 + 临床医生
**业务目标**：识别病历缺项、时限超时，给出整改建议
**演示数据**：[sample_emr_qc_case.json](../../ai-dev-input/06_samples/sample_emr_qc_case.json)

### 3.1 输入要点

患者 `P_EMR_001` 入院 24 小时，病历存在：

- 主诉缺失。
- 入院记录未在 24 小时内完成。
- 现病史时长 < 50 字。
- 既往史空白。

### 3.2 操作步骤

| # | 操作 | 期望屏幕字段 |
|---|---|---|
| 1 | "演示与校验 → 病历质控演示"，选择场景 `EMR_QC_MISSING_FIELDS` | 表单显示病历摘要 |
| 2 | 选择规则包 `PKG_EMR_QC` 版本 `2026.05` | 显示规则数量、医学来源完整性 |
| 3 | 点击"运行 dry-run" | 调 `POST /api/rule-engine/evaluate` (scenario_code=`EMR_QC`) |
| 4 | 查看命中清单 | 命中 4 条：缺主诉、入院记录超时、现病史过短、既往史空白 |
| 5 | 点击任一命中规则 | 打开来源卡片：医学/院内制度依据 + 章节 + 审核人 |
| 6 | 导出质控报告 | 生成 PDF/HTML，包含命中列表、整改建议、规则版本、traceId |

### 3.3 验收口径

- [ ] 命中规则数 = 4，严重级别分布合理（≥ 1 高，≥ 1 中）。
- [ ] 每条命中规则都有来源标题 + 章节 + 摘要 + 审核人。
- [ ] 导出报告中显示规则包版本 `2026.05`、字典版本、组织上下文。
- [ ] 缺来源的规则在发布阶段被阻断（验证用例 EMR-QC-NEG-001）。
- [ ] dry-run 不写正式患者状态。

## 4. 剧本三：医保质控

**目标角色**：医保办
**业务目标**：识别费用规则风险、限定支付违规、给出申诉材料建议
**演示数据**：[sample_insurance_qc_case.json](../../ai-dev-input/06_samples/sample_insurance_qc_case.json)

### 4.1 输入要点

患者 `P_INS_001` 急性冠脉综合征住院，费用清单包含：

- 限定二线用药未走特批流程。
- 单次检查频次超过医保规定。
- 高价耗材使用与诊断不匹配。

### 4.2 操作步骤

| # | 操作 | 期望屏幕字段 |
|---|---|---|
| 1 | "演示与校验 → 医保质控演示"，选择 `INSURANCE_QC` 场景 | 表单显示费用清单 |
| 2 | 选择规则包 `PKG_INSURANCE_QC` 版本 `2026.05` | 政策来源标题 + 版本 |
| 3 | 运行 dry-run | 调 `POST /api/rule-engine/evaluate` (scenario_code=`INSURANCE_QC`) |
| 4 | 查看风险清单 | 3 条风险，标注金额影响、政策依据条款 |
| 5 | 点击"生成申诉材料" | 输出申诉摘要 + 政策原文 + 患者事实快照 |
| 6 | 导出医保风险报告 | PDF 包含风险等级、金额、规则版本、政策来源、申诉建议 |

### 4.3 验收口径

- [ ] 风险数 = 3，每条都有"金额影响"字段（数值 + 单位）。
- [ ] 政策来源条款可一键展开（标题/机构/版本/章节/摘要）。
- [ ] 申诉材料可下载 PDF。
- [ ] 风险评估写入审计 + 内存评估结果，可通过 `GET /api/rule-engine/results/{resultId}` 复查。
- [ ] 同样输入在两次运行间结果稳定（无随机性）。

## 5. 剧本四：医嘱安全

**目标角色**：临床医生 + 药师
**业务目标**：识别用药禁忌、相互作用、适应症不符、检查不合理
**演示数据**：[sample_order_safety_case.json](../../ai-dev-input/06_samples/sample_order_safety_case.json)

### 5.1 输入要点

患者 `P_ORD_001` 急诊冠脉综合征，医嘱包含：

- 阿司匹林 + 氯吡格雷（双抗联用，合理）。
- 氯吡格雷 + 奥美拉唑（药物相互作用提醒）。
- 万古霉素无肌酐清除率调整（强提醒）。
- 患者既往哮喘，开了非选择性 β 阻滞剂（阻断）。
- 影像 CT 重复开单 3 次（重复检查警告）。

### 5.2 操作步骤

| # | 操作 | 期望屏幕字段 |
|---|---|---|
| 1 | "演示与校验 → 医嘱安全演示"，选择 `ORDER_SAFETY` 场景 | 表单显示医嘱清单 |
| 2 | 运行 dry-run | 调 `POST /api/rule-engine/evaluate` (scenario_code=`ORDER_SAFETY`) |
| 3 | 查看 4 条命中 | 1 条 REMIND（PPI 提醒）、1 条 STRONG_REMIND（万古剂量）、1 条 BLOCK（哮喘β阻滞剂）、1 条 STRONG_REMIND（重复 CT） |
| 4 | 尝试在 BLOCK 项点击"忽略" | 必须输入医生姓名 + 上级授权 + 覆盖原因，记录 `override_reason` |
| 5 | 查看每条命中规则来源 | 药品说明书 / 指南 / 院内制度 来源卡片 |
| 6 | 导出医嘱安全报告 | 包含 4 类提醒级别、覆盖记录、规则版本 |

### 5.3 验收口径

- [ ] 命中规则分级正确：REMIND/STRONG_REMIND/BLOCK 至少各 1 条。
- [ ] BLOCK 类规则必须显式弹出"上级授权"流程，不能默认通过。
- [ ] 覆盖原因写入审计 + `doctor_decision` 字段。
- [ ] 标准化层：奥美拉唑 / 万古霉素 / 美托洛尔 等院内名称已归一到标准码 (`PPI`、`VANCOMYCIN`、`BETA_BLOCKER_NONSELECTIVE`)。
- [ ] 重复 CT 命中"3 次以上"逻辑，时间窗 24 小时。

## 6. 剧本五：配置发布

**目标角色**：信息科 + 医学/医保审核
**业务目标**：导入配置包 → review → publish → 回滚的完整闭环
**演示数据**：[sample_config_package.json](../../ai-dev-input/06_samples/sample_config_package.json)

### 6.1 操作步骤

| # | 操作 | 期望屏幕字段 |
|---|---|---|
| 1 | "配置包中心 → 导入"，上传 `sample_config_package.json` | 列表显示 `PKG_AMI_RULE_CONFIG / 2026.05.01 / DRAFT` |
| 2 | 点击"Review" | 显示 manifest、hash、组织范围、问题列表、来源完整性、ready_to_publish |
| 3 | 故意把样例 hash 改 1 字符再导入 | review 返回 `HASH_MISMATCH`，发布按钮禁用 |
| 4 | 恢复正确 hash，点击"Publish" | 必须填 `approved_by`，提交后状态变 `PUBLISHED`，审计写入 |
| 5 | 点击"Export" | 下载完整 snapshot JSON，hash 与发布一致 |
| 6 | 尝试在同一版本号下重复导入不同内容 | 返回错误，不静默覆盖 |
| 7 | 切到"已发布"列表，找到该包，点击"回滚" | 上一版本变回 ACTIVE，本版本 RETIRED，审计写入 |

### 6.2 验收口径

- [ ] hash 不一致时不可发布。
- [ ] 同 `package_code + package_version` 内容不同不能覆盖。
- [ ] 发布、回滚均写审计（`engineType=CONFIG, actionType=PUBLISH/ROLLBACK`）。
- [ ] 缺来源/未审核的配置 review 返回阻断（PROV-004 落地后）。
- [ ] Export 内容可重新 Import 到同库或异库。

## 7. 剧本六：院级驾驶舱

**目标角色**：院领导
**业务目标**：5 分钟回答院级运营 7 个核心问题
**演示数据**：FE-007 + QC-001 落地后的看板

### 7.1 7 个核心问题

| # | 问题 | 期望看板 |
|---|---|---|
| 1 | 哪些规则今天命中了多少次？ | 规则命中率排行榜（top 10）|
| 2 | 哪些科室路径偏离最多？ | 科室变异率热力图 |
| 3 | 哪些医保风险最高、金额影响多大？ | 医保风险列表（按金额降序）|
| 4 | 哪些来源文献更新会影响现有规则？ | 来源影响分析页（PROV-006）|
| 5 | 哪些规则被医生频繁忽略？ | 规则忽略率排行（按 override_reason 聚合）|
| 6 | 哪些适配器不稳定？ | 适配器调用错误率（ADAPT-004）|
| 7 | 哪个版本什么时候由谁发布，出问题如何回滚？ | 发布审计台账 + 回滚目标版本 |

### 7.2 操作步骤

| # | 操作 | 期望屏幕字段 |
|---|---|---|
| 1 | "院级驾驶舱 → 总览" | 集团/医院/院区/科室 4 级 KPI |
| 2 | 切换时间范围近 7 天 | 数字动态刷新 |
| 3 | 下钻到"心内科" | 显示心内科 AMI 路径入径率、节点完成率、节点滞留 |
| 4 | 下钻到具体医生 | 显示医生的规则忽略率、变异记录 |
| 5 | 切回"医保风险" | 显示金额 top 5 风险 |
| 6 | 点击其中一条，下钻到"政策来源" | 来源卡片完整展示 |
| 7 | 切到"发布台账" | 显示近 30 天发布记录 + 回滚按钮 |

### 7.3 验收口径

- [ ] 7 个问题在 5 分钟内全部可回答（看板 + 下钻）。
- [ ] 任意指标可下钻至少 3 层（集团 → 医院 → 院区 → 科室 → 医生）。
- [ ] 每个指标有口径说明（鼠标悬停或问号）。
- [ ] 发布台账记录责任链：起草人 / 医学审核 / 医保审核 / 信息科审核 / 总审批 / 上线时间 / 回滚目标。

## 9. 剧本维护

- 新增业务场景（如儿科 / 中医 / 慢病）→ 在本文新增"剧本 N"段。
- 新增样例文件 → 在剧本"演示数据"段写明路径。
- 修改剧本验收口径 → 提交 DOC-xxx 任务并在最终回复说明影响。
- 客户实际验收过程中发现的差异 → 反馈到对应 RULE/PATH/FE 任务。
