# 前端产品交互与视觉规范

## 1. 文档定位

本文是医疗智能引擎平台前端产品交互、视觉语言和高保真原型优化的深度参考。后续任何 AI 或开发者设计页面、修改原型、实现 React 前端前，默认先阅读 `00_总入口与AI接手导航.md` 和 `07_前端开发规范.md`；只有做复杂页面、原型、视觉体系或客户演示体验时，再查本文。

```text
zy-engine-mvp/docs/00_总入口与AI接手导航.md
zy-engine-mvp/docs/07_前端开发规范.md
```

本文解决三个问题：

- 让不同 AI 不再各自设计一套页面结构、按钮口径、状态标签和发布流程。
- 让界面完美融入客户现有业务系统的“天蓝色菜单栏 + 白色背景”风格。
- 让高保真原型、React 工程、验收脚本和客户演示使用同一套交互规范。

核心原则：

```text
客户系统融合优先于视觉炫技。
工作台效率优先于营销表达。
白色工作区承载业务，天蓝菜单栏承载系统身份。
所有关键结论必须可解释、可追溯、可审核。
AI 生成内容必须标识，不能替代人工确认。
```

## 2. 产品体验定位

这是医院内网生产工具，不是独立品牌官网。界面要像客户业务系统里的一个自然模块，而不是突然跳进另一个产品。

体验关键词：

```text
清爽
可信
高密度
可扫描
可追溯
少打扰
像院内系统
```

不允许：

- 大面积深色侧栏。
- 大面积渐变背景。
- 营销式 hero 首屏。
- 装饰性插画、浮动卡片、低信息密度留白。
- 只有颜色没有文字/图标的状态表达。
- 用 AI 对话框替代结构化配置流程。

## 3. 视觉系统

### 3.1 颜色基线

客户业务系统为天蓝菜单栏和白色背景。本系统必须跟随该气质。

```css
:root {
  --zy-menu-bg: #2fa7e8;
  --zy-menu-bg-hover: #178fd0;
  --zy-menu-bg-active: #0b78bd;
  --zy-menu-text: #ffffff;
  --zy-page-bg: #ffffff;
  --zy-page-bg-soft: #f5fbff;
  --zy-panel-bg: #ffffff;
  --zy-border: #d8e8f3;
  --zy-border-strong: #b8d8ec;
  --zy-text-primary: #1f2d3d;
  --zy-text-secondary: #52677a;
  --zy-text-tertiary: #7c8b99;
  --zy-primary: #168bd3;
  --zy-primary-hover: #0f73b6;
  --zy-primary-soft: #e8f6ff;
  --zy-success: #12966e;
  --zy-warning: #c78200;
  --zy-danger: #d33f49;
  --zy-info: #168bd3;
}
```

使用规则：

- 主导航必须使用 `--zy-menu-bg`，文字白色。
- 页面背景默认白色；只允许用极浅蓝 `--zy-page-bg-soft` 区分工具栏、筛选区和表头。
- 主按钮使用 `--zy-primary`，危险操作使用红色，成功/警告/失败必须有文字和图标。
- 不使用深蓝黑侧栏，不使用紫色主视觉，不使用大面积灰底。

### 3.2 布局基线

默认采用“客户系统融合壳”：

```text
顶部天蓝菜单栏 52px
左侧浅色二级导航 216px，可折叠
右侧白色业务工作区
页面标题区
筛选/上下文条
主内容表格/画布/表单
右侧抽屉或底部结果区
```

如果客户系统已经提供顶部菜单栏，本系统嵌入时只保留：

```text
左侧二级导航 + 白色内容区
```

高保真原型必须同时说明两种模式：

- `standalone`：自带天蓝顶部菜单栏。
- `embedded`：嵌入客户系统，只显示二级导航和内容。

### 3.3 字体和密度

字体：

```css
font-family: "Microsoft YaHei", "PingFang SC", "Source Han Sans CN", "Noto Sans CJK SC", sans-serif;
```

字号：

```text
页面标题：20px
模块标题：16px
正文/表格：14px
辅助信息：12px
数字指标：24px，使用等宽数字
```

密度：

- 表格行高 40px 至 48px。
- 工具栏高度 48px 至 56px。
- 卡片圆角不超过 6px。
- 页面左右留白 16px 或 24px，不做大留白。

## 4. 信息架构

主导航必须按工作任务组织，不按技术模块堆叠。

推荐一级菜单：

```text
工作台
演示与校验
配置中心
标准化中心
来源追溯
质控运营
发布与同步
审计运维
系统设置
```

二级菜单：

```text
配置中心
  配置包
  路径配置
  规则配置
  图谱配置
  Dify 契约
  适配器绑定

标准化中心
  字典导入
  映射审核
  未映射治理
  AI 候选配置
  映射包发布

质控运营
  路径质控
  病历质控
  医保质控
  医嘱安全
  报表导出

审计运维
  Provider 状态
  同步任务
  执行日志
  审计日志
  降级记录
```

原则：

- 页面入口按用户任务命名，避免直接暴露技术名。
- `Dify`、`Neo4j` 可以出现在二级配置或 Provider 状态里，但不能成为业务首屏主线。
- 规则引擎核心能力在无 Dify/无 Neo4j 时也要完整展示。

## 5. 页面模板

### 5.1 工作台

工作台是登录后的第一屏，不做欢迎页。

必须包含：

- 当前组织上下文。
- 运行模式：`DB_ONLY / HYBRID / FULL_INTEGRATION / IN_MEMORY_DEMO`。
- Provider 状态。
- 待审核配置包。
- 未映射字典。
- 来源缺失/过期风险。
- 规则执行异常。
- 发布与同步失败任务。
- 今日核心指标。

工作台不允许：

- 大图背景。
- 宣传口号。
- 长篇说明文字。
- 只有图表没有待办入口。

### 5.2 列表页

结构：

```text
页面标题
组织上下文条
筛选工具栏
批量操作栏
数据表格
分页
右侧详情抽屉
```

列表页必须有：

- 加载态。
- 空态。
- 错误态。
- 服务端校验错误。
- traceId。
- 导出/刷新/重试入口。

### 5.3 配置编辑页

结构：

```text
基础信息
组织范围
来源证据
结构化编辑器
JSON/DSL 高级编辑
校验结果
diff
dry-run
提交审核
```

保存草稿不等于发布。发布必须走：

```text
编辑 -> 校验 -> diff -> dry-run -> 审核 -> 发布 -> 同步 -> 激活
```

### 5.4 演示与校验工作台

结构：

```text
场景库
患者上下文
输入数据编辑
运行 dry-run
结果区
证据/来源
标准化差异
traceId
报告导出
```

结果区必须可切换：

```text
命中规则
未命中规则
缺失事实
标准化差异
来源证据
执行日志
```

### 5.5 审核页

适用于配置包、字典映射、AI 候选规则、来源证据。

结构：

```text
左侧：待审核列表
中间：候选内容/差异/测试结果
右侧：来源证据/审核记录/影响范围
底部：通过、修改后通过、驳回、要求补证据
```

审核动作必须要求填写意见：

- 驳回必须填原因。
- 修改后通过必须记录修改摘要。
- 高风险规则必须显示二次审核状态。

## 6. 组件规范

### 6.1 顶部菜单栏

规范：

- 高度 52px。
- 背景天蓝。
- 左侧显示系统名或客户系统模块名。
- 中间显示一级菜单。
- 右侧显示组织上下文、用户、消息、帮助。
- 当前菜单使用更深蓝底或白色下划线。

### 6.2 左侧二级导航

规范：

- 宽度 216px。
- 背景白色或极浅蓝。
- 当前项使用左侧 3px 天蓝色指示条。
- 分组标题使用 12px 灰蓝文字。
- 支持折叠，但折叠后图标必须有 tooltip。

### 6.3 状态标签

状态必须三重编码：

```text
颜色 + 图标/圆点 + 文案
```

配置状态：

```text
DRAFT       灰色 草稿
REVIEWED    蓝色 已审核
PUBLISHED   青色 已发布
SYNCED      绿色 已同步
ACTIVE      绿色 生效中
RETIRED     灰色 已退役
BLOCKED     红色 已阻断
```

严重级别：

```text
INFO       蓝色
LOW        灰蓝
MEDIUM     黄色
HIGH       橙色
CRITICAL   红色
```

### 6.4 AI 标识

所有 AI 生成或 AI 辅助内容必须显示：

```text
AI 生成
模型/工作流
生成时间
来源证据
审核状态
```

AI 内容不允许直接显示成“已生效结论”。必须处在：

```text
候选
待审核
已人工确认
已驳回
```

### 6.5 来源卡片

来源卡片必须包含：

- 来源标题。
- 发布机构。
- 版本/发布日期。
- 引用章节/页码/条款。
- 证据等级。
- 审核人。
- 影响资产。

来源缺失、过期、未审核时，发布按钮必须阻断或进入风险确认流程。

### 6.6 表格

表格是主要工作区，不是装饰组件。

必须支持：

- 服务端分页。
- 组织范围筛选。
- 状态筛选。
- 批量操作。
- 固定操作列。
- 空态和错误态。
- 列宽不抖动。
- 关键列可复制。

### 6.7 抽屉

抽屉用于：

- 详情。
- 审核。
- diff。
- 来源证据。
- 执行日志。

抽屉宽度：

```text
详情：560px
审核/diff：720px
复杂配置：960px
```

移动端或窄屏改为全屏。

## 7. 关键流程

### 7.1 配置发布流程

```text
草稿保存
-> 自动校验
-> 来源检查
-> diff 预览
-> dry-run
-> 提交审核
-> 专家审核
-> 发布
-> 同步
-> 激活
-> 审计可查
```

每一步都必须有：

- 当前状态。
- 负责人。
- 时间。
- 结果。
- traceId 或审计编号。

### 7.2 字典映射流程

```text
院内字典导入
-> 候选映射生成
-> 置信度排序
-> 冲突处理
-> 专业审核
-> 映射包发布
-> 运行时标准化
-> 未映射治理
```

有 Dify/AI 时，只增强候选生成和复核。没有 Dify/AI 时，流程仍必须完整可用。

### 7.3 规则校验流程

```text
选择组织
-> 选择场景
-> 选择规则包
-> 编辑患者上下文
-> dry-run
-> 查看命中
-> 查看来源
-> 导出报告
```

规则结果必须优先展示确定性规则命中，不把 AI 文本放在第一位。

### 7.4 异常处理流程

错误展示必须包含：

- 错误码。
- 用户可理解说明。
- 可恢复动作。
- traceId。
- 是否已写审计。

禁止只显示：

```text
操作失败
系统异常
请联系管理员
```

## 8. 高保真原型优化规范

当前 `origin/main:frontend-prototype/` 已有静态 HTML 原型。后续优化必须遵守：

### 8.1 必改视觉方向

- `.app-sidebar` 从深色侧栏改为天蓝顶部菜单 + 白色/浅蓝二级导航。
- `body` 背景改为白色或极浅蓝，不使用深灰蓝底。
- `--color-brand` 调整为天蓝体系。
- 卡片保留白底，边框使用浅蓝灰。
- 当前菜单项使用天蓝高亮，不使用深色块。

### 8.2 原型页面目标

必须优化这些页面：

```text
frontend-prototype/index.html
frontend-prototype/workbench.html
frontend-prototype/demo-validation.html
frontend-prototype/config-package-center.html
frontend-prototype/provenance.html
frontend-prototype/hospital-dashboard.html
frontend-prototype/styles/base.css
```

### 8.3 原型验收

原型完成后必须检查：

- 1366x768 首屏不出现横向滚动。
- 1440x900 信息密度适中。
- 390px 移动宽度不遮挡文字。
- 菜单栏为天蓝色。
- 主背景白色。
- 关键按钮、状态、错误、AI 标识、来源卡片一致。
- 无 Neo4j/无 Dify 状态在工作台和演示页可见。

### 8.4 原型补丁

由于当前本地 `main` 落后 `origin/main`，不要直接在当前工作区新增 `frontend-prototype/`。应先在最新 `origin/main` 基线上改原型，生成补丁，再在合入远端后应用。

推荐流程：

```powershell
git fetch --all --prune
git worktree add --detach C:\tmp\zyengine-fe-prototype origin/main
# 在 C:\tmp\zyengine-fe-prototype 修改 frontend-prototype
git -C C:\tmp\zyengine-fe-prototype diff -- frontend-prototype > C:\tmp\zyengine-fe-prototype-blue.patch
git -C C:\tmp\zyengine-fe-prototype apply --check C:\tmp\zyengine-fe-prototype-blue.patch
```

## 9. React 工程落地规范

React 工程必须把本文 token 落到：

```text
frontend/src/styles/global.css
frontend/src/layouts/AppLayout.tsx
frontend/src/router/routes.tsx
```

Ant Design 主题：

```ts
{
  token: {
    colorPrimary: "#168bd3",
    colorBgLayout: "#ffffff",
    colorBorder: "#d8e8f3",
    borderRadius: 6,
    fontSize: 14
  }
}
```

工程页面必须复用同一组业务组件：

```text
OrgContextBar
ProviderStatusStrip
SourceEvidenceCard
RuleHitList
StatusTag
AiGeneratedBadge
TraceIdText
ReviewDrawer
DiffViewer
DryRunResultPanel
```

不允许每个页面自己重新画一套状态标签、来源卡片和错误提示。

## 10. AI 开发约束

后续 AI 做前端任务必须先回答：

```text
页面服务哪个角色？
用户在这个页面完成哪个工作任务？
是否需要组织上下文？
是否涉及配置发布生命周期？
是否需要来源追溯？
是否需要 dry-run？
是否有 DB-only / 无 Dify 状态？
是否显示 traceId？
是否有 loading/error/empty/data 四态？
是否符合天蓝菜单栏和白色背景？
```

前端任务完成前必须提供：

- 页面截图或原型链接。
- 关键状态截图。
- 移动宽度检查。
- `git diff --check`。
- 设计规范符合性自检。

## 11. 红线

- 不允许把前端做成营销首页。
- 不允许深色侧栏作为默认高保真方向。
- 不允许同一状态在不同页面用不同颜色或文案。
- 不允许发布按钮绕过 review/dry-run/来源检查。
- 不允许把 AI 生成内容当作临床最终结论。
- 不允许只在前端做权限控制。
- 不允许错误提示没有 traceId。
- 不允许新增页面没有空态、错误态和加载态。
- 不允许在当前脏工作区直接新增远端已有的 `frontend/` 或 `frontend-prototype/` 目录。
