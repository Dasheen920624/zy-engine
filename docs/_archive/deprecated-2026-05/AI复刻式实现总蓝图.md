# AI 复刻式实现总蓝图

> 本文是后续 AI “按同一套体系复刻式实现”的实施总控。任务状态以 `02_任务台账.md` 为准；具体领域细节以专项详设为准；旧长文只作为历史背景，不再作为实施权威。

## 1. 本次梳理修正的问题

| 问题 | 风险 | 修正方式 |
|---|---|---|
| 文档权威链不清 | AI 会在旧总纲、新详设、任务台账之间摇摆 | 本文定义权威顺序；`00_总入口` 同步修正 |
| 旧产品总纲仍称自己为总纲 | 后续 AI 可能按旧优先级开发 | 已降级为历史基线，最新实施口径看本文 |
| 新增 9 项、原有能力、已实现改造分散 | AI 不知道一个任务该读哪份详设 | 本文给出任务前缀到详设文档的映射 |
| 已实现能力缺统一改造节奏 | 新功能会叠在不一致底座上 | `REFIT-*` 任务作为统一改造主线 |
| P0 能力之间依赖顺序不够显式 | 并行开发可能先做上层功能，后补底座返工 | 本文给出阶段化执行顺序 |
| 验收证据和开发完成口径可能混淆 | 功能“代码完成”但无法客户验收 | 本文规定所有客户可见/高风险能力必须进入 feature acceptance |

## 2. 权威顺序

发生冲突时按以下顺序裁决：

1. `02_任务台账.md`：任务状态、优先级、依赖、独占文件唯一权威。
2. `00_总入口与AI接手导航.md`：接手流程、并发规则、DoD、验证命令唯一权威。
3. `AI复刻式实现总蓝图.md`：体系边界、实现顺序、复刻模板唯一权威。
4. 专项详设文档：对应领域的数据模型、API、页面、验收细节权威。
5. 旧长文档：历史背景和补充材料，不作为默认实施依据。

## 3. 最小阅读路径

每个 AI 接手任务时只读以下内容：

```text

1. 00_总入口与AI接手导航.md
```

## 5. 目标体系总图

```text
外网通用服务 / 院内部署 / 混合交付
        |
租户 SEC_TENANT + 组织 ORG_UNIT + 平台用户 SEC_USER
        |
统一权限 / 身份绑定 / 服务账号 / API 签名
        |
统一待办 WF + 审计 AUDIT + 来源 PROV + traceId
        |
配置包生命周期 PKG
        |
规则 RULE / 路径 PATH / 字典 TERM / 图谱 GRAPH / Dify AI / 知识 AIK
        |
CDSS / 智能评估 EVAL / 报表 RPT / 运维 OPS
        |
前端工作台 / 审核台 / 配置中心 / 验收中心
        |
feature acceptance + 多角色 review + GOLD/SILVER/BRONZE/REJECTED
```

## 6. 不变量

任何新代码或改造都必须保持以下不变量：

- 每个业务对象都有 `tenant_id` 和必要组织范围。
- 每个人类操作都有平台内部 `user_id`；外部账号只作为身份绑定。
- 每个服务账号有 `client_id`、密钥 hash、权限范围、审计。
- 医学、医保、质控、临床建议、AI 候选必须有来源证据。
- 发布、回滚、导入、导出、审核、权限变更、同步任务必须写审计。
- API 必须返回 `success/code/message/data/trace_id` 或兼容现有统一结构。
- DDL 必须同步 Oracle、达梦、PostgreSQL/Kingbase、LOCAL_H2_FILE。
- 前端必须有加载态、空态、错误态、无权限态和标准中文文案。
- Neo4j、Dify、大模型不可用时，DB-only 核心流程仍可验收。
- 客户可见或高风险功能没有验收证据不得标记 DONE。

## 8. 复刻式实现模板

每个 AI 实施任务时，必须按以下顺序产出。

### 8.1 开工前

```text
1. git pull --ff-only origin main
2. git status -sb
3. check-ai-collaboration.ps1
4. 读取 00、02 目标任务行、本文、对应专项详设
5. 创建 active claim 并推送
6. 确认独占文件和依赖已完成
```

### 8.2 设计落点

在最终实现前先明确：

- 本任务属于哪个租户/组织/身份/来源/审计/数据库/前端/验收底座。
- 需要新增或修改哪些表、字段、索引、注释。
- 需要新增或修改哪些 Controller、Service、Repository、Provider。
- 需要新增或修改哪些前端页面、组件、路由、API client 类型。
- 需要新增哪些样例、测试、验收记录。

### 8.3 代码实现顺序

```text
DDL / migration
-> domain model
-> repository / persistence
-> service
-> controller / API
-> audit / provenance / permission / traceId
-> frontend API client / types
-> frontend page / component
-> samples
-> tests
-> feature acceptance
-> docs
```

纯后端任务可跳过前端，但必须说明为什么。

### 8.4 验收证据

至少包含：

- 单元测试或契约测试。
- DB-only 验证；无生产库时说明 Oracle/DM/PG 补验计划。
- 权限或组织隔离验证。
- 审计或 traceId 验证。
- 客户可见功能截图或 E2E 证据。
- `ai-dev-input/13_feature_acceptance/` 记录。

## 9. 任务提示词模板

后续给 AI 分配任务时，建议直接使用：

```text
请领取 <TASK_ID>。

硬性要求：
1. 先读 00_总入口与AI接手导航.md、02_任务台账.md 的目标任务行、AI复刻式实现总蓝图.md。
2. 按任务前缀补读总蓝图第 4 节指定的专项详设。
3. 创建并推送 active claim 后再改业务文件。
4. 严格只改任务台账声明的独占文件；需要共享文件先登记说明。
5. 所有新增表同步 Oracle、达梦、PostgreSQL/Kingbase、LOCAL_H2_FILE，并补中文注释。
6. 后端必须有契约测试；前端必须有浏览器/E2E 或截图验证；客户可见功能必须有 feature acceptance。
7. 完成后提交 review，归档 claim/review/acceptance，commit + push 到 main。

目标：
按行业最高标准完成 <TASK_ID>，不要只做原型；必须形成可交付、可验收、可追溯的实现。
```

## 10. 复刻质量检查表

| 检查项 | 必须回答 |
|---|---|
| 租户 | 数据是否按 `tenant_id` 隔离 |
| 组织 | 是否按集团/医院/院区/科室过滤 |
| 身份 | 操作人是否是平台 `user_id` 或服务 `client_id` |
| 权限 | 后端是否强制鉴权，前端是否有无权限态 |
| 来源 | 医学/医保/质控资产是否有来源 |
| 审计 | 发布、回滚、审核、导入导出、权限变更是否写审计 |
| 数据库 | 四类数据库 DDL 是否同步 |
| 降级 | Neo4j/Dify/AI 不可用时核心流程是否可用 |
| 文案 | 用户可见文案是否标准中文 |
| 验收 | 是否有自动化测试和 feature acceptance |

## 11. 禁止事项

- 不允许绕过任务台账直接“顺手实现”未登记需求。
- 不允许只写文档宣布功能完成。
- 不允许只跑单元测试就声称数据库功能完成。
- 不允许用院内外部账号直接作为系统审计主体。
- 不允许新增表只做 Oracle 或只做 H2。
- 不允许把 AI 输出直接作为临床高风险结论。
- 不允许把旧长文中的过期优先级作为当前开发依据。

## 12. 完成口径

一个任务只有同时满足以下条件才可以标记 DONE：

```text
claim: DONE
review: APPROVED
feature_acceptance: GOLD 或明确 N/A
tests/build/checks: PASS 或有明确非阻断说明
database: 生产库/开发库验证计划明确
docs: 受影响入口和详设同步
git: commit + push 到 main
```
