# 已实现能力统一改造优化方案

> 结论：已经实现的能力必须统一改造优化，但不是推倒重写。原则是先盘点、建基线、补验收，再按统一底座逐步重构，最终形成一套一致的产品体系。

## 1. 为什么必须统一改造

当前已实现能力覆盖配置包、组织、来源、规则、路径、图谱、Dify、前端原型、审计、测试等多个方向，但它们由不同批次、不同 AI 实现，存在天然不一致风险：

- 有的能力已接组织上下文，有的仍可能是内存态或跨租户共享。
- 有的能力有生产库/开发库 DDL，有的只有内存或单库实现。
- 有的能力写审计，有的只返回结果。
- 有的能力有来源证据，有的只保存业务字段。
- 有的前端页面已接真实 API，有的仍是占位或局部演示。
- 有的能力有测试，有的没有 feature acceptance 和客户验收证据。

如果不统一改造，后续新增 9 项能力会叠在不一致的底座上，最终系统会变成“功能能跑，但体系不统一、无法稳定交付”的状态。

## 2. 改造原则

| 原则 | 说明 |
|---|---|
| 不推倒重写 | 保留已通过验证的模型、API 和测试，采用兼容迁移和小步改造 |
| 先建验收基线 | 改造前先记录当前行为，避免优化过程中破坏已有功能 |
| 以底座统一为主 | 统一租户、身份、组织、来源、审计、数据库、API、前端状态 |
| 生产库和开发库同步 | Oracle、达梦、PostgreSQL/Kingbase、LOCAL_H2_FILE 同步演进 |
| 医疗安全优先 | 医学/医保/质控相关能力先补来源、审核、发布阻断和审计 |
| AI 并发友好 | 每个改造任务必须有独占文件、claim、review、feature acceptance |

## 4. 统一体系目标

所有已实现和后续新增功能最终都必须接入以下 9 个统一底座：

| 底座 | 统一要求 |
|---|---|
| 租户和组织 | 每个业务对象、查询、发布、执行、审计都带 `tenant_id` 和组织范围 |
| 用户和权限 | 所有人类用户、服务账号、外部身份都归并到本系统身份中心 |
| 配置包生命周期 | 规则、路径、图谱、Dify、字典、适配器、知识包都可注册、review、发布、回滚 |
| 来源和证据 | 医学/医保/质控资产必须有来源，运行结果必须能查证据 |
| 审计和 trace | 发布、回滚、导入、导出、审核、权限变更、同步、模型调用均有审计 |
| 多数据库 | 生产库 Oracle/达梦/PG-Kingbase 与开发库 LOCAL_H2_FILE 结构同步 |
| API 契约 | 响应、错误码、分页、幂等、traceId、权限错误、中文消息统一 |
| 前端交互 | 天蓝白底、角色工作台、四态、无权限态、来源卡片、AI 标识统一 |
| 验收证据 | 每个客户可见功能都有 feature acceptance、截图、API/DB/审计证据 |

## 5. 改造阶段

### 5.1 阶段 0：冻结基线和盘点

目标：知道当前系统到底有哪些已实现能力、哪些风险、哪些行为不能破坏。

交付：

- 已实现能力清单。
- 当前 API 列表、页面列表、表列表、测试列表。
- 每个能力的租户、组织、来源、审计、数据库、前端、验收状态矩阵。
- P0/P1/P2 改造 finding。

任务：`REFIT-001`。

### 5.2 阶段 1：核心底座贯通

目标：让已实现能力先统一接入最关键的底座。

交付：

- 租户/组织/身份上下文贯通。
- 来源、审计、traceId、发布门禁统一。
- 多数据库 DDL、中文注释和持久化差异补齐。

任务：`REFIT-002`、`REFIT-003`、`REFIT-004`。

### 5.3 阶段 2：契约和体验统一

目标：统一 API 和前端体验，让后续 AI 开发不会继续扩散风格。

交付：

- API 响应、错误码、分页、幂等、traceId、message_key 统一。
- 前端页面四态、无权限态、状态标签、来源卡片、AI 标识、中文文案统一。
- 已有页面接入角色工作台和权限模型。

任务：`REFIT-005`、`REFIT-006`。

### 5.4 阶段 3：验收证据和专业优化

目标：把已实现功能全部纳入正式验收，未达标的进入优化池。

交付：

- 每个已实现客户可见功能有 feature acceptance。
- 每个核心能力至少有一条 AI 自动验收路径。
- 未达到 `GOLD` 的功能进入 `needs_optimization/` 和任务台账。

任务：`REFIT-007`。

## 8. 改造方法

### 8.1 不破坏兼容

- 保留已有 API 路径，必要时新增 v2 或兼容参数。
- 数据库迁移必须可重复执行，已有数据要有默认组织和租户迁移策略。
- 已有测试先记录为基线，再逐步加强断言。
- 对高风险改造使用 feature flag 或配置开关。

### 8.2 先补测试再改核心

每个 `REFIT-*` 任务必须先补：

- 当前行为回归测试。
- 至少 1 条 DB-only 验收。
- 至少 1 条权限或组织隔离测试。
- 高风险能力补审计断言。

### 8.3 改造后必须归档证据

每个能力改造完成后，必须在 `ai-dev-input/13_feature_acceptance/` 记录：

- 改造前问题。
- 改造后行为。
- 截图或 API 响应。
- 数据库验证。
- 审计验证。
- 剩余风险和优化项。

## 9. 完成标准

统一改造完成后，系统应达到：

- 新增 9 项能力和原有能力共享同一套身份、租户、组织、来源、审计、数据库、API、前端、验收体系。
- 任意功能都能回答：属于哪个租户、哪个组织、谁操作、依据什么来源、发布了哪个版本、如何回滚、如何验收。
- 无 Oracle/DM/PG 环境时，开发库 LOCAL_H2_FILE 可做等价验证；有生产库环境时必须补真实 smoke。
- 客户可见功能不再只以“代码完成”为标准，而以 `GOLD/SILVER/BRONZE/REJECTED` 验收等级为标准。
