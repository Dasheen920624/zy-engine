# 用户体系与身份联邦详细设计方案

> 状态：已形成产品与实现方案，具体编码按 `02_任务台账.md` 中 `SEC-001/006/007/008/011/012` 认领实施。

## 1. 结论

本系统必须有自己的独立用户体系。院内 HIS、EMR、OA、统一身份平台、LDAP/AD、CAS、OIDC、SAML 只能作为外部身份源或同步源，不能替代本系统用户主数据。

原因：

- 产品面向 N 个医院、集团、院区、科室和外网客户，不能被某一家医院的账号模型锁死。
- 外网基准服务、客户试用、实施交付、专家审核、运维支持、API 客户端都需要不依赖院内身份源即可运行。
- 院内用户同步会出现重名、换岗、离职、跨院区、多身份源冲突，必须由平台账号统一归并、授权和审计。
- 医疗安全、发布、审核、回滚、知识包导入导出都必须追溯到平台内部 `user_id`，外部账号只作为绑定证据。

## 2. 身份边界

| 身份类型 | 是否本系统主账号 | 来源 | 典型场景 |
|---|---|---|---|
| 平台运营账号 | 是 | 本系统创建 | SaaS 运维、知识包基准维护、客户支持 |
| 客户租户管理员 | 是 | 本系统创建或邀请 | 外网开通、院内初始化、租户配置 |
| 院内业务用户 | 是 | 院内同步后生成或绑定 | 医生、护士、药师、质控、医保、信息科 |
| 外部专家/审核员 | 是 | 本系统邀请 | 指南审核、规则复核、知识包评审 |
| 第三方系统客户端 | 是，服务账号 | 本系统签发 | API 调用、院内适配器、同步任务 |
| 院内 SSO 身份 | 否 | CAS/OIDC/SAML/LDAP/AD | 作为登录凭证和身份绑定来源 |

统一原则：

- 每个可审计主体都必须有平台内部 `platform_user_id` 或 `client_id`。
- 外部身份通过绑定表关联，不直接成为权限判断主键。
- 权限判断只看本系统租户、角色、权限、组织范围、数据范围、责任链。
- 院内身份源失效时，系统仍允许保留应急本地管理员、服务账号和只读审计入口。

## 3. 产品形态

### 3.1 外网通用服务

外网服务用于产品基准运营、客户试用、知识包生成、跨客户交付和通用管理。

必须支持：

- 客户租户开通、暂停、禁用、到期、扩容。
- 租户管理员邀请、密码登录、MFA、找回密码、登录风控。
- 平台运营人员按客户租户授权，不允许越权查看院内敏感数据。
- 外网知识包、规则包、字典包生成和导出。
- 外网不得接收院内真实患者隐私；院内上传治理包必须先脱敏并有授权记录。

### 3.2 院内部署服务

院内服务用于内网落地、业务系统集成、临床审核和发布使用。

必须支持：

- 本地超级管理员初始化租户、组织、身份源和同步策略。
- 院内用户全量同步、增量同步、手动同步、停用同步。
- 院内 SSO 登录后绑定到平台内部用户。
- 无 SSO 或院内身份源不可用时，可使用本地应急管理员和服务账号完成恢复。
- 院内真实业务数据不依赖外网服务即可完成核心验收。

### 3.3 混合交付

```text
外网基准服务
-> 生成标准知识包 / 规则包 / 字典包 / 评测包
-> 签名导出
-> 院内导入 dry-run
-> 院内审核
-> 院内发布
-> 院内反馈脱敏治理包
-> 外网生成增量包
```

身份要求：

- 外网和院内可以是不同租户实例，但必须通过租户授权、包签名和审计关联。
- 外网用户不能自动拥有院内权限。
- 院内用户不能默认登录外网，除非租户管理员显式邀请或绑定。

## 4. 核心数据模型

| 表 | 用途 | 关键字段 |
|---|---|---|
| `SEC_TENANT` | 客户租户和外网客户 | `tenant_id`、`tenant_code`、`tenant_type`、`status`、`license_id` |
| `SEC_USER` | 平台内部用户主表 | `user_id`、`tenant_id`、`login_name`、`display_name`、`user_type`、`status` |
| `SEC_LOCAL_CREDENTIAL` | 本地登录凭证 | `user_id`、`password_hash`、`mfa_enabled`、`password_updated_at` |
| `SEC_IDENTITY_PROVIDER` | 身份源配置 | `provider_id`、`tenant_id`、`provider_type`、`priority`、`status` |
| `SEC_IDENTITY_BINDING` | 外部身份绑定 | `binding_id`、`user_id`、`provider_id`、`external_subject`、`external_org_code` |
| `SEC_ROLE` | 角色 | `role_id`、`tenant_id`、`role_code`、`role_name`、`role_type` |
| `SEC_PERMISSION` | 权限点 | `permission_id`、`permission_code`、`resource_type`、`action_code` |
| `SEC_USER_ROLE` | 用户角色关系 | `user_id`、`role_id`、`scope_type`、`scope_id` |
| `SEC_USER_ORG_SCOPE` | 组织数据范围 | `user_id`、`tenant_id`、`org_unit_id`、`scope_level` |
| `SEC_SESSION` | 登录会话 | `session_id`、`user_id`、`login_method`、`expires_at`、`revoked_at` |
| `SEC_API_CLIENT` | 服务账号和 API 客户端 | `client_id`、`tenant_id`、`client_type`、`secret_hash`、`status` |
| `SEC_INVITATION` | 外网邀请和客户开通 | `invitation_id`、`tenant_id`、`email_or_mobile`、`role_code`、`expires_at` |

DDL 要求：

- Oracle、达梦、PostgreSQL/Kingbase、LOCAL_H2_FILE 必须同步。
- 生产库表和关键字段必须有中文注释。
- `tenant_id + provider_id + external_subject` 必须唯一，避免同一外部账号绑定多个平台用户。
- `login_name` 只在租户内唯一，不做全局唯一，避免 N 医院同名冲突。

## 5. 登录与同步流程

### 5.1 本地账号登录

```text
输入账号密码
-> 校验租户状态和用户状态
-> 校验密码 / MFA / 风控
-> 创建 SEC_SESSION
-> 加载角色、权限、组织范围
-> 写登录审计
```

### 5.2 院内 SSO 登录

```text
跳转院内 SSO
-> 获取外部 subject
-> 查 SEC_IDENTITY_BINDING
-> 找到平台 SEC_USER
-> 同步必要的姓名、科室、岗位快照
-> 创建 SEC_SESSION
-> 加载本系统权限
```

如果没有绑定：

- 有自动开户策略：创建 `SEC_USER`，绑定外部身份，进入待授权或默认角色。
- 无自动开户策略：拒绝登录，并生成待处理身份绑定申请。

### 5.3 院内用户同步

```text
读取院内用户源
-> 标准化用户、岗位、科室、状态
-> 匹配 SEC_IDENTITY_BINDING
-> 新用户创建平台用户
-> 已有用户更新快照和组织范围
-> 离职/停用用户禁用登录但保留审计主体
-> 输出同步报告
```

冲突处理：

- 同一外部 subject 绑定多个用户：阻断并进入人工合并。
- 同一人员多个身份源：允许绑定到同一平台用户，按身份源优先级处理。
- 调科或跨院区：更新组织范围，保留历史审计。
- 离职：禁用登录，不删除用户，不删除历史审计。

## 6. 权限模型

权限判断由 5 层组成：

1. 租户状态：租户未启用、过期、暂停时阻断。
2. 用户状态：锁定、停用、离职、密码过期时阻断。
3. 角色权限：角色包含菜单、按钮、接口、动作权限。
4. 组织范围：集团、医院、院区、站点、科室范围过滤。
5. 数据策略：患者隐私、模型输出、导出、审计、知识包按策略二次过滤。

前端权限只做体验控制，后端必须强制校验。

## 7. 实现拆解

| 阶段 | 任务 | 交付 |

| 1 | `SEC-001` 平台独立用户、租户和角色中心 | 本系统用户主数据、角色、权限、组织范围、会话、DDL、基础 API |
| 2 | `SEC-011` 外网通用服务账号和客户租户开通 | 外网租户开通、客户管理员邀请、本地登录、MFA、服务账号 |
| 3 | `SEC-006` 院内用户体系同步 | HIS/EMR/OA/统一身份平台同步、差异报告、停用处理 |

## 8. 验收标准

- 无任何院内身份源时，外网服务可完成客户租户开通、管理员登录、知识包管理和服务账号调用。
- 院内部署时，可通过本地管理员初始化身份源并同步院内用户。
- SSO 登录后，用户权限来自本系统，不直接信任院内角色文本。
- 同一用户可绑定多个身份源，但审计主体始终是同一个平台内部用户。
- 离职或停用用户不能登录，但历史发布、审核、回滚、模型调用审计仍能追溯。
- API 客户端与人类用户分离，密钥不以明文保存。
- 所有用户、角色、权限、身份源、绑定、会话、服务账号表完成多数据库 DDL 和中文注释。
