# MedKernel v1.0 GA 产品收口方案

> 版本：1.0 · 2026-05-23  
> 适用：v1.0 GA 产品定义、菜单信息架构、AI 团队拆单、客户验收。  
> 核心原则：**最少入口，完整闭环；客户先看业务主线，专家再进高级能力。**

---

## 1. 一句话

MedKernel 不是“功能集合”，而是一条医院能听懂、能试点、能验收的闭环：

> **把指南、路径、规则和院内数据接起来，在临床现场提醒医生，在质控侧形成整改闭环，并留下合规证据。**

面向客户时只讲四件事：

| 顺序 | 客户问题 | 产品回答 |
|---|---|---|
| 1 | 我怎么开始试点？ | 试点准备：建租户、接系统、导入路径和规则 |
| 2 | 医生怎么真正用起来？ | 临床运行：患者入径、节点流转、医嘱提醒、待办处理 |
| 3 | 管理层怎么看效果？ | 质控改进：预警、医保审核、评估结果、整改追踪 |
| 4 | 能不能放心上线？ | 合规运维：身份、审计、安全、监控、备份、SLA |

所有技术名词（图谱、AI 工作流、Provider、OpenAPI、国密、Flyway）都必须服务这四件事，不能成为客户第一层导航。

---

## 2. 产品设计理念

### 2.1 最精简化

- 一个主线：试点准备 → 临床运行 → 质控改进 → 合规运维。
- 一个默认登录方式：账号密码；统一身份认证作为医院配置后的增强入口。
- 左侧侧边栏保持为主菜单，不改成顶部导航；顶部只放品牌、面包屑、折叠按钮和用户信息。
- 左侧一级分组不超过 6 组；每组按客户工作顺序排列。
- 客户演示只走 6 个剧本，不逐项介绍 20+ 功能。
- 首屏不展示内部任务编号、技术债、Provider 名称和实现细节。

### 2.2 功能最全

精简不是删功能，而是分层：

| 层级 | 面向谁 | 内容 |
|---|---|---|
| L1 主路径 | 院长 / 医务处 / 信息科主任 | 试点、临床、质控、合规四组入口 |
| L2 专业工作台 | 信息科 / 质控 / 路径专家 | 路径、规则、字典、适配器、审核、报表 |
| L3 高级工具 | 架构师 / 实施 / 运维 | 图谱、AI 工作流、Provider、OpenAPI、监控、迁移 |

功能可以完整，但展示必须有主次。

## 3. v1.0 GA 菜单信息架构

菜单形态固定为**左侧侧边栏 SideMenu**。顶部 Header 不承载主菜单，只负责品牌、当前位置、组织/用户入口和折叠控制；移动端可退化为抽屉，但信息架构仍按左侧菜单组织。

### 3.1 一级菜单

| 顺序 | 一级菜单 | 客户能理解的话 |

## 4. 登录页收口

### 4.1 默认形态

登录页只保留一个主动作：**账号登录**。

| 元素 | 处理 |
|---|---|
| 账号密码 | 默认展示，兼容医院工号/管理员账号 |
| 统一身份认证 | 作为次级按钮区，按医院配置显示 CAS/OIDC/SAML |
| 手机短信 | 不作为独立登录 Tab；仅作为二次验证或找回密码能力 |
| LDAP 域账号 | 不单独暴露给客户；并入统一身份认证配置 |
| MFA / 国密 / 审计 | 作为安全承诺展示，不让用户手动选择算法或认证器 |
| 演示账号 | 仅 demo profile 折叠展示，生产环境不可见 |

### 4.2 中国国情底线

登录页必须保留：ICP备案、公安备案、用户协议、隐私政策、会话超时提示、审计留痕说明。依据包括《个人信息保护法》《网络安全法》《数据安全法》《医疗卫生机构网络安全管理办法》《数据出境安全评估办法》和商用密码应用安全性评估要求。

参考官方来源：

- [个人信息保护法](https://www.npc.gov.cn/npc/c2/c30834/202108/t20210820_313088.html)
- [网络安全法](https://www.miit.gov.cn/ztzl/rdzt/tdzzyyhlwsdrhfzjkjstggyhlwpt/zcfb/art/2020/art_41be9e94ecc5433899ca88a0339a38b6.html)
- [数据安全法](https://www.miit.gov.cn/zwgk/zcwj/flfg/art/2022/art_284b390b84484f10b0e43eeafaad0f6d.html)
- [医疗卫生机构网络安全管理办法](https://app.www.gov.cn/govdata/gov/202208/31/488953/article.html)
- [数据出境安全评估办法](https://www.gov.cn/zhengce/2022-07/07/content_5728937.htm)
- [商用密码应用安全性评估管理办法](https://www.gov.cn/zhengce/202501/content_6997433.htm)

## 5. 功能取舍标准

任何新功能进入 v1.0 GA 前，必须回答 5 个问题：

| 问题 | 通过标准 |
|---|---|
| 客户是否一听就懂？ | 能用一句医院业务话说明，不依赖技术名词 |
| 是否服务四段主线？ | 属于试点准备、临床运行、质控改进、合规运维之一 |
| 是否能被验收？ | 有明确输入、操作、输出、证据 |
| 是否符合中国国情？ | 数据本地化、等保、国密、审计、备案、个保法边界清楚 |
| 是否能渐进展示？ | 初级用户不被复杂配置打扰，专家能进入高级能力 |

不能通过的功能进入 `v1.1+` 或隐藏到高级工具，不进入客户主路径。

## 6. 全功能极简交互规格

这条规则适用于所有功能，不只适用于登录页。**最精简化**指默认只让用户做最少、最明确的选择；**功能最全**指高级能力完整保留，但通过模板、推荐、折叠、向导和权限分层渐进展示。

### 6.1 通用页面模型

| 页面层 | 默认展示 | 高级能力放哪里 | 验收标准 |
|---|---|---|---|
| 首屏 | 1 个主目标、1 个主按钮、最多 3 个常用筛选 | 更多筛选 / 批量操作 / 调试参数放“高级选项”折叠区 | 新用户 10 秒内知道下一步点哪里 |
| 列表 | 名称、状态、影响范围、更新时间、下一步动作 | 技术编码、traceId、版本 hash、Provider 细节放详情页 | 列表不横向滚动即可完成常见判断 |
| 详情 | 业务摘要、校验结果、影响范围、审计证据 | 原始 JSON / DSL / 节点边表放“技术详情” | 医务/信息科能看懂风险和结果 |
| 编辑 | 模板优先、默认值优先、自动校验优先 | 手写 DSL、图谱节点、复杂表达式只给专家角色 | 普通用户不需要理解内部数据结构也能完成配置 |
| 发布 | 校验通过 → 影响评估 → 人工确认 → 审计留痕 | 回滚、灰度、跨环境同步放发布向导后段 | 每次变更可解释、可回滚、可追溯 |

### 6.2 配置类功能统一流程

路径、规则、图谱、字典、适配器、配置包都必须使用同一条客户能理解的流程：

```text
选模板 / 导入 → 填最少必要信息 → 自动校验 → 看影响 → 提交审核 → 发布 → 运行证据 → 可回滚
```

任何 AI 不得给不同配置功能各自发明一套入口、状态、按钮和审核语言。

### 6.3 关键功能收口标准

| 功能 | 默认最简形态 | 完整服务能力 | 易用选择 |
|---|---|---|---|
| 配置包中心 | “导入配置包”“查看校验结果”“发布到本院”三件事 | 跨环境同步、冲突分析、来源审核、导出、回滚、审计 | 默认按“待处理 / 已发布 / 生效中”分组，高级资产类型筛选折叠 |
| 路径配置 | 从专病模板开始，先填科室、入径条件、关键节点 | 版本对比、节点任务、变异、图谱/规则绑定、灰度发布 | 普通模式用表单 + 时间线；专家模式再打开画布和 JSON |
| 规则库 | 从“医保审核 / 医嘱安全 / 路径质控”模板创建 | DSL、严重度、动作模式、来源、试运行、版本发布 | 默认推荐规则模板和示例事实；手写 DSL 放专家区 |
| 图谱配置 / 查询 | 默认只做“输入症状/体征 → 看候选和证据” | 节点、边、版本、证据链、Neo4j/DB 降级、图谱发布 | 节点/边明细默认折叠为技术详情，不干扰客户主流程 |
| 字典映射 | 默认导入院内码并自动匹配标准码 | 批量审核、冲突处理、版本化、跨院区差异 | 只把“未匹配 / 有冲突 / 已确认”作为首层任务 |
| 适配器中心 | 默认展示系统接入状态和下一步配置 | HIS/EMR/LIS/PACS、CDS Hooks、重试、熔断、字段映射 | 普通用户看“接通/异常/待配置”，接口参数放高级配置 |
| 患者路径 | 默认看当前节点、下一步任务、变异原因 | 入径推荐、节点推进、任务执行、医生确认、审计 | 医护只看到与当前患者相关的动作，不展示配置细节 |
| 临床提醒治理 | 默认看提醒、建议、来源和医生确认 | 疲劳治理、阻断规则、覆盖理由、追溯日志 | 只给“采纳/不采纳并说明”两类主动作 |
| 质控改进 | 默认看问题、责任科室、整改状态、趋势 | 指标库、预警、医保审核、报告导出、派单闭环 | 首屏按“待整改 / 整改中 / 已闭环”组织 |
| 合规运维 | 默认看风险项、证据缺口和整改动作 | 等保、国密、审计验签、备份恢复、SLA、监控告警 | 合规项显示“已满足 / 缺证据 / 需整改”，不展示算法选择 |

### 6.4 禁止的复杂化模式

| 禁止模式 | 替代做法 |
|---|---|
| 把左侧菜单改成顶部横向导航 | 保持左侧 SideMenu；顶部只放品牌、面包屑、用户和折叠按钮 |
| 首屏同时出现 5 个以上同权按钮 | 只保留 1 个主按钮，其余进“更多”或详情页 |
| 让用户手动选择国密算法、MFA 方式、Provider 链路 | 系统按医院策略自动选择，页面只显示安全承诺和状态 |
| 默认展示 DSL、JSON、节点边表、traceId | 放入专家模式 / 技术详情 / 审计详情 |
| 每个功能自造状态机 | 配置类统一使用草稿、待审核、已发布、生效中、已下线 |
| 把客户不理解的英文缩写放一级菜单 | 改成业务语言，技术缩写放说明或高级工具 |
| 为了“全”把所有筛选都摊开 | 默认 3 个常用筛选，其余放高级筛选 |

## 7. v1.0 GA 验收标准

| 验收对象 | 标准 |
|---|---|
| 客户演示 | 30 分钟跑完 6 大剧本，客户能复述“准备、运行、质控、合规”四段价值 |
| 菜单 | 左侧 SideMenu 保持固定；一级分组不超过 6 组；无客户可见 PlaceholderPage；每个菜单有明确业务动词 |
| 登录 | 默认只有账号登录；统一身份认证按医院配置展示；合规信息完整但不干扰登录 |
| 全功能易用性 | 所有客户可见页面符合 §6：1 个主目标、1 个主按钮、最多 3 个默认筛选，高级参数折叠 |
| 试点准备 | 能建租户、接系统、导入路径/规则/字典、发布配置包 |
| 临床运行 | 能完成患者入径、节点流转、规则校验、医生确认、待办处理 |
| 质控改进 | 能发现问题、派单整改、查看趋势、导出报告 |
| 合规运维 | 能证明身份权限、审计留痕、安全基线、监控告警、备份恢复 |
| 中国国情 | 等保 2.0 三级、国密、个保法、数据安全、网络安全、医疗机构网络安全要求均有证据链 |

## 8. AI 团队执行要求

AI 团队后续所有拆单必须按“主路径优先”排序：

1. 先修客户看不懂的问题：菜单、登录、首页、剧本。
2. 再补客户试点必须完成的闭环：接入、路径、规则、临床提醒、质控整改。
3. 再补 GA 证据：测试覆盖、合规评测、监控、手册、SLA。
4. 最后做高级工具增强：图谱、AI 工作流、自动化、商业化扩展。

任何 AI 不得为了“功能完整”把二级/高级能力重新塞回一级菜单。
