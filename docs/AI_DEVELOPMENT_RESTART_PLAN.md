# AI 研发重启执行方案

> 版本：1.0 · 2026-05-31
> 状态：执行闸门
> 定位：本文不替代施工卡，不新增功能事实源；它规定 AI 如何按现有权威文档高质量执行，防止再次出现登录不可用、界面混乱、假闭环和 done 注水。
> 权威边界：产品事实以 [CONSTITUTION](CONSTITUTION.md) + [cards/](cards/) + [EXPERIENCE_CONTRACT](EXPERIENCE_CONTRACT.md) + [backlog](backlog.md) + [质量基线](audit/质量基线.md) 为准。业务范围核查见 [业务实现范围核查方案](BUSINESS_IMPLEMENTATION_SCOPE_AUDIT.md)。本文只管研发节奏、核查闸门和 AI 交付纪律。

---

## 1. 当前判定

本项目进入**研发重启**：现有代码只能作为可核查素材，不再作为完成证明。任何 `done`、旧计划、旧审计结论或页面肉眼可见效果，都必须重新用施工卡、测试、门禁和真实运行证据证明。

本次核查发现的立即风险：

- 活动 OpenSpec `engine-authenticity-remediation` 曾把字典 LCS 写成修复目标，但核心 §7 与质量基线 §4.3 明确要求医学语义映射、禁止字符 LCS。
- [质量基线](audit/质量基线.md) 仍残留“旧详规是实现权威”“E0–E6”等旧口径，容易误导 AI 继续读旧巨物。
- 现有前端仍有多处 `eslint-disable medkernel/no-page-mock`、函数包装规避门禁、`Math.random()` 造数和假 hash；登录页样式仍有硬编码 hex / rgba / px 阴影，说明设计 token 与 T-GATE 尚未真正闭环。
- [docs/_HANDOFF](_HANDOFF.md) 曾显示 wave2 批 2-4 待合，但 `origin/main` 已包含 #169，交接状态必须及时纠偏，避免下个 AI 重复做已合工作。
- [业务实现范围核查](BUSINESS_IMPLEMENTATION_SCOPE_AUDIT.md) 显示：S0–S40 在卡索引中已有承接，但覆盖矩阵仍存在大量 `待迁`，且 D2 存在“28 张物理卡 / 30 个逻辑交付项”的计数口径差异，必须在执行前显性化。

结论：先修**执行系统**，再修业务功能。没有闸门，AI 会继续把局部看起来能跑的东西做成不可验收的假产品。

## 2. 不可妥协原则

1. **从 D0 登录域重新证明系统可用**：登录页可渲染、可提交、可报错、可跳转、可登出、会话过期可回登录，且 13 角色菜单按 RBAC 呈现；D0 未过域级验收前不得进入 D1。
2. **一卡一 PR，一域一验收**：默认每个 PR 只交付一张施工卡；大卡按卡内“大卡工序”拆 PR。一个域全部卡完成后，必须再跑域级 E2E。
3. **先核业务范围，再核代码实现**：每域开工前必须确认 S 场景、菜单、页面、接口、数据、B0 主链路和 wave2 消费点均能追溯到施工卡。
4. **AI 不直接相信旧代码**：开工必须先用搜索、测试、页面打开和代码事实确认现状；禁止凭文件名、旧注释、旧审计结论判断“已有实现”。
5. **TDD 与证据优先**：每张卡先写失败测试或失败 E2E，再最小实现，再跑全量门禁；没有命令输出、截图、traceId、审计记录和 PR 证据，不得说完成。
6. **无模型 B0 先行**：模型、Dify、图投影、外部系统全部关闭时，主链路必须诚实返回 B0 / NOT_CONNECTED / NOT_SYNCED / MODEL_DISABLED，且可继续完成核心人工流程。
7. **医学安全宁可慢，不可假**：字典映射、CDSS、推荐、质控和知识生成必须走来源、版本、人工确认、风险标识和回滚；禁止字符 LCS 作为医学语义判断依据。

## 3. AI 开工标准流程

每个 AI 接到任务后按下列顺序执行，不得跳步：

1. 读 [docs/_HANDOFF](_HANDOFF.md)，确认当前工作线、分支、下一步。
2. 拉取最新 `origin/main`，从最新 `main` 建短分支；Codex 分支用 `codex/` 前缀。
3. 只读必要权威文档：核心 + 对应域简报 + 对应施工卡；页面卡加读体验契约；按需查质量基线、卡索引和术语表。
4. 建立现状证据：搜索涉及文件、接口、路由、迁移、测试；记录“已存在 / 不存在 / 存在但不可信”。
5. 建立范围证据：对照 [业务实现范围核查方案](BUSINESS_IMPLEMENTATION_SCOPE_AUDIT.md)，确认该卡属于哪个 S 场景、哪个菜单/页面、哪条 B0 主链路、哪些接口/数据与测试。
6. 写红灯测试：后端单测/契约测试，前端组件测试或 Playwright 可打开性测试；页面卡必须包含真实浏览器截图或 E2E 证据。
7. 实现最小代码；禁止顺手重构无关模块。
8. 跑本卡最小门禁，再跑域相关门禁；PR 前跑全量 T-GATE、构建、类型、测试。
9. PR 描述用中文列出变更范围、业务范围映射、验证结果、未完成事项、医疗安全影响、数据迁移影响和降级结论。
10. CI 通过后合并远程 `main`，再确认 `origin/main` 包含合并提交，并更新 `_HANDOFF`。

## 4. 阶段闸门

### G0 · 治理纠偏闸门

目标：清除会误导 AI 的旧口径与错误计划。

- [ ] 活动 OpenSpec 不再出现“LCS 是修复目标”“E5/E6 当前执行”等错误口径。
- [ ] 质量基线只承认核心 + 域简报 + 施工卡，不再写旧详规权威口径。
- [ ] README / docs README / backlog 均指向本执行方案和卡体系。
- [ ] `_HANDOFF` 与 `origin/main` 真实状态一致。

### G1 · 代码债务盘点闸门

目标：把现有低质量代码显性化，禁止继续隐藏。

- [ ] 生成当前假闭环清单：`eslint-disable medkernel/*`、`Math.random()` 业务造数、假 hash、硬编码医学常量、前端假成功 catch、页面内 mock 数据。
- [ ] 清单按施工卡归属到 D0–D6 / wave2 / ga；没有归属的先挂 [BASE-09](cards/D0/BASE-09.md)。
- [ ] 每一项债务只允许两种状态：已被本 PR 修复，或已登记到对应卡的未完成事项；不得在 PR 中轻描淡写为“后续优化”。

### G1.5 · 业务实现范围核查闸门

目标：避免“做了很多代码，但业务范围缺块、页面缺块、接口缺块”。

- [ ] S0–S40 共 41 个场景均有存在的主卡，且主卡/辅助卡边界清楚。
- [ ] 27 个客户二级菜单 + 5 个高级工具均有页面卡或合并说明；任何实际路由必须能回指施工卡。
- [ ] 每张卡进入实现前都有“卡到代码映射”：前端路由/组件/hooks、后端 Controller/Service/DTO、迁移、测试、E2E。
- [ ] D0–D6 每域都有 B0 主链路脚本或明确补建任务。
- [ ] wave2 每张卡都有 B0 消费点；没有 B0 消费点的 AI 能力不得实现。
- [ ] 覆盖矩阵关键业务锚点不再 `待迁`，或已登记为范围治理缺口，不得被当作已完成。
- [ ] D2 计数口径统一为“28 张物理卡 / 30 个逻辑能力或页面交付项”，避免重复建卡或误判缺卡。

### G2 · D0 登录域救援闸门

目标：先让系统入口真实、稳定、可验收。

执行顺序：

1. [INFRA-01](cards/D0/INFRA-01.md) / [INFRA-02](cards/D0/INFRA-02.md)：门禁先以“阻断新增 + 阻断 touched file + 输出存量清单”落地；[BASE-09](cards/D0/BASE-09.md) 清完存量后切到全量硬阻断。
2. [BASE-10](cards/D0/BASE-10.md)：设计 token 与 stylelint；登录页及 AppLayout 必须先通过。
3. [AUTH-01](cards/D0/AUTH-01.md) / [AUTH-02](cards/D0/AUTH-02.md) / [AUTH-03](cards/D0/AUTH-03.md)：双模登录、httpOnly cookie、CSRF、真实错误、MFA/首登改密。
4. [BASE-02](cards/D0/BASE-02.md) / [INFRA-05](cards/D0/INFRA-05.md)：五维 RBAC 与 13 角色菜单矩阵。
5. [INFRA-04](cards/D0/INFRA-04.md) / [INFRA-08](cards/D0/INFRA-08.md)：登出、401、会话过期、多 tab 同步。
6. [BASE-06](cards/D0/BASE-06.md)：27 二级 + 5 高级路由可打开到六态空态。
7. D0-验收：13 角色登录、菜单、路由、跨租户拒绝、审计、退出、T-GATE 全绿。

D0 过闸前，任何 D1–D6 功能 PR 只允许修测试、修文档或修 D0 依赖，不允许扩新功能。

### G3 · 单卡实现闸门

每张卡必须满足：

- [ ] 功能要求 FR 与验收 AC 一一对应。
- [ ] 有红灯测试，且能证明原问题会失败。
- [ ] 后端新增公共类/方法 Javadoc 与迁移注释为简体中文。
- [ ] 页面卡有六态、1 主按钮、≤3 默认筛选、服务端分页、token 样式、中文文案、错误 traceId、真实数据来源。
- [ ] 所有外部依赖断开时诚实降级，不伪造成功。
- [ ] PR 证据可让 reviewer 不运行代码也能判断范围，但 reviewer 仍要抽查本地复跑。

### G4 · 域级验收闸门

一个域全部卡完成后，必须单独开域级验收 PR 或在最后一张卡 PR 中包含域级验收段：

- [ ] 逐页打开无 Hook 错误、空白页、404、null reference。
- [ ] 跑本域 B0 主链路 E2E。
- [ ] T-GATE、构建、类型、测试全部通过。
- [ ] 关键页面提供浏览器截图；登录域必须提供登录页、工作台、无权限、错误态截图。
- [ ] owner ≠ reviewer 签字；高风险医疗域双签。

## 5. 禁止清单

- 禁止把旧巨物当实现权威；旧巨物只可作为覆盖矩阵溯源材料。
- 禁止在业务页面用 `eslint-disable medkernel/no-page-mock` 逃避整改。
- 禁止用 `Math.random()` 生成健康分、RTT、采纳率、证据数、hash、trace 或业务 ID。
- 禁止用 UUID / 时间戳伪造数据完整性 hash；证据 hash 必须来自物理字节流，按配置使用 SM3 或 SHA-256。
- 禁止 catch 后 `message.success` 或返回成功对象掩盖失败。
- 禁止默认展示 JSON / DSL / trace / prompt；这些只能在专家模式且有权限时显示。
- 禁止模型供应商直连业务代码；所有模型能力经统一模型能力网关。
- 禁止字符 LCS、编辑距离或简单字符命中作为医学字典语义映射的充分依据；只能作为低权重召回信号，并必须经同义词典、编码交叉表、高危负样本和人工确认门禁裁决。

## 8. 完成定义

一个任务只有在以下条件同时满足时才可称为完成：

- 代码合入远程 `main`；
- 对应施工卡 AC 全部有证据；
- 自动化测试和手工页面核查都有本轮证据；
- 文档、卡、backlog、handoff 已同步；
- reviewer 按质量基线签字；
- 没有新增假闭环、无证据成功或旧口径回流。

这套方案的核心不是“写更多文档”，而是让每个 AI 在最小上下文内得到清晰边界，并被真实测试、真实页面、真实审计证据约束住。
