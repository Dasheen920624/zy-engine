# MedKernel SLA 服务等级协议

> 版本：1.0（初稿）· 2026-05-24
> 任务：GA-LEGAL-01
> 状态：法务初稿，需专业律师审核

---

## 1. 服务概述

本 SLA 适用于 MedKernel 医疗知识引擎软件的 SaaS/私有化部署服务。

## 2. 服务可用性

### 2.1 可用性承诺

| 服务等级 | 月度可用性 | 年度可用性 | 适用场景 |
|----------|-----------|-----------|----------|
| 标准版 | 99.0% | 99.0% | 非关键业务 |
| 专业版 | 99.5% | 99.5% | 临床辅助决策 |
| 旗舰版 | 99.9% | 99.9% | 核心临床系统 |

### 2.2 可用性计算

```
月度可用性 = (月度总分钟数 - 不可用分钟数) / 月度总分钟数 × 100%
```

### 2.3 不可用定义

以下情况不计入不可用时间：
- 计划内维护（提前 48 小时通知）
- 甲方网络或设备故障
- 不可抗力事件
- 甲方未按规范操作导致的故障

## 3. 响应时间

### 3.1 故障响应

| 故障级别 | 定义 | 响应时间 | 恢复时间 |
|----------|------|----------|----------|
| P0 紧急 | 系统完全不可用 | 15 分钟 | 4 小时 |
| P1 严重 | 核心功能不可用 | 30 分钟 | 8 小时 |
| P2 一般 | 非核心功能异常 | 2 小时 | 24 小时 |
| P3 轻微 | 体验性问题 | 4 小时 | 72 小时 |

### 3.2 服务时间

| 服务类型 | 服务时间 | 响应方式 |
|----------|----------|----------|
| 标准支持 | 工作日 9:00-18:00 | 电话/邮件/工单 |
| 7×12 支持 | 每日 8:00-20:00 | 电话/邮件/工单 |
| 7×24 支持 | 全天候 | 电话/邮件/工单 |

## 4. 性能指标

| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| 规则评估延迟 P95 | < 500ms | Prometheus 指标 |
| 路径操作延迟 P95 | < 1s | Prometheus 指标 |
| 页面加载时间 | < 3s | 前端性能监控 |
| API 响应时间 P95 | < 2s | APM 监控 |

## 5. 数据安全

### 5.1 数据保护承诺

| 项目 | 承诺 |
|------|------|
| 数据加密 | SM4-CBC / AES-256-GCM 字段级加密 |
| 传输加密 | TLS 1.2+ |
| 访问控制 | RBAC + 数据权限 + 组织范围 |
| 审计日志 | 180 天保留，SHA-256 链式防篡改 |
| 密钥管理 | AES-256-GCM + 密钥轮换 + 24h 宽限期 |
| 备份恢复 | 每日全量备份，30 天保留 |

### 5.2 安全合规

- 等保 2.0 三级
- 国密算法（SM2/SM3/SM4）
- 《个人信息保护法》合规
- 《数据安全法》合规

## 6. 违约赔偿

### 6.1 可用性违约

| 月度可用性 | 赔偿比例 |
|-----------|---------|
| < 99.5% 但 ≥ 99.0% | 月度服务费的 10% |
| < 99.0% 但 ≥ 95.0% | 月度服务费的 25% |
| < 95.0% | 月度服务费的 50% |

### 6.2 响应时间违约

| 违约情形 | 赔偿 |
|----------|------|
| P0 故障响应超时 | 月度服务费的 5%/次 |
| P0 故障恢复超时 | 月度服务费的 10%/次 |

### 6.3 赔偿上限

违约赔偿总额不超过甲方当月支付的服务费用。

## 7. 变更管理

7.1 乙方如需进行计划内维护，应提前 48 小时通知甲方。

7.2 紧急安全补丁可不经提前通知部署，但应在 24 小时内书面告知甲方。

7.3 版本升级应提前 30 天通知甲方，并提供升级方案和回滚预案。

## 8. 报告

8.1 乙方每月提供《服务月报》，包括：
- 可用性统计
- 故障处理记录
- 性能指标数据
- 安全事件汇总
- 变更记录

8.2 乙方每季度提供《安全评估报告》。

---

> **法务审核备注**：
> 1. 可用性目标需根据实际部署环境调整
> 2. 赔偿比例需与财务部门确认
> 3. 医疗场景的 SLA 需考虑临床安全要求
> 4. 建议增加灾难恢复（DR）条款
