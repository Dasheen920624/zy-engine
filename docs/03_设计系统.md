## 1. 设计哲学

### 1.1 五条铁律

1. **融合优先于美学** — 看起来像客户业务系统的一部分，不是独立品牌
2. **节奏优先于密度** — 不同产品节奏决定不同密度，不强求统一
3. **可信优先于美观** — 来源、版本、审核、AI 标识在任何 UI 元素之前
4. **静音优先于通知** — 不打扰是最高美德，每个通知都要付通知税
5. **可达优先于精致** — 颜色+图标+文字三重表达，无障碍是底线

### 1.2 反模式（明确禁止）

```
❌ 大面积深色侧栏（与客户业务系统不符）
❌ 大面积渐变背景
❌ 营销式 hero 首屏
❌ 装饰性插画、浮动卡片、低信息密度留白
❌ 只有颜色没有文字/图标的状态表达
❌ 用 AI 对话框替代结构化配置
❌ 用 emoji 表达功能（emoji 只用于状态标签和品牌标识）
❌ 长文案弹窗（弹窗 > 3 行必须改抽屉）
❌ 多层级嵌套弹窗（弹窗内不允许再开弹窗）
❌ 用红色（danger）表达"非紧急但需要注意"
```

## 3. 三产品密度模式

### 3.1 三产品的不同间距和字号

同样的 Token，三产品用不同的"密度模式"组合：

| Token | A 配置工厂 | B 临床嵌入器 | C 质控驾驶舱 |
|---|---|---|---|
| 表格行高 | 48px（舒展） | — | 36px（紧凑） |
| 表格字号 | `--mk-text-base` (14px) | — | `--mk-text-sm` (13px) |
| 卡片内边距 | `--mk-space-6` (24px) | `--mk-space-3` (12px) | `--mk-space-4` (16px) |
| 区块间距 | `--mk-space-7` (32px) | `--mk-space-3` (12px) | `--mk-space-4` (16px) |
| 主操作按钮高度 | 36px（标准） | 32px（紧凑） | 32px（紧凑） |
| 表单字段间距 | `--mk-space-5` (20px) | `--mk-space-3` (12px) | — |
| 关键数字字号 | `--mk-text-lg` (18px) | `--mk-text-md` (16px) | `--mk-text-2xl` (32px) |

### 3.2 落地方式

```css
/* 在容器上加 data-product 属性切换密度 */
[data-product="factory"] .mk-table { --row-height: 48px; }
[data-product="embed"]   .mk-table { /* 不用 */ }
[data-product="cockpit"] .mk-table { --row-height: 36px; }
```

React 实现：

```tsx
<AppLayout product="cockpit">
  <Dashboard />  {/* 自动继承 cockpit 密度 */}
</AppLayout>
```

### 4.2 C01 StatusBadge

**目的：** 统一全平台状态展示，强制图标+文字+颜色三重表达。

```tsx
type StatusKey = 
  | 'draft'        // 草稿 - 灰
  | 'reviewed'     // 待发布 - 蓝
  | 'published'    // 已发布 - 绿
  | 'synced'       // 已同步 - 绿浅
  | 'active'       // 运行中 - 绿
  | 'retired'      // 已下线 - 灰
  | 'rejected'     // 已拒绝 - 红
  | 'pending'      // 待审核 - 橙
  | 'processing'   // 处理中 - 蓝
  | 'error'        // 错误 - 红
  | 'missing_source' // 来源缺失 - 橙红
  | 'ai_candidate' // AI 候选 - 紫
  | 'success'      // 成功 - 绿
  | 'warning'      // 警告 - 橙
  | 'danger';      // 危急 - 红

interface StatusBadgeProps {
  status: StatusKey;
  size?: 'sm' | 'md';        // 默认 'md'
  showIcon?: boolean;        // 默认 true（强制）
  showText?: boolean;        // 默认 true（强制）
  text?: string;             // 自定义文字，否则用预设
  dotOnly?: boolean;         // 紧凑模式只显示色点（必须配 title）
  title?: string;            // 鼠标悬浮提示
}

// 使用示例
<StatusBadge status="published" />               // 默认完整
<StatusBadge status="published" size="sm" />     // 表格内紧凑
<StatusBadge status="error" text="发布失败" />   // 自定义文字
```

**强制规则：**
- `dotOnly` 模式必须带 `title`（无障碍）
- 不允许只传 `status` 不显示文字（除非 `dotOnly=true`）
- 颜色绝不能是唯一区分（图标和文字必须存在）

### 4.3 C02 SourceInfo

**目的：** 任何医学/医保/质控内容必须挂这个组件，体现不变量 #8。

```tsx
interface SourceInfoProps {
  source: {
    documentName: string;          // 如"2023 ACC/AHA AMI 指南"
    documentId: string;            // 后端 doc id
    section?: string;              // 如"§4.2"
    publishYear?: number;
  };
  citation?: {                     // 引用片段
    id: string;
    excerpt: string;               // 引用原文段落
    pageNumber?: number;
  };
  review: {
    status: 'reviewed' | 'pending' | 'rejected' | 'missing';
    reviewerName?: string;
    reviewedAt?: string;
  };
  version: string;                 // 来源版本
  variant?: 'inline' | 'card' | 'compact';   // 三种展示
  onClickDocument?: () => void;    // 点击查看原文
}

// 使用示例
<SourceInfo
  source={{ documentName: '2023 ACC/AHA AMI 指南', documentId: 'doc-001', section: '§4.2', publishYear: 2023 }}
  review={{ status: 'reviewed', reviewerName: '陈主任', reviewedAt: '2026-05-17 14:30' }}
  version="v2.1"
  variant="inline"
  onClickDocument={() => openPdf('doc-001')}
/>
```

**强制规则：**
- `review.status === 'missing'` 时必须显示橙红警告底
- 必须提供"查看原文"入口（即使可选）
- compact 模式不允许隐藏 reviewer 信息

### 4.4 C03 AiBadge

**目的：** 任何 AI 生成内容必须挂这个组件，体现不变量 #9。

```tsx
interface AiBadgeProps {

  reviewStatus: 'pending' | 'accepted' | 'rejected' | 'modified';

// card 模式：完整操作卡
<AiBadge 
  confidence={92} 
  model="Dify AMI v2.1" 
  generatedAt="..." 
  reviewStatus="pending"
  variant="card"
  onAccept={...}
  onModify={...}
  onReject={...}

```

**强制规则：**
- 必须显示 🤖 图标 + "AI 候选" 文字
- 置信度颜色按 §2.1 ai-confidence-* 自动映射
- 未审核状态不允许"采纳"，只允许"待审"
- card 模式必须提供采纳/修改/拒绝三个操作

### 4.5 C04 OrgContextSelector

**目的：** A 配置工厂、C 质控驾驶舱顶部必须有，全局组织上下文切换。

```tsx
interface OrgContext {
  tenantId: string;
  groupCode?: string;
  hospitalCode: string;
  campusCode?: string;
  siteCode?: string;
  departmentCode?: string;
}

interface OrgContextSelectorProps {
  current: OrgContext;
  allowedScopes: OrgContext[];     // 当前用户有权访问的组织范围
  onChange: (next: OrgContext) => void;
  level?: 'hospital' | 'department';  // 默认到科室级
  variant?: 'inline' | 'dropdown';

```

**强制规则：**
- 必须在 React Context 中持久化
- 切换后必须刷新所有列表（react-query invalidateQueries）
- 切换写入 sessionStorage（不写 localStorage 避免跨用户污染）

### 4.7 C06 DangerConfirm

**目的：** 危险操作三级防护。

```tsx
type DangerLevel = 'low' | 'medium' | 'high';

interface DangerConfirmProps {
  level: DangerLevel;
  title: string;
  description: string;
  consequences: string[];          // 后果列表
  confirmText?: string;            // medium/high 级需要用户输入的文字
  reasonRequired?: boolean;        // high 级必须填原因
  irreversibleNote?: string;       // high 级特殊提示，如"此操作不可撤销"
  onConfirm: (input?: { reason?: string }) => Promise<void>;
  onCancel: () => void;
}

// low：单纯确认
<DangerConfirm level="low" title="停用规则" description="..." consequences={[...]} onConfirm={...} onCancel={...} />

// medium：输入对象名确认
<DangerConfirm level="medium" title="回滚配置包" description="..." consequences={[...]} 
  confirmText="AMI_STEMI_PKG" onConfirm={...} onCancel={...} />

// high：输入对象名 + 必填原因 + "不可撤"提示
<DangerConfirm level="high" title="发布到生产环境" description="..." consequences={[...]} 
  confirmText="AMI_STEMI_PKG" reasonRequired irreversibleNote="此操作不可撤销"
  onConfirm={...} onCancel={...} />
```

**强制规则：**
- 按钮文字必须明示后果（"确认发布"而不是"确定"）
- 危险按钮必须用 danger 色 + 必须有"取消"二级按钮
- high 级必须二次确认动画延迟（防止误点）

### 4.8 C07 StepWizard

**目的：** 多步向导，所有 A 类配置/发布操作必须用。

```tsx
interface StepWizardStep {
  key: string;
  title: string;
  description?: string;
  render: (ctx: StepContext) => React.ReactNode;
  validate?: (ctx: StepContext) => Promise<ValidationResult>;
  canSkip?: boolean;
}

interface StepWizardProps {
  steps: StepWizardStep[];
  draftKey?: string;               // 用于保存草稿到 localStorage
  onComplete: (data: any) => Promise<void>;
  onCancel?: () => void;
  initialStep?: number;
}

interface ValidationResult {
  ok: boolean;
  errors?: string[];
  warnings?: string[];             // 警告不阻止下一步
  blockers?: string[];             // 阻断（如 MISSING_SOURCE）
}
```

**强制规则：**
- 步骤数控制在 3-6 步
- 每步必须有独立 validate
- blockers 必须阻止"下一步"按钮
- 草稿保存到 localStorage（key 含用户 ID + draftKey）
- URL 必须含步骤号（`?step=3`），支持刷新回到同一步

### 4.9 C08 EmptyState / C09 ErrorState

```tsx
interface EmptyStateProps {
  icon?: React.ReactNode;          // 默认 InboxOutlined
  title: string;                   // 必填，中文
  description?: string;
  action?: { text: string; onClick: () => void };
}

interface ErrorStateProps {
  errorCode?: string;              // 如 'CONFIG_NOT_FOUND'
  title: string;                   // 必填，中文
  description: string;             // 具体错误描述
  traceId?: string;                // 便于排查
  onRetry?: () => void;
}
```

**强制规则：**
- title 必填且必须中文
- 错误描述必须具体（"该字段有误"❌、"版本号只能含数字和点"✅）
- 必须提供 retry 入口（除非真的不可重试）

### 4.10 C10 AuditTrail

**目的：** 审计时间轴，A 配置工厂的详情页必须有。

```tsx
interface AuditEvent {
  id: string;
  timestamp: string;
  actor: { userId: string; displayName: string; role: string };
  action: string;                  // 如 "publish" "review_approve"
  target?: string;
  details?: Record<string, any>;
  traceId?: string;
}

interface AuditTrailProps {
  events: AuditEvent[];
  loading?: boolean;
  onLoadMore?: () => void;
  maxItems?: number;               // 默认 10，"展开更多"
}
```

### 4.11 C11 RuleDslEditor

**目的：** 规则 DSL 编辑器，双模式（可视化 / 代码）。

```tsx
interface RuleDslEditorProps {
  value: RuleDsl;
  onChange: (next: RuleDsl) => void;
  mode?: 'visual' | 'code';
  onModeChange?: (mode: 'visual' | 'code') => void;
  availableFields: FieldDef[];     // 字段补全
  availableActions: ActionDef[];
  onTest?: (input: TestInput) => Promise<TestResult>;  // 单规则测试
  readOnly?: boolean;
}

interface RuleDsl {
  when: ConditionTree;             // AND/OR/NOT
  then: ActionList;
  unless?: ConditionTree;          // 例外
}
```

**强制规则：**
- 双模式可切换，切换时校验当前内容
- 代码模式用 Monaco Editor + JSON Schema 校验
- 可视化模式拖拽 + 表单
- 必须支持"测试"按钮（dry-run）

### 4.12 C12 PathwayCanvas

**目的：** 路径模板画布，AntV X6 封装。

```tsx
interface PathwayCanvasProps {
  pathway: PathwayDef;
  mode: 'edit' | 'view' | 'diff';
  onChange?: (next: PathwayDef) => void;
  onNodeSelect?: (nodeId: string) => void;
  diffWith?: PathwayDef;           // mode='diff' 时对比版本
  readOnly?: boolean;
}
```

**强制规则：**
- 节点拖拽必须实时校验流转条件
- diff 模式必须高亮新增/删除/修改的节点
- 必须支持缩放、平移、自动布局

### 4.13 C13 EmbeddedAlert

**目的：** B 临床嵌入器的核心组件。

```tsx
type AlertSeverity = 'info' | 'warning' | 'danger' | 'success';

interface EmbeddedAlertProps {
  severity: AlertSeverity;
  title: string;                   // 短，≤ 20 字
  evidence: string;                // 命中证据，≤ 50 字
  source?: SourceInfoProps['source'];   // 来源（必须，除非系统消息）
  ruleRef?: { code: string; version: string };
  confidence?: number;             // 0-100，AI 候选时显示
  actions: Array<{
    text: string;
    intent: 'primary' | 'secondary' | 'tertiary';
    onClick: () => void;
  }>;                              // 2-3 个，不超过 3 个
  onClose?: () => void;            // 可关闭
  autoHide?: number;               // ms，自动消失（仅 success 用）
}
```

**强制规则：**
- 高度固定 ≤ 80px（不可超过医生 HIS 屏幕的 1/8）
- actions 最多 3 个
- danger 必须是医疗安全 P0 红线场景（不允许滥用）
- 必须有来源条（即便折叠）

## 5. 7 个全局模式

模式 = 跨组件的标准做法。

### 5.1 模式 P01 — 列表 + 详情 + 编辑

适用于：路径模板、规则、配置包等几乎所有 A 类对象。

```
左侧列表（45%）  ←→  右侧详情（55%，可切换全屏编辑）
↓ 单击列表项     ↓ 切换详情内容
↓ 双击或点编辑   ↓ 进入全屏编辑器（保留返回按钮）
```

### 5.2 模式 P02 — 看板 + 钻取

适用于：质控驾驶舱、医保看板。

```
KPI 卡阵列（顶）
↓
分布图 / 趋势图（中）
↓
明细表（底，可下钻）
```

### 5.3 模式 P03 — 嵌入条 + 抽屉

适用于：所有 B 临床嵌入器场景。

```
嵌入条（默认折叠态，紧凑）
↓ "查看详情"
抽屉（480px，从右滑入）
↓ 决策操作
toast 反馈（3 秒消失）
```

### 5.4 模式 P04 — 向导式发布

适用于：配置包发布、路径版本发布、批量字典映射。

```
Step 1 → Step 2 → Step 3 → ... → Confirm
每步：
  - 顶部进度条
  - 中部表单/校验/预览
  - 底部"上一步"+"下一步"+"取消"
```

### 5.5 模式 P05 — 双模式编辑器

适用于：规则 DSL、JSON 配置。

```
[可视化模式]  [代码模式]   ← 顶部切换
↓
内容区域
↓
[测试运行]  [保存草稿]  [提交审核]   ← 底部固定操作条
```

### 5.6 模式 P06 — 审核流（异步协作）

适用于：路径版本审核、规则审核、知识审核。

```
列表（待审 / 已审 / 已拒）
↓ 点列表项
详情（含变更对比 + 来源审核 + 审计轨迹）
↓
底部固定：[通过] [驳回（必填理由）] [转人工]
```

### 5.7 模式 P07 — 时间轴回放

适用于：审计日志、规则命中历史、患者路径回放。

```
左侧时间轴（垂直）
  ●─────  事件 1 - 时间 - 操作人
  │
  ●─────  事件 2 - 时间 - 操作人
  │
  ●─────  事件 3
右侧选中事件的详情面板
```

## 6. 全局规范

### 6.1 六态实现强制

任何列表页和详情页必须实现：

```tsx
function MyPage() {
  const { data, isLoading, isError, error } = useQuery(...)
  
  if (isLoading) return <PageSkeleton />                  // 1. 加载
  if (isError)   return <ErrorState ... />                // 2. 错误
  if (!hasPermission) return <NoPermissionState />        // 3. 无权限
  if (!data?.length) return <EmptyState ... />            // 4. 空
  
  return (
    <>
      {isMutating && <Spin />}                            // 5. 处理中
      {data.map(...)}
      {mutationSuccess && <Toast />}                       // 6. 成功反馈
    </>
  )
}
```

### 6.2 中文文案规范

```
状态：草稿 / 待审核 / 审核中 / 已发布 / 已同步 / 运行中 / 已停用 / 已回滚
操作：新建 / 编辑 / 保存 / 提交审核 / 通过 / 驳回 / 发布 / 回滚 / 停用 / 删除
结果：操作成功 / 保存成功 / 发布成功 / 操作失败，请重试
提示：确认要[动作][对象]吗？此操作[后果说明]
空态：暂无[对象]，[CTA 文字]
加载：加载中... / 正在[动作]...
错误：[具体错误]，[建议操作]
```

### 6.3 国际化预留

```tsx
import { useTranslation } from 'react-i18next'

const { t } = useTranslation('pathway')
<Button>{t('action.publish')}</Button>
```

资源文件结构：
```
frontend/src/locales/
├── zh-CN/
│   ├── common.json
│   ├── pathway.json
│   ├── rule.json
│   └── ...
└── en-US/  (P2 阶段补)
```

### 6.4 无障碍底线（A11Y-001）

- 颜色对比度 ≥ WCAG AA（正文 4.5:1，大字 3:1）
- 所有交互元素 keyboard accessible
- 表单字段必须有 `<label>`
- 所有图标按钮必须有 `aria-label`
- 状态色不可作为唯一信号（必须图标 + 文字）

### 6.5 性能基线

| 场景 | P95 | P99 |
|---|---|---|
| 首屏渲染 | < 1.5s | < 3s |
| 列表加载 | < 800ms | < 2s |
| 表单提交 | < 500ms | < 1.5s |
| 嵌入条出现（B） | < 1.5s | < 3s |
| 看板加载（C） | < 2s | < 5s |
| 抽屉滑入动画 | 200ms | — |
