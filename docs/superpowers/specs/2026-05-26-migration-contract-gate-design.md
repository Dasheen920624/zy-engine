# 数据迁移合同门禁设计

> 日期：2026-05-26
> 任务：`GA-ENG-BASE-05` 数据迁移门禁收口
> 决策依据：项目尚未上线，按当前 `main` 基线直接建立强门禁，不保留旧阶段口径。

## 目标

将当前已经具备的 H2、PostgreSQL、Oracle、达梦和人大金仓五套 `V1` 至 `V6` 迁移，从“文件存在且部分环境可执行”提升为可持续阻断漂移的基线合同。后续任何表族、索引、状态约束、版本字段或审计字段被误删，普通 CI 即可失败。

## 已核实边界

- 五方言当前均包含同序 `V1` 至 `V6`，覆盖组织审计、知识资产、字典映射、审计链头和身份权限表族。
- H2 可在无容器环境执行全部六条迁移；PostgreSQL 与 Oracle 由存在容器运行条件时的 smoke 执行；达梦和金仓保留静态合同门禁并进入后续国产化环境烟测。
- Oracle、达梦以 `NUMBER(1)` 表达候选冲突布尔值，需要额外 `ck_mapping_candidate_conflict` 约束；H2、PostgreSQL、金仓采用布尔类型，不要求同名约束。两者是等价业务约束，不属于漂移。
- 追加型证据或关系记录只要求创建审计信息；会被修改的主数据与权限配置必须具备 `created_at`、`created_by`、`updated_at`、`updated_by` 四列。
- 运行作业 `knowledge_export_job` 以 `requested_by`、`started_at`、`completed_at`、`expires_at` 记录作业生命周期，动作另进入审计事件；`audit_chain_head` 是每租户链头锁锚点，只保存最新更新时间。两类技术状态不伪装成可配置主数据的四列审计模型。

## 方案比较

| 方案 | 优点 | 缺点 | 结论 |
|---|---|---|---|
| 仅将 H2 断言从 `V4` 提升到 `V6` | 变更最少 | 无法阻断其他方言漏表、漏索引和漏约束 | 不采用 |
| 五数据库全部在公开 CI 执行 | 运行真实性最高 | 达梦和金仓当前缺少可用公开运行环境，无法稳定落地 | 留给 `GA-ENG-QA-02` |
| 可执行 smoke + 静态合同门禁 | 普通 CI 即可阻断结构漂移，并允许合法方言差异 | 不能替代目标数据库最终烟测 | 采用 |

## 设计

### 迁移序列门禁

新增测试读取五个迁移目录，要求每个目录恰好包含当前权威序列 `V1__init.sql` 至 `V6__security_permission_baseline.sql`。新增迁移时必须在同一 PR 更新合同、五方言脚本和任务口径，防止单方言先行。

### 结构合同门禁

测试从 SQL 文件提取具名表、索引和约束，要求：

- 五方言具有相同的 22 张业务/底座表和相同的共同索引集合。
- 五方言至少具备相同的共同状态、风险、唯一性与权限约束集合。
- Oracle、达梦额外具备数值布尔约束；其他三方言不得被迫添加该物理表达。

### 字段门禁

测试按表块检查：

- 除平台元数据表外，所有当前业务/底座表都含 `tenant_id`，确保租户隔离字段不会被遗漏。
- 会更新的主数据与安全配置表必须具有四列审计字段；运行作业和链头按上文专属审计字段执行。
- 需要状态机或版本语义的表必须存在对应 `status` 或 `version_no` 字段。

### 可执行 smoke 收紧

- H2 smoke 必须确认实际执行全部六个版本，并确认最新版本为 `6`。
- 可运行的 PostgreSQL、Oracle 与 H2 多方言 smoke 必须确认执行完整六版本，而不是仅检查 `V1` 存在。
- 达梦、金仓在本阶段受静态合同覆盖；真实运行验证仍由国产化环境验收承接。

## 验证与风险

- 运行新增迁移合同测试和现有 Flyway smoke。
- 运行后端完整测试，确认门禁未改变业务运行逻辑。
- 本任务只强化迁移合同与测试，不新增数据表、不改变已有数据库内容，因此无新增医疗数据迁移风险。
- 若合同发现现存脚本不一致，将在项目上线前直接修正当前基线迁移，并在 PR 中写明变更原因，不制造历史补丁包袱。
