# MedKernel 故障应急手册

> 版本：1.0 · 2026-05-24
> 适用：MedKernel v1.0 GA
> 受众：医院信息科运维人员

## 1. 应急响应流程

```
发现故障 → 初步判断 → 影响评估 → 应急处置 → 恢复验证 → 事后复盘
   5min       10min       15min       30min       10min       24h内
```

### 1.1 故障分级

| 级别 | 定义 | 响应时间 | 示例 |
|------|------|----------|------|
| P0 紧急 | 系统完全不可用 | 5 分钟 | 数据库宕机、服务无法启动 |
| P1 严重 | 核心功能不可用 | 15 分钟 | 规则引擎故障、登录失败 |
| P2 一般 | 非核心功能异常 | 1 小时 | 图谱查询超时、LLM 降级 |
| P3 轻微 | 体验性问题 | 4 小时 | 页面加载慢、样式异常 |

## 2. 常见故障处置

### 2.1 服务无法启动

**现象**：systemctl start medkernel 后服务立即退出

**排查步骤**：

```bash
# 1. 查看启动日志
journalctl -u medkernel --since "5 minutes ago"

# 2. 常见原因
#    - 端口被占用
ss -tlnp | grep 18080
#    - 配置文件错误
java -jar medkernel-mvp.jar --validate-config
#    - JDK 版本不匹配
java -version
#    - 磁盘空间不足
df -h /opt/medkernel

# 3. 应急处置
#    - 端口占用：kill 占用进程或修改端口
#    - 配置错误：恢复备份配置
#    - 磁盘满：清理日志/备份文件
```

### 2.2 数据库连接失败

**现象**：API 返回 500 错误，日志显示 "Connection refused" 或 "Login failed"

**排查步骤**：

```bash
# 1. 检查数据库进程
ps aux | grep mysql   # 生产
ps aux | grep h2      # 开发

# 2. 检查连接
mysql -u root -p -e "SELECT 1"   # 生产
ls -la /opt/medkernel/data/       # 开发

# 3. 检查 Actuator
curl -s http://localhost:18081/medkernel/actuator/health | jq '.components.db'

# 4. 应急处置
#    - MySQL：重启 MySQL 服务
systemctl restart mysqld
#    - H2：删除 lock 文件
rm -f /opt/medkernel/data/medkernel.lock.db
#    - 降级为内存模式
# 修改 application.yml: medkernel.database.mode=in-memory
```

### 2.3 登录失败

**现象**：用户无法登录，提示"用户名或密码错误"或"账户已锁定"

**排查步骤**：

```bash
# 1. 检查用户状态
curl -s "http://localhost:18080/api/security/admin/users?username=xxx" \
  -H "Authorization: Bearer $ADMIN_TOKEN" | jq '.locked_until'

# 2. 解锁用户
curl -X POST "http://localhost:18080/api/security/admin/users/{userId}/unlock" \
  -H "Authorization: Bearer $ADMIN_TOKEN"

# 3. 重置密码
curl -X POST "http://localhost:18080/api/security/admin/users/{userId}/reset-password" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -d '{"new_password": "TempP@ss123"}'
```

### 2.4 规则引擎无响应

**现象**：规则评估请求超时，返回 504

**排查步骤**：

```bash
# 1. 检查规则引擎状态
curl -s http://localhost:18080/api/system/providers | jq '.providers.database'

# 2. 检查规则评估延迟
curl -s http://localhost:18081/medkernel/actuator/prometheus | grep medkernel_rule_evaluation_duration

# 3. 应急处置
#    - 重启服务
systemctl restart medkernel
#    - 检查是否有死循环规则
curl -s "http://localhost:18080/api/rule-engine/rules?status=ERROR" \
  -H "Authorization: Bearer $TOKEN"
```

### 2.5 LLM 服务降级

**现象**：AI 功能返回"服务暂时不可用"

**排查步骤**：

```bash
# 1. 检查 Provider 状态
curl -s http://localhost:18080/api/system/providers | jq '.providers.model_gateway'

# 2. 检查 LLM 错误率
curl -s http://localhost:18081/medkernel/actuator/prometheus | grep medkernel_llm_call_errors

# 3. 应急处置
#    - 系统自动降级为规则引擎模式
#    - 检查 Dify/LLM 服务是否正常
#    - 如需禁用 LLM：修改 application.yml: medkernel.dify.enabled=false
```

### 2.6 磁盘空间不足

**现象**：日志报 "No space left on device"

**排查步骤**：

```bash
# 1. 检查磁盘使用
df -h
du -sh /opt/medkernel/logs/*
du -sh /opt/medkernel/backup/*

# 2. 清理日志
find /opt/medkernel/logs -name "*.log" -mtime +30 -delete

# 3. 清理旧备份
find /opt/medkernel/backup -name "*.sql" -mtime +90 -delete

# 4. 清理 Docker 资源（如有）
docker system prune -f
```

### 2.7 证书过期

**现象**：浏览器提示"证书无效"

**排查步骤**：

```bash
# 1. 检查证书有效期
openssl x509 -in /etc/nginx/ssl/medkernel.crt -noout -dates

# 2. 更新证书
cp new-cert.crt /etc/nginx/ssl/medkernel.crt
cp new-key.key /etc/nginx/ssl/medkernel.key
nginx -t && nginx -s reload
```
