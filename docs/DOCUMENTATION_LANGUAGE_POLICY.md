# 文档语言与 AI 协作规范

> 状态：当前有效规范
> 适用对象：所有开发者、AI 助手、产品、测试、实施、运维和临床评审人员

## 1. 总原则

MedKernel 的文档默认读者是中文医疗软件交付团队。为了降低交接成本、避免多 AI 产出风格分裂，当前有效文档必须使用简体中文表达，且优先使用医院、信息科、医务处、质控办和研发团队都能理解的业务语言。

## 2. 必须使用中文的内容

- 根目录、模块、部署、运维、测试、产品、设计、OpenSpec、计划、任务台账和运行手册。
- PR 说明、Issue 模板、验收清单、发布说明和变更记录。
- AI 协作规则、任务认领说明、实现计划、评审意见和交付总结。
- 页面文案、错误说明、验收口径和业务流程说明。

## 3. 可以保留英文的内容

以下内容可以保留英文，但需要在正文中用中文解释其业务含义：

- 代码标识符、类名、函数名、接口路径、配置键、数据库字段和枚举值。
- 命令、环境变量、日志片段、HTTP 状态、JSON / YAML / SQL / TypeScript / Java 代码块。
- 第三方产品名、开源项目名、标准缩写和行业约定名称，如 Docker、Dify、OpenSpec、JWT、API、DDL、SLA。
- 外部规范或官方文档标题；必要时可保留原文链接。

## 4. 新增或修改文档的门禁

提交前必须自检：

- 文档主体是否为中文。
- 英文术语是否有中文解释。
- 是否误把旧分支、旧任务锁、旧模板或旧版本说明当成当前规则。
- 是否把新的实现细节补到了权威文档或任务台账，而不是新建并行事实源。
- PR 描述是否为中文，且说明验证结果和远程 `main` 合并状态。

## 5. 历史材料处理

项目尚未正式上线，不保留历史包袱。历史归档目录、旧任务卡、旧 AI 锁、旧分支策略、旧模板和旧计划应从当前仓库移除；需要了解过往提交时使用 Git 历史，不在当前工作树中保留第二套事实源。

## 6. AI 输出要求

任何 AI 助手在本仓库工作时必须：

- 默认用中文沟通和写文档。
- 完成功能后创建 PR，等待远端检查通过，并合并到 GitHub 远程 `main`。
- 不把英文模板、英文 OpenSpec 或英文计划作为新增交付物。
- 若必须引用英文原文，先给中文结论，再给英文标识或链接。
