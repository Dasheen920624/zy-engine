# MedKernel v0.3 演示版重设计规格

> 版本：1.0 · 2026-05-21（基于 v0.2-demo 实地核查）  
> 目标：把 v0.2-demo 的「能跑但混乱」升级到「完整、自洽、符合中国国情、可面向集团医院 CIO/CMO 演示」的 v0.3-demo。  
> 适用读者：产品/设计 AI（生成 Figma 规格）+ 前后端开发 AI（实现）+ 实施工程师（演示）。
>
> **配套文档**：本规格聚焦「演示体验 + 中国国情合规 + 视觉/路径补全」；工程债时间表（KD-001~007 修复 + owner + 估时）见 [`engineering/2026-05-21-roadmap-v0.3.md`](engineering/2026-05-21-roadmap-v0.3.md)。两份互补，AI 接 PR-V3-* 前必须两份都读。

---

## 0. 为什么要重做

v0.2-demo 已经把「**能力面**」做出来了（50 Controller / 81 路由 / 6 大剧本），但「**演示面**」存在 4 类系统性问题：

| 问题类 | 实地证据 | 影响 |
|---|---|---|
| **数据浏览路径不全** | 7 个前端入口仍占位（`/adapter/hub` `/dify/workflows` `/mpi/patients` `/tenant/onboarding` `/admin/users` `/admin/audit` `/rule/definitions/:code/edit` 等），点击只显示「该功能规划中」 | 客户点进去 = 翻车 |
| **风格不统一** | Login.tsx 和 SsoLogin.tsx 同样的 LDAP 表单写了两遍；Login 用硬编码 fallback，SsoLogin 用 API；样式都用 `className="mk-login-shell"` 但内联 `style={{ background: "var(--mk-bg-secondary)", ... }}` 满天飞 | 改一处不影响另一处，永远改不齐 |
| **颜色基线不对** | tokens.css 主色 `#0b8fa6`（青蓝偏冷），与国内医疗行业主流「医蓝」`#1976d2` / 卫健委蓝 `#0066b3` 都不一致，客户演示时 PPT 配色和系统配色对不上 | CIO 觉得「不像医院系统」 |
| **不符合中国国情** | 登录页没有：① ICP/公安备案位 ② 手机号+短信验证码 ③ 国密 SM 算法选项 ④ MFA 二次验证 ⑤ 用户协议/隐私政策勾选；演示账号 `zhao01 / demo123` 直接明文写在登录卡上 | 等保三级过不了；信通院评测过不了；客户合规官第一眼否决 |

## 1. v0.3 总原则（不变量，所有 PR 必须遵守）

### 1.1 设计原则

| # | 原则 | 说明 |
|---|---|---|
| P1 | **三产品分层各有节奏** | A 配置工厂用「Notion 范」、B 临床嵌入器用「Apple Health 范」、C 质控驾驶舱用「Grafana 范」。三种密度模式（`[data-product]`）必须各自有完整样例页 |
| P2 | **任何路由必须可达完整页面** | 砍掉所有 PlaceholderPage 入口 — 没准备好的功能直接从 SideMenu 隐藏，不留「该功能规划中」 |
| P3 | **任何颜色都来自 token** | 内联 `style={{ background: "..." }}` 必须为 token；`#xxx` / `rgb()` 只能出现在 `styles/tokens.css` / `styles/tokens.ts` |
| P4 | **不写中国国情功能 ≠ 没做** | 没做的功能必须显式声明「规划中 v0.4」，不能默认假装是国际通用方案 |
| P5 | **演示账号 / Mock 数据必须按 tenant 隔离** | 任何 demo 数据走 `tenant_id=DEMO`，生产打开 `medkernel.security.demo-account-enabled=false` 直接禁用 |

### 1.2 视觉基线（强制更新 tokens.css）

| 旧值 | 新值 | 理由 |
|---|---|---|
| `--mk-brand-primary: #0b8fa6`（青蓝） | `#1565c0`（医蓝） | 国家卫健委「健康码」「医保电子凭证」均采用 `#1565c0` 系蓝色，国内医院 CIO 第一眼有信任感 |
| `--mk-brand-primary-hover: #12a8c5` | `#1976d2` | 与主色同色系，Material Design 主蓝 |
| `--mk-brand-primary-active: #066979` | `#0d47a1` | 深医蓝，按下态稳重 |
| `--mk-brand-primary-soft: #e6f8fb` | `#e3f2fd` | 信息提示底色 |
| 字体顺序：`-apple-system, ... "PingFang SC", "Microsoft YaHei"` | 中文优先：`"PingFang SC", "Microsoft YaHei", "Source Han Sans CN", -apple-system, ...` | 中文优先级前置，避免英文字体覆盖中文渲染异常 |

### 1.3 命名基线（统一）

- **产品名**：对外宣传统一称 **「集团医疗智能中枢 · MedKernel」**（CIO 友好），不再说「医疗智能引擎平台 / 治理控制台」（工程化措辞）
- **控制台名**：「治理控制台」→「管理工作台」（中文 IT 行业更熟悉）
- **菜单分组名**（4 大治理域沿用，但措辞统一）：
  - 「配置治理」→「**知识工厂**」（对应三产品 A）
  - 「运营治理」→「**质控驾驶舱**」（对应三产品 C）
  - 「用户与组织」→「**用户与组织**」
  - 「系统」→「**系统设置**」

## 2. 中国国情合规清单（强制 v0.3 落地）

### 2.1 登录页（重写 Login.tsx + 砍 SsoLogin.tsx 合并）

| # | 必须有 | 法规依据 | v0.2-demo 状态 | v0.3 要求 |
|---|---|---|---|---|
| L1 | **ICP 备案号** 显示在页脚 | 《互联网信息服务管理办法》第四条 | ❌ 无 | 配置项 `medkernel.compliance.icp-number`，登录页页脚显示「京ICP备XXXXXXXX号-X」 |
| L2 | **公安联网备案号** | 《计算机信息网络国际联网安全保护管理办法》第十一条 | ❌ 无 | 配置项 `medkernel.compliance.psb-number`，登录页页脚显示「京公网安备 11000000000000 号」 |
| L3 | **手机号 + 短信验证码** 登录 | 国内主流登录方式（95% 应用支持）；《个人信息保护法》允许的最小化身份核验 | ❌ 无 | Tab：账号密码 / 手机短信 / SSO / 域账号 共 4 种，默认手机短信 |
| L4 | **国密算法**（SM2/SM3/SM4）选项 | 《商用密码管理条例》+ 等保 2.0 三级要求 | ❌ 无 | application.yml 加 `medkernel.security.crypto-suite: SM2`（可选 RSA），登录密码前端用 SM2 加密传输 |
| L5 | **MFA 二次验证**（短信 / TOTP / U2F） | 等保 2.0 三级 §8.1.4.1 / 《关键信息基础设施安全保护条例》| ❌ 无 | 管理员账户必启用；普通账户 application.yml 可配 `medkernel.security.mfa-required-roles` |
| L6 | **用户协议 / 隐私政策勾选** | 《个人信息保护法》第十三条 + 网信办「APP 收集个人信息基本规范」 | ❌ 无 | 登录按钮上方加 `<Checkbox>` + 链接 `/legal/user-agreement` `/legal/privacy-policy`，未勾选时登录按钮 disabled |
| L7 | **图形验证码 / 滑块验证** 防暴力 | 等保 2.0 三级 §8.1.4.1 身份鉴别 | ❌ 无 | 连续 3 次失败启用，集成腾讯/阿里云行为验证 |
| L8 | **演示账号下移到合规位置** | 演示账号不能与生产 UI 混淆 | ⚠️ 「zhao01 / demo123」直接显示 | 仅在 `medkernel.profile=demo` 时展示，且放在小字提示区+灰色折叠 |
| L9 | **登录失败次数锁定** | 等保 2.0 三级 | ✅ 已有 `lock-threshold: 5` / `lock-duration-minutes: 30` | 在登录页 UI 显示「还可尝试 N 次」 |
| L10 | **密码强度策略**显示 | 等保 2.0 三级 § 8.1.4.1 | ❌ 无规则提示 | 输入密码时实时显示「需 8 位 + 含大小写 + 数字 + 特殊字符」 |
| L11 | **忘记密码 / 重置密码** 入口 | 国内基础体验 | ❌ 无 | 链接到 `/forgot-password`，走手机短信验证 |
| L12 | **会话超时 + 自动登出**提示 | 等保 2.0 三级 § 8.1.4.4 | ⚠️ 后端 JWT 已有，前端无提示 | 距离过期 2 分钟时弹「续期吗？」对话框 |

### 2.2 全局合规（非登录页）

| # | 必须有 | 法规依据 | v0.3 要求 |
|---|---|---|---|
| G1 | **页脚版权 + 版本号** | 国内 IT 行业惯例 | `© 2026 MedKernel v0.3-demo · powered by Zoesoft` |
| G2 | **数据出境提示** | 《数据出境安全评估办法》 | 涉及调用 Dify（大模型）等外部 API 时显式提示「调用 XX 服务，可能涉及数据出境」 |
| G3 | **个人信息字段最小化** | 《个人信息保护法》§6 | 患者数据所有展示页强制：身份证 4+4 脱敏、手机号 3+4 脱敏、姓名 1+* 脱敏（除非显式有「查看完整」权限） |
| G4 | **审计可溯源** | 等保 2.0 三级 § 8.1.4.3 | 所有医疗决策/规则触发必须有 `traceId`，前端在每个 AI 推荐结果下方显示「追溯 ID: traceId | 查看详情」 |

| G6 | **客服 / 故障联系方式** | 卫健委系统要求 | 全局 footer 有 `400-XXX-XXXX` + 实施工程师二维码 |
| G7 | **无障碍**（WCAG AA） | 《互联网信息服务无障碍环境建设管理办法》（2023 年生效） | 按钮对比度 4.5:1+；所有图标 `aria-label`；表单 `label htmlFor` |

## 4. 风格统一改造（前端）

### 4.1 Login.tsx + SsoLogin.tsx 合并（PR-V3-LOGIN）

当前问题：
- 两文件总共 ~400 行，60% 代码重复（SSO 列表、LDAP 表单、错误处理、handleCallback 都各写一遍）
- Login.tsx 有硬编码 SSO_PROVIDERS fallback，SsoLogin.tsx 没有 → 行为不一致
- 内联样式 11 处（`style={{ ... }}`）

v0.3 设计：
```
LoginPage (单一组件)
├─ Header（产品名 + 副标题 + 医院 logo 占位）
├─ Tabs：[ 账号密码 | 手机短信 | SSO | 域账号 ]   ← 默认 Tab 由 medkernel.login.default-method 决定
│   ├─ TabPassword：用户名 + 密码 + 验证码（>=3 次失败）+ MFA Tab
│   ├─ TabSms：手机号 + 短信验证码（60s 倒计时）+ 用户协议勾选
│   ├─ TabSso：动态拉 SsoProviders，按 type 分组渲染（CAS / OIDC / SAML / OAuth2）
│   └─ TabLdap：域账号 + 域密码 + 域名 dropdown
├─ Footer
│   ├─ 用户协议链接 | 隐私政策链接 | 忘记密码
│   ├─ ICP 备案 | 公安备案 | 版本号
│   └─ Demo 提示折叠（仅 profile=demo）
```

新文件结构：
```
frontend/src/pages/auth/
├─ LoginPage.tsx              （Tab 容器，~80 行）
├─ tabs/
│   ├─ PasswordTab.tsx        （账号密码，~60 行）
│   ├─ SmsTab.tsx             （手机短信，~80 行）
│   ├─ SsoTab.tsx             （SSO，~60 行）
│   └─ LdapTab.tsx            （域账号，~50 行）
├─ ComplianceFooter.tsx       （合规页脚，~40 行）
└─ DemoHint.tsx               （演示账号提示，~20 行）

总计 ~390 行，但每个文件单一职责，且无重复。
```

### 4.2 颜色基线迁移（PR-V3-COLOR）

操作：
1. `frontend/src/styles/tokens.css` 第 14-17 行替换为新医蓝（见 §1.2）
2. `frontend/src/styles/tokens.ts` 同步替换 hex
3. 跑 `npm run build` 看截图回归（人工对比 5 个关键页面）

### 4.3 删除所有内联样式（PR-V3-INLINE）

策略：
1. 全仓 grep `style={{` 数量（v0.2 估计 100+ 处）
2. 每处迁移到对应组件的 `.module.css` 或全局 `mk-*` 类
3. 验证：grep `style={{` 在 `frontend/src` 下 ≤ 10 处（保留 dynamic style 必须用 inline 的场景，如 `transform: rotate(${deg}deg)`）

### 4.4 加 ESLint 规则 `no-inline-style`（PR-V3-LINT）

```js
// frontend/eslint-rules/no-inline-style.js
// 禁止 JSX 使用 style={{ ... }}，除非紧邻 // eslint-disable-next-line + 理由
```
