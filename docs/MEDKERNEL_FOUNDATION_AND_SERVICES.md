## 0. 当前重启口径

“0 业务”不是只搭空框架，也不是做无业务意义的壳。它的意思是：

> **先不按试点准备、临床运行、质控改进、合规运维这些业务服务拆分团队和排期，而是先把医疗智能引擎的全能力链路做通。**

引擎全能力上线后，再把能力包装成客户能理解的业务服务。

| 阶段 | 做什么 | 不做什么 |
|---|---|---|
| 0 业务引擎全能力 | 打通知识、字典、规则、路径、推荐、评估、随访、包发布、嵌入、模型网关、审计证据和降级链路 | 不按菜单拆业务域、不做单病种硬编码、不用 mock 冒充完成 |
| 业务服务包装 | 把已上线的引擎能力组合成试点准备、临床运行、质控改进、合规运维和专业领域服务包 | 不绕过引擎另起业务实现 |

## 1. 两层模型

| 层 | 一句话 | 验收口径 |
|---|---|---|
| 基础底座 | 组织、权限、数据、版本、审计、产品体验、部署、监控和安全 | 能稳定运行、可追踪、可灰度、可回滚、可降级、可大规模操作 |
| 引擎服务能力 | 知识、字典、规则、路径、推荐、评估、随访、包发布、嵌入和模型增强 | 能端到端执行、审核、发布、同步、反馈和留证 |

业务服务不是第三套实现，只是对引擎能力的组合包装。

---

## 2. 基础底座

| 底座 | 解决什么 | 最小上线能力 |
|---|---|---|
| 组织与租户 | 多集团、多医院、多院区、多科室和专业包如何继承 | 组织上下文、继承覆盖、影响范围、租户生命周期 |
| 身份权限 | 谁能访问、审核、发布和导出 | 用户、角色、菜单权限、动作权限、数据范围、身份绑定 |
| 数据与集成 | 院内系统如何进入统一上下文 | 适配器、事件、批量、FHIR 门面、标准临床上下文 |
| 资产版本 | 知识、规则、路径、指标如何版本化 | 草稿、审核、发布、生效、下线、归档、历史重放 |
| 发布同步 | 资产如何进入院内运行 | 配置包、知识包、灰度、全量、回滚、同步日志 |
| 审计证据 | 每一步如何可追溯 | 来源、生成、审核、发布、运行、反馈、整改、回滚证据 |
| 模型治理 | AI 如何增强但不绑死业务 | 模型能力网关、结构化输出、路由策略、B0 无模型降级 |
| 运行交付 | 系统如何上线和运维 | 5 方言迁移、监控、备份恢复、内外网双形态、国产化自检 |
| 产品体验 | 用户如何在复杂系统中保持清楚、可信、低打扰 | 一页一目标、角色默认视图、大规模分页、低打扰嵌入、可信解释、体验门禁 |
| 前端外壳 | 用户如何保持简单一致 | 5+1 菜单、六态、1 主按钮、3 默认筛选、7 步流、状态机、详情抽屉、异步导出 |

## 3. 引擎服务能力

引擎能力必须先端到端走通，然后再包装业务服务。

| 引擎能力 | 核心链路 | 最小上线标准 |
|---|---|---|
| 知识资产引擎 | 来源登记 → 解析 → 去重 → 版本识别 → 审核 → 发布 → 替换旧版 | 当前有效知识唯一；海量知识支持分页、筛选、审核队列、异步导出；旧版只用于历史审计和重放 |
| 字典映射引擎 | 标准字典 → 院内字典 → 候选映射 → 冲突处理 → 发布映射包 | 未映射项、冲突项、人工确认和回滚闭环可用 |
| 规则引擎 | 规则定义 → 测试样例 → 影响分析 → 审核 → 发布 → 执行解释 | 能输入标准上下文并返回命中、风险、解释和动作 |
| 路径引擎 | 路径模板 → 专病包 → 患者入径 → 节点推进 → 变异 → 随访接续 | 支持分型、关键时钟、组织覆盖、仿真和历史回放 |
| 推荐/CDSS 引擎 | 事件触发 → 规则/路径/知识综合 → 推荐卡 → 医师反馈 | 医师确认前不进入医嘱或病历，采纳/拒绝都留痕 |
| 评估质控引擎 | 指标配置 → 病例命中 → 问题生成 → 整改 → 复核 | 指标、结果、问题、整改和证据链完整 |
| 随访引擎 | 出院/路径节点 → 计划生成 → 任务/问卷 → 异常回院 | 可生成计划并把异常事件回流评估和临床链路 |
| 包发布引擎 | 资产组合 → 校验 → 灰度 → 全量 → 同步 → 回滚 | 知识包和配置包可导入导出、同步和留证 |
| 嵌入引擎 | launch token → iframe/SDK/API → 卡片展示 → 回调反馈 | 医生工作站接入不阻断主流程，降级可审计 |
| 模型能力网关 | 能力代码 → 路由 → 脱敏 → 结构化输出 → 审计 → 降级 | 业务模块不直接绑定模型厂商或 Dify |

## 4. 业务服务包装

业务服务只在引擎全能力上线后进入实现排期。

| 服务能力 | 由哪些引擎能力组合 |
|---|---|
| 试点准备 | 组织、适配器、知识、字典、规则、路径、包发布 |
| 临床运行 | 事件、上下文、规则、路径、推荐、嵌入、反馈、随访 |
| 质控改进 | 规则、评估、质控问题、整改、医保/病案、证据 |
| 合规运维 | 身份权限、审计、数据安全、Provider、监控、备份、证据包 |
| 专业领域服务包 | 护理、报告、药事、手术麻醉输血、急重症、中医药等复用同一引擎链路 |

### 4.1 业务服务包标准结构

每个业务服务包都只是引擎能力的组合，不允许另起一套业务实现。

| 包内标准件 | 含义 |
|---|---|
| 角色 | 谁使用、谁审核、谁发布、谁负责结果 |
| 数据 | 需要哪些院内系统、标准上下文和组织范围 |
| 资产 | 来源、字典、规则、路径、指标、随访、表单和配置包 |
| 流程 | 默认工作流、7 步发布流、异常分支和降级路径 |
| 页面 | 默认视图、专家模式、分页、筛选、详情、证据入口 |
| 接口 | 同步 API、异步事件、批量任务、嵌入组件和回调 |
| 证据 | 来源、审核、发布、运行、反馈、整改、回滚和导出 |
| 验收 | B0 无模型、无 Dify、无图投影、安全边界和性能 |

### 4.2 业务医疗服务包目录

| 服务包 | 子包 | 最小业务闭环 |
|---|---|---|
| 试点准备服务包 | 租户开通、组织权限、适配器接入、数据质量、知识包、配置包、字典映射、路径/规则准备 | 新医院或新科室能完成接入、校验、灰度、全量和证据导出 |
| 临床运行服务包 | 患者主索引、患者路径、关键时钟、CDSS 提醒、规则校验、待办通知、护理/报告/床旁知识、随访触发 | 医生在院内系统内看到低打扰建议，采纳/不采纳均回流并留证 |
| 质控改进服务包 | 院级驾驶舱、质控预警、病历内涵质控、医保 DRG/DIP、评估指标、整改复核、价值指标 | 问题能发现、派单、整改、复核、统计和导出证据 |
| 合规运维服务包 | 用户身份、数据权限、审计日志、安全基线、Provider 状态、模型治理、备份恢复、证据包、离线许可 | 信息科和审计能看到系统安全、可用、可追溯、可回滚 |
| 专病路径服务包 | 胸痛/心梗、卒中、肿瘤、慢病、感染、围手术期、妇儿、急重症、基层双向转诊、中医药 | 专病能从来源到路径、规则、推荐、质控、随访和证据端到端运行 |
| 专业协同服务包 | 护理、药事、医技报告、手术麻醉输血、营养康复心理疼痛、院感公卫、科研真实世界、区域互认远程协同 | 专业团队能使用同一引擎资产和审核发布链路，不产生孤岛 |

### 4.3 业务包装启动门槛

| 门槛 | 必须满足 |
|---|---|
| 引擎能力 | E1-E5 通过，核心引擎链路真实可运行 |
| 数据契约 | 标准上下文、事件、分页、批量和证据 API 稳定 |
| 页面基线 | PageShell、六态、7 步流、详情抽屉、异步导出可复用 |
| 安全边界 | AI 候选、医师确认、高风险审核、降级和回滚已验证 |
| 代码净化 | 业务主链路不再依赖 mock、裸 Map、硬编码单病种或前端假闭环 |

## 5. AI 团队交付契约

AI 团队只按任务台账领单。每个任务必须同时交付代码、文档、测试和证据，不允许只交页面或只交说明。

| 交付物 | 最小要求 |
|---|---|
| 文档 | 更新对应权威文档或补充详细规范，不新增并行方案 |
| 后端 | DTO、校验、ApiResult/ProblemDetail、权限、审计、组织上下文、迁移和服务实现 |
| 前端 | 路由元数据、PageShell、六态、分页/筛选、详情抽屉、权限态、证据入口 |
| 数据 | 表族、索引、状态、版本、审计字段、5 方言迁移或明确不落库原因 |
| 测试 | 单测、接口契约、状态机、降级、安全边界、必要 E2E |
| 证据 | 运行截图/日志、traceId、导出样例、失败场景说明和回滚点 |

## 6. 实施顺序

| 顺序 | 目标 | 交付判断 |
|---|---|---|
| 1 | 文档与计划清场 | 只保留当前权威文档和台账，旧计划和无关入口清除 |
| 2 | 基础底座上线 | 组织、权限、上下文、审计、版本、发布、降级、监控可运行 |
| 3 | 引擎接口上线 | 十大引擎能力有 DTO、API、状态、表族、事件和 OpenAPI |
| 4 | 引擎执行上线 | 知识、字典、规则、路径、推荐、评估、随访能跑通最小端到端 |
| 5 | 发布与证据上线 | 7 步流、灰度、全量、同步、回滚和证据包全链路可用 |
| 6 | 嵌入与模型上线 | iframe/API 接入、模型网关、B0/B1/B2 降级和审计可用 |
| 7 | 引擎全能力验收 | 关闭模型、关闭 Dify、关闭图投影后，主链路仍能运行 |
| 8 | 业务服务包装 | 按试点、临床、质控、合规和专业领域包拆业务上线 |

---

## 7. 纯粹规则

| 规则 | 含义 |
|---|---|
| 先引擎，后业务 | 先把全能力引擎链路做通，再做业务菜单包装 |
| 一个事实源 | 实现细节只进唯一详细规范，不再分散建专项方案 |
| 一个状态体系 | 配置、变更、待办、告警只用 4 套状态机 |
| 一个发布流程 | 配置类能力都走 7 步流 |
| 一个模型入口 | 所有 AI 能力都经模型能力网关 |
| 一个证据链 | 来源、生成、审核、发布、运行、反馈、回滚全留证 |
| 一个体验门禁 | 全系统页面和交互遵守产品体验固定规范 |
| 一个任务台账 | AI 团队只按当前 backlog 领取、交付和验收 |
