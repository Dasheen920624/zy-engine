# 分支策略与发布管理

> 生效日期：2026-05-19
> 适用范围：本仓库所有人（用户 + 全部 AI 协作角色）

## 1. 目标

- **main 永远稳定**：任意时刻 checkout `main` 都能跑通、能部署。
- **develop 是唯一开发合入线**：所有 AI 领任务、编码、认领锁、业务提交都必须以 `develop` 为基线和合入目标。
- **多 AI 并行无冲突**：不同 AI 同时领任务时通过 claim 系统约束独占文件，不互相覆盖。
- **任意变更可回滚**：每个进入 main 的里程碑都有 `stable-*` tag。

## 2. 分支模型（简化 Git Flow）

```
main          ←── 仅接受 develop → main 的 PR（用户 approve + CI PASS）
                  每次 merge 后立即打 stable-YYYY-MM-DD-<short-hash> tag

develop       ←── 所有日常开发推到这里
                  AI 可以直接 push（小变更）或开 feature 分支 PR（大变更）
                  push 前必须本地跑通 .\scripts\verify-pr.ps1

feature/<TASK-ID>/<slug>  ←── 可选，仅高风险或大型变更使用
                              基于 develop 创建，完成后 PR → develop
```

## 3. 谁动哪个分支

| 分支 | 谁能 push | 谁能 merge | 触发 CI |
|---|---|---|---|
| `main` | 仅 GitHub merge button | 用户（你）approve PR 后 | PR + push |
| `develop` | 所有 AI、用户 | 直接 push 或 PR | push |
| `feature/<TASK-ID>/<slug>` | 该 AI 自己 | PR → develop | PR + push |

**核心规则**：
- 任何 AI **不得**在 `main` 领任务、开发、提交、push，或开业务 PR 直接合入 `main`。
- `main` 只接受用户主导的 `develop -> main` 发布 PR；其它 head 分支到 `main` 一律拒绝。
- 任何 AI 在 develop 上 push 之前必须本地 `verify-pr.ps1` FAIL=0。
- 任何 AI 在改动业务文件前必须先创建 active claim（参考 [00_总入口与AI接手导航.md](00_总入口与AI接手导航.md)）。

## 4. 分支命名规范

| 用途 | 格式 | 示例 |
|---|---|---|
| 主干 | `main` | — |
| 集成 | `develop` | — |
| 任务卡（推荐） | `ai/<TASK-ID>/<slug>` | `ai/PR-V2-01/design-tokens` |
| 高风险大型变更 | `feature/<TASK-ID>/<slug>` | `feature/PHASE-E/db-schema-migration` |
| 紧急修复 | `hotfix/<slug>` | `hotfix/ddl-encoding` |
| Claude Code 自动 worktree | `claude/<adjective>-<name>-<hash>` | 默认值；接到任务后建议 rename 为 `ai/<TASK-ID>/<slug>` |

## 5. 标准 AI 工作流（从领任务到合入）

```
1. 同步 develop 并跑接手前自检
   git fetch origin
   git checkout develop
   git pull --ff-only origin develop
   .\scripts\verify-task-prereq.ps1 -TaskId <TASK-ID> -Level <junior|middle|senior>
2. 在 docs/engineering/02_任务台账.md 找到任务编号 → TODO 改 IN_PROGRESS
3. 创建 ai-dev-input/10_task_claims/active/<TASK-ID>-S01.md（声明独占文件）
4. 同一提交创建 ai-dev-input/10_task_claims/active_locks/<TASK-ID>.lock（同任务唯一锁）
5. 立即提交并推送 claim + lock 到 develop；push 成功才算占领任务
   git add docs/engineering/02_任务台账.md ai-dev-input/10_task_claims/active/<TASK-ID>-S01.md ai-dev-input/10_task_claims/active_locks/<TASK-ID>.lock
   git commit -m "<TASK-ID>: claim task lock"
   git push origin HEAD:develop
6. 基于最新 develop 拉 worktree
   git fetch origin
   git worktree add ../wt-<TASK-ID> -b ai/<TASK-ID>/<slug> origin/develop
7. 写代码 + commit（commit message 含 TASK-ID 前缀）
8. 本地跑 .\scripts\verify-pr.ps1 -TaskId <TASK-ID>
   要求 FAIL=0；WARN 自行评估
9. push 到 develop
   小变更：git push origin HEAD:develop
   大变更/危险变更：git push -u origin ai/<TASK-ID>/<slug>，然后开 PR → develop
10. 完成后归档 claim（active → archive/YYYYMMDD/）+ 删除 active_locks/<TASK-ID>.lock + 任务台账 DONE
```

如果 `develop` 受分支保护不能直接推送，第 5 步必须先开一个只包含任务台账、active claim、task lock 的小 PR 到 `develop`；该 PR 合入成功后才允许进入业务代码开发。

禁止从 `main` 开始上述流程；如果当前分支是 `main`，`verify-task-prereq.ps1` 会直接 FAIL。

## 6. 发布流程（develop → main）

由用户主导，建议节奏：**每完成一个大 PR 或一组相关 PR 后**。

```
1. 切到 main 同步：git checkout main && git pull origin main
2. 开 PR：from develop to main（用户 approve、CI PASS）
3. Merge（采用 squash merge 或 merge commit，看变更粒度）
4. 立即打 tag：
   git tag -a stable-2026-05-19-<short> <merge-commit> -m "<summary>"
   git push origin stable-2026-05-19-<short>
5. CHANGELOG.md 加一条 release 记录
```

## 7. 回滚

```bash
# 列出所有稳定基线
git tag --list "stable-*"

# 回到某个基线（main）
git checkout main
git reset --hard stable-2026-05-19
git push --force-with-lease origin main   # 仅用户能做，需手工解锁 branch protection
```

⚠️ `--force` 推 main 是破坏性操作，仅在重大事故时使用。**优先考虑 `git revert`**：

```bash
git revert <bad-commit>      # 生成反向 commit
git push origin main         # 通过正向 commit 撤销
```

## 8. 多 AI 并行避免冲突的硬约束

| 约束 | 强制位置 |
|---|---|
| 独占文件声明 | `ai-dev-input/10_task_claims/active/<TASK-ID>-S01.md` 的 `write_scope` 字段 |
| 同任务唯一 Git 锁 | `ai-dev-input/10_task_claims/active_locks/<TASK-ID>.lock`，必须和 active claim 同提交创建，归档 claim 时同提交删除 |
| 同任务 ID 不可重复 IN_PROGRESS | [02_任务台账.md](02_任务台账.md) 第 23 行 |
| commit message 含 TASK-ID 前缀 | [PR template](../../.github/pull_request_template.md) §6 |
| 提交前禁用命名检查 | `.\scripts\verify-pr.ps1` 检查 #2.2 |
| 提交前路径断链检查 | `.\scripts\verify-pr.ps1` 检查 #2.3 |
| AI 协作冲突自检 | `.\medkernel-mvp\scripts\check-ai-collaboration.ps1` |

## 9. CI 设计

[`.github/workflows/ci.yml`](../../.github/workflows/ci.yml) 在以下事件触发：

| 事件 | jobs |
|---|---|
| push to `main` | `backend-build-test` |
| push to `develop` | `backend-build-test` |
| pull_request to `develop` | `backend-build-test` + `guard-rules` |
| pull_request to `main` | 仅允许 `develop -> main`，跑 `backend-build-test` + `guard-rules` |

`guard-rules` job 跑 `verify-pr.ps1 -SkipFrontend -SkipBackend`，仅验 ADR 不变量 / 禁用命名 / 路径断链 / UTF-8 / AI 协作。
