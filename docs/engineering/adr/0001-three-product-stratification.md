# ADR-0001: 三产品分层架构（A 配置工厂 / B 临床嵌入器 / C 质控驾驶舱）

- 状态：Accepted
- 日期：2026-05-18
- 决策者：产品 + 架构 + 用户
- 涉及范围：全平台

## 上下文

V1 阶段把项目称为"三大引擎"（路径/规则/图谱），所有功能用一套视觉规范、一套交互模式实现。V2 升级后产生新矛盾：

- 临床路径专家在"配置工厂"创作路径，需要 Notion / Figma 那样的创作感
- 主治医师在 HIS 旁边看预警，需要 Linear / Apple Health 那样的"不打扰、关键时刻显现"
- 医务质控员看院级数据，需要 Datadog / Grafana 那样的高密度可钻取

三类用户的工作节奏、信息密度、设计取向**根本不同**。强行用一套规范要么牺牲创作体验、要么牺牲信号准确性、要么牺牲数据密度。

## 决策

**平台本质是三个不同节奏的产品共用一套底座。**

| 产品 | 代号 | 用户 | 节奏 | 信息密度 | 设计取向 |
|---|---|---|---|---|---|
| A 配置工厂 | A | 医学专家、信息科 | 周/月（创作型） | 中 | Notion / Figma 范 |
| B 临床嵌入器 | B | 医生、护士、药师 | 秒/分钟（事件触发） | 极低 | Linear / Apple Health 范 |
| C 质控驾驶舱 | C | 管理层、质控、医保 | 日/周（巡检型） | 极高 | Datadog / Grafana 范 |

**共用底座：** 三大引擎 + 配置数据（字典/来源/配置包） + 基础设施（组织/身份/审计/traceId） + 设计 Token + 22 不变量。

## 不变量

- I-1：每个页面、每个功能、每个 API 必须明确归属 A/B/C 之一（或共用底座）
- I-2：三产品**不允许各自重复实现 Provider 抽象层**，必须共用
- I-3：B 临床嵌入器**可独立部署**（如 embed.js），但**必须通过同一套 API 网关**
- I-4：三产品的密度模式不同，但 **CSS Token、品牌色、字号梯度统一**
- I-5：兼具多重身份的对象（如规则），**拆成 A 视角和 B 视角两个页面/接口**，不一锅炖

## 替代方案及拒绝原因

- **全统一一套设计** → 拒绝：B 临床嵌入器塞高密度会被医生骂"打扰"；C 驾驶舱套创作密度信息量不足
- **三个独立产品** → 拒绝：用户体系、组织、审计、配置等会重复建设
- **只做 A，B 和 C 后续做** → 拒绝：B 是医生触达点，A 没有 B 就只是"高级 Word"

## 影响

正面：
- 三类用户都能得到最合适的体验
- 共用底座保证数据一致、维护成本低
- 设计有清晰边界，AI 不会越界

负面：
- 设计文档变厚（每个组件要标注密度模式）
- 前端需要支持 `data-product` 属性切换密度
- 文档说"三产品分层"时需要解释清楚

## 强制方式

- 任何页面/组件必须在 V2 [`04_页面规格书.md`](../../04_页面规格书.md) 中标注归属（A/B/C）
- [`03_设计系统.md §3`](../../03_设计系统.md) 强制密度模式 CSS 切换
- ESLint 规则 `require-product-attr.js`：根级 `<AppLayout>` 必须有 `product` prop
- 架构总图 [`05_架构总图与服务边界.md §1.1`](../05_架构总图与服务边界.md) 同步说明
