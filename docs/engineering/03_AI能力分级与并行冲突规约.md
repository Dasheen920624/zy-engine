# 03 AI 能力分级与并行冲突规约

## 0. 用途

本文把"不同能力 AI 怎样领任务、能改什么、不能改什么、撞车了怎么仲裁"做成可机判规则。目标：

- 让能力较弱的 AI **也能做出合格交付**（不出 Bug、不毁配置、不踩边界）。
- 让能力较强的 AI **不过度扩大范围**（不顺手重构、不擅自抽象）。
- 让多个 AI 并行时 **不互相覆盖**。

任何 AI 接到任务前，必须先回答：

1. 我属于哪一档能力？
2. 本任务的最低能力等级要求是多少？我够吗？
3. 本任务的"独占文件"是什么？我会不会动到它之外？
4. 我开工前，[02_任务台账.md](02_任务台账.md) 是否已经把状态改成 `IN_PROGRESS` 且 push 了？

## 1. 能力分级定义

### 1.1 初级（L1）

- 能精确执行明确的、范围窄的、不跨模块的指令。
- 不擅长设计抽象、不擅长跨服务协调、不擅长权衡折中。
- 适合：文档/样例/单测/小字段补充/接口示例。

### 1.2 中级（L2）

- 能在一个模块内完成完整功能（设计 + 实现 + 测试 + 文档）。
- 能阅读相关代码后做小范围抽象。
- 适合：单接口/单表新增、Provider 实现、单页面 + 接口对接。

### 1.3 高级（L3）

- 能跨多个模块协调（如 Oracle 持久化 + 服务 + API + 测试 + 文档）。
- 能在不破坏边界的前提下做架构内重构。
- 适合：发布生命周期改造、来源追溯底座、灰度发布、E2E + 可访问性。

## 2. 各等级允许做 / 禁止做

### 2.1 L1 初级 AI

**允许做（白名单）**

- 改 1 个模块内的 ≤ 3 个 Java 文件。
- 新增/修改 1 张样例 JSON。
- 新增/修改 1 条 API 示例（`api-examples.http`）。
- 新增/修改 1 个 smoke 脚本中的 1 段（不整体重写脚本）。
- 新增 1 个或 1 组契约测试用例。
- 修复字面错别字、文案、注释（中文）。

**禁止做（黑名单）**

- 跨模块改动（如同时改 `rule/` 和 `pathway/`）。
- 修改任何 DDL 文件。
- 修改任何 PowerShell 脚本入口（`scripts/*.ps1/*.cmd`）。
- 修改 `pom.xml` / `package.json` / CI workflow。
- 修改组织模型 / 配置包生命周期 / 来源追溯模型 / Provider 接口。
- 删除已有代码（只能增、不能删）。
- 修改其它 AI 未提交的工作。

**完成口径**

- `run-tests.ps1` PASS。
- 改动 ≤ 50 行（不含测试 + 注释）。
- 最终回复必须列出"未涉及的范围"以证明边界控制。

### 2.2 L2 中级 AI

**允许做**

- 一个模块内的完整小功能：服务类 + DTO + Controller + 测试 + 文档 + 样例。
- 一张新表 + 对应 Repository + Oracle/达梦/PostgreSQL-Kingbase/LOCAL_H2_FILE DDL + 中文备注 + 开发库验证 + 生产库 smoke 计划。
- 一个新接口及其前端调用对接（接口 + 类型 + 简单页面）。
- 一个 Provider 实现（按已有接口）。
- 修改 1 个 PowerShell 脚本但不改入口约定。

**禁止做**

- 同时改前端 + 后端（必须拆任务）。
- 修改 `ApiResult`、错误码枚举、统一审计模型等 **横切类**。
- 修改 Provider 接口签名（只能新增实现）。
- 修改组织模型五段式定义。
- 跨多个泳道协调（必须拆任务给 L3）。
- 在没有生产库 smoke 或 LOCAL_H2_FILE 开发库验证说明的情况下声称落库任务完成。

**完成口径**

- `run-tests.ps1` + `build.ps1` PASS。
- 涉及 Oracle 时 `run-oracle-org-smoke.ps1` PASS 或显式说明未验证原因。
- 改动 ≤ 800 行。
- 契约测试覆盖新接口的成功路径 + ≥ 1 个失败路径。

### 2.3 L3 高级 AI

**允许做**

- 跨多个模块的完整改造（如 PROV-001 来源底座 + 4 张表 + 多服务接入）。
- Provider 抽象演进（在保持向后兼容前提下增加方法或拆分实现）。
- 发布生命周期 / 灰度 / 回滚等架构级特性。
- 前端工程脚手架级别的搭建（FE-002）。
- 跨泳道协调（但必须拆成多个有依赖的子任务）。

**禁止做**

- 在一次提交中做"顺手重构"——任何超出任务卡范围的优化必须拆为独立任务。
- 删除其它任务的产出。
- 修改任何"红线"边界（参见 00_总入口 §4）。
- 修改 `pom.xml` 主版本号 / Spring Boot 主版本号 / JDK 主版本号。

**完成口径**

- `run-tests.ps1` + `build.ps1` + `git diff --check` PASS。
- 涉及 Oracle → smoke PASS。
- 涉及前端 → 浏览器手动验证 PASS。
- 改动 ≤ 3000 行，超过必须先拆任务。
- 必须为下一个 AI 留"下一步建议 + 风险清单"。

## 3. 任务等级如何判定

| 信号 | 判级 |
|---|---|
| 文档/样例/单测/单字段 | L1 |
| 1 个模块、1 张表、1 个接口、1 张页面 | L2 |
| 多模块、多张表、跨泳道、架构改造、发布生命周期、安全权限 | L3 |
| 涉及生产库/开发库 DDL + 持久化 + smoke | 至少 L2 |
| 涉及 Provider 接口签名 | L3 |
| 涉及前端工程脚手架 | L3 |
| 涉及 E2E + 可访问性 | L3 |

如果你不确定，**默认按更高等级处理**，并在任务卡里说明判定依据。

## 4. 并行冲突保护

### 4.1 独占文件规则

每个任务在 [02_任务台账.md](02_任务台账.md) 中声明 "独占文件 glob"。规则：

- **独占文件**：本任务唯一允许修改的文件范围；其它 AI 视为锁定。
- **共享文件**：跨任务都可能改动的文件（如 `README.md`、`docs/api-examples.http`、`07_tests/test_case_matrix.md`）。共享文件需在最终回复中显式声明本次改了哪几段，并尽量保持小幅度增量。
- **禁改文件**：除非任务专门针对（如 PKG-004），否则禁止改动：
  - `.editorconfig`、`.gitignore`、`pom.xml`
  - `.github/workflows/ci.yml`
  - `medkernel-mvp/scripts/run-tests.ps1`、`build.ps1`、`verify-encoding.ps1`
  - 已发布配置包样例（`sample_config_package.json` 在 PKG-004 完成前不再改基础结构）

### 4.2 共享文件并发约定

下表中的共享文件采用"追加 + 不重排"原则：

| 文件 | 共享方式 | 约定 |
|---|---|---|
| `README.md`（根） | 任务自由追加 | 在对应章节末尾追加，不重排已有段落 |
| `medkernel-mvp/README.md` | 任务自由追加 | 在"当前能力"列表末尾追加新能力，不重写历史项 |
| `docs/engineering/api-examples.http` | 任务自由追加 | 在文件末尾追加新接口示例 |
| `docs/01_产品事实源.md`、`docs/04_页面规格书.md`、`docs/05_AI实施手册.md` | 仅特定 DOC-V2 任务 | 改动属于 DOC-V2-*，需在最终回复说明 |
| `docs/engineering/02_任务台账.md` | 每个任务都改 | **只改自己那一行**；状态/AI/commit hash 字段先行 |
| `ai-dev-input/07_tests/test_case_matrix.md` | 任务自由追加 | 在对应章节末尾追加用例行 |

### 4.3 撞车仲裁

两个 AI 同时占领同一编号、或独占文件冲突时：

1. **先 push 的胜出**（看 git 推送时间）。
2. 后者必须立即停止，把自己的工作改为另一个未占领的任务。
3. 双方在 [02_任务台账.md](02_任务台账.md) 第 5 节"冲突仲裁记录"补一行（被让步方填写），便于复盘。
4. 如果两者都未 push 但都已 IN_PROGRESS → 由用户裁定。

### 4.4 任务交接

当一个 AI 完成本任务但任务还有"第 N+1 批"时：

- 在最终回复中显式写："下一步建议：编号 XXX 第 N+1 批，独占文件 ..."。
- 把台账中该任务的"第 N 批"标 DONE，"第 N+1 批"建 TODO 行。
- 让下一个 AI 来领。

## 5. 测试与验证最低门槛（按等级）

| 等级 | 必跑 | 强烈建议 | 涉及 Oracle 时 |
|---|---|---|---|
| L1 | run-tests | verify-encoding | — |
| L2 | run-tests + build + git diff --check | smoke 中相关脚本 | run-oracle-ddl + run-oracle-org-smoke |
| L3 | run-tests + build + git diff --check | smoke 全部 + 前端浏览器验证 + Oracle smoke | 必跑 + 编写新 smoke 段 |

## 6. 检查清单（每次提交前 self-check）

L1 AI：

- [ ] 我只改了 1 个模块吗？
- [ ] 改动 ≤ 50 行吗？
- [ ] 没动 DDL / 脚本 / pom.xml 吗？
- [ ] 没删代码吗？

L2 AI：

- [ ] 改动 ≤ 800 行吗？
- [ ] 我没同时改前端 + 后端吗？
- [ ] Oracle 相关 → smoke 跑了吗？
- [ ] 共享文件改动是"追加"还是"重排"？必须是追加。

L3 AI：

- [ ] 改动 ≤ 3000 行吗？超过有先拆任务吗？
- [ ] 我没在这次提交里"顺手优化"无关代码吗？
- [ ] 下一步建议写清楚了吗？
- [ ] 红线一条都没碰吗？

所有等级：

- [ ] [02_任务台账.md](02_任务台账.md) 已更新状态、AI 代号、commit hash？
- [ ] 最终回复符合 00_总入口 §3.4 模板？
- [ ] commit 后立即 push 了吗？

## 7. 例外处理

以下情况允许越级，但必须在最终回复中显式说明：

- **修 bug**：发现影响主流程的 bug 时，可在不扩大范围前提下顺手修；必须独立小提交。
- **补缺测试**：发现现有代码缺测试时，可补；必须独立小提交。
- **修文档错别字**：可顺手改；不计入改动行数限制。

以下情况禁止越级：

- 重构（除非任务卡明确要求）。
- 改架构边界。
- 改 Provider 接口签名。
- 删除任何文件。
