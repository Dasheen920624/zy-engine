# 09 内网部署与版本管理

## 0. 用途

本文是 `deploy/` 目录的伴随说明、版本管理约定、内网部署流程图。任何运维 AI、工程 AI、信息科实施人员在做"打包 → 部署 → 升级 → 回滚 → 灾备"时必读。

## 1. 总体流程

```
开发分支              测试 / UAT                     内网生产
─────────             ─────────                     ─────────
git push origin       构建发布包                     离线安装
    ↓                 deploy/build-release.{ps1,sh}      ↓
合并 main             ↓                              install-online.sh
    ↓                 medkernel-vX.Y.Z-<hash>.tar.gz       OR
打 tag vX.Y.Z         （含 jar + db + frontend +     install-offline.sh
    ↓                  scripts + docs + manifest）         ↓
触发 CI                ↓                              systemd 启动
    ↓                 内部 UAT 跑 4 类剧本                  ↓
产出工件               ↓                              healthcheck.sh
    ↓                 客户验收报告（04_客户验收剧本）        ↓
归档 nexus             ↓                              监控接入
                      传到院方运维通道                       ↓
                                                       灾备：upgrade.sh
                                                            rollback.sh
```

详细每一步见 §3–§7。

## 2. 版本号约定（SemVer + git）

```
MAJOR.MINOR.PATCH[-rc.N][+<gitShortHash>]
  例：1.0.0
       1.2.3-rc.1
       2.0.0+a1b2c3d
```

| 段 | 含义 | 何时变 |
|---|---|---|
| MAJOR | 不兼容变更 | 数据库表删除 / API 删除 / 字段含义变更 |
| MINOR | 向后兼容新增 | 新增模块 / 新增接口 / 新增字段 |
| PATCH | bug 修复 / 内部优化 | 不影响 API 与 DDL |
| `-rc.N` | release candidate | 发版前的预发布 |
| `+gitShortHash` | 构建标识 | 每次 `deploy/build-release` 自动注入 |

**发布 tag**：`git tag -a v1.2.3 -m "..."`，CI 监听 tag 触发完整构建。

**当前版本注入**：

- 后端：`pom.xml <version>` + `Application.properties` 注入 `git.commit.id.abbrev`（maven git-commit-id-plugin）。
- 前端：`package.json version` + Vite `define` 注入。
- `/api/system/version` 返回完整版本对象：

```json
{
  "version": "1.2.3",
  "git_hash": "a1b2c3d",
  "build_time": "2026-05-17T08:30:00+08:00",
  "spring_boot": "2.7.18",
  "jdk": "1.8.0_402"
}
```

## 5. 离线安装流程

### 5.1 准备

```bash
# 部署机：医院内网某台 Linux/Windows
# 1) 将 medkernel-v1.2.3-a1b2c3d.tar.gz 通过审批方式（U 盘 / 跳板机）传入
# 2) sha256 校验
sha256sum -c checksums.sha256 < medkernel-v1.2.3-a1b2c3d.tar.gz

# 3) 解压
mkdir -p /zoesoft/medkernel
tar -xzvf medkernel-v1.2.3-a1b2c3d.tar.gz -C /zoesoft/medkernel --strip-components=1
```

### 5.2 选择 profile

```bash
cd /zoesoft/medkernel
cp profiles/kylin-aarch64-dm.env conf/medkernel.env
vim conf/medkernel.env          # 填入数据库地址 / 用户名 / 密码 / 端口
```

### 5.3 数据库初始化

```bash
# 首次部署
./scripts/install-offline.sh --init-db

# 升级老库
./scripts/install-offline.sh --migrate-db
```

脚本会按 `MEDKERNEL_DB_DIALECT` 自动选 `db/oracle/` / `db/dm/` / `db/postgres/`。

### 5.4 启动

```bash
# 注册 systemd
sudo cp systemd/medkernel.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now medkernel
sudo systemctl status medkernel

# 健康检查
./scripts/healthcheck.sh
# [OK] /api/health 返回 200
# [OK] /api/system/providers ready=true（Database/Adapter）
# [OK] traceId 链路串通
```

### 5.5 前端 Nginx

```bash
sudo cp nginx/medkernel.conf /etc/nginx/conf.d/medkernel.conf
sudo nginx -t && sudo systemctl reload nginx
```

### 5.6 访问

```text
http://<部署机内网IP>/
```

## 6. 升级流程

```
升级窗口（医院维护时段，通常夜间 2 小时）
├─ 1) 信息科确认：影响范围、回滚目标版本、回滚耗时上限
├─ 2) sudo systemctl stop medkernel
├─ 3) 备份：./scripts/upgrade.sh --backup-only
│       → /zoesoft/medkernel.bak/<timestamp>/
├─ 4) ./scripts/upgrade.sh --to v1.3.0
│       会自动：
│       - 备份 jar / dist / db dump（如配置）
│       - 解压新包到 /zoesoft/medkernel
│       - 执行 db migration（幂等）
│       - 重启服务
├─ 5) ./scripts/healthcheck.sh
├─ 6) 跑客户验收剧本（[04_客户验收剧本]）
│       - AMI 推荐 / EMR_QC / INSURANCE_QC / ORDER_SAFETY 四类各 1 例
└─ 7) 失败 → ./scripts/rollback.sh --to v1.2.3
```

**关键约束**：

- 升级前必须备份 → `backup` 步骤失败禁止升级。
- DB migration 必须幂等可重入。
- 升级失败必须能 ≤ 5 分钟回滚到上一版本。

## 7. 回滚流程

```bash
# 自动回滚（推荐）
./scripts/rollback.sh --to v1.2.3

# 等价于：
# 1) systemctl stop medkernel
# 2) 把 /zoesoft/medkernel 内容用 /zoesoft/medkernel.bak/<旧版本> 覆盖
# 3) （可选）DB 回滚 → 一般 DB 改动是"追加列 + 兼容旧 jar"，不需回滚 DB
# 4) systemctl start medkernel
# 5) ./scripts/healthcheck.sh
```

**DB 回滚原则**：

- 默认 **不回滚 DDL**。所有迁移脚本必须"加列不删列、加表不删表"。
- 老 jar 兼容新增列（必须 nullable 或有 default）。
- 例外：若必须删列，需独立 ARCH 任务评审 + 客户书面同意 + 完整 DB 备份。

## 8. 备份策略

### 8.1 应用备份

- `upgrade.sh --backup-only` 把 `/zoesoft/medkernel/{lib,frontend,conf}` 打包到 `/zoesoft/medkernel.bak/<timestamp>/`。
- 保留 5 个版本，FIFO 清理。

### 8.2 数据库备份

由 DBA 按医院规章备份。**应用不内置 DB dump**，避免与 DBA 冲突。但提供建议：

```bash
# Oracle
expdp medkernel/$PASSWORD@$CONNECT DIRECTORY=DATA_PUMP_DIR DUMPFILE=medkernel_%U.dmp SCHEMAS=medkernel

# DM
dexp USERID=medkernel/$PASSWORD@host:5236 SCHEMAS=medkernel FILE=medkernel.dmp

# PG
pg_dump -h host -U medkernel -F c -b -v -f medkernel.backup medkernel
```

### 8.3 日志归档

`/zoesoft/medkernel/logs/` 由 logback 滚动；30 天后自动删除（[06_后端开发规范.md](06_后端开发规范.md) §6.4）。重要 traceId 应归档到 SIEM / ELK（如有）。

## 9. 灾备恢复

发布包结构本身就是灾备基线。完全裸机恢复：

```bash
# 1) 新机器安装 OS + JDK + DB
# 2) DB：从 DBA 备份恢复
# 3) 应用：./scripts/install-offline.sh --skip-init-db
# 4) 验证：./scripts/healthcheck.sh
```

灾备 RTO 目标：**4 小时**；RPO 取决于 DB 备份频率（建议 ≤ 1 小时）。

## 10. CHANGELOG 维护

`CHANGELOG.md`：

- 每次合 main 时增加一节 "Unreleased"。
- 打 tag 时把 "Unreleased" 改为 `[1.2.3] - 2026-05-17`。

## 11. 健康检查与监控

### 11.1 健康检查端点

```text
GET /medkernel/api/health
  → 200 + {success: true, code: "SUCCESS", data: {status: "UP"}}

GET /medkernel/api/system/providers
  → 200 + Provider 列表

GET /medkernel/api/system/version    （ARCH-003 后）
  → 200 + 版本信息
```

### 11.2 systemd 自愈

`medkernel.service` 已含 `Restart=on-failure RestartSec=10`，进程崩溃自动拉起。连续失败 ≥ 5 次告警。

### 11.3 监控接入（医院已有监控时）

- Zabbix / Prometheus 接 `/api/health` 与 `/api/system/providers`。
- OPS-003 落地后暴露 `/actuator/prometheus`（Micrometer）。
- 日志接 ELK / 国产 SIEM 时统一 traceId 字段名 `trace_id`。

### 11.4 SLO 定义与监控 (GA-OPS-01)

| SLO | 目标 | 计算方式 | 月度预算 |
|---|---|---|---|
| 可用性 | 99.9% | 非 5xx 请求 / 总请求 | ≤ 43.8 分钟停机 |
| 延迟 (P95) | < 300ms | histogram_quantile(0.95, ...) | 5% 请求可超 300ms |
| 错误率 | < 0.1% | 5xx 请求 / 总请求 | 0.1% 错误率 |
| 饱和度 | 连接池 < 80% | active / max connections | 20% 余量 |

**SLO 证据采集**：

1. Prometheus 自动采集 `http_server_requests_seconds_*` 和 `hikaricp_*` 指标
2. Grafana SLO 看板实时展示 SLO 达成率和预算消耗
3. SLO burn rate 告警在预算快速消耗时自动通知
4. healthcheck 脚本增强检查 actuator/health 和 Prometheus 告警状态

**告警分级**：

| 严重度 | 响应时间 | 通知方式 |
|---|---|---|
| critical | ≤ 15 分钟 | 电话 + 短信 + IM |
| warning | ≤ 1 小时 | 短信 + IM |
| info | 下一工作日 | IM |

**监控看板**：

| 看板 | UID | 说明 |
|---|---|---|
| MedKernel SLO | medkernel-slo | SLO 概览 + 错误率/延迟趋势 + 活跃告警 |
| MedKernel Security | medkernel-security | 安全概览 + 认证/权限/脱敏/审计 |

**运维检查命令**：

```bash
# 健康检查（含 actuator + SLO）
./deploy/scripts/healthcheck.sh

# Prometheus 规则校验
promtool check rules deploy/monitoring/prometheus/medkernel-alert-rules.yml
```

## 12. 安全检查（部署后）

- [ ] `application-local.yml` 已设值；密码不在 git 历史
- [ ] `/zoesoft/medkernel/conf/medkernel.env` 权限 600，owner medkernel
- [ ] 日志目录 `/zoesoft/medkernel/logs/` 权限 750
- [ ] 防火墙仅暴露 80 / 443（公网）+ 18080（仅内部）
- [ ] Nginx 启用 TLS（即使内网，建议自签证书）
- [ ] systemd 用 `User=medkernel` 非 root
- [ ] OS 安全补丁已应用（联系信息科）
- [ ] DB 用户最小权限（不给 DBA / SYSDBA）

## 13. 不允许

- 在生产机 `git clone` / `npm install` / `mvn package`（违反内网约束）
- 用 `nohup java -jar &` 替代 systemd
- 在生产机改源码 hotfix（必须经 main → tag → 发布包流程）
- 关掉 SELinux 或防火墙图省事
- 用 root 启动应用
- 在日志输出数据库密码或 API Key
- 不做备份直接升级
- 跳过 healthcheck 就交付给业务

## 14. 与现有脚本的关系

- `medkernel-mvp/scripts/*` 是 **开发期** 脚本（run-tests / build / start-memory / start-oracle / smoke）
- `deploy/scripts/*` 是 **部署期** 脚本（install / upgrade / rollback / healthcheck）
- 两者不互相依赖，但 `deploy/build-release.{ps1,sh}` 会调用 `medkernel-mvp/scripts/build.ps1`

## 15. 演进

修改本文属于 DOC / ARCH 任务。每个新 OS 支持加入时，需在 §3 发布包结构里加 `jdk/<os>-<arch>/`，并在 `deploy/profiles/` 加 env 模板。
