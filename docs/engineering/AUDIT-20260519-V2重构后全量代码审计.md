### 1.2 `any` 类型未注释（§2.1）— 必修

```
frontend/src/components/DryRunResultPanel/DryRunResultPanel.types.ts:6

frontend/src/pages/WorkflowTodos.tsx:328
```

### 1.3 Controller 使用裸 Map（§3.2 + §3.3 引申 + 06 §3 命名约定）— 必修

下列 Controller 全部使用 `Map<String, Object>` 作为 Request/Response，违反 06 §3 DTO 命名（`XxxRequest`/`XxxResponse`）与 §10.2 契约测试要求（无字段校验无法做参数校验测试）：

- `WorkflowController` 9 个方法全部 `@RequestBody Map<String, Object>`
- `AuthController.login` `@RequestBody Map<String, String>` / 返回 `Map<String, Object>`
- `WorkflowTodoService` 9 个 public 方法返回 `Map<String, Object>`

**修复建议：** 至少为 `login`、`approve`、`reject`、`delegate`、`cancel`、`addSign`、`createTodo` 7 个写接口建 DTO + `@Valid` 校验。即便接收端用 Map，Controller 入参/出参也应是 DTO。

### 1.4 JSON 字段命名策略缺失（06 §3）— 必修

规范明示：

```yaml
spring:
  jackson:
    property-naming-strategy: SNAKE_CASE
    default-property-inclusion: NON_NULL
```

实测：`application.yml` 完全无 jackson 配置；全局 grep 整个 `medkernel-mvp` 找不到 `SNAKE_CASE` / `PropertyNamingStrategy`。

**直接后果：**
- `ApiResult.traceId` Java 字段名 → 序列化后是 `traceId`，而前端 `api/types.ts` 与 `client.ts` 全部读 `trace_id`：

```ts
// client.ts:63
body.trace_id || (resp.headers["x-trace-id"] as string) || ""
```

实际拿到 `undefined`，traceId 透传链路在 envelope 层断裂；只有 `data` 内的 `trace_id` 因为 Service 自己写的字符串 key 才正确（这就是测试一直没暴露的原因，测试只断言了 `data.get("trace_id")`）。

- `SecurityFilter.java:74-76` 手写 JSON 用 `"traceId"` 反而是错的——如果 Jackson 启用 SNAKE_CASE，envelope 标准就是 `trace_id`。

**修复建议：**
1. 在 `application.yml` 加 Jackson 配置（按规范）。
2. 在 `ApiResult` 字段加 `@JsonProperty("trace_id")` 显式锁定（双保险，避免后续 Jackson 配置被改）。
3. `SecurityFilter.java` 手写的 JSON 改为 `"trace_id"`。
4. 在 `EngineApiContractTests` 新增 envelope-level `trace_id` 断言（防止再次回退）。

### 1.5 `console.warn` 留在 store（§8.1 + 07 §15）— 观察

```
frontend/src/store/orgContext.ts:21      console.warn(VITE_DEFAULT_TENANT_ID...)
frontend/src/components/StatusBadge/StatusBadge.tsx:133  console.warn([StatusBadge] Unknown status: ...)
```

### 1.6 历史命名 `zy-engine` / `ZyEngine`（§7.1）— 已收敛

### 2.2 Repository 抽象未沉淀（§2 包结构）— 应修

规范：`persistence` 是唯一允许 `import javax.sql.*` / `JdbcTemplate` 的包。

### 2.3 无连接池（§8 + §11 性能基线）— 必修

`DriverManager.getConnection` 每次都打开**新物理连接**。每次登录流程触发 ≥6 次（findByUsername、findUserRoles、findUserPermissions、findUserOrgScopes、updateLoginStatus、writeAuditLog），每次都建/断 TCP+Oracle 会话。

实测后果：
- 性能：远超 §11 `/api/health P99 < 50ms` 的余量被消耗在连接握手。
- 稳定性：高并发下会被 Oracle `processes` 上限挡掉。
- 事务：`@Transactional` 当前完全失效（每个调用都是独立连接，不在同一事务）。

**修复建议：** 引入 Spring Boot 自带的 HikariCP（已在 spring-boot-starter-jdbc 中），改 Service 注入 `DataSource`，全部替换为 `dataSource.getConnection()`。

### 2.4 异常体系不完整（§5.2）— 应修

规范要求的异常类清单：

| 类名 | 状态 |
|---|---|
| `ValidationException` | ❌ 缺失 |
| `ConfigNotFoundException` | ❌ 缺失 |
| `DataMissingException` | ❌ 缺失 |

| `AdapterTimeoutException` | ❌ 缺失 |

**修复建议：** 新建 `common/exception/` 子包，按规范创建 6 个 RuntimeException 子类（仅 5–10 行），同时扩展 `GlobalExceptionHandler` 增加分支映射。

### 2.5 SecurityExceptionHandler HTTP 状态错绑（§5.2 表格）— 必修

```java
// SecurityExceptionHandler.java:25
@ResponseStatus(HttpStatus.UNAUTHORIZED)
public ApiResult<Void> handleAuthException(AuthException ex) {
```

但 `AuthException` 在 `AuthService` 里同时被用于：
- `LOGIN_FAILED` → 应 401 ✅
- `FORBIDDEN`（"账户已禁用"）→ 实际应 403，现回 401 ❌
- `USER_LOCKED` → 实际应 423/403，现回 401 ❌
- `UNAUTHORIZED`（"未登录"）→ 401 ✅

**修复建议：** 根据 `ex.getErrorCode()` 动态选 HTTP 状态：

```java
HttpStatus status;
switch (ex.getErrorCode()) {
    case FORBIDDEN: status = HttpStatus.FORBIDDEN; break;
    case USER_LOCKED: status = HttpStatus.LOCKED; break; // 423
    default: status = HttpStatus.UNAUTHORIZED;
}
return ResponseEntity.status(status).body(ApiResult.failure(...));
```

### 2.7 时间 API 用 `new Date()`（§12.2）— 应修

`security/JwtTokenProvider.java:51, 115`：

```java
Date now = new Date();
Date expiry = new Date(now.getTime() + TOKEN_VALIDITY_MS);
```

`io.jsonwebtoken` 的 `setIssuedAt` 必须吃 `java.util.Date`，所以这是接口约束不能 100% 消除，但内部计算与比较可改 `Instant`：

```java
Instant nowInstant = Instant.now();
Date now = Date.from(nowInstant);
Date expiry = Date.from(nowInstant.plusMillis(TOKEN_VALIDITY_MS));
// validateToken:
return claims.getExpiration().toInstant().isAfter(Instant.now());
```

### 2.8 SQL 反模式（§15）— 必修

`config/ConfigPackageRepository.java:51`：

```sql
INSERT INTO cfg_config_package (...) VALUES (...)
ON DUPLICATE KEY UPDATE status = VALUES(status), ...
```

**`ON DUPLICATE KEY UPDATE` 是 MySQL 专有语法。** 项目支持的 5 个方言（Oracle / DM / PostgreSQL / KingbaseES / H2）**没有一个支持**。这条 INSERT 在生产 Oracle / DM 上跑会直接抛 `ORA-00933` 或类似错误。

**修复建议：** 改为：
- Oracle / DM：`MERGE INTO ... USING ... ON ... WHEN MATCHED ... WHEN NOT MATCHED ...`
- PostgreSQL / KingbaseES：`INSERT ... ON CONFLICT (...) DO UPDATE SET ...`
- H2：兼容 PG 语法

或更稳的做法是在 Service 层先 SELECT 再 INSERT/UPDATE 二选一（虽然有竞态，但配合 unique 约束 + 重试即可）。

`SELECT *` 也建议改为显式列名，避免列增/减时映射代码静默失败。

### 2.9 ID 生成（§8.1）— ✅ 合规

`persistence/Ids.java` 应用层生成、`@PostConstruct` 注入，无 `AUTO_INCREMENT` 依赖。✅

### 3.2 OrgContext 类型双定义 + 持久化双写不一致 — 必修

| 路径 | 字段命名 | 持久化 |
|---|---|---|
| `api/types.ts:31-38 OrgContext` | snake_case（`tenant_id`, `hospital_code`...）| `store/orgContext.ts` → localStorage key `medkernel.orgContext` |
| `components/OrgContextSelector/OrgContextSelector.types.ts:1-8 OrgContext` | camelCase（`tenantId`, `hospitalCode`...）| `OrgContextSelector.tsx:173` → sessionStorage key `mk-org-context` |

**两套字段命名 + 两套存储 key + 两套介质**——若用户用 `OrgContextSelector` 切换组织，写入 sessionStorage 的对象不会被 `store/orgContext.ts` 读到（既不是同一 key，也不是同一格式）。`http.interceptors.request.use(applyOrgHeaders)` 取的是 `store/orgContext.ts` 的旧值，导致**切换组织在 UI 上生效但请求 Header 不变**，是非常隐蔽的 Bug。

**修复建议：**
1. 删除 `OrgContextSelector.types.ts` 内的 `OrgContext`，统一引用 `api/types.ts`。
2. `OrgContextSelector.tsx` 改为调用 `setOrgContext(...)`（来自 `store/orgContext.ts`），让所有写入都走同一通道。
3. 持久化只在 store 一处做（决定用 localStorage 或 sessionStorage 一种，建议 sessionStorage 以避免跨标签污染）。
4. 加一个集成测试覆盖"切换组织 → 下一个 GET 请求带新 hospital_code Header"。

### 3.3 TodoTask 等 DTO 字段错走 camelCase（07 §4.2 + 06 §3）— 必修

```ts
// api/types.ts:388
export interface TodoTask {
  taskCode: string;
  businessType: BusinessType;
  ...
}
```

07 §4.2 明示："类型字段命名跟随后端（snake_case），不做 camelCase 转换"。

但 `TodoTask`、`TodoTaskDetail`、`TodoSummary`、`ApprovalAction`、`ApprovalRequest`、`DelegateRequest` 全部用 camelCase。后端 `WorkflowTodoService` 写出来的 Map 也是 camelCase——这是两边都违反规范的"对称错误"，所以表面看起来跑得通。

**根因：** 06 §3 要求 Jackson `SNAKE_CASE`，开启后这套都会自动断（详见 §1.4）。

**修复路径：**
- A. 全部前端改 snake_case + 后端 service 改 snake_case + 开 Jackson SNAKE_CASE（一致性最高，推荐）。
- B. 在该模块加 `@JsonNaming(PropertyNamingStrategies.LowerCamelCaseStrategy.class)` 局部豁免，并在前端类型保留 camelCase（违反规范但落地最小）。

**归属：** WF-001 self check 范围。

## 5. 安全（06 §7）

### 5.1 默认 JWT 密钥可被源码读取 — 必修

```java
// security/SecurityProperties.java:14
private String jwtSecret = "medkernel-default-jwt-secret-please-change-in-production-env-2026";
```

```yaml
# application.yml
jwt-secret: ${MEDKERNEL_JWT_SECRET:medkernel-default-jwt-secret-please-change-in-production-env-2026}
```

**风险：** 任何人 clone 此 repo 都拥有"如果运维忘配 `MEDKERNEL_JWT_SECRET` 时"的有效签名密钥，能伪造任意 user_id 的 JWT。06 §7.3 明示"密钥禁止入 Git"。

**修复建议：**
1. `SecurityProperties.jwtSecret` 字段初始值改为 `null`。
2. `JwtTokenProvider.init()` 加：
   ```java
   if (properties.getJwtSecret() == null || properties.getJwtSecret().length() < 32) {
       throw new IllegalStateException(
         "MEDKERNEL_JWT_SECRET must be set (≥32 chars) before starting in non-test profile");
   }
   ```
3. `application.yml` 删默认值。`application-test.yml`（如有）可以保留一个测试专用值，profile 隔离。

### 5.2 SecurityFilter 401 响应 JSON 不规范 — 必修

```java
// SecurityFilter.java:74-76
httpResponse.getWriter().write("{\"success\":false,\"code\":\"" + ...
    + "\",\"traceId\":\"" + traceId + "\"}");
```

- 用 `traceId` 而非 `trace_id`（同 §1.4 envelope 一致性问题）。
- 字符串拼接 + 没有转义（若 `getMessage()` 含 `"` 直接破坏 JSON）。

**修复建议：** 改用 `ObjectMapper.writeValue(httpResponse.getWriter(), ApiResult.failure(...))`。

### 5.3 登录失败重试统计的竞态 — 观察

`AuthService.login` 在密码错误分支：

```java
int attempts = user.getLoginAttempts() + 1;
persistenceService.updateLoginStatus(user.getId(), false, ip);
if (attempts >= securityProperties.getLockThreshold()) {
    persistenceService.lockUser(...);
}
```

`user.getLoginAttempts()` 是登录前快照，`updateLoginStatus` 落库时并发场景下计数会被覆盖；但锁判定用的是本地快照 +1。两个并发失败请求都会算 `attempts=2` 而不是 `2` 和 `3`。

**修复建议：** 把锁判定下放到 SQL（`UPDATE ... SET login_attempts = login_attempts + 1` 然后 `SELECT login_attempts`），或者在事务里加行锁 `SELECT ... FOR UPDATE`。

### 5.4 IP 取值未做长度/合法性校验 — 观察

`getClientIp` 直接取 `X-Forwarded-For` 第一段。攻击者可控的 Header，可能塞超长字符串。建议截断 + IP 正则校验后再入库。

### 5.5 BCryptPasswordEncoder 在构造函数 new — 应修

每个 `AuthService` 实例 new 一个 `BCryptPasswordEncoder`，并发下没事但浪费内存且不能在测试中替换。应：

```java
@Bean
public BCryptPasswordEncoder passwordEncoder() { return new BCryptPasswordEncoder(); }
```

注入到 `AuthService`。
