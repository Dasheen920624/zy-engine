## 1. P0 主线：消化 7 个已知缺陷（KD-001..007）

> 这 7 条是**演示时必须回避、试点时必须修复**的硬骨头。任何一条不修，v0.3 不能上线。

### P0-1 · KD-001 · WF-001 待办中心接通持久化

**现状**：`WorkflowTodoService` 9 个 public 方法全部返回硬编码示例数据，approveTask/rejectTask 不落库。
**影响**：演示时只能看不能点，否则用户体验崩溃。
**修复**：
- 设计 `workflow_todo_task` + `workflow_approval_action` 两张表 + 4 方言 DDL
- 实现 WorkflowTodoRepository（参考 ConfigPackageRepository 的 SELECT→UPDATE/INSERT 模式）
- WorkflowTodoService 改为真实 CRUD + 状态机（PENDING → APPROVED/REJECTED/CANCELLED/EXPIRED）
- approveTask 中检查 platform_user_id + 写 ENGINE_AUDIT_LOG
- 加 9 个契约测试（每个端点 1 个 success + 1 个 fail）
**owner candidate**: WF-001 原 owner（CodeBuddy）回头补，或新分 WF-002
**估时**：3-4 天
**验收**：演示中点"通过" → 列表里这条任务状态变 `APPROVED`，再刷新仍是 `APPROVED`；契约测试 9/9 PASS

### P0-2 · KD-002 + KD-003 · 双 Controller 副本 ADR 决策

**现状**：
- `patient/MpiController` (`/api/v1/mpi/*` 17 端点) vs `patientindex/controller/MpiController` (`/api/mpi/*` 7 端点)
- `security/UserSyncController` (`/api/security/sync` provider 模型) vs `security/usersync/UserSyncController` (`/api/user-sync` source/task 模型)

两套设计方向不同、字段不互通、台账没记录哪个是权威。

**修复**：
- 发 `ADR-0008-mpi-controller-unification.md` 决策保留哪套
- 发 `ADR-0009-usersync-controller-unification.md` 同上
- ADR 通过后：删除被弃用的那套（含 service、entity、DDL），更新所有 import + 前端 api 文件
- 数据迁移：若两套都有产线数据，写 migration script 把弃用版数据搬到保留版

**owner candidate**: 架构组（建议非 MPI/SEC 单一 owner 主导，由架构师中立决策）
**估时**：ADR 2 天 + 删/迁移 3 天 = 5 天
**验收**：`@RequestMapping` 全代码 grep 没有重复 prefix；任务台账更新 KD-002/003 已关闭

### P0-3 · KD-004 · HikariCP 连接池接入

**现状**：16 处 `DriverManager.getConnection()` 散落在 5+ 持久化类，每次请求开新物理连接，演示禁止压测。
**影响**：试点时 50 个医生并发就会打挂 Oracle session 上限。
**修复**：
- application.yml 加 `spring.datasource.hikari.*` 配置（max-pool-size、connection-timeout、leak-detection-threshold）
- 把 5 个 Repository / Service 改注入 `DataSource`，去掉 `private Connection connection()` 私有方法
- 验证：`@Transactional` 真正生效（同一 service 内多个 DAO 调用走同一 connection）
- 性能基线测试：100 个并发 / 持续 30 min，无 connection leak
**owner candidate**: REFIT-002 owner（其已认领 `com.medkernel.**`）或单开 REFIT-005
**估时**：4 天
**验收**：Hikari pool metrics 正常 + 100 并发压测无 timeout

### P0-4 · KD-005 · Jackson SNAKE_CASE 全局启用

**现状**：DTO 类用 camelCase 字段（PatientPathwayInstance.instanceId 等），Jackson 默认 camelCase 输出；前端 api/types.ts 标 snake_case 但实际收到 camelCase。
**影响**：v0.2 演示功能不受影响（前端代码读 camelCase 也能跑），但任何严格按 types 走的下游集成会断。
**修复**：
- application.yml 启用 `spring.jackson.property-naming-strategy: SNAKE_CASE`
- 全文 grep `(String) data.get("camelCaseKey")` → 改为 `snake_case_key`（影响 EngineApiContractTests 约 30 处断言）
- 前端 api/types.ts 验证所有 DTO 类型字段对得上
- 跑全套契约测试 → 通过
**owner candidate**: 后端架构 + 前端基建
**估时**：5 天（测试断言改写最耗时）
**验收**：全 248 个 test 仍 PASS + 前端 `npm run typecheck` PASS

### P0-5 · KD-006 · OrgContext 类型彻底统一

**现状**：`api/types.ts.OrgContext` (snake_case) vs `OrgContextSelector.types.ts.OrgContext` (camelCase) 两套；已修 latent header refresh bug，但类型未合并（PR-V2-06/PR-V2-11 锁 types.ts 期间没法做）。
**修复**：
- 等 PR-V2-06/11 归档（应该已完成）→ 验证 active claim 释放
- 删除 `OrgContextSelector.types.ts.OrgContext`，统一引用 `api/types.ts`
- OrgContextSelector.tsx 删 `toStoreShape` 适配函数（直接传递）
- 加集成测试："切换组织 → 下一个 GET 请求带新 X-Hospital-Code Header"
**owner candidate**: 前端架构
**估时**：1 天
**验收**：grep 全前端 `interface OrgContext` 只出现 1 次

### P0-6 · KD-007 · 5 个 PlaceholderPage 入口补真实页面

**现状**：dashboard 上 5 张卡 PENDING 状态（adapter/dify/mpi/tenant/admin）+ 菜单里另有几个 placeholderHint。点了显示 `PlaceholderPage`。
**影响**：客户演示时强迫销售跳过这些菜单，可信度受损。
**修复**：按客户优先级排：

| 入口 | v0.3 优先级 | 估时 | 后端是否就绪 |
|---|---|---|---|
| `/mpi/patients` 患者主索引管理 | **必做** | 5 天 | ✅ MPI-001 |
| `/tenant/onboarding` 租户开通向导 | **必做**（销售必演）| 4 天 | ✅ SEC-011 |
| `/admin/users` 用户管理 | 必做 | 3 天 | ✅ SEC-001..007 |
| `/admin/audit` 审计日志查询 | 必做 | 3 天 | ✅ AUDIT-001/SEC-010 |
| `/adapter/hub` 适配器中心 | v0.3 末期或 v0.4 | 4 天 | ✅ ADAPT-001/INTEROP-001 |
| `/dify/workflows` Dify 工作流 | v0.4 | 3 天 | ✅ DIFY-002 |
| `/rule/definitions` 规则库 + DSL 编辑器 | **必做**（演示重头）| 8 天 | ✅ RULE-001..008 |
| `/graph/explore` 图谱查询工作台 | v0.4 | 6 天 | ✅ GRAPH-001..005 |

**owner candidate**: 前端组 + 各业务 owner 配合
**v0.3 内必做合计**：5 个页面 × 平均 4.5 天 = ~22 天工作量（2 个前端 ~ 2 周并行）
**验收**：dashboard PENDING 卡变 READY，菜单 placeholderHint 移除
