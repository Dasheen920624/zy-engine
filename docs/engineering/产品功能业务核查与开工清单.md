## 1. 核查结论

当前产品方向可以进入 AI 自主开发阶段，但必须按“业务闭环优先”推进，不能只堆接口或只补底层表。

本次业务核查结论：

```text
产品定位：通过。
核心用户角色：通过。
集团化/多院区业务模型：通过。
配置发布治理：通过，但需继续补 Oracle 持久化、回滚、跨环境迁移。
来源追溯和医学可信：方向通过，属于最高优先级待完善底座。
规则、路径、标准化、适配器、图谱、Dify 边界：通过。
外网/SaaS 基准服务定位：方向通过，但当前 MVP 只能做受控演示、知识包运营和脱敏协作，不能裸露公网承载真实患者数据。
客户可视化验收：方向通过，但前端工程和 UAT 剧本仍需尽快落地。
质控/医保/医嘱安全业务包：方向通过，需补样例规则、来源、报告和复核闭环。
生产试点能力：未完成，需权限、安全、运维、部署、监控、回滚继续建设。
```

最终判断：

```text
可以让 AI 开始干活。
第一阶段不要随机扩展功能，必须围绕 P0/P1 客户验收闭环推进。
```

## 2. 产品一句话

本产品不是单个“路径推荐接口”，而是面向集团化医院和多院区部署的医疗智能引擎平台：

```text
用 Oracle/关系型数据库治理配置、版本、审计和运行记录；
用规则、路径、图谱、标准化和 Dify 工作流支撑临床、质控、医保和运维场景；
用来源追溯、配置发布、组织隔离和前端验收工作台保证可解释、可审核、可交付。
```

## 3. 产品边界

### 3.1 产品必须做

- 支持集团、医院、院区、卫生所/站点、科室多级组织。
- 支持配置资产版本化、审核、发布、回滚和审计。
- 支持路径推荐、医生确认入径、节点流转和变异记录。
- 支持规则导入、发布、模拟、执行、批量评估和结果回查。
- 支持病历质控、医保质控、医嘱安全、用药适应症、检查合理性等规则场景。
- 支持字典标准化和未映射治理。
- 支持适配器定义、Mock 验证和组织绑定。
- 支持图谱版本、节点、边、证据和 Neo4j 可选投影。
- 支持 Dify 工作流契约、降级输出、调用统计和回放。
- 支持来源文档、引用片段、资产绑定、来源审核、发布阻断和运行证据链。
- 支持 DB-only 环境完整验收，不能强依赖 Neo4j、Dify 或公司内网 Oracle。
- 支持前端配置、演示校验、规则 dry-run、质控看板和运维追溯。
- 支持外网基准知识包和客户租户开通能力，但真实患者数据必须优先留在院内/专网实例。

### 3.2 产品当前不应做

- 不直接替代医生做最终诊疗决策。
- 不把 AI/Dify 输出作为未审核的医疗结论。
- 不直接连接客户生产库做不可逆操作。
- 不把 Neo4j 或 Dify 当作配置主数据源。
- 不把单个医院、院区、科室或厂商逻辑硬编码进核心代码。
- 不做没有来源、没有审核、没有版本、没有审计的医学/医保/质控配置发布。
- 不把演示数据和正式患者数据混用。
- 不把未完成强认证、MFA、租户隔离、审计和网关/WAF 的 MVP 直接暴露到公网。

## 4. 用户角色和业务目标

| 角色 | 核心目标 | 必须可完成的业务动作 | 验收口径 |
|---|---|---|---|
| 医生 | 看懂推荐并保留最终决策权 | 查看路径推荐、命中原因、来源摘要；确认入径；覆盖或忽略建议 | 医生知道为什么推荐、能确认/覆盖、能记录原因 |
| 临床专家 | 审核路径、规则和医学来源 | 审核规则/路径来源、适用人群、排除条件、证据等级 | 高风险规则有审核人、来源和适用范围 |
| 质控办 | 发现问题并推动整改 | 运行病历质控、路径质控、查看缺项/时限/变异、导出报告 | 质控问题可复核、可追溯、可整改 |
| 医保办 | 发现费用和政策风险 | 运行医保质控、查看政策依据、费用风险、复核状态 | 政策版本清楚、申诉材料有据 |
| 信息科 | 稳定部署和治理配置 | 导入配置包、review、publish、rollback、看 Provider 状态和审计 | 发布可审计、失败可定位、回滚可执行 |
| 实施工程师 | 完成院内适配 | 导入组织、字典映射、适配器定义、演示数据和 dry-run | 不改代码也能完成基础配置和验证 |
| 院领导 | 看管理价值 | 查看路径执行、质控缺陷、医保风险、科室排行和趋势 | 能看出风险变化、科室差异和管理抓手 |
| 第三方系统 | 标准调用规则能力 | 调用 `/rule-engine/evaluate`、批量评估、回查结果 | 接口稳定、错误可读、trace 可追踪 |

## 5. 必须打通的业务闭环

### 5.1 配置发布闭环

```text
导入配置
-> 校验
-> 查看 diff/hash/manifest
-> 来源完整性检查
-> dry-run
-> 审核
-> 发布
-> 激活
-> 审计
-> 回滚
```

AI 开发重点：

- 配置包生产库（Oracle/达梦/PostgreSQL-Kingbase）与开发库（LOCAL_H2_FILE）持久化。
- active 指针和回滚。
- 跨环境导入导出。
- 前端配置包中心。

### 5.2 来源可信闭环

```text
来源文档
-> 引用片段
-> 资产绑定
-> 来源审核
-> 发布阻断
-> 运行结果展示来源
-> 来源更新影响分析
```

AI 开发重点：

- `SRC_DOCUMENT`、`SRC_CITATION`、`SRC_ASSET_BINDING`、`SRC_REVIEW_RECORD`、`SRC_RUNTIME_EVIDENCE`。
- 缺来源、来源过期、未审核来源不得发布。
- 规则命中、路径推荐、质控结论能展示来源摘要。

### 5.3 临床路径闭环

```text
患者事实
-> 标准化
-> 候选路径
-> 规则/图谱证据
-> 医生确认
-> 节点任务
-> 变异记录
-> 质控指标
```

AI 开发重点：

- 路径运行组织上下文。
- 路径配置来源字段。
- 节点完成率、滞留时长、变异分析。
- 医生覆盖/忽略原因。

### 5.4 病历质控闭环

```text
病历/诊疗事实
-> 标准化
-> 质控规则评估
-> 缺陷清单
-> 来源依据
-> 整改建议
-> 复核状态
-> 报告导出
```

AI 开发重点：

- `EMR_QC` 样例规则包。
- 结果持久化和复核状态。
- 报告导出和前端质控工作台。

### 5.5 医保质控闭环

```text
诊断/医嘱/费用
-> 医保规则评估
-> 风险金额/政策依据
-> 申诉材料建议
-> 人工复核
-> 结果归档
```

AI 开发重点：

- `INSURANCE_QC` 样例规则包。
- 医保政策来源版本。
- 风险金额、复核状态和导出报告。

### 5.6 医嘱安全闭环

```text
医嘱原文
-> 标准化
-> 禁忌/重复/适应症/检查合理性规则
-> 提醒/强提醒/阻断/需确认
-> 医生处理
-> 审计和复盘
```

AI 开发重点：

- `ORDER_SAFETY`、`DRUG_INDICATION`、`EXAM_RATIONALITY` 场景。
- action_mode 和 clinical_severity。
- 医生覆盖原因和提醒疲劳分析。

### 5.7 运维交付闭环

```text
环境检测
-> 数据库模式识别
-> 本地 H2 或 Oracle 启动
-> Provider 状态
-> smoke 验证
-> 日志/审计/trace
-> 升级回滚
```

AI 开发重点：

- `OPS-001` 初始化检查。
- Oracle/达梦升级和回滚手册。
- Provider 状态页。
- 监控指标和同步任务。

## 7. AI 开工后的业务优先级

后续 AI 自主开发时，业务优先级按以下顺序：

1. `PROV-001/002/003/004` 来源追溯和发布阻断闭环。
2. `PKG-004/005/006` 配置包持久化、回滚和跨环境迁移。
3. `ORG-003/004/005` 组织持久化、继承覆盖、配置包组织覆盖。
4. `RULE-005/008` 规则来源字段和评估结果持久化。
5. `TEST-001` 组织、来源、发布、数据库模式回归矩阵。
6. `FE-002/003/004` 前端工程、演示校验、配置包中心。
7. `RULE-006/006B/006C` 病历质控、医保质控、医嘱安全规则包。
8. `TERM-001/002/003` 标准化、未映射治理和版本影响。
9. `QC-001/002/003/004/005` 质控和医保看板、报告导出。
10. `SEC-001/002/003` 角色、权限、签名和防重放。
11. `OPS-001/002/003/004` 初始化、同步任务、监控、离线部署和回滚。

## 11. AI 开工红线

后续 AI 开工时禁止：

- 只实现接口但不说明业务角色和验收场景。
- 只做内存演示但不维护 Oracle/达梦/PostgreSQL-Kingbase/LOCAL_H2_FILE 结构。
- 做规则、质控、医保、医嘱安全但不绑定来源。
- 做发布、回滚、同步但不写审计。
- 做前端页面但不支持空态、错误态、组织上下文和 traceId。
- 做 Dify 解释但脱离规则、来源和人工确认。
- 直接使用真实患者隐私做测试样例。

## 12. 最终开工建议

可以开工，但第一批 AI 不要分散。

建议第一轮自主开发只允许从以下方向领取任务：

```text
PROV 来源追溯底座
PKG 配置包持久化和回滚
ORG 组织持久化和继承覆盖
RULE 规则来源和结果持久化
TEST 业务验收矩阵
FE 演示校验和配置包中心
```

如果多个 AI 同时开工，推荐分工：

```text
AI-1 数据库/来源：PROV-001/002/003/004
AI-2 后端/配置：PKG-004/005/006
AI-3 后端/组织：ORG-003/004/005
AI-4 规则/测试：RULE-005/008 + TEST-001
AI-5 前端/验收：FE-002/003/004 + TEST-003
AI-6 运维/安全：OPS-001/005 + SEC-001/002
```

每个 AI 必须按自主运行、任务认领、数据库识别、质量门禁执行。
