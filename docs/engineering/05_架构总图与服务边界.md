# 05 架构总图与服务边界

## 0. 用途

本文是项目 **系统架构总纲**。所有后端、前端、DBA、运维 AI 都必须先理解本文的分层、Provider 抽象、跨数据库/跨 OS/跨 CPU 适配位置，再去看具体模块。

冲突仲裁：业务边界冲突 → [`../01_产品事实源.md`](../01_产品事实源.md)；架构边界冲突 → 本文为准。

## 1. 一句话

四层架构（接入 / 应用 / Provider / 数据）+ 五段组织上下文 + 六类资产配置包 + 七类目标部署环境（OS × CPU × DB 组合）+ 全链路 traceId + **三产品分层（A 配置工厂 / B 临床嵌入器 / C 质控驾驶舱）**。

## 1.1 三产品分层与本文的关系（V2 体系，2026-05-18 引入）

V2 体系把平台明确为**三个不同节奏的产品共用一套底座**。本文（架构总图）只讲底座；三产品的定义、用户、节奏、设计取向见 [`../01_产品事实源.md §3`](../01_产品事实源.md#3-三产品分层v2-核心决策)。

三产品在架构上的位置：

| 产品 | 部署位置 | 占用哪些应用层模块 | 占用哪些 Provider |
|---|---|---|---|
| **A 配置工厂** | 浏览器 + 平台主 Web | pathway / rule / graph / terminology / config / aik / security | DatabaseProvider / SourceProvider / AuditProvider |
| **B 临床嵌入器** | HIS/EMR iframe 或独立 Web | rule（实时评估）/ pathway（流转）/ adapter | AdapterProvider / WorkflowProvider / NotifyProvider |
| **C 质控驾驶舱** | 平台主 Web | quality / audit + 聚合查询 | DatabaseProvider（只读）/ AuditProvider |

**架构层面的关键约束：**

1. 三产品**不允许各自重复实现 Provider 抽象层**，必须共用 §3 的 Provider 接口
2. B 临床嵌入器**可独立部署**（如打成 embed.js 给 HIS 嵌入），但**必须通过同一套 API 网关**
3. C 质控驾驶舱**只读查询**为主，必须支持读写分离（生产库主从）的能力（可后续接入）
4. 三产品的前端**密度模式不同**（详见 [`../03_设计系统.md §3`](../03_设计系统.md#3-三产品密度模式)），但**视觉 Token 统一**

详细的应用层模块清单见本文 §3。

## 3. 五段组织上下文

```
集团 group ─┬─ 医院 hospital ─┬─ 院区 campus ─┬─ 卫生所/站点 site ─┬─ 科室 department
            │                  │                │                    │
            └ tenant_id 在最外层（同一系统部署可承载多个集团 / 商务实体）
```

- 配置继承顺序：科室 > 站点 > 院区 > 医院 > 集团 > 系统内置默认（`PLATFORM/DEFAULT`，不是真实组织）
- 上下文 Header 全集：`X-Tenant-Id / X-Group-Code / X-Hospital-Code / X-Campus-Code / X-Site-Code / X-Department-Code`，Body 同名字段优先级最高（Body > Query > Header > Default > None）
- 所有运行实例、配置、审计、日志必须落 `tenant_id + 5 段` 9 列结构化字段
- `org_source` 取值：`HEADER / QUERY / BODY / DEFAULT / NONE`

## 5. Provider 抽象（核心）

Provider 层是系统能在 **不同数据库 / 不同图谱 / 不同 AI 网关 / 不同医院信息系统** 之间切换的关键。**任何业务服务都不允许直接 import 具体实现**，必须通过 Provider 接口注入。

### 5.1 DatabaseProvider（关系型）

```java
// 概念示意，落地见 src/main/java/com/medkernel/persistence
public interface DatabaseProvider {
    String dialect();                  // "oracle" / "dm" / "postgres" / "kingbase"
    boolean ready();
    void executeMigration(Path script);
    <T> List<T> query(String sql, RowMapper<T> mapper, Object... args);
    int update(String sql, Object... args);
    // 方言差异：clob 大字段、分页 syntax、序列/IDENTITY、时间类型、布尔类型
}
```

方言差异处理位置：

| 差异点 | 处理位置 | 备注 |
|---|---|---|
| 主键 ID 生成 | **应用层生成**（雪花/UUID）| 禁止依赖 AUTO_INCREMENT / SEQUENCE，避免方言锁死 |
| 分页 | `LimitOffsetClause` 工具类 | Oracle `OFFSET … ROWS FETCH …` / 达梦 `LIMIT … OFFSET` / PG `LIMIT … OFFSET` |
| CLOB / TEXT 大字段 | DAO 层判定类型 | Oracle `CLOB`、达梦 `CLOB`、PG `TEXT`、Kingbase `CLOB` |
| 布尔 | 统一用 `CHAR(1) Y/N` | Oracle 无原生 boolean；PG 用 boolean 但 mapper 翻译 |
| 时间戳 | `TIMESTAMP WITH TIME ZONE` 优先 | 涉及跨时区时统一存 UTC，展示层加时区 |
| 中文字符集 | 库级 AL32UTF8 / UTF8 / GB18030 | 见 §8 与 [编码与中文备注规范.md](编码与中文备注规范.md) |

### 5.2 GraphProvider

```text
Neo4jProvider          → 内网 Neo4j 5.x 可用时
DBFallbackGraphProvider → 把 Oracle/达梦/PG 中 GE_* 表当图查询源
HeuristicGraphProvider  → DB-only 启发式兜底（如 AMI 内置规则）
```

切换由 `medkernel.graph.enabled` 与运行时探测决定，**关闭 Neo4j 后 `/api/graph/*` 仍可返回有意义结果**。

### 5.3 WorkflowProvider（AI）

```text
DifyProvider           → 内网 Dify 部署
LocalFallbackProvider  → 按 workflow_code 返回 degraded_outputs 模板
```

切换由 `medkernel.dify.enabled` 决定。

### 5.4 AdapterProvider（院内外部系统）

每家医院的 HIS / EMR / LIS / PACS / 医保接口都不同。Adapter 层把"协议差异"沉淀为**可配置的查询定义**：

```text
ADP_ADAPTER_DEF：base_url / auth_type / timeout_ms / retry / 组织绑定
ADP_QUERY_DEF：query_code / SQL / REST path / 字段映射 / mock 行集
```

业务服务调用：`adapterProvider.query(adapter_code, query_code, params) → List<Map>`。

新医院接入只改 `ADP_*` 表 + 写测试，不改 Java 代码。

## 9. 内网环境约束

医院内网通常具备以下"反默认"特征，所有设计必须接受：

1. **无公网访问**：禁止运行时 `curl github.com`、`npm install` 拉取、`docker pull`。所有依赖必须预打入发布包或私有镜像源。
2. **离线 Maven**：`medkernel-mvp/m2repo/` 已纳入 .gitignore，但发布包必须自带"离线 Maven 镜像 tar"或预热的本地仓库。
3. **离线 npm**：前端构建必须可指向私有 verdaccio / nexus；`frontend/.npmrc.offline` 模板见批次 3。
4. **HTTPS 证书**：内网可能用自签证书；Java 需 `cacerts` 导入，前端 `axios` 需 `rejectUnauthorized: false`（仅内网）。
5. **代理穿透**：部分医院走 squid 代理上网；JVM 用 `-Dhttp.proxyHost`，需在启动脚本中预留参数。
6. **端口受限**：常见允许 80 / 443 / 8080 / 18080；其它端口需走 OA 提单。**默认端口必须可在 `application.yml` 改**，不要硬编码。
7. **时间同步**：内网 NTP 可能漂移；所有审计字段写"应用端 + 数据库端"双时间戳，便于排查。
8. **磁盘配额**：日志和审计可能很快撑爆 /var；日志必须 rolling + 压缩 + 保留天数可配。
9. **第三方系统跨网段**：HIS/EMR 常在不同 VLAN，适配器需配独立超时、独立重试、独立熔断。
10. **运维窗口**：医院"维护窗口"通常只有夜间 2 小时；升级、回滚、迁移脚本必须 **幂等可重入**。

## 10. 服务边界（再次明确）

| 模块 | 必须做 | 禁止做 |
|---|---|---|
| 路径引擎 | 候选/入径/节点流转/变异 | 直接读 HIS / 写 Cypher / 调 Dify prompt 字面拼接 |
| 规则引擎 | DSL 解析/执行/日志/包发布 | 写患者状态 / 直接查 Oracle 业务表 |
| 图谱引擎 | 候选召回/证据查询/版本 | 暴露任意 Cypher / 让外部传 Cypher |
| Dify 工作流 | 编排 + 生成解释 | 保存核心业务状态 |
| 标准化中心 | 字典映射 + 标准码 | 在业务代码硬编码院内文本 |
| 适配器中心 | 协议封装 + 字段映射 | 让外部传任意 SQL |
| 配置发布 | review / publish / 审计 / 回滚 | 静默覆盖已发布版本 |
| 审计 | 写 ENGINE_AUDIT_LOG + traceId 串联 | 输出密码/密钥/患者全量隐私 |
| 前端 | 配置 + 演示 + 校验可视化 | 替代后端权限 / 保存敏感凭据 / 直接 dry-run 写正式状态 |

## 11. 全链路 traceId

```
浏览器（fe-{ts}-{rand}）→ Nginx（X-Trace-Id）→ Spring Boot Filter → 业务方法 MDC →
  ├─ DB SQL 注释 /* trace=... */
  ├─ Provider 调用 Header
  ├─ Audit 日志
  └─ 异常堆栈 first line

排错路径：前端 Devtools Network → 后端日志 grep → DB 审计表 LIKE → 第三方系统日志
```

任何 Provider 调用都必须把 traceId 透传，第三方接口失败时 traceId 可定位到具体患者/包/版本。
