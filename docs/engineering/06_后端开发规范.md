# 06 后端开发规范

## 0. 用途

本文是 `medkernel-mvp` 后端开发的 **强制规范**。任何 L1–L3 AI 写后端代码前必须读。审查代码也以本文为准。

冲突仲裁：架构边界 → [05_架构总图与服务边界.md](05_架构总图与服务边界.md)；完成口径 → [00_总入口](00_总入口与AI接手导航.md) §3；本文规范本身为最具体的执行细则。

## 1. 技术栈固化

| 项 | 选型 | 不可改 |
|---|---|---|
| JDK | 1.8（兼容 OpenJDK / Bisheng / Temurin / Oracle JDK 8）| ✅ |
| 构建 | Maven 3.6+ | ✅ |
| 框架 | Spring Boot 2.7.x（不升 3.x，3.x 需 JDK 17）| ✅ |
| Web | Spring MVC（非 WebFlux）| ✅ |
| 序列化 | Jackson（默认）| ✅ |
| 测试 | JUnit 5 + Spring Boot Test + MockMvc | ✅ |
| 数据库驱动 | ojdbc8 + DmJdbcDriver18 + postgresql + kingbase8 jdbc | 按需启用 |
| 日志 | SLF4J + Logback | ✅ |
| 校验 | jakarta.validation（javax 兼容）| ✅ |

> 升级主版本属于架构变更，需独立 ARCH 任务讨论。

## 2. 包结构（com.medkernel）

```
com.medkernel
├── MedKernelApplication            主入口
├── common                          ApiResult / 错误码 / 校验 / 通用 DTO
├── system                          /api/system/* 健康 / Provider 状态 / 组织上下文入口
├── organization                    组织目录 + OrganizationContextService
├── config                          配置包（PKG-xxx）
├── provenance                      来源追溯（PROV-xxx，批次 1 落地后）
├── pathway                         路径引擎（PATH-xxx）
├── rule                            规则引擎内部配置（/api/rules）
│   └── engine                      第三方规则引擎入口（/api/rule-engine）—— 与配置层解耦
├── graph                           图谱引擎（GRAPH-xxx）
├── dify                            Dify/AI 工作流（DIFY-xxx）
├── terminology                     标准化中心（TERM-xxx）
├── adapter                         适配器中心（ADAPT-xxx）
├── quality                         质控聚合（QC-xxx）
├── audit                           ENGINE_AUDIT_LOG
├── security                        鉴权 / 签名 / 数据权限（SEC-xxx）
├── persistence                     DAO + 方言 + Repository
├── dto                             跨模块共享 DTO（小心，慎用）
└── util                            通用工具
```

**包级别强约束**：

- 业务包（path/rule/graph/...）**不允许** 跨包 import 对方的 internal 类。需要协作时通过事件、Provider 或 dto 包。
- `persistence` 是唯一允许 `import javax.sql.*` / `JdbcTemplate` 的包；其它包必须通过 Repository。
- `common` / `util` **不允许** 反向依赖业务包。

## 3. 命名约定

| 类型 | 命名 | 示例 |
|---|---|---|
| Controller | `XxxController` | `RuleEngineController` |
| Service 接口 | `XxxService` | `RuleEngineService` |
| Service 实现 | `XxxServiceImpl` 或单实现直接命名 | `RuleEngineService` 含 `@Service` |
| Repository 接口 | `XxxRepository` | `ConfigPackageRepository` |
| DTO | `XxxRequest` / `XxxResponse` / `XxxView` | `EvaluateRequest` |
| 实体 | `XxxDefinition` / `XxxRecord` / `XxxEntry` | `RuleDefinition` |
| 枚举 | `XxxType` / `XxxStatus` | `ScenarioCode` |
| 异常 | `XxxException`（继承 `RuntimeException`）| `ConfigNotFoundException` |
| Provider 接口 | `XxxProvider` | `GraphProvider` |
| Provider 实现 | `XxxNeo4jProvider` / `XxxDbFallbackProvider` | `Neo4jGraphProvider` |

### 字段命名

- Java 字段：`camelCase`
- JSON 字段：`snake_case`（与 ai-dev-input 样例一致）
- 数据库列：`SNAKE_CASE`（大写 + 下划线）
- URL 路径：kebab-case（`/api/rule-engine`、`/api/config-packages`）
- 包名：全小写（`com.medkernel.ruleengine`，但已有 `rule.engine` 子包，保持现有约定）

Jackson 配置在 `application.yml`：

```yaml
spring:
  jackson:
    property-naming-strategy: SNAKE_CASE
    default-property-inclusion: NON_NULL
    serialization:
      write-dates-as-timestamps: false
```

## 4. API 设计规范

### 4.1 URL

```
GET    /api/{resource}                     列表（分页）
GET    /api/{resource}/{id}                详情
POST   /api/{resource}                     创建 / 批量导入
PUT    /api/{resource}/{id}                整体更新（少用）
POST   /api/{resource}/{id}/{action}       状态动作（review/publish/rollback/sync/export）
DELETE /api/{resource}/{id}                删除（极少用，倾向 RETIRED 软停用）
```

- 资源名用 kebab-case 复数：`/config-packages`、`/rule-engine`、`/patient-pathways`
- 动作用动词：`review` / `publish` / `rollback` / `export` / `evaluate` / `simulate`
- 嵌套资源最多 2 层：`/patient-pathways/{id}/nodes/{nodeCode}/complete`

### 4.2 ApiResult 统一返回

所有接口（含异常分支）必须返回：

```json
{
  "success": true,
  "code": "SUCCESS",
  "message": "OK",
  "data": { ... },
  "trace_id": "fe-1f4d-...-server-2c9a"
}
```

错误码集（**唯一来源 [00_总入口](00_总入口与AI接手导航.md) §9**）：

```
SUCCESS                 业务成功
VALIDATION_ERROR        参数校验失败、组织校验失败、hash 不一致
DATA_MISSING            必要数据缺失（如 patient.facts 没有 arrival_time）
CONFIG_NOT_FOUND        配置/版本不存在
ENGINE_TIMEOUT          引擎执行超时
ADAPTER_TIMEOUT         第三方适配器超时
DIFY_TIMEOUT            Dify 超时（含降级）
DB_ERROR                数据库异常
UNKNOWN_ERROR           兜底
NO_RULES_MATCHED        规则评估时无匹配规则（warning，不抛异常）
PENDING_MAPPING         标准化未映射
MISSING_SOURCE          缺来源（PROV-004 阻断发布）
```

**新增错误码必须在 [00_总入口](00_总入口与AI接手导航.md) §9 登记**，禁止业务模块私自扩展。

### 4.3 traceId

- 入口 Filter 读 `X-Trace-Id` Header；无则生成 `srv-{timestamp}-{rand}`。
- 写入 `MDC.put("traceId", ...)`，Logback `%X{traceId}` 输出。
- 响应头回写 `X-Trace-Id`。
- ApiResult.trace_id 与 MDC 同源。

### 4.4 分页

统一 query 参数：

```
?limit=100        默认 100，最大 1000
&offset=0         默认 0
&order_by=created_time
&order=desc       desc / asc
```

返回结构（`data` 字段为分页 envelope 时）：

```json
{
  "items": [...],
  "total": 1234,
  "limit": 100,
  "offset": 0
}
```

### 4.5 组织上下文

所有列表 / 评估 / 配置查询接口必须支持组织上下文过滤：

- Header / Query / Body 三方合并（Body 优先）
- Header 全集见 [05_架构总图](05_架构总图与服务边界.md) §3
- 评估、运行实例、审计写入时落 9 段字段 + `org_source`

工具类：`OrganizationContextService.resolveWithBody(headers, query, body)` → `OrganizationContext`。

## 5. 异常处理

### 5.1 异常分类

```java
// 业务可恢复异常（用户能修复）
throw new ValidationException("hash 不一致", "VALIDATION_ERROR");

// 配置缺失（系统状态问题）
throw new ConfigNotFoundException("PKG_AMI_RULE_CONFIG", "2026.05.01");

// 外部依赖问题（可重试或降级）
throw new EngineTimeoutException("Dify call timeout after 30s");
```

### 5.2 全局 ExceptionHandler

由 `@RestControllerAdvice GlobalExceptionHandler` 统一捕获，转换为 ApiResult：

| 异常类型 | code | HTTP 状态 |
|---|---|---|
| `ValidationException` / `MethodArgumentNotValidException` | `VALIDATION_ERROR` | 400 |
| `ConfigNotFoundException` | `CONFIG_NOT_FOUND` | 404 |
| `DataMissingException` | `DATA_MISSING` | 400 |
| `EngineTimeoutException` | `ENGINE_TIMEOUT` | 504 |
| `AdapterTimeoutException` | `ADAPTER_TIMEOUT` | 504 |
| `DifyTimeoutException` | `DIFY_TIMEOUT` | 504 |
| `DataAccessException` / SQL 异常 | `DB_ERROR` | 500 |
| 其它 `RuntimeException` | `UNKNOWN_ERROR` | 500 |

异常 message 必须**面向用户可读**，且**不泄露**：SQL、堆栈、内部表名、密钥、患者隐私明文。

### 5.3 不允许吞异常

禁止 `catch (Exception e) { /* ignore */ }`。最少 `log.warn("...", e)` + 必要降级返回。

## 6. 日志规范

### 6.1 级别

| 级别 | 用法 |
|---|---|
| `TRACE` | 详细函数入参出参；生产关闭 |
| `DEBUG` | 中间状态；UAT 开 |
| `INFO` | 关键业务事件（入径、发布、评估、降级）|
| `WARN` | 可恢复异常、降级、外部超时 |
| `ERROR` | 不可恢复异常、需告警 |

### 6.2 格式

`logback-spring.xml` 模板：

```xml
<pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level [%X{traceId:-}] %logger{36} - %msg%n</pattern>
```

每行必须含 `traceId`，便于 grep。

### 6.3 脱敏

**禁止输出**：

- 密码、API Key、Token、JWT、Session ID
- 数据库连接字符串中的密码段
- 患者身份证号全量、手机号全量、住院号原文（可输出后 4 位）
- Dify API Key、HIS/EMR/LIS 接口凭据
- 完整患者上下文（可输出 `patient_id` + 摘要字段）

工具类：`SensitiveMasker.maskIdNumber(id)` / `maskPhone(p)` 等，统一调用。

### 6.4 滚动与保留

`application.yml`：

```yaml
logging:
  file:
    path: ${MEDKERNEL_LOG_PATH:./logs}
  logback:
    rollingpolicy:
      max-file-size: 100MB
      max-history: 30
      total-size-cap: 10GB
      file-name-pattern: "${LOG_PATH}/medkernel.%d{yyyy-MM-dd}.%i.log.gz"
```

内网磁盘紧张时可调小 `total-size-cap`。

## 7. 安全规范

### 7.1 OWASP Top 10 防护

| 风险 | 防护位置 |
|---|---|
| Injection（SQL / Cypher）| 全部用 `PreparedStatement` / Cypher 参数化；外部输入禁止拼 SQL；GraphProvider 不暴露 raw query |
| Broken Authentication | SEC-001 用户角色 + SEC-003 API 签名 + nonce + 时间戳 |
| Sensitive Data Exposure | §6.3 脱敏 + TLS（即使内网建议启用） |
| XML External Entities | Jackson 默认禁用；如解析 XML 用 `XMLInputFactory` 显式 disable XXE |
| Broken Access Control | SEC-002 数据权限按组织过滤；前端权限只是体验 |
| Security Misconfiguration | actuator 端点限制；默认密码禁用 |
| XSS | 前端 React 默认转义；后端不返回 raw HTML |
| Insecure Deserialization | 仅信任 Jackson；不使用 Java 原生 readObject |
| Components with Known Vulnerabilities | 定期 `mvn dependency:tree` 扫描；CVE 跟踪 |
| Insufficient Logging | §6 强制 traceId + 关键操作审计 |

### 7.2 第三方调用

`/api/rule-engine/*` 等对外暴露接口必须（SEC-003）：

- Header `X-Api-Key`：客户端身份
- Header `X-Signature`：HMAC-SHA256(method + path + body + timestamp + nonce, secret)
- Header `X-Timestamp`：±5 分钟容忍
- Header `X-Nonce`：内存 10 分钟去重

签名失败、时间超界、nonce 重复 → `VALIDATION_ERROR` + 审计写入。

### 7.3 密钥管理

- 数据库密码、Dify API Key、HIS 接入凭据 **禁止入 Git**。
- 通过环境变量传入（`MEDKERNEL_DB_PASSWORD` 等）。
- 内网生产环境使用 OS 级 secrets 文件（Linux `chmod 600`），启动脚本读取后注入。
- 日志、API 响应、审计都不能回显凭据原文。

## 8. 数据库访问规范

### 8.1 ID 生成

**应用层生成**，禁止依赖 `AUTO_INCREMENT` / `SEQUENCE`。原因：方言锁死 + 集群难。

实现：

```java
// util/IdGenerator.java
public static String next(String prefix) {
    return prefix + "-" + Long.toUnsignedString(System.currentTimeMillis(), 36)
                  + "-" + ThreadLocalRandom.current().nextInt(0x10000);
}
// 例：PKG-l8z2x-3a7f
```

### 8.2 方言差异处理

不要写 `if (oracle) ... else if (dm) ...`。把差异封装在：

- `PersistenceDialect` 接口（每方言一个实现）
- DDL 文件分目录（`db/oracle/` / `db/dm/` / `db/postgres/`）
- 用 JPA / MyBatis 等 ORM 时配置方言

### 8.3 CLOB / TEXT

JSON 大字段统一 CLOB（Oracle/达梦/Kingbase）/ TEXT（PG）。**查询关键字段必须结构化为独立列**（可索引）。

读取：

```java
// 避免 ResultSet.getString 在 Oracle 11g 上返回 null
String json = rs.getString("MANIFEST_JSON");
if (json == null) {
    Clob clob = rs.getClob("MANIFEST_JSON");
    if (clob != null) {
        json = clob.getSubString(1, (int) clob.length());
    }
}
```

### 8.4 事务

- 写操作必须 `@Transactional`。
- 默认 `Propagation.REQUIRED`，特殊场景显式说明。
- 跨表更新放在 Service 层，DAO 不开事务。
- **禁止**在事务内调用外部 HTTP / Dify / Neo4j（事务不会回滚远程副作用，且会拉长持锁时间）。

### 8.5 DDL 与迁移

- DDL 必须**幂等可重入**：
  - 建表用 `BEGIN ... EXCEPTION WHEN OTHERS THEN NULL; END;`（Oracle）或 `IF NOT EXISTS`（PG）。
  - 加列前先 `SELECT column_name FROM user_tab_columns ...` 判断。
- 迁移脚本命名：`medkernel_<purpose>_migration.sql`，可重复执行。
- 每条 DDL 同步 `db/oracle/`、`db/dm/`（批次 2 后）、`db/postgres/`（批次 2 后）。
- 中文备注用 `UNISTR` 注入（已有 `medkernel_comments_unistr.sql` 模式）。
- 表名 ≤ 30 字符（Oracle 11g 约束），字段名 ≤ 30 字符。

## 10. 测试规范

### 10.1 测试金字塔

```
契约测试（MockMvc）         ←  必须覆盖所有 Controller 接口
└─ 服务/领域单元测试         ←  规则 DSL / 配置 hash / 标准化等纯逻辑
   └─ smoke 脚本             ←  PowerShell + curl，跑真实进程
      └─ Oracle smoke         ←  涉及 DDL / 落库的任务
         └─ UAT 剧本           ←  非开发人员可执行（[04_客户验收剧本](04_客户验收剧本与报告模板.md)）
```

### 10.2 契约测试硬性要求

每个 Controller 必须有：

1. **成功路径**：返回 200 + `success=true` + `data` 结构正确。
2. **至少 1 个失败路径**：参数缺失 / 组织不存在 / 配置版本不匹配。
3. **traceId 透出**：Header 与 ApiResult.trace_id 一致。
4. **组织上下文**：传不同 hospital_code 返回不同结果或正确过滤。

测试类命名：`{Module}ApiContractTests`，统一在 `src/test/java/com/medkernel/`。

### 10.3 Oracle smoke

涉及 DDL / 持久化 / 落库的任务必须扩展 `scripts/run-oracle-org-smoke.cmd`，校验：

- DDL 已建表 / 加列。
- API 写入后能从 Oracle SQLPlus 读到。
- 组织字段全部正确写入。

### 10.4 禁止

- 长期 `@Disabled` / `@Ignore` 不解释。
- mock 掉所有外部，然后说"测试通过"。
- 用内存测试代替真实 Oracle 落库（仅限不涉及 DB 的任务）。

## 11. 性能基线

| 指标 | 目标 | 测量方式 |
|---|---|---|
| `/api/health` | P99 < 50ms | 启动后 ab/wrk |
| 规则评估单条 | P99 < 200ms | RULE-008 持久化后 metrics |
| 路径候选推荐 | P99 < 500ms | 含规则 + 图谱降级 |
| 配置包导入（10 条规则）| < 2s | smoke 计时 |

超过目标必须有 issue 跟踪或 OPS-003 监控告警。

## 12. 代码质量约束

### 12.1 静态约束

- 类长度：单文件 ≤ 600 行。超过须拆分。
- 方法长度：单方法 ≤ 50 行（getter/setter/case 块除外）。
- 圈复杂度：单方法 ≤ 10。
- 参数个数：单方法 ≤ 5，超过用 Request DTO 包装。
- 嵌套：if/for 嵌套 ≤ 3 层。

### 12.2 禁用

- `System.out.println` / `e.printStackTrace()`（用 SLF4J）
- `new Date()` / `Calendar.getInstance()`（用 `java.time` LocalDateTime / Instant）
- `SimpleDateFormat`（线程不安全，用 `DateTimeFormatter`）
- `Random`（用 `ThreadLocalRandom`；密码相关用 `SecureRandom`）
- `synchronized` 围在大方法上（粒度过粗，改用 `ConcurrentHashMap` / `ReentrantLock`）
- 反射调用业务方法（除非 Spring 框架本身需要）

### 12.3 注释

- 公共 API（Controller / Service 接口）必须 Javadoc。
- 复杂算法、为什么这样写、踩过什么坑：写注释。
- "做了什么"（代码自解释）：不写注释。
- TODO 必须带 issue 编号或日期 + 计划。

### 12.4 中文备注

- DDL 表 / 列备注、JSON sample 中可读字段（`message_template` / `note`）必须中文。
- 接口 / 字段命名、枚举值必须英文。

## 13. Git 与提交

### 13.1 单任务单 commit

- 一个 commit 解决一个任务编号。
- 跨任务的混合改动 → 拆 commit。

### 13.2 提交信息

格式：`动词 + 模块 + 能力`，中文短句。

```
增加规则医学来源字段
补充病历质控规则包样例
修正配置包发布回滚审计字段
```

禁止：

```
update                ← 无信息
WIP                   ← 不应提交进 main
fix bug               ← 无信息
update some files     ← 无信息
```

### 13.3 推送

完成立即 `git push origin claude/<branch>:main`（按 memory `feedback_auto_push_main`）。

## 14. 模板：新增一个 Controller 的清单

新增一个 `/api/foo/bar` 接口，必须同步：

- [ ] `FooController.bar()` 方法 + Javadoc
- [ ] `FooService.bar()` 接口 + 实现
- [ ] 请求 DTO `BarRequest` + 校验注解
- [ ] 响应 DTO `BarResponse` 或共享 View
- [ ] `FooApiContractTests.testBar_success` + `testBar_validationError` 至少 2 条
- [ ] `ai-dev-input/06_samples/sample_bar.json`
- [ ] `medkernel-mvp/docs/api-examples.http` 末尾追加示例
- [ ] `medkernel-mvp/README.md` "当前能力"列表末尾追加一行
- [ ] 涉及 DB → 同步生产库 DDL（Oracle/达梦/PostgreSQL-Kingbase）和开发库 DDL（LOCAL_H2_FILE）
- [ ] 涉及 DB → `run-oracle-org-smoke.cmd` 扩展校验段
- [ ] 涉及组织上下文 → 实现 Header/Query/Body 合并 + 落 9 段字段
- [ ] 涉及来源追溯 → 调用 PROV-004 阻断检查
- [ ] 涉及审计 → 写 `ENGINE_AUDIT_LOG`

## 15. 反模式（出现即驳回）

- `if (hospital.equals("ZYHOSPITAL")) ...` — 硬编码医院
- `String sql = "SELECT * FROM ... WHERE name='" + userInput + "'"` — SQL 拼接
- `throw new RuntimeException("xxx")` 顶层抛 — 应该用具体异常类
- `return null;` 当数据缺失 — 用具体异常或 `Optional`
- `// TODO 后面再做` — 必须带 issue + 计划
- `private static List<String> hospitals = Arrays.asList("HOSP_A", "HOSP_B");` — 配置写死
- `restTemplate.exchange(url, ..., String.class)` 不带超时 — 必须显式 timeout
- `@RestController` 同时含业务逻辑 — 必须委托 Service
