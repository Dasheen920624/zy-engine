# 业务实现范围核查方案

> 版本：1.0 · 2026-05-31
> 状态：研发重启范围闸门
> 定位：本文用于核查“到底要实现哪些业务、页面、接口、B0 主链路和 AI 加深能力”。它不替代施工卡，只定义范围核查方法和当前发现。

---

## 1. 核查结论

业务实现范围必须按三层锁定：

1. **业务场景锁**：S0–S40 共 41 个业务场景，每个场景必须映射到至少一张施工卡。
2. **菜单页面锁**：5+1 一级菜单、27 个客户二级菜单 + 5 个高级工具不得净增；每个可见页面必须有页面卡或被明确并入引擎卡。
3. **交付链路锁**：D0–D6 第一波 B0 主链路先真实跑通，wave2 只在 B0 之后加深，GA 最后用 A1–A9 统一验收。

当前代码与文档只能说明“范围已有卡体系承接”，不能说明“业务已经实现”。所有状态仍以 `pending` 为默认，必须逐卡举证。

## 2. 当前范围盘点

### 2.1 场景覆盖

- `docs/cards/_index.md` 当前有 S0–S40 共 **41 行场景映射**。
- 其中 S0–S16 主要由 D0–D6 第一波承接，S17–S40 主要由 wave2 领域门面、首发知识资产和 D3/D4 基线能力承接。
- 这只证明“有卡承接”，不证明“实现完成”。

### 2.3 覆盖矩阵风险

`docs/cards/_coverage-matrix.md` 仍有大量 `待迁`，包括：

- S17–S40 旧详规场景锚点；
- 全医疗专业知识领域覆盖矩阵；
- 病历内涵质控复杂规则；
- 模型能力、AI 工厂、电子病历评级、最终验收等横切章节；
- 落地规划、FOUNDATION、旧体验规范的多处总览锚点。

因此：**不得宣称 P8 巨物退役已完成**，也不得宣称业务范围“零遗漏”。当前只能说：卡索引已有承接路径，覆盖矩阵仍需回填审计。

## 3. 范围核查闸门

### S-GATE-1 · 场景到卡

每个 S0–S40 必须满足：

- [ ] `_index.md` 有卡链接；
- [ ] 链接文件存在；
- [ ] 卡内有该场景的功能要求与验收；
- [ ] 若该场景被多个卡共同承接，必须说明主卡和辅助卡，避免多源冲突。

### S-GATE-2 · 菜单到页面

每个客户二级菜单和高级工具必须满足：

- [ ] 菜单项在核心 §2 固定树内；
- [ ] 有页面卡或合并说明；
- [ ] 路由元数据包含 sectionKey / menuKey / requiredPermissions / requiredRoles；
- [ ] 页面可打开到六态，不得白屏、404 或调用不存在端点；
- [ ] 普通用户不暴露 JSON / DSL / trace / prompt。

### S-GATE-3 · 卡到代码

每张施工卡进入实现前，必须补一张“卡到代码映射”：

| 项 | 必填 |
|---|---|
| 前端 | 路由、页面组件、hooks、测试文件 |
| 后端 | Controller、Service、Repository、DTO、错误码、审计事件 |
| 数据 | 迁移脚本、表、索引、状态机字段、审计字段 |
| 验证 | 单测、契约测试、E2E、截图、traceId、降级验证 |

没有代码映射的卡不能进入实现；已有代码但不可信时，必须标注“待净化 / 待重写 / 可复用”。

### S-GATE-4 · B0 主链路

每个域必须有一条 B0 主链路，不依赖模型、不依赖图投影、不依赖外部系统成功连接：

- D0：建组织 → 建用户 → 绑角色 → 登录 → 菜单呈现 → 跨租户拒绝 → 审计 → 退出。
- D1：角色首屏 → 演示校验 → 阻塞项定位。
- D2：导入/配置规则路径知识字典 → 7 步流 → 灰度 → 全量 → 回滚。
- D3：建患者 → 入径 → 节点推进 → 确定性提醒 → 采纳/拒绝 → 待办闭环。
- D4：配指标 → 评估命中 → 问题 → 整改 → 复核 → 驾驶舱下钻。
- D5：管权限 → 身份绑定 → 查审计 → 导出证据包 → Provider 断连诚实降级。
- D6：来源追溯 → 图谱投影可查/可降级 → 国产化自检 → 开发者控制台。

### S-GATE-5 · wave2 加深范围

wave2 只做 B0 之后的 AI 加深，不得抢跑：

- AIK：来源、解析、候选、门禁、审核分流、包同步。
- LLM：统一模型网关、B0/B1/B2、真实 provider、评测、数据最小化。
- KNOWGEN：首发知识资产生成候选，不直接成为事实。
- DOMAIN：护理、报告、床旁知识、药事、急危重症、特殊人群、围术期、肿瘤、照护、中医药、公卫、基层慢病、区域协同、扩展专科、科研数据等专业门面。

每个 wave2 卡必须回指一个或多个 D0–D6 B0 消费点；没有 B0 消费点的 AI 能力不得实现。

## 4. 当前必须补的范围治理任务

1. **覆盖矩阵回填**：把 S17–S40 与 wave2 卡的旧锚点全部回填，消灭“场景已建卡但矩阵仍待迁”的矛盾。
2. **D2 计数口径统一**：统一写成“28 张物理卡，30 个逻辑能力/页面交付项”，或把 RULE/PATH 页面拆卡；推荐前者，避免重复事实源。
3. **卡到代码矩阵**：D0 第一闸先建；后续每域开工前补该域矩阵。
4. **页面路由盘点**：实际 `frontend/src/pages` 与 27+5 菜单逐项对照，任何未入卡页面必须删除、隐藏或补卡。
5. **接口盘点**：实际后端 Controller 与施工卡 API 契约逐项对照，任何旧接口必须标注保留、净化、删除或迁移。
6. **B0 链路验收脚本**：每域至少一个 Playwright / 后端 E2E / 集成测试组合，不能只靠单测。

---

## 5. 完成定义

业务实现范围核查只有在以下条件满足时才能标为通过：

- S0–S40 全部有存在的主卡；
- 27+5 菜单全部有页面卡或合并说明；
- D0–D6 每域都有 B0 主链路验收脚本；
- wave2 每张卡都有 B0 消费点；
- 覆盖矩阵关键业务锚点不再 `待迁`；
- 实际前端路由、后端接口、数据库迁移、测试文件均能反向追溯到施工卡；
- 未实现项明确保持 `pending`，不允许“范围覆盖”被误写成“实现完成”。
