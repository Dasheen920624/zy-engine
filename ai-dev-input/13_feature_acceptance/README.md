# 功能验收与专业优化目录

本目录用于逐功能验收，解决“不同 AI 交付质量参差不齐”的问题。任何面向客户、医学/医保/质控、数据库持久化、前端页面、发布同步或安全权限的功能，都必须建立功能验收记录。

目录约定：

```text
pending/             待验收或验收中的功能。
needs_optimization/  未达到质量目标，需专业人员优化。
approved/            已通过验收，可作为稳定能力对外演示或进入后续集成。
```

验收角色按需参与：

- 产品负责人：业务闭环、客户价值、验收故事线。
- 架构负责人：模块边界、Provider、生产库/开发库、可演进性。
- 后端负责人：接口、错误码、traceId、审计、降级。
- 前端负责人：交互、可访问性、状态、真实 API / MSW 契约。
- 数据库负责人：表/字段注释、索引、约束、Oracle/DM/PostgreSQL-Kingbase/LOCAL_H2_FILE 同步。
- 测试负责人：契约测试、smoke、异常场景、回归矩阵。
- 医学/医保/质控专家：来源、适用范围、风险阻断、人工复核。
- 安全/运维负责人：权限、脱敏、日志、部署、回滚。

质量等级：

```text
GOLD      可对客户演示、可集成、无 P0/P1/P2 开放问题。
SILVER    可内部联调，有 P2 优化项但不阻断主流程。
BRONZE    仅原型或早期能力，不允许标记为 DONE。
REJECTED  不符合产品/架构/安全/医学/数据库底线，必须重做或拆分。
```

硬性要求：

- 未通过功能验收的功能不得作为客户演示卖点。
- 新增或修改的数据库表和关键字段必须有中文 `COMMENT ON` 或等价备注脚本；H2 只承担开发库结构等价，不替代生产库备注。
- 公共 Controller、Service、Provider、复杂校验、发布/回滚/同步、医学/医保/质控判断必须有简洁注释说明意图；禁止无意义逐行注释。
- P0/P1/P2 开放问题必须进入 `needs_optimization/`，并在任务台账登记后续优化编号。
