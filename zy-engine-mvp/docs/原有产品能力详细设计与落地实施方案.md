# 原有产品能力详细设计与落地实施方案

> 本文补齐原有产品能力的统一实施说明。新增 9 项能力看 `九大产品能力详细设计与落地实施方案.md`；原有配置包、组织、来源、规则、路径、字典、图谱、Dify、适配器、前端、审计运维能力看本文。

已实现能力的体系化改造和优化节奏见：`已实现能力统一改造优化方案.md`。

## 0. 统一实现口径

原有产品能力已经有不少代码和文档，但后续 AI 不能只看历史描述。每个原有能力继续开发时，必须按同一模板交付：

- 产品目标：服务哪个角色、解决哪个客户场景。
- 核心流程：从输入、校验、审核、发布、执行到审计闭环。
- 数据模型：生产库 Oracle/达梦/PostgreSQL/Kingbase 与开发库 LOCAL_H2_FILE 同步。
- 后端服务/API：Controller、Service、Repository、Provider 边界明确。
- 前端页面：角色入口、列表、详情、审核、diff、dry-run、错误态、无权限态。
- 测试验收：单元测试、契约测试、DB-only 验证、生产库 smoke 计划、feature acceptance。
- 任务映射：以 `02_任务台账.md` 编号为唯一实施依据。

通用红线：

- 不允许硬编码单医院、单院区、单厂商、单数据库。
- 不允许 Neo4j、Dify、外部大模型不可用时导致核心规则、路径、配置、验收不可用。
- 不允许医学、医保、质控配置缺来源、过期来源或未审核来源仍可发布。
- 不允许发布、回滚、导入、导出、权限变更绕过审计。
- 不允许用 H2 验证冒充生产库 Oracle/达梦/PostgreSQL/Kingbase 验收。

## 1. 组织模型与多租户上下文

### 1.1 产品目标

支撑集团、医院、院区、站点、科室、外网租户和多客户并行运行。所有配置、规则、路径、图谱、知识包、日志、审计、报表都必须带组织上下文。

### 1.2 核心流程

```text
创建租户/组织目录
-> 导入集团/医院/院区/科室
-> 解析请求组织上下文
-> 计算默认值和继承覆盖
-> 查询、发布、执行时按组织范围过滤
-> 审计记录组织快照
```

### 1.3 数据模型

| 模型 | 说明 | 任务 |
|---|---|---|
| `ORG_UNIT` | 组织目录树 | `ORG-002/003` |
| `OrganizationContext` | 请求上下文解析结果 | `ORG-001` |
| `OrgOverrideEntry` | 组织继承覆盖 | `ORG-004` |
| 配置业务表组织字段 | `tenant_id/group_code/hospital_code/campus_code/site_code/department_code/scope_level/scope_code` | `ORG-*`、各业务任务 |

### 1.4 后端/API

- `GET /api/system/org-context`：解析 Header/Query/Body 组织上下文。
- `POST /api/organizations/import`：导入组织目录。
- `GET /api/organizations/tree`：树形查询。
- `OrganizationContextService`：所有业务服务统一调用。
- `OrganizationPersistenceService`：生产库/开发库持久化。

### 1.5 验收标准

- 不同医院同编号资源互不覆盖。
- 查询、发布、执行、审计均能按组织范围过滤。
- 缺组织上下文时使用明确默认值，不静默落到错误租户。
- `ORG-005` 继续补配置包组织覆盖，后续图谱、Dify、适配器也必须接组织隔离。

## 2. 配置包与发布治理

### 2.1 产品目标

统一规则、路径、图谱、Dify、字典、适配器、知识包等资产的导入、审核、发布、导出、回滚、跨环境迁移和影响分析。

### 2.2 核心流程

```text
导入配置包
-> 校验 manifest/hash/schema/组织范围
-> review 检查来源、依赖、冲突、安全
-> dry-run
-> 发布
-> 写审计
-> 导出 / 回滚 / 跨环境迁移
```

### 2.3 数据模型

| 模型 | 说明 | 任务 |
|---|---|---|
| `CFG_PACKAGE` | 包主表、版本、状态、hash、组织范围 | `PKG-001/004` |
| `CFG_PACKAGE_ASSET` | 包内资产清单 | `PKG-004/007` |
| `CFG_PACKAGE_REVIEW` | review 结果 | `PKG-001/003` |
| `CFG_PACKAGE_PUBLISH` | 发布记录 | `PKG-003/005` |
| `CFG_ASSET_REGISTRY` | 统一资产注册表 | `PKG-007` |
| `OPS_SYNC_TASK` | 跨环境导入导出和异步同步 | `OPS-002`、`PKG-006` |

### 2.4 后端/API

- `POST /api/config-packages/import`
- `POST /api/config-packages/{id}/review`
- `POST /api/config-packages/{id}/dry-run`
- `POST /api/config-packages/{id}/publish`
- `POST /api/config-packages/{id}/rollback`
- `GET /api/config-packages/{id}/diff`
- `POST /api/config-packages/export`

实现要求：

- `ConfigPackageService` 只负责包生命周期，不直接保存各业务资产细节。
- 各资产类型通过 `ConfigAssetHandler` 接入校验、应用、回滚。
- review 必须聚合来源追溯、权限、组织范围、依赖、冲突、schema、hash。
- 发布和回滚必须写 `ENGINE_AUDIT_LOG`。

### 2.5 前端页面

| 页面 | 能力 |
|---|---|
| 配置包列表 | 筛选、状态、组织范围、版本、最近发布 |
| 配置包详情 | manifest、资产、hash、来源、review |
| 差异页 | 版本 diff、资产 diff、风险提示 |
| 发布页 | dry-run、确认、审计、发布结果 |
| 回滚页 | 目标版本、影响分析、确认、审计 |

### 2.6 验收标准

- 同一 `package_code + version` 内容不同必须阻断。
- 缺来源或来源未审核的医学/医保/质控资产不得发布。
- 回滚能恢复 active version，并保留审计。
- 跨环境导入导出必须有 hash、签名、dry-run 和错误报告。

## 3. 来源追溯与医学可信

### 3.1 产品目标

让规则、路径、图谱、Dify、字典、知识包、适配器、评估结果都能追溯到来源文档、引用片段、版本、审核人和适用组织范围。

### 3.2 核心流程

```text
来源文档登记
-> 引用片段结构化
-> 资产绑定来源
-> 发布前来源检查
-> 运行结果携带证据
-> 来源变更影响分析
```

### 3.3 数据模型

| 模型 | 说明 | 任务 |
|---|---|---|
| `SRC_DOCUMENT` | 来源文档、版本、机构、有效期、授权 | `PROV-001` |
| `SRC_CITATION` | 章节、页码、条款、引用片段 | `PROV-002/002F` |
| `SRC_ASSET_BINDING` | 资产和来源绑定 | `PROV-003/003F` |
| `SRC_REVIEW_STATUS` | 来源审核状态 | `PROV-004` |
| `SRC_IMPACT_RESULT` | 来源影响分析结果 | `PROV-006` |

### 3.4 后端/API

- `POST /api/provenance/documents`
- `POST /api/provenance/citations`
- `POST /api/provenance/asset-bindings`
- `GET /api/provenance/assets/{assetType}/{assetCode}/sources`
- `POST /api/provenance/publish-check`
- `GET /api/provenance/documents/{id}/impact`

实现要求：

- 来源服务必须独立于规则、路径、图谱等业务服务。
- 业务服务只保存 asset identity，证据查询走 `SourceEvidenceService`。
- `PROV-002F/003F` 必须修复当前持久化未接通和字段错位。

### 3.5 前端页面

- 来源库：文档、版本、机构、有效期、授权。
- 引用片段：章节、页码、条款、摘要。
- 资产绑定：规则、路径、图谱、Dify、字典的来源卡片。
- 影响分析：来源变更影响哪些资产和发布包。

### 3.6 验收标准

- 发布前可阻断缺来源、过期来源、未审核来源。
- 运行结果能显示来源摘要和 traceId。
- 来源变更可反查受影响资产。

## 4. 规则引擎、质控和医嘱安全

### 4.1 产品目标

提供本地可运行、DB-only 可验收、可第三方调用的规则引擎，支撑病历质控、医保质控、医嘱安全、路径推荐和发布前校验。

### 4.2 核心流程

```text
导入规则包
-> review 来源和 DSL
-> 发布规则版本
-> 输入患者/病历/医保/医嘱事实
-> 标准化字典
-> evaluate / batch evaluate
-> 输出命中、证据、建议动作、缺失事实
-> 写执行日志和审计
```

### 4.3 数据模型

| 模型 | 说明 | 任务 |
|---|---|---|
| `RULE_DEFINITION` | 规则定义、版本、场景、来源 | `RULE-001/005` |
| `RULE_EXEC_LOG` | 单条规则执行日志 | `RULE-002/004` |
| `RE_RULE_EVAL_RESULT` | 批量评估结果 | `RULE-008` |
| 样例包 | 病历质控、医保质控、医嘱安全样例 | `RULE-006/006B/006C` |

### 4.4 后端/API

- `POST /api/rule-engine/evaluate`
- `POST /api/rule-engine/batch-evaluate`
- `GET /api/rule-engine/results`
- `POST /api/rules/import`
- `POST /api/rules/{id}/review`
- `POST /api/rules/{id}/publish`

实现要求：

- 规则执行不能依赖 Dify 或 Neo4j。
- DSL 扩展必须有单元测试和错误提示。
- 规则只使用标准码，院内原文必须先经过 `TERM-*` 标准化。
- 第三方调用必须接 `SEC-003` API Key、签名、nonce、防重放。

### 4.5 前端页面

- 规则配置器：条件树、动作、严重等级、来源卡片。
- dry-run 工作台：输入样例、选择规则包、查看命中。
- 结果列表：场景、组织、命中、严重等级、traceId。

### 4.6 验收标准

- 无 Dify/Neo4j 时规则 evaluate、batch、results 可用。
- 每个命中结果有规则版本、输入快照、证据和审计。
- 医嘱安全高风险规则不得由 AI 实时输出直接替代。

## 5. 临床路径引擎

### 5.1 产品目标

管理临床路径配置、候选推荐、医生入径、节点流转、变异记录、版本 diff 和来源绑定。

### 5.2 核心流程

```text
导入路径配置
-> 校验节点、流转、来源
-> 发布路径版本
-> 按诊断/手术/病种推荐路径
-> 医生确认入径
-> 节点流转
-> 记录变异
-> 生成路径执行结果和审计
```

### 5.3 数据模型

| 模型 | 说明 | 任务 |
|---|---|---|
| `PATHWAY_CONFIG` | 路径配置、版本、组织范围 | `PATH-001` |
| `PATHWAY_NODE` | 节点、时限、动作 | `PATH-001/002` |
| `PATHWAY_INSTANCE` | 患者入径实例 | `PATH-004/005` |
| `PATHWAY_VARIATION` | 变异记录 | `PATH-006` |
| `PATHWAY_VERSION_DIFF` | 版本差异 | `PATH-007` |
| 来源字段 | 节点和流转绑定来源 | `PATH-008` |

### 5.4 后端/API

- `POST /api/pathways/import`
- `POST /api/pathways/{id}/review`
- `POST /api/pathways/{id}/publish`
- `POST /api/pathways/recommend`
- `POST /api/pathway-instances`
- `POST /api/pathway-instances/{id}/transitions`
- `POST /api/pathway-instances/{id}/variations`
- `GET /api/pathways/{id}/diff`

实现要求：

- 医生入径必须人工确认。
- 路径状态持久化必须按租户和组织隔离。
- 路径推荐可使用规则增强，但核心推荐不可依赖大模型。

### 5.5 前端页面

- 路径列表和版本。
- 路径画布：节点、流转、时限、来源。
- 入径确认和流转。
- 变异记录和统计。

### 5.6 验收标准

- 路径导入、校验、发布、推荐、入径、流转、变异闭环可跑通。
- 节点和流转可查来源。
- 多医院同患者号或路径号互不覆盖。

## 6. 标准化中心和字典治理

### 6.1 产品目标

解决院内诊断、医嘱、药品、检验、项目名称和标准编码不一致导致规则、路径、医保质控不可用的问题。

### 6.2 核心流程

```text
导入院内字典
-> 导入标准字典
-> 自动匹配和人工审核
-> 未映射进入治理队列
-> 发布字典版本
-> 规则/路径执行只使用标准码
```

### 6.3 数据模型

| 模型 | 说明 | 任务 |
|---|---|---|
| `TERM_STANDARD_CODE` | 标准编码 | `TERM-001` |
| `TERM_LOCAL_CODE` | 院内编码 | `TERM-001` |
| `TERM_MAPPING` | 映射关系、版本、审核 | `TERM-001` |
| `TERM_PENDING_MAPPING` | 未映射治理队列 | `TERM-002` |
| `TERM_IMPACT_RESULT` | 字典版本影响分析 | `TERM-003` |

### 6.4 后端/API

- `POST /api/terminology/standard/import`
- `POST /api/terminology/local/import`
- `POST /api/terminology/mappings/review`
- `GET /api/terminology/pending-mappings`
- `POST /api/terminology/pending-mappings/{id}/resolve`
- `GET /api/terminology/normalize`
- `GET /api/terminology/versions/{id}/impact`

实现要求：

- 规则和路径不得直接依赖院内原文。
- 未映射时输出明确 `PENDING_MAPPING`，不能静默跳过。
- AI 候选映射必须经过人工审核。

### 6.5 前端页面

- 标准字典、院内字典、映射表。
- 未映射治理队列。
- 映射审核和影响分析。

### 6.6 验收标准

- 未映射医嘱进入治理队列。
- 已发布规则只使用标准码。
- 字典版本变更可提示影响哪些规则、路径、知识包。

## 7. 医学知识图谱

### 7.1 产品目标

以生产库保存图谱主数据，Neo4j 作为可选投影，支撑关系查询、证据查询、版本发布、回滚和同步。

### 7.2 核心流程

```text
导入图谱包
-> 校验节点、边、来源、组织范围
-> 发布图谱版本
-> 查询证据和关系
-> 可选同步 Neo4j
-> 失败可回滚到上一版本
```

### 7.3 数据模型

| 模型 | 说明 | 任务 |
|---|---|---|
| `GE_GRAPH_VERSION` | 图谱版本 | `GRAPH-001` |
| `GE_NODE` | 图谱节点 | `GRAPH-001` |
| `GE_EDGE` | 图谱关系 | `GRAPH-001` |
| 来源绑定 | 节点/边绑定来源 | `GRAPH-003` |
| 同步任务 | Neo4j 投影同步 | `GRAPH-005`、`OPS-002` |

### 7.4 后端/API

- `POST /api/graph/packages/import`
- `POST /api/graph/versions/{id}/publish`
- `POST /api/graph/versions/{id}/rollback`
- `GET /api/graph/query`
- `GET /api/graph/evidence`
- `POST /api/graph/sync/neo4j`

实现要求：

- GraphService 所有方法必须按 `tenant_id` 过滤，完成 `GRAPH-006`。
- Neo4j 不可用时必须仍可查询生产库主数据。
- 图谱同步必须 dry-run、重试和审计。

### 7.5 前端页面

- 图谱版本列表。
- 节点/关系查询。
- 证据卡片。
- 同步状态和回滚入口。

### 7.6 验收标准

- 生产库图谱主数据可独立验收。
- Neo4j 关闭时核心查询不崩。
- 回滚后 active version 正确。

## 8. Dify 与 AI 工作流契约

### 8.1 产品目标

管理 Dify/AI 工作流契约、参数、来源绑定、调用回放和降级策略。Dify 只做候选、解释和辅助，不保存正式配置主数据。

### 8.2 核心流程

```text
登记 Dify 契约模板
-> 绑定真实 app 或本地 Provider
-> 校验输入输出 schema
-> 调用生成候选或解释
-> 记录输入输出 hash 和耗时
-> 失败降级到本地规则/外网基准包/人工审核
```

### 8.3 数据模型

| 模型 | 说明 | 任务 |
|---|---|---|
| `SRC_DIFY_TEMPLATE` | Dify 契约模板 | `DIFY-001/002` |
| `DIFY_APP_BINDING` | 环境和真实 app 绑定 | `DIFY-001` |
| `DIFY_CALL_LOG` | 调用回放、输入输出 hash、耗时 | `DIFY-004` |
| Provider 配置 | Dify、本地模型、外网包、人工 | `AIK-005`、`AI-GOV-001` |

### 8.4 后端/API

- `POST /api/dify/templates`
- `GET /api/dify/templates`
- `POST /api/dify/templates/{id}/validate`
- `POST /api/dify/call`
- `GET /api/dify/calls/{id}/replay`

实现要求：

- `DIFY-002` 必须补生产库/开发库持久化。
- Dify API Key 不得入日志，不得明文存储。
- Dify 不可用时核心规则、路径、知识包流程不能失败。

### 8.5 前端页面

- Dify 契约模板列表。
- schema 校验。
- 调用回放。
- Provider 状态和降级说明。

### 8.6 验收标准

- 无 Dify 环境可以完成 DB-only 验收。
- 调用日志可回放但不泄露密钥和隐私。
- Dify 输出只能进入候选或解释，不直接发布为正式配置。

## 9. 第三方适配器

### 9.1 产品目标

以适配器方式接入 HIS、EMR、LIS、PACS、医保、OA、统一身份平台和外部规则调用方，避免业务服务硬编码厂商协议。

### 9.2 核心流程

```text
定义适配器
-> 绑定组织范围
-> 配置认证和接口协议
-> 测试连接
-> 执行同步或调用
-> 记录运行日志、错误码、重试和审计
```

### 9.3 数据模型

| 模型 | 说明 | 任务 |
|---|---|---|
| `ADAPTER_DEFINITION` | 适配器定义 | `ADAPT-001` |
| `ADAPTER_BINDING` | 组织绑定、环境、启停 | `ADAPT-001` |
| `ADAPTER_AUTH_CONFIG` | API Key、证书、OAuth、签名配置 | `SEC-003`、`ADAPT-001` |
| `ADAPTER_RUN_LOG` | 调用日志、错误码、耗时、重试 | `ADAPT-004` |

### 9.4 后端/API

- `POST /api/adapters`
- `POST /api/adapters/{id}/bindings`
- `POST /api/adapters/{id}/test-connection`
- `POST /api/adapters/{id}/sync`
- `GET /api/adapters/run-logs`

实现要求：

- 业务服务只依赖 `AdapterProvider` 接口。
- 外部系统不得传任意 SQL/Cypher 到核心引擎执行。
- 适配器运行必须有 traceId、审计和脱敏日志。

### 9.5 前端页面

- 适配器列表。
- 组织绑定。
- 测试连接。
- 同步任务和运行日志。

### 9.6 验收标准

- 不同组织可绑定不同适配器。
- 连接失败有中文错误和 traceId。
- 重试、超时、错误码和审计可查询。

## 10. 前端配置平台与演示验收

### 10.1 产品目标

提供可让客户、实施、医学审核、信息科直接操作的配置、演示、审核和验收界面，而不是只靠 API 和 Postman。

### 10.2 页面矩阵

- E2E 能打开核心页面并截图。

## 11. 审计、运维、数据库和测试底座

### 11.1 产品目标

让系统具备可上线、可追溯、可回滚、可运维、可验收的工程底座。

### 11.2 能力矩阵

| 能力 | 任务 | 必须交付 |
|---|---|---|
| 统一审计 | `AUDIT-001`、`SEC-010` | 操作人、组织、动作、对象、traceId、防篡改 |
| 初始化检查 | `OPS-001` | JDK、数据库、端口、依赖检查 |
| 同步任务 | `OPS-002` | 异步任务、重试、状态、审计 |
| 监控指标 | `OPS-003` | 错误率、命中率、降级率、调用耗时 |
| 离线部署 | `OPS-004` | 内网部署、升级、回滚脚本 |
| 灾备回滚 | `OPS-006` | 备份、恢复、演练报告 |
| 性能容量 | `OPS-007` | 压测、容量基线、阈值 |
| 归档策略 | `OPS-008` | 日志、模型调用、审计归档 |
| 回归测试 | `TEST-001/002/003` | 回归矩阵、DB-only、功能验收 |

### 11.3 数据库要求

- 所有生产表必须有 Oracle、达梦、PostgreSQL/Kingbase DDL。
- LOCAL_H2_FILE 是开发等价验证，不是生产替代。
- DDL 必须包括表注释、关键字段注释、索引、唯一约束。
- 持久化服务必须避免硬编码单一 SQL 方言。

### 11.4 验收标准

- `check-ai-collaboration.ps1` 能识别 claim/review/acceptance 状态。
- 新增 DB 能力无 Oracle 时必须跑 LOCAL_H2_FILE，最终说明生产库补验计划。
- 发布、回滚、导入导出、权限变更均可查审计。
