# 00 总入口与 AI 接手导航

## 0. 一句话

本项目是面向集团化医院、多院区、卫生所/站点和科室的 **医疗智能引擎平台**（路径 + 规则 + 图谱 + 标准化 + 适配器 + Dify/AI 工作流 + 质控 + 配置发布 + 前端配置/演示校验）。生产库与开发库必须严格区分：Oracle 是当前生产权威验收库，达梦 / PostgreSQL / KingbaseES 是生产交付兼容库，`LOCAL_H2_FILE` 只作为无内网数据库时的开发本地文件库。Neo4j 是图谱投影，Dify 是 AI 工作流执行目标，DB-only 模式必须独立可验收。

后续会有 **多个 AI（不同能力）和多位工程师并行开发**。本文是任何 AI、任何能力等级接手时必须先读的 **唯一入口**。

### Step 5（硬门禁）— 生产库 / 开发库分层

开发前必须先确认当前任务运行在哪一类数据库上：

| 类型 | 允许用途 | 禁止事项 |
|---|---|---|
| 生产权威库：Oracle | 正式验收、真实落库、上线迁移、审计追责 | 不允许用 H2 结果冒充 Oracle smoke |
| 生产兼容库：达梦 / PostgreSQL / KingbaseES | 国产化和客户部署交付 DDL、迁移脚本、差异说明 | 不允许只维护 Oracle，不同步兼容库结构 |
| 开发本地库：`LOCAL_H2_FILE` | 无 Oracle/内网环境时的 AI 开发、DB-only 验证、本地文件持久化 | 不允许作为生产替代库，不允许提交本地数据文件 |
| `IN_MEMORY_DEMO` | 快速演示、纯单测兜底 | 不允许作为持久化、DDL、跨实例验收依据 |

任何新增表、索引、约束、持久化 SQL 或落库行为，都必须说明“生产库如何验收”和“开发库如何验证”。无 Oracle 时也必须跑 `LOCAL_H2_FILE`，并把生产库 smoke 标为待有环境补验。

## 2. 开发硬性 6 条（任何任务都适用）

| 编号 | 规则 | 解释 |
|---|---|---|
| H1 | **只改任务台账声明的独占文件** | 共享文件需先在台账登记申请，避免与其它 AI 冲突 |
| H2 | **不硬编码单医院 / 单院区 / 单系统厂商逻辑** | 差异沉淀为配置包、字典映射、适配器绑定、Provider |
| H3 | **Neo4j / Dify 不可用时核心路径/规则必须仍可用** | DB-only 模式必须可独立完成验收 |
| H4 | **关键医疗 / 医保 / 质控配置必须有来源追溯** | 缺来源、来源过期、未审核不得发布 |
| H5 | **临床建议必须由医生最终确认** | 系统给建议，医生做决定；阻断类规则必须院方授权 |
| H6 | **生产库和开发库分离** | Oracle/DM/PG/Kingbase 是生产交付口径；LOCAL_H2_FILE 只做开发等价验证 |

## 2.1 并发开发硬机制

为解决 AI 未认领、状态不及时、git 状态不同步的问题，任何代码、DDL、脚本、样例或核心文档改动前必须满足：

| 阶段 | 必做动作 | 阻断条件 |
|---|---|---|
| 开发前 | `git pull --ff-only origin main` + `git status -sb` + `check-ai-collaboration.ps1` | 工作树不干净且无法说明来源 |
| 认领前 | 创建并推送 `ai-dev-input/10_task_claims/active/<claim_id>.md` | 没有远端 active claim 不允许改业务文件 |
| 开发中 | 超过 60 分钟更新 `last_heartbeat` 并推送 | 4 小时无心跳视为可能 STALE |
| 提交前 | claim 状态、任务台账状态、review 状态、git diff 范围四项对账 | 任何一项不一致不得提交 |
| 推送后 | 再跑 `git status -sb`，把 commit hash 写回台账/claim/review | main 未推送成功不得口头宣布完成 |

冲突处理：

- active claim 写入范围重叠 → 停止开发，拆分任务或认领集成任务。
- 发现他人未提交改动 → 不回滚、不覆盖；只处理自己的写入范围。
- claim 超过 24 小时无心跳且无提交 → 新 AI 必须创建接管 claim，记录原 claim 和接管原因。
- 业务代码可以滞后于 claim，但 claim、review、feature acceptance 的状态不能滞后于实际进度。

## 2.2 功能验收与专业优化机制

为解决功能质量参差不齐的问题，以下功能必须建立 `ai-dev-input/13_feature_acceptance/` 验收记录：

- 客户可见页面、演示工作台、配置中心、质控看板。
- 医学、医保、质控、规则命中、路径推荐、Dify 解释。
- 数据库持久化、生产库/开发库 DDL、发布/回滚/同步、权限安全。
- 对外接口、第三方适配器、审计运维能力。

验收结果分级：

| 等级 | 含义 | 是否可对客户演示 |
|---|---|---|
| GOLD | 业务闭环、代码、测试、注释、数据库、交互均达标 | 可以 |
| SILVER | 主流程可用，有非阻断优化项 | 内部联调可以，客户演示需确认 |
| BRONZE | 原型或早期能力 | 不可以 |
| REJECTED | 不满足产品/架构/安全/医学/数据库底线 | 不可以 |

专业优化安排：

- P0/P1/P2 问题必须进入 `needs_optimization/`，并在任务台账登记后续任务。
- 医学/医保/质控功能必须有领域 Reviewer。
- 前端页面必须有前端 Reviewer 或浏览器验证证据。
- 数据库变更必须由数据库 Reviewer 检查表/字段注释、索引、约束和多数据库同步。
- 公共 Controller / Service / Provider、复杂规则、发布回滚同步、医学/医保判断必须有简洁意图注释；禁止无意义逐行注释。

## 4. 红线（违反即必须停止）

| 红线 | 例子 |
|---|---|
| 不允许硬编码单医院逻辑 | `if (hospital.equals("ZYHOSPITAL"))` 形式禁止进入业务服务 |
| 不允许把 Neo4j / Dify 作为测试环境强依赖 | 测试不可在 Neo4j 关闭时失败 |
| 不允许外部系统传任意 SQL / Cypher 到核心引擎执行 | 必须走 Adapter / GraphProvider 受控接口 |
| 不允许配置静默覆盖已发布版本 | 同 `package_code + package_version` 内容不同必须返回错误 |
| 不允许关键医疗建议绕过医生确认 | 阻断类规则必须有 `action_mode + doctor_decision` 字段 |
| 不允许发布、回滚、同步绕过审计 | 必须写入 `ENGINE_AUDIT_LOG` |
| 不允许缺来源的医学/医保/质控配置发布 | review 必须返回 `MISSING_SOURCE` 阻断 |
| 不允许日志输出数据库密码 / Dify API Key / 患者完整隐私 | 日志必须脱敏 |
| 不允许回滚 / 覆盖其它 AI 未提交的工作 | 不确定的未提交改动先问用户 |

## 5. 多 AI 并行三句话

1. **领任务前**：先 `git pull`，再检查 claim/review/feature acceptance 队列，创建并推送 active claim；没有远端 claim 不允许改业务文件。
2. **开发中**：只动 claim 和台账声明的独占文件；超过 60 分钟同步心跳；需要碰共享文件先更新 claim 并协调。
3. **完成后**：通过 review 和必要功能验收，再 commit + push，把台账/claim/review/feature acceptance 写入提交 hash 后归档。
