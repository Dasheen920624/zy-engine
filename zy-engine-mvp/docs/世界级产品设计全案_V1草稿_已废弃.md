## 1. 产品愿景与核心价值

### 1.1 一句话定义

> **让每一条临床路径都有规则保障、知识支撑和可追溯闭环，让医院可以独立配置、持续迭代、生产级运行。**

### 1.2 产品定位矩阵

```
目标用户层：集团化医院 / 多院区 / 专科医联体
部署环境：医院内网 + 国产化优先 + 离线可用
产品形态：SaaS 底座（外网租户开通） + 院内私有化交付（离线包 + 配置包同步）
核心竞争力：三大引擎独立服务化 + 配置驱动 + 来源可审计 + AI 知识工厂
```

### 1.3 三大引擎定义

| 引擎 | 核心问题 | 独立能力 | 消费者 |
|---|---|---|---|
| **路径引擎** | 患者在哪条路径、哪个节点、下一步做什么 | 路径版本管理、入径、节点流转、变异记录 | 医生工作站、随访平台、质控平台 |
| **规则引擎** | 当前事实是否违反规则、需要提醒/拦截/质控 | 规则 DSL、适配器取数、字典映射、实时预警 | 路径引擎、医生工作站、病历质控 |
| **图谱引擎** | 疾病/症状/检查/治疗/证据之间有什么关系 | 图谱版本、关系查询、候选召回、来源追溯 | 路径引擎、规则引擎、Dify、推荐服务 |

### 1.4 不做什么（边界）

- 不做 HIS / EMR / LIS 系统本体；只做接入适配层
- 不做 AI 聊天机器人入口；AI 作为后台能力服务
- 不替代医生判断；所有临床建议必须由医生最终确认
- 不做大数据 BI 报表平台；仅提供质控和监控看板

## 4. 核心模块功能详细规格

### 4.1 路径模板编辑器

**目标体验：** 临床路径专家用"路径图"构建路径，不依赖 IT 协助。

**核心功能：**

| 功能 | 规格 | 实现方式 |
|---|---|---|
| 阶段/节点配置 | 拖拽 + 表单，支持多阶段嵌套节点 | AntV X6 画布 + Ant Design 表单 |
| 节点任务绑定 | 每节点可绑定规则、图谱查询、Dify 工作流、适配器取数 | 配置面板抽屉 |
| 入径条件配置 | 支持字段条件组合（AND/OR/NOT） | 规则 DSL 可视化编辑器 |
| 节点流转条件 | 同上，支持"自动流转"和"手动流转" | 同上 |
| 版本对比 | 左右双栏展示两版本差异（字段级 diff） | 自定义 diff 渲染 |
| 发布前校验 | 来源审核状态、条件完整性、依赖版本 | 后端 review API → 前端展示 |

**页面布局：**

```
┌─ 顶部栏 ──────────────────────────────────────────────────────┐
│  路径名称  版本 v3.1 (草稿)  [来源状态: 已审核✓]  [发布] [保存] │
├─ 左侧树 ─┬─ 中央画布 ─────────────────┬─ 右侧配置面板 ──────────┤
│ 阶段列表  │ [阶段1：入院]               │  节点：血常规检查         │
│ > 阶段1   │   ○ 入院评估 ─────────────→│  类型：检查任务           │
│   > 节点1 │   ○ 血常规检查              │  绑定规则：检验时限规则    │
│   > 节点2 │   ○ 床位安排 ─────────────→│  图谱查询：AMI 相关检查   │
│ > 阶段2   │ [阶段2：治疗]               │  Dify 工作流：无          │
│           │   ○ 手术评估               │  流转条件：[编辑]         │
│           │   ...                     │  变异记录：必填            │
└───────────┴────────────────────────────┴────────────────────────┘
```

### 4.2 规则 DSL 编辑器

**目标体验：** 规则工程师用可视化方式编写规则，医疗专家无需理解代码即可理解规则逻辑。

**DSL 编辑器双模式：**

```
[可视化模式] ←→ [代码模式]
```

**可视化模式布局：**

```
┌─ 规则基本信息 ──────────────────────────────────────────────┐
│  规则编号  规则名称  规则类型（质控/预警/路径质控）  优先级      │
│  适用科室  适用路径（可选）  触发时机                           │
├─ 条件区（WHEN）── AND/OR 分组 ───────────────────────────────┤
│  [+ 添加条件]                                                 │
│  ├─ 患者年龄 ≥ 65 岁                           [×]           │
│  └─ AND：血红蛋白 < 70 g/L                    [×]           │
├─ 动作区（THEN）──────────────────────────────────────────────┤
│  [+ 添加动作]                                                 │
│  ├─ 预警：输血指征评估提醒  级别：⚠️警告  要求医生确认           │
│  └─ 记录：质控问题"贫血未评估"  责任科室：血液科               │
├─ 例外区（UNLESS）── 可选 ────────────────────────────────────┤
│  └─ 已存在输血医嘱  [×]                                       │
└─────────────────────────────────────────────────────────────┘
```

**代码模式：** Monaco Editor，语法高亮+自动补全（字段名、操作符、函数）。

### 4.5 配置包发布流程

**流程步骤（向导式）：**

```
步骤 1：基本信息   → 步骤 2：资产清单   → 步骤 3：来源审核   → 步骤 4：影响分析   → 步骤 5：发布确认
```

每步展示：

| 步骤 | 展示内容 | 验证逻辑 |
|---|---|---|
| 基本信息 | 包名、版本、目标组织、描述 | 版本号唯一性 |
| 资产清单 | 路径×条、规则×条、字典×条…全量列表 | 资产完整性检查 |

## 6. 交互设计规范

### 6.1 页面必需状态

所有列表页和详情页必须处理六种状态：

| 状态 | 展示方式 | 技术实现 |
|---|---|---|
| **加载中** | 骨架屏（Skeleton）或 Spin，不用全屏遮罩 | `isLoading` |
| **空状态** | 图标 + 中文说明 + 主要操作按钮 | `data.length === 0` |
| **错误** | 错误图标 + 错误描述（中文）+ 重试按钮 | `isError` |
| **无权限** | 锁形图标 + "您没有该功能的访问权限" | 后端 403 → 前端拦截 |
| **处理中** | 按钮变 Loading 态，禁止重复点击 | 操作 pending |
| **成功反馈** | Toast（2-3秒）或 Banner（需后续操作时） | 操作 success |

### 6.2 表单设计规范

```
字段顺序：重要/必填字段在前，可选/高级字段折叠
字段标签：左对齐，必填字段加红色 * 号
字段校验：blur 时即时校验，不等 submit 才报错
错误文案：描述具体问题，不用 "该字段有误"（改为 "版本号只能包含数字和.字符"）
分组：复杂表单用分节标题，相关字段卡在一起
弹窗宽度：窄（480px）/ 标准（640px）/ 宽（900px）按内容量选
```

### 6.3 危险操作防护

三级防护机制：

```
一级（轻度危险）：确认弹窗，明确说明后果
二级（中度危险）：弹窗 + 输入确认文字（如"请输入路径名称以确认"）
三级（高度危险）：二级确认 + 审计通知相关人 + 操作后不可撤销提示
```

适用范围：
- 一级：停用规则/路径、忽略预警
- 二级：回滚配置包、删除映射规则
- 三级：删除已发布路径版本、强制清除患者路径记录

### 6.4 AI 内容标识规范

所有 AI 生成/候选内容必须：

```
1. 显示 🤖 AI 候选 标签（紫色浅背景）
2. 显示置信度（百分比 + 进度条）
3. 显示生成模型版本（如：Dify AMI 路径工作流 v2.1）
4. 提供三种操作：[✅ 采纳] [✏ 修改] [❌ 拒绝]
5. 采纳后自动记录审核人、审核时间、原始 AI 输出
6. 不允许 AI 内容直接发布为正式版本，必须经过人工审核
```

### 6.5 多步骤操作向导

适用场景：配置包发布、路径版本发布、批量字典映射等。

```
规范：
- 步骤数控制在 3-6 步
- 每步标题 + 简要说明（一行）
- 每步有独立验证，不通过不允许进入下一步
- 最后一步是"确认"页，展示所有信息摘要
- 整个向导过程可以保存草稿，下次继续
- 进入流程后，路由 URL 带步骤参数，支持刷新回到同一步
```

### 6.6 数据表格规范

```
行高：40px（标准）/ 48px（含状态徽章）
分页：每页 20/50/100 行可选，默认 20
列宽：固定列宽（不允许无限拉伸）
排序：点列标题排序，支持多列排序
筛选：列级筛选 + 顶部全局筛选条（二者互补）
行操作：主要操作直接显示（查看/编辑），次要操作进下拉菜单
批量操作：选中行后顶部浮现操作条
表格空态：必须有说明和主要 CTA 按钮
```

## 7. 核心流程端到端设计

### 7.1 路径从创建到患者使用的全流程

```
[临床路径专家]
    │
    ▼ 创建路径模板草稿
    ├─ 填写基本信息（疾病、适用人群、科室）
    ├─ 在画布上配置阶段和节点
    ├─ 为节点绑定规则/图谱查询/Dify工作流
    └─ 配置入径条件和节点流转条件
    │
    ▼ 提交来源审核
    ├─ 系统扫描所有节点的医学依据
    ├─ 标记未注册来源的内容（阻断发布）
    └─ 知识工程师完成来源注册
    │
    ▼ 提交版本审核（→ [主任医师]）
    ├─ 主任查看路径全貌和来源信息
    ├─ 可提意见或批注
    └─ 批准 / 驳回
    │
    ▼ 发布版本（→ [临床路径专家] 触发）
    ├─ 系统执行发布前完整性校验
    ├─ 写入审计日志
    └─ 发布成功，进入 PUBLISHED 状态
    │
    ▼ 患者入径（→ [主治医师]）
    ├─ 在工作站查找适配路径（系统根据诊断推荐）
    ├─ 确认入径（系统创建患者路径实例）
    └─ 路径进入 ACTIVE 状态
    │
    ▼ 节点执行（→ [主治/护士/药师]）
    ├─ 当前节点任务提示（待执行项）
    ├─ 规则引擎实时检查（有违规 → 预警）
    ├─ 执行节点任务（操作 HIS/EMR 或手动标记）
    └─ 节点完成 → 自动或手动流转到下一节点
    │
    ▼ 变异处理（→ [主治医师]）
    ├─ 医生判断无法按路径执行
    ├─ 记录变异原因（系统预设 + 自由文本）
    └─ 路径进入变异状态（继续执行或退出）
    │
    ▼ 路径完成 / 退出
    ├─ 完成时生成路径执行报告
    └─ 质控系统汇总分析
```

### 7.2 规则引擎实时执行流程

```
临床事件触发（如：检验结果回报）
    │
    ▼ 事件进入临床事件总线
    ├─ 附带患者 ID、就诊上下文、事件类型、事件数据
    └─ 附带组织上下文（tenant_id + 科室 + 院区）
    │
    ▼ 规则引擎评估（POST /api/rule-engine/evaluate）
    ├─ 从适配器取数（HIS/EMR/LIS 相关字段）
    ├─ 字典映射（院内编码 → 标准编码）
    ├─ 执行规则集（按优先级，AND/OR/NOT）
    └─ 返回命中结果（规则列表 + 动作列表 + 证据链）
    │
    ▼ 结果处理
    ├─ 无命中：记录评估日志，返回
    ├─ 有命中：
    │   ├─ 预警级：推送实时通知 → 医生工作站显示预警条
    │   ├─ 拦截级：阻断操作，要求医生确认 + 记录决策
    │   └─ 质控级：写入质控问题队列，通知质控管理员
    └─ 全程写入 ENGINE_AUDIT_LOG
```

### 7.3 字典映射处理流程

```
院内系统数据入库（如：检验结果含院内检验项目码）
    │
    ▼ TerminologyService 扫描未映射项
    ├─ 查找 TERM_MAPPING 表
    ├─ 有匹配：直接转换，继续
    └─ 无匹配：记录"未映射问题"
    │
    ▼ 未映射问题处理（→ [信息科]）
    ├─ 工作台展示所有未映射院内编码
    ├─ 信息科确认标准码对应关系
    ├─ 支持：手动映射 / AI 候选建议（来自图谱）/ 批量导入
    └─ 保存映射规则后，系统自动重新处理历史未映射数据
    │
    ▼ 映射规则生命周期
    ├─ 创建（DRAFT）→ 审核（需来源注册）→ 发布（PUBLISHED）
    └─ 变更时版本递增，历史版本保留
```

### 7.4 配置包跨环境同步流程

```
源环境（如：开发/测试）
    ├─ 配置包发布 → PUBLISHED
    └─ 执行导出（/api/config-package/{id}/export）
        ├─ 生成 ZIP 包（JSON 配置 + 哈希摘要 + 数字签名）
        └─ 下载到本地 / 传输到目标环境
    │
目标环境（如：生产）
    ├─ 上传配置包 ZIP
    ├─ 系统执行 dry-run（校验哈希 + 冲突检测 + 来源审核）
    │   ├─ 有冲突 → 展示冲突详情，不允许强制覆盖
    │   └─ 通过 → 展示变更预览
    ├─ 信息科/运维确认导入
    └─ 写入审计日志（含导入人、来源包 hash、变更摘要）
```

### 8.2 API 接口规范

**统一返回结构：**

```json
{

  "message": "操作成功",
  "data": { ... },

**错误码规范：**

|---|---|---|

| DB_ERROR | 数据库错误 | 500 |
| UNKNOWN_ERROR | 未知系统错误 | 500 |

### 8.3 安全架构

```
身份认证：
  人：本地账号（密码 bcrypt + pepper）/ SSO（CAS/OIDC/SAML2）/ LDAP
  系统：HMAC-SHA256 签名 + nonce + 时间戳（±5分钟）

授权模型：
  RBAC（角色-权限-资源）+ 数据权限（组织范围过滤）
  所有查询接口自动附加 tenant_id + org_scope 过滤

传输安全：
  内网 TLS 1.2+（若医院有要求）
  不允许跨租户数据访问

审计：
  所有写操作写入 ENGINE_AUDIT_LOG（不可删除）
  审计表防篡改（触发器 + 序列校验，SEC-010）

脱敏：
  日志不得输出密码 / API Key / 患者完整身份信息
  患者 ID 在日志中截断（仅保留前4位）
```

### 8.4 可观测性设计

```

引擎指标：
  - 规则引擎执行次数/耗时/命中率（按科室）
  - 路径引擎当前活跃路径数、今日入径数、变异率
  - 图谱引擎查询次数/耗时/缓存命中率
  - 字典映射未映射率趋势
  - 适配器调用失败率
日志：
  - traceId 全链路串联（前端→网关→业务→数据库→外部系统）
  - 结构化 JSON 日志（level/time/service/traceId/message/data）
告警：
  - 规则引擎累计失败 > 阈值 → 通知运维
  - Provider 心跳失败 → 自动切降级模式 + 告警
```

## 12. 验收与质量基线

### 12.1 功能验收等级

| 等级 | 标准 | 可否客户演示 |
|---|---|---|
| **GOLD** | 业务闭环、代码、测试、注释、数据库、交互均达标 | 可以 |
| **SILVER** | 主流程可用，有非阻断优化项 | 内部联调可以，客户演示需确认 |
| **BRONZE** | 原型或早期能力 | 不可以 |
| **REJECTED** | 不满足产品/架构/安全/医学/数据库底线 | 不可以 |

### 12.2 每个功能的验收维度

```
☐ 后端 API 有契约测试（@SpringBootTest + MockMvc）
☐ 接口返回统一 ApiResult 结构（含 trace_id）
☐ 所有写操作写入 ENGINE_AUDIT_LOG
☐ 涉及医学/医保/质控内容有来源审核检查
☐ DDL 同步 Oracle / 达梦 / PostgreSQL / LOCAL_H2_FILE
☐ 前端实现加载/空/错误/无权限/处理中/成功六种状态
☐ AI 生成内容有 🤖 标签和人工确认流程
☐ 危险操作有三级防护确认
☐ 已创建 ai-dev-input/13_feature_acceptance/ 验收记录
```

### 12.3 硬性红线（违反即停止）

1. 不允许硬编码单医院/单院区逻辑（if hospital.equals("XXX")）
2. 不允许 Neo4j / Dify 成为测试环境强依赖（关闭时测试必须能过）
3. 不允许配置静默覆盖已发布版本（同 code+version 内容不同 → 错误）
4. 不允许关键医疗建议绕过医生确认
5. 不允许发布/回滚/审核绕过审计
6. 不允许缺来源的医学/医保/质控配置发布（MISSING_SOURCE 阻断）
7. 不允许日志输出密码 / API Key / 患者完整身份
8. 不允许回滚/覆盖其它 AI 未提交的工作
