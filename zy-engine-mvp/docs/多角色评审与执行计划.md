# 多角色评审与执行计划

## 1. 文档定位

本文把当前项目从产品、架构、研发、测试、工程、医疗、行政和运营视角做一次综合评审，并把后续功能补充为可执行任务池。

本文不替代：

- `产品化方案与AI开发编排.md`
- `AI接手执行手册.md`
- `前端配置平台规划与开发验证.md`

后续代码开发仍以任务编号、契约测试、样例和文档同步为准。

## 2. 当前判断

项目已经从“AMI 样例演示”推进到“DB-only 可验收的医疗智能引擎后端骨架”：

- 路径、规则、图谱、Dify、字典、适配器、质控、审计均有基础接口。
- Neo4j 和 Dify 不再是核心流程强依赖。
- 规则包、路径版本、图谱版本、Dify 模板、字典和适配器已经有局部版本化能力。
- 前端配置平台仍处在规划阶段，业务人员仍需要依赖接口和脚本验收。
- 多组织、多院区、配置包统一发布、安全权限、医嘱治理和第三方规则引擎 API 是后续产品化关键路径。

## 3. 多角色评审

### 3.1 产品设计师视角

优点：

- 产品边界清楚：引擎平台不是单医院定制系统。
- DB-only 验收原则正确，便于客户测试、交付和内网部署。
- 前端规划已经覆盖配置、演示、规则校验、质控运营和运维追溯。

必须补齐：

- 配置包中心：业务人员需要看到包状态、版本、组织范围、hash、diff、review、publish、export。
- 演示场景库：AMI、病历质控、医保质控、医嘱安全必须可一键 dry-run。
- 发布影响分析：发布前明确影响哪些路径、规则、组织范围和适配器。
- 质控看板：从接口指标升级为临床/质控办可读的漏斗、超时、变异和规则命中视图。

### 3.2 架构师视角

优点：

- Oracle/关系型数据库作为配置主数据源的原则是正确的。
- Provider 状态探测和外部依赖降级已经建立。
- 当前模块边界基本清晰。

必须补齐：

- 统一配置包模型：所有资产进入同一生命周期，不再各模块各管各的发布。
- 组织上下文：`tenant_id/group_code/hospital_code/campus_code/site_code/department_code` 需要成为通用上下文。
- 发布链路：`DRAFT -> REVIEWED -> PUBLISHED -> SYNCED -> ACTIVE -> RETIRED` 要有统一审计和回滚。
- 同步任务：Oracle 到 Neo4j、Dify binding、前端导出导入需要异步任务状态。

### 3.3 软件开发师视角

优点：

- Spring Boot + MockMvc 契约测试适合快速迭代。
- Java 1.8 兼容性已固定，便于医院旧环境部署。
- 各模块已遵循统一 `ApiResult`。

必须补齐：

- 独立规则引擎 API：`/api/rule-engine/evaluate` 支持病历质控、医保质控、医嘱安全。
- 配置校验抽象：路径、规则、图谱、Dify、字典、适配器需要统一校验结果结构。
- API 契约整理：为前端生成类型、错误码和样例。
- 代码层 Provider 接口化：避免服务内部长期堆积外部系统细节。

### 3.4 测试人员视角

优点：

- 已有端到端契约测试覆盖核心接口。
- DB-only、Neo4j 降级、Dify 降级已经被纳入验收口径。

必须补齐：

- 配置包生命周期测试：重复导入、hash 不一致、review 阻断、发布审计、导出一致性。
- 多组织隔离测试：同一配置包在集团、医院、科室范围的继承和覆盖。
- 安全测试：越权、日志脱敏、接口签名、敏感字段泄露。
- 前端 E2E：演示场景、规则校验、发布 review、质控看板的视觉与交互验收。

### 3.5 工程与运维视角

优点：

- 脚本入口完整：测试、构建、启动、DDL、smoke、编码验证都有基础脚本。
- Provider 状态接口对部署排障有价值。

必须补齐：

- 离线部署包和升级脚本。
- 初始化检查：JDK、Oracle、Neo4j、Dify、端口、编码、DDL 版本。
- 监控指标：接口错误率、Dify 降级率、Neo4j 同步状态、规则命中率、配置发布失败率。
- 灾备恢复：通过 `full_snapshot + content_hash` 重建配置。

### 3.6 医疗人员视角

优点：

- 临床建议要求医生确认，方向正确。
- 推荐结果包含证据、缺失资料和建议动作。
- 路径变异和节点滞留适合质控复盘。

必须补齐：

- 医嘱安全拦截必须清楚区分“提醒、强提醒、阻断、需上级确认”。
- 病历质控和医保质控结果必须展示规则依据、命中事实、版本和申诉/复核入口。
- 路径推荐不能只给分数，必须给“为什么推荐、缺什么资料、下一步如何确认”。
- 临床规则和路径内容需要院方医学、医保、质控人员最终审核。

### 3.7 行政与管理人员视角

优点：

- 集团、多院区、卫生所/站点的目标已经写入总纲。
- 配置包和组织范围适合集团统一治理。

必须补齐：

- 集团级配置下发、医院级覆盖、科室级例外审批。
- 发布审批流：起草、医学审核、医保审核、信息科审核、上线批准。
- 运营报表：各院区路径入径率、变异率、规则命中率、整改闭环率。
- 权限与责任归属：谁改配置、谁审核、谁发布、谁回滚必须可追溯。

## 5. 本轮执行计划

本轮先执行 `PKG-001` 第一批。

范围：

- 新增配置包数据模型。
- 新增 `/api/config-packages` 导入、列表、详情接口。
- 新增 review 接口，输出 manifest、hash、问题列表和是否可发布。
- 新增 publish 接口，要求审批人并写审计。
- 新增 export 接口，返回可跨环境迁移的完整快照。
- 新增样例 JSON、API 示例、契约测试和文档说明。

本轮暂不做：

- Oracle 配置包正式表。
- 图谱/Dify 真实同步任务。
- 前端页面实现。
- 多级组织继承计算。

## 6. 验收标准

- DB-only 模式可导入、review、publish、export 配置包。
- hash 不一致的配置包不能发布。
- 同一 `package_code + package_version` 内容不同不能静默覆盖。
- 发布必须记录审计日志。
- 契约测试、构建、`git diff --check` 通过。
