## 1. 为什么有 V2

### 1.1 V1 之前的问题

V1 之前是 35 份独立 markdown 文档堆砌而成：

- **同概念多份定义**：菜单结构、角色定义、版本流程在 5+ 份文档里各说各话。
- **优先级飘移**：同一任务编号在不同文档里 P0/P1 标定不一致。
- **架构图风格差异**：四层架构在 `05_架构总图`、`产品化方案`、`顶级多角色评审` 各画一遍且细节不同。
- **缺少 PR 级颗粒**：任务台账只到泳道级，AI 拿到任务还要再做一轮拆分。
- **缺少产品取向**：试图用一套视觉规范覆盖三种完全不同节奏的用户。

### 1.2 V2 的取向决策

V2 做了三个**不可逆的产品判断**：

| 决策 | 内容 | 出处 |
|---|---|---|
| **三产品分层** | 平台本质是 **配置工厂**（B2B SaaS 创作） + **临床嵌入器**（HIS/EMR 旁挂预警） + **质控驾驶舱**（管理层钻取看板）三个不同节奏的产品共用底座 | §3 |
| **彻底消解** | 30 份历史文档降级为 `historical/`，5 份金本位 + 5 份核心工程规范是唯一权威 | §4 |
| **PR 级实施** | AI 实施手册精确到接下来 12 个 PR，每个 PR 给出独占文件清单和验收脚本 | `05_AI实施手册.md` |

## 3. 三产品分层是什么

```
                       医疗智能引擎平台
                              │
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
   ┌────▼─────┐         ┌─────▼─────┐         ┌────▼─────┐
   │ A 配置工厂 │         │B 临床嵌入器│         │C 质控驾驶舱│
   │           │         │           │         │           │
   │ 给医学专家 │         │ 给医生护士 │         │ 给管理层  │
   │ 创作路径、 │  共用   │ 在 HIS/EMR │  共用   │ 看全院质量│
   │ 规则、知识 │  底座   │ 上看实时预警│  底座   │ 钻取问题  │
   │           │ ────►  │            │ ◄────  │           │
   │ 创作感    │         │ 不打扰     │         │ 高密度    │
   │ Notion 范 │         │ Linear 范  │         │ Grafana 范│
   └───────────┘         └────────────┘         └───────────┘
        │                     │                     │
        └──── 共用 ────┬──── 三大引擎 ────┬──── 共用 ────┘
                       │                  │
                  路径 / 规则 / 图谱
                  字典 / 来源 / 审计
                  组织 / 身份 / 配置包
```

### 3.1 三产品的差异

| 维度 | A 配置工厂 | B 临床嵌入器 | C 质控驾驶舱 |
|---|---|---|---|
| 典型用户 | 临床路径专家、规则工程师、知识工程师、信息科 | 主治医师、护士、药师 | 主任医师、医务质控、医保审核、院级管理 |
| 使用节奏 | 周/月级（创作型） | 秒/分钟级（事件触发） | 日/周级（巡检型） |
| 信息密度 | 中（创作时聚焦单对象） | 极低（≤3 条/屏，强信号弱噪声） | 极高（多指标钻取） |
| 主导色调 | 蓝白（创作环境，注意力久） | 中性灰 + 重点语义色 | 数据色（图表色阶） |
| 主导交互 | 拖拽 / 编辑 / 版本对比 | 通知 / 一键确认 / 一键忽略 | 筛选 / 钻取 / 下钻 |
| 错误代价 | 低（草稿可回滚） | 高（直接影响诊疗） | 中（统计偏差） |
| 性能要求 | 编辑流畅、保存可异步 | P95 < 500ms（关键路径同步） | 看板 P95 < 2s（异步可接受） |

### 3.2 共用底座

不管哪个产品，都共用：

- **三大引擎**：路径 / 规则 / 图谱
- **配置数据**：字典 / 来源 / 配置包
- **基础设施**：组织 / 身份 / 审计 / traceId
- **设计 Token**：颜色、字号、间距、动效曲线（[设计系统.md](03_设计系统.md) §2）
- **质量约定**：22 个不变量（[产品事实源.md](01_产品事实源.md) §7）

### 3.3 怎么选产品归属

每个页面、每个功能、每个 API 都必须明确归属到 A/B/C 之一：

- 创作、编辑、版本、发布、配置 → **A 配置工厂**
- 实时事件、预警、医嘱校验、患者节点流转 → **B 临床嵌入器**
- 统计、看板、报表、问题追踪、整改 → **C 质控驾驶舱**

当一个对象兼具多种身份（如"规则"既要被 A 创作又要被 B 触发），把 A/B 视为两个不同视角的页面/API，而不是一个页面塞所有功能。

## 6. 维护规则

- 任何 V2 文档的修改都属于 `DOC-V2-xxx` 任务，必须在任务台账登记。
