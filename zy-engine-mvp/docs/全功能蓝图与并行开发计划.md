# 全功能蓝图与并行开发计划

## 1. 文档定位

本文是面向客户持续扩展需求后的全功能蓝图。目标不是把某个单点需求做大，而是把平台最终应具备的功能、边界、验收标准和并行开发切片一次性约定清楚。

客户提出“规则和知识需要有来源查阅”，本蓝图按以下口径处理：

- 规则必须能说明来源于哪一份医学文献、指南、专家共识、医保政策或院内制度。
- 规则结果必须能展示规则版本、规则包版本、医学文献来源摘要和审核人。
- 来源追溯平台可以作为正式模块建设，但第一阶段以规则医学文献来源字段和发布检查先落地，避免阻塞主线。
- 后续逐步扩展为来源文档、引用片段、资产绑定、审核和影响分析的完整来源追溯平台。

## 2. 总体产品形态

```text
集团化医疗智能引擎平台
= 组织与权限
+ 配置包中心
+ 路径引擎
+ 规则引擎
+ 图谱引擎
+ Dify/AI 工作流
+ 标准化中心
+ 适配器中心
+ 质控运营
+ 审计运维
+ 前端配置与演示校验平台
```

总体目标：

- 支持集团、医院、院区、卫生所/站点、科室多级组织。
- 支持 DB-only 环境完整验收，Neo4j 和 Dify 不作为强依赖。
- 支持配置导入、校验、review、发布、回滚、审计。
- 支持规则、路径、图谱、字典、适配器和 Dify 模板按组织范围继承与覆盖。
- 支持业务人员通过前端完成配置、演示、规则校验、质控复盘和运营追踪。
- 支持多个工程师或 AI 按模块并行开发，互不踩边界。

## 3. 全局约定

### 3.1 数据主源

- Oracle/关系型数据库是配置、版本、审计和运行记录主数据源。
- Neo4j 是图谱查询投影。
- Dify 是 AI 工作流执行目标。
- HIS/EMR/LIS/PACS/医保系统通过适配器读取，不直接耦合核心引擎。

### 3.2 组织范围

所有核心资产逐步支持：

```text
tenant_id
group_code
hospital_code
campus_code
site_code
department_code
scope_level
scope_code
```

配置继承顺序：

```text
科室 > 卫生所/站点 > 院区 > 医院 > 集团 > 系统内置默认
```

### 3.3 发布状态

统一配置生命周期：

```text
DRAFT -> REVIEWED -> PUBLISHED -> SYNCED -> ACTIVE -> RETIRED
```

任何发布都必须具备：

- 配置版本。
- 配置 hash。
- 组织范围。
- review 结果。
- 审批人。
- 审计日志。
- 回滚方案。

### 3.4 规则医学文献来源

规则类配置必须补充以下元数据：

```text
reference_title       医学文献/指南/政策标题
reference_type        GUIDELINE / CONSENSUS / POLICY / DRUG_LABEL / HOSPITAL_RULE / EXPERT_REVIEW
reference_org         发布机构
reference_version     来源版本或发布日期
reference_section     章节、页码、条款或摘要
evidence_level        证据等级，可为空但要预留
reviewed_by           医学/医保/质控审核人
reviewed_time         审核时间
```

发布检查：

- 病历质控、医保质控、医嘱安全、路径入径规则必须填写来源。
- 来源缺失时允许保存草稿，不允许发布。
- 来源内容不做全文管理，但必须能让客户看到“该规则依据哪份文献/指南/政策”。

## 5. 多方评审结论

### 5.1 产品设计师

- 必须把客户演示从 Postman 变成可视化工作台。
- 规则结果必须展示“为什么命中”和“依据哪份文献/指南/政策”。
- 发布前必须能看到版本 diff、影响范围和风险提示。

### 5.2 架构师

- Oracle 是唯一配置主数据源，Neo4j/Dify 只做投影和执行。
- 组织上下文必须横贯配置、运行、日志、审计和报表。
- 来源追溯平台可以建设；短期先落规则医学文献来源字段、发布检查和结果展示，中期扩展到来源文档、引用片段、资产绑定和影响分析。

### 5.3 软件开发师

- 每个模块必须有清晰接口、样例、测试和文档。
- 新增字段要考虑 Oracle/达梦兼容。
- API 响应保持向后兼容，新增能力通过可选字段扩展。

### 5.4 测试人员

- DB-only、Neo4j 降级、Dify 降级必须持续覆盖。
- 规则发布必须测试“缺医学来源不能发布”。
- 组织隔离、配置包 hash、审计日志、Oracle 落表要形成回归矩阵。

### 5.5 医疗与医保人员

- 规则必须能看到医学文献、指南、医保政策或院内制度来源。
- 医嘱安全需要区分提醒、强提醒、阻断、上级确认。
- 医保质控结果要支持申诉和复核材料导出。

### 5.6 行政与运维人员

- 发布责任要清楚：谁起草、谁审核、谁发布、谁回滚。
- 多院区运营指标要能汇总也能下钻。
- 部署、升级、回滚和灾备必须脚本化。

## 6. 并行开发路线

### P0：底座完善

可并行任务：

1. `PKG-003` 配置包 Oracle/达梦正式表。
2. `ORG-003` 组织目录持久化。
3. `PROV-001` 来源追溯平台最小底座：来源文档、引用片段、资产绑定和发布检查。
4. `RULE-005` 规则医学文献来源字段、发布检查和结果展示。
5. `TEST-001` 回归测试矩阵补齐组织、发布、医学来源检查。
6. `DOC-001` 客户验收手册补充“规则来源于哪份医学文献”的验收口径。

验收：

- 规则缺医学文献来源时不能发布。
- 规则结果能显示来源标题、版本、章节或摘要。
- 可通过来源追溯平台查询规则引用的医学文献、指南、政策或院内制度。
- DB-only 测试和构建通过。

### P1：业务能力扩展

可并行任务：

1. `TERM-001/002` 医嘱标准化与未映射治理。
2. `RULE-006/007` 病历质控、医保质控、医嘱安全规则包。
3. `PATH-007/008` 路径版本 diff 和路径关键配置依据来源字段。
4. `GRAPH-001/003` 图谱版本和证据查询。
5. `ADAPT-001/002` 适配器定义和查询定义。

验收：

- 三类规则场景可 dry-run。
- 路径、规则、图谱、字典都能进入配置包 review。
- 客户能查看规则依据来源。

### P2：前端产品化

可并行任务：

1. `FE-002` 前端工程脚手架。
2. `FE-003` 演示与规则校验工作台。
3. `FE-004` 配置包中心。
4. `FE-005` 规则配置器展示医学文献来源字段。
5. `QC-001/002` 质控看板基础报表。

验收：

- 非开发人员不用 Postman 也能完成演示和规则校验。
- 规则命中结果能显示医学来源、规则版本、traceId。
- 发布 review 可视化展示问题和风险。

### P3：生产治理

可并行任务：

1. `SEC-001/002` 权限和组织数据范围。
2. `OPS-002/003/004` 同步任务、监控、离线部署。
3. `DIFY-003/004/005` 真实 Dify 绑定、回放和解释来源展示。
4. `GRAPH-005/006` Neo4j 同步和图谱画布。
5. `DOC-002` 管理员、医学审核、实施交付手册。

验收：

- 可按组织范围进行生产发布和回滚。
- 关键运行指标可监控。
- 客户 UAT 有完整操作手册和验收矩阵。

## 8. Definition of Done

每个明确任务完成前必须满足：

1. 只修改当前任务相关文件。
2. 新增接口有契约测试。
3. 新增配置有样例 JSON 或 API 示例。
4. 需要时更新 Oracle/达梦 DDL 和迁移脚本。
5. 更新 README、产品总纲或本蓝图。
6. `run-tests.ps1` 通过。
7. `build.ps1` 通过。
8. `git diff --check` 通过。
9. 完成后提交并推送，保证其它 AI 能获取最新项目。
