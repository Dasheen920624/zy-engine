# 前端配置平台规划与开发验证

## 1. 文档定位

本文规划医疗智能引擎平台的前端配置界面、功能演示界面、规则校验工作台、设计原则、信息架构、技术选型、开发路线和验证流程。

当前仓库尚未建立独立前端工程。现阶段前端相关资料主要是早期 HTML 原型和后端 API 示例。后续需要将“配置中心、规则中心、图谱中心、Dify 契约、发布运维、标准化治理、功能演示、规则校验”做成可交付的内网管理台。

本文是 `FE-xxx` 系列任务的总入口。

## 2. 设计参考

前端设计不直接复制任何产品，但参考以下成熟体系：

- Ant Design / Ant Design Pro：适合企业级中后台产品，Ant Design 官方定位为企业级 Web 应用 UI 体系，Ant Design Pro 提供企业应用的 out-of-box 模板、表单、列表、工作台和测试实践。参考：`https://ant.design/docs/react/introduce/`、`https://github.com/ant-design/ant-design-pro`
- IBM Carbon Data Table：适合数据密集型后台，强调表格工具栏、分页、排序、批量操作、骨架屏、AI 标识和可访问性。参考：`https://carbondesignsystem.com/components/data-table/usage/`
- NHS App Design System：医疗服务强调可访问性、复用组件、经过验证的模式和从他人经验中学习。参考：`https://design-system.nhsapp.service.nhs.uk/`
- GOV.UK Design System Validation：复杂配置表单要有清晰错误摘要、字段级错误、服务端校验，不能只依赖浏览器默认校验。参考：`https://design-system.service.gov.uk/patterns/validation/`
- NHS Accessibility：医疗系统应从项目开始就考虑可访问性，持续做审计和测试，目标至少对齐 WCAG AA。参考：`https://www.nhs.uk/accessibility-statement/`
- AntV X6：适合 DAG、流程图、ER 图、血缘图等图编辑场景，可用于路径节点和图谱关系可视化。参考：`https://x6.antv.antgroup.com/en/tutorial/about`
- Apache ECharts：适合质控、命中率、同步状态、运营指标可视化。参考：`https://echarts.apache.org/en/feature.html`
- Monaco Editor：适合 JSON、规则 DSL、Cypher 预览、Dify 输入输出契约编辑。参考：`https://github.com/microsoft/monaco-editor`

## 3. 产品目标

前端配置平台要服务以下用户：

- 集团管理员：管理组织、权限、集团级配置包和全局质控。
- 医院管理员：维护本院路径、规则、字典、适配器和发布。
- 院区/卫生所管理员：维护院区能力、转诊路径、接口绑定。
- 质控办：配置病历质控、路径质控、查看指标和追溯。
- 医保办：配置医保质控、费用规则、目录映射。
- 临床专家：审核路径、规则、图谱证据和 Dify 解释模板。
- 实施工程师：导入配置、处理未映射字典、执行 dry-run。
- 运维人员：查看 Provider、同步任务、审计日志、发布状态。

核心目标：

- 让配置从“手写 JSON”升级为“结构化编辑 + JSON 可回退”。
- 让发布从“直接保存”升级为“校验、review、diff、dry-run、publish、sync、active、rollback”。
- 让多医院差异通过组织 scope 和继承覆盖管理。
- 让 DB-only 测试环境和内网 Neo4j/Dify 环境使用同一套界面。
- 让每次配置变更可追溯、可解释、可回滚。
- 让规则校验、病历质控、医保质控、医嘱安全拦截可以由业务人员在界面上选择样例、输入数据、运行模拟、查看命中依据。
- 让演示和 UAT 不依赖 Postman 或开发人员手动调用接口，而是通过可视化流程完成。

## 4. 设计原则

### 4.1 内网医疗后台，不做营销页

首屏必须是可操作工作台，不做品牌展示页。界面要安静、密集、有层次，适合反复配置、核对和审批。

### 4.2 最精确上下文优先

所有列表、编辑器、发布动作必须显示当前组织上下文：

```text
集团 / 医院 / 院区 / 卫生所或站点 / 科室
```

如果用户正在修改院区配置，界面必须清楚提示该配置是否继承自集团或医院，以及本次修改是否形成覆盖。

### 4.3 配置必须有状态

所有配置界面必须统一展示：

```text
DRAFT / REVIEWED / PUBLISHED / SYNCED / ACTIVE / RETIRED
```

状态使用一致的状态标签和筛选器。

### 4.4 编辑和发布分离

保存草稿不等于发布。发布必须经过：

```text
编辑 -> 校验 -> 差异对比 -> 模拟/ dry-run -> 审核 -> 发布 -> 同步 -> 激活
```

### 4.5 错误必须可修复

表单错误要包含：

- 页面顶部错误摘要。
- 字段级错误。
- 可定位到 JSON/DSL 行号。
- 可跳转到流程图节点或表格行。
- 服务端校验结果优先。

### 4.6 AI 生成内容必须标识

如果前端显示 Dify 或大模型生成内容，必须：

- 标记为 AI 生成或 AI 辅助。
- 显示输入来源、模板版本、调用时间。
- 提供人工确认和编辑入口。
- 不把 AI 内容直接作为临床结论。

### 4.7 可访问性和安全

目标：

- 键盘可操作。
- 表格、表单、弹窗有清晰焦点。
- 颜色不能作为唯一状态表达。
- 至少按 WCAG AA 做设计和测试。
- 不在界面泄露 API Key、数据库密码、患者完整隐私。

### 5.2 演示与校验

该模块是面向客户演示、UAT、质控办、医保办、临床专家和实施人员的可视化操作入口。它不是营销页，而是一个可运行的沙箱工作台。

页面：

- 演示场景库。
- 患者上下文构建器。
- 路径推荐演示。
- 规则校验工作台。
- 病历质控演示。
- 医保质控演示。
- 医嘱安全校验演示。
- Dify 解释回放。
- 执行结果追溯。

演示场景建议内置：

```text
AMI/STEMI 入径推荐
胸痛路径节点流转
病历质控规则命中
医保费用规则命中
医嘱安全拦截
院内医嘱名称变化后的标准化映射
Neo4j/Dify 不可用时的降级演示
```

规则校验工作台必须支持：

- 选择组织范围、规则包、规则版本、场景码。
- 选择样例患者或手工粘贴 JSON。
- 可视化编辑患者上下文、医嘱、诊断、检查检验、费用项目。
- 一键运行 dry-run，不写入正式业务状态。
- 展示命中规则、未命中规则、严重级别、建议动作、证据、耗时、traceId。
- 展示标准化前后的编码和文本，突出未映射项。
- 展示规则 DSL/条件树命中路径，能定位到具体条件。
- 支持导出本次校验报告，供医学、质控、医保人员复核。

病历质控和医保质控演示必须支持：

- 按场景输入病历摘要、诊断、医嘱、费用、检查检验。
- 运行 `EMR_QC`、`INSURANCE_QC`、`ORDER_SAFETY` 等规则场景。
- 输出问题清单、风险等级、依据、建议处理动作。
- 追溯到规则包版本、字典版本、路径版本和输入快照。

安全边界：

- 默认使用演示数据或脱敏数据。
- 演示操作默认不激活配置、不写正式患者路径状态。
- 如果需要保存演示记录，必须标记 `DEMO` 或 `SANDBOX`。
- 不允许在浏览器保存数据库密码、Dify API Key 或患者完整隐私。

对应后端：

```http
POST /api/patient-pathways/candidates
POST /api/rules/simulate
POST /api/rule-engine/evaluate
POST /api/rule-engine/batch-evaluate
POST /api/terminology/normalize
POST /api/dify/workflows/run
GET  /api/rules/exec-logs
GET  /api/rules/exec-logs/{logId}
```

其中 `/api/rule-engine/*` 是 `RULE-001` 后续要补齐的第三方规则引擎 API。

## 6. 页面模式

### 6.1 列表页模式

统一结构：

```text
标题 + 组织上下文
筛选区
主表格
批量操作
分页
右侧详情抽屉
```

表格要求：

- 列名短，必要时 tooltip。
- 状态标签统一。
- 支持搜索、排序、过滤、分页。
- 批量操作只用于低风险动作。
- 发布、回滚、同步必须二次确认。

### 6.2 详情页模式

统一结构：

```text
基础信息
状态和版本
配置内容
校验结果
影响范围
审计历史
操作区
```

### 6.3 编辑器模式

适用：

- 路径 DSL。
- 规则 DSL。
- 图谱节点边。
- Dify input mappings。
- 适配器参数映射。

统一能力：

- 左表单/右 JSON 或上表单/下 JSON。
- JSON 可导入导出。
- 结构化编辑和源码编辑互相同步。
- 服务端校验。
- 错误定位。
- 保存草稿。

### 6.4 Review 模式

统一展示：

- diff。
- 校验问题。
- 影响范围。
- 相关规则/路径/字典。
- dry-run 结果。
- 审核意见。

### 6.5 发布模式

发布前必须显示：

- 当前版本。
- 目标版本。
- scope。
- content hash。
- 发布人。
- 风险项。
- 回滚目标。

### 6.6 演示校验模式

统一结构：

```text
场景选择
组织和版本上下文
输入数据编辑区
一键运行
命中结果
证据和解释
标准化差异
trace 和审计
报告导出
```

适用：

- 规则校验。
- 病历质控。
- 医保质控。
- 医嘱安全。
- 路径推荐。
- Dify 解释回放。

要求：

- 默认 dry-run。
- 清晰区分演示数据和正式数据。
- 支持复制 traceId 给后端排查。
- 支持把失败输入保存为回归测试样例。

## 7. 技术选型建议

### 7.1 前端工程

建议：

```text
React + TypeScript + Ant Design / ProComponents
```

脚手架可选：

- Ant Design Pro：适合快速建立企业级管理台。
- Vite + React + Ant Design：适合更轻量、可控的内网管理台。

项目应支持内网离线构建和私有 npm 源。

### 7.2 状态和数据请求

建议：

- TanStack Query 或同类请求缓存库。
- OpenAPI 生成 API Client。
- 统一 request interceptor 处理 traceId、token、错误码。

### 7.3 表格和表单

建议：

- ProTable / Table。
- ProForm / Form。
- Schema 驱动表单作为后续增强。

### 7.4 画布和编辑器

建议：

- AntV X6：路径画布、图谱关系编辑。
- Monaco Editor：JSON、DSL、映射表达式编辑。
- Apache ECharts：质控和运营图表。

### 7.5 权限

前端需要支持：

- 菜单权限。
- 按钮权限。
- 数据范围提示。
- 组织上下文切换。

真正权限必须由后端校验，前端只做体验控制。

- 规则命中结果能看到规则包版本、规则版本、标准化前后值、命中条件、建议动作和 traceId。
- DB-only 环境可使用 mock/degraded 数据完成演示。

## 10. 验证计划

### 10.1 原型验证

参与人：

- 产品。
- 实施。
- 后端。
- 质控办代表。
- 医保办代表。
- 临床专家代表。

方法：

- 用 AMI 样例走查路径配置。
- 用规则包走查 review 和 publish。
- 用医嘱变更场景走查标准化中心。
- 用无 Neo4j/Dify 场景走查 Provider 状态和降级。

### 10.2 开发验证

每批前端任务必须：

- 有页面截图或录屏。
- 有 mock 数据。
- 有接口契约说明。
- 有至少一个 E2E 或组件测试。
- 通过构建。

### 10.3 UAT 验证

UAT 场景：

- 集团管理员发布集团级规则包。
- 医院管理员覆盖本院路径。
- 院区管理员维护院区能力。
- 实施人员导入医嘱映射。
- 临床专家使用 AMI 样例演示路径推荐和解释。
- 质控办使用界面运行病历质控规则校验。
- 医保办使用界面运行医保质控规则校验。
- 质控办查看路径质控。
- 运维查看 Dify 降级和 Neo4j 同步失败。

## 11. 前端 Definition of Done

每个前端任务完成前必须满足：

- 页面和路由可访问。
- 空态、加载态、错误态齐全。
- 表单有字段级错误和错误摘要。
- 操作有成功/失败反馈。
- 高风险操作有二次确认。
- 后端错误码能被正确展示。
- 有组织上下文展示。
- 不暴露敏感配置。
- 有测试或明确验证记录。
- 文档和 API 示例同步更新。

## 12. 当前结论

当前方案此前主要覆盖后端引擎和产品化底座，前端配置平台尚未形成完整开发计划。后续需要把前端作为正式产品模块推进，优先从 `FE-001 信息架构与高保真原型`、`FE-002 前端工程脚手架`、`FE-003 功能演示与规则校验工作台` 开始。
