# 产品能力缺口评估与多角色完善方案

## 1. 文档定位

本文针对 2026-05-18 用户提出的 9 项产品核查问题，统一给出：

- 当前能力评估。
- 缺陷和风险扩展。
- 产品设计方案。
- 后续 AI 开发任务拆分。
- 多角色评审结论。

本文不是替代 `00_总入口与AI接手导航.md` 和 `02_任务台账.md`，而是把新增产品诉求压缩成一份可执行方案。后续开发仍以任务台账为唯一状态源。

9 项能力的详细产品设计、数据模型、API、页面、数据库、测试和验收拆解见：`九大产品能力详细设计与落地实施方案.md`。

## 2. 总体结论

当前系统已经具备医疗智能引擎平台的核心雏形：组织上下文、配置包、规则、路径、来源追溯、前端工程、生产库/开发库分层、AI 协作和功能验收机制均已建立。但若要满足医院客户“生产试点 + 可持续知识更新 + 全功能验收 + 多院区治理”的要求，还存在 9 类关键缺口：

| 序号 | 结论 | 级别 |
|---|---|---|
| 1 用户体系 | 现有任务只覆盖 RBAC 和第三方签名，缺院内用户同步、SSO、责任链和会话治理 | P0 |
| 2 界面交互 | 已有规范和部分页面，但缺统一交互验收、角色工作台和组件一致性落地 | P0 |
| 3 AI 医疗知识 | 已有知识工厂方案，但缺用户订阅知识需求、定时/手动同步、前台审核、知识包院内同步的台账任务 | P0 |
| 4 AI 全功能验证 | 已有功能验收目录，但缺 AI 浏览器验收机器人、全链路 UAT 证据和验收中心 | P0 |
| 5 中英文混杂 | 技术字段必须英文，但面向用户的文案、状态、错误、菜单需要统一中文标准 | P0 |
| 6 临床决策支持 | 具备规则/路径/来源基础，但缺 CDSS 触发点、医生覆盖闭环、风险分级、标准接口对标 | P0 |
| 7 智能评估系统 | 有规则和质控方向，但缺统一评估指标模型、评分、复核、整改、报告闭环 | P0 |
| 8 多语言 | 前端规范仅预留 i18n，缺后端消息、配置包、字典、审计和报表的多语言资源框架 | P1 |
| 9 产品文档 | 文档很多且有历史重复，缺精简版产品事实源、冲突消解和淘汰规则 | P0 |

## 3. 逐条评估与设计方案

### 3.1 用户体系管理、院内同步和单点登录

当前覆盖：

- `SEC-001` 用户角色、`SEC-002` 数据权限、`SEC-003` 第三方鉴权已在台账中。
- 组织模型已有集团、医院、院区、站点、科室上下文。

不足：

- 缺本系统独立用户主数据和外网通用服务账号体系；现有描述容易被误解为完全依赖院内用户。
- 缺院内 HIS/EMR/OA/统一身份平台用户同步模型。
- 缺 SSO 协议支持：CAS、OAuth2/OIDC、SAML2、LDAP/AD 至少要有适配策略。
- 缺医生、护士、药师、质控、医保、信息科、管理员与组织范围的映射。
- 缺账号生命周期：入职、调科、停用、离职、锁定、密码策略、会话失效。
- 缺责任链：谁起草、谁审核、谁发布、谁回滚、谁复核、谁处理告警。

设计方案：

```text
本系统独立身份中心
-> 租户 / 用户 / 角色 / 权限 / 组织范围 / 服务账号
-> 外网通用服务登录与客户开通
-> 院内身份源绑定
-> 用户同步适配器
-> SSO 登录
-> 后端鉴权和数据权限
-> 前端菜单/按钮权限
-> 审计责任链
```

必须支持：

- 本系统必须拥有 `platform_user_id` 作为唯一审计主体，院内账号、SSO subject、LDAP DN、API client 都只是绑定身份。
- 外网通用服务必须支持客户租户开通、平台运营账号、客户管理员、本地密码/MFA、服务账号和 API 客户端。
- 院内用户全量同步、增量同步、手动同步。
- 多身份源优先级：本系统用户主数据为准；登录凭证可来自本地账号、院内 SSO、LDAP/AD 或 API Key 客户端。
- 组织范围绑定：用户可以跨集团、医院、院区、科室授权。
- 登录态和第三方 API 调用分离：人用 Session/Token，系统调用用 API Key + 签名。

详细设计见：`用户体系与身份联邦详细设计方案.md`。

新增任务：`SEC-001`、`SEC-006`、`SEC-007`、`SEC-008`、`SEC-009`、`SEC-011`、`SEC-012`。

### 3.2 界面功能交互和完整方案

当前覆盖：

- `07_前端开发规范.md` 和 `前端产品交互与视觉规范.md` 已定义天蓝白底、工作台、列表页、审核页、状态标签、AI 标识、来源卡片。
- `FE-002/003/004` 已完成前端脚手架、演示工作台、配置包中心。

不足：

- 任务台账未把完整交互验收、角色工作台、组件一致性、可用性走查排成高优先级。
- 部分页面仍是占位或局部功能，没有按角色工作任务串成完整客户路径。
- 还缺“无权限态、空态、错误态、加载态、降级态、traceId”的统一验收。

设计方案：

- 建立统一产品壳：顶部天蓝导航 + 左侧二级导航 + 组织上下文条 + 白色工作区。
- 建立 8 个一级入口：工作台、演示与校验、配置中心、标准化中心、来源追溯、质控运营、发布与同步、审计运维。
- 所有页面按角色任务命名，不按技术模块命名。
- AI 生成内容必须显示候选、来源、模型/工作流、审核状态，不能显示为最终结论。

新增任务：`FE-011`、`FE-012`、`FE-013`、`FE-014`。

### 3.3 AI 自动生成医疗知识、用户约定知识需求和知识包同步

当前覆盖：

- `AI医疗知识工厂与字典映射方案.md` 已设计来源接入、AI 抽取、候选映射、规则候选、图谱断言、知识包导入。
- `全功能蓝图与并行开发计划.md` 已列出 `AIK-*`、`PKG-AI-*`、`FE-AI-*` 方向。

不足：

- 台账未正式纳入这些任务。
- 缺“用户约定想获取哪些医疗知识”的订阅模型。
- 缺定时同步、手动同步、同步差异预览和失败重试。
- 缺前台审核台：审核 AI 生成的来源、规则、图谱、字典映射和测试样例。
- 缺医疗知识包从外网基准版同步到院内系统的完整闭环。
- 缺授权和版权治理，无法判断知识是否允许商业使用、分发、导出。

设计方案：

```text
知识需求订阅
-> 来源注册和授权校验
-> 定时/手动同步任务
-> AI 抽取与候选生成
-> 自动质检
-> 前台专家审核
-> 医疗知识包发布
-> 离线导出
-> 院内导入 dry-run
-> 院内审核激活
```

知识订阅支持：

- 按疾病、科室、指南、医保政策、药品、检查项目、质控主题订阅。
- 设置同步频率、来源白名单、语言、地区、证据等级。
- 手动触发同步和定时同步并存。
- 每次同步产生差异报告、风险报告、授权报告和回滚点。

新增任务：`AIK-001`、`AIK-002`、`AIK-003`、`AIK-004`、`AIK-005`、`PKG-AI-001`、`FE-AI-001`、`OPS-AI-001`。

### 3.4 AI 全面功能验证和最终项目验收

当前覆盖：

- 已建立 `ai-dev-input/13_feature_acceptance/` 功能验收目录。
- 已有 `TEST-003` 全功能逐项验收与专业优化执行任务。

不足：

- 缺 AI 自动浏览器验收机器人。
- 缺全功能验收中心，无法按功能、角色、页面、接口、数据库、来源、安全汇总证据。


### 3.5 中英文混杂与标准化中文统一

当前覆盖：

- `编码与中文备注规范.md` 已约定面向院方评审、产品设计、实施配置的文档使用中文；接口字段、枚举、表字段保持英文。
- 数据库备注要求中文。

不足：

- 前端菜单、按钮、状态、错误、空态、表格列名尚未建立统一词表。
- API 错误码为英文，缺面向用户的中文消息表。
- 文档和样例中存在 dry-run、review、publish、Provider、traceId 等混排词，需定义标准中文译名或保留规则。

统一原则：

| 类型 | 规则 |
|---|---|
| 面向用户文案 | 标准中文 |
| 页面菜单和按钮 | 标准中文，必要时括号说明技术词 |
| 错误提示 | 中文解释 + traceId |
| 技术字段/API/枚举 | 保持英文，不翻译 |
| 数据库表名/列名 | 保持英文，备注中文 |
| 文档首次出现技术词 | 中文名 + 英文原词 |

新增任务：`I18N-001`、`DOC-007`、`TEST-006`。

### 3.6 临床决策支持系统底座

当前覆盖：

- 规则引擎、路径引擎、来源追溯、医生确认原则已有。
- 文档已引用 CDS Hooks 方向。

不足：

- 缺 CDSS 事件触发点：开医嘱、写病历、入径、检验检查申请、医保结算前。
- 缺动作分级：提示、强提醒、阻断、需上级确认、仅记录。
- 缺医生覆盖/忽略原因、二次确认、提醒疲劳治理。
- 缺与院内业务系统的嵌入方式：API、iframe、CDS Hooks、消息队列、页面跳转。
- 缺医疗安全评审：证据等级、适用人群、排除条件、禁忌、人工确认。

设计方案：

```text
院内业务事件
-> CDSS Trigger
-> 患者事实标准化
-> 规则/路径/图谱评估
-> 来源证据和风险等级
-> 医生确认/覆盖
-> 审计和复盘
```

新增任务：`CDSS-001`、`CDSS-002`、`CDSS-003`、`CDSS-004`。

### 3.7 智能评估系统底座

当前覆盖：

- 规则评估、质控方向、医保方向、路径指标已有规划。

不足：

- 缺统一评估对象：患者、病历、科室、规则包、路径包、医保风险、配置质量。
- 缺评估指标模型、评分规则、权重、阈值和版本。
- 缺评估结果复核、整改、再评估和报告导出。
- 缺模型/规则效果评估：准确率、召回率、人工驳回率、误报率、漏报率。

设计方案：

```text
评估对象
-> 指标集
-> 评分和风险等级
-> 证据和来源
-> 人工复核
-> 整改任务
-> 再评估
-> 报告和看板
```

新增任务：`EVAL-001`、`EVAL-002`、`EVAL-003`、`EVAL-004`。

### 3.8 多语言系统要求

当前覆盖：

- 前端规范要求第一版 zh-CN，未来预留 i18n。

不足：

- 后端错误消息、状态文案、审计文案、配置包名称、报表模板、知识来源语言没有统一多语言设计。
- 缺 `locale` 透传、语言资源表、导出报表语言选择。
- 缺多语言知识包：中文指南、英文指南、国家/地区差异。

设计方案：

- 第一阶段以中文为主，但引入资源键和中文默认资源。
- 前端所有可见文案逐步从资源文件读取。
- 后端错误码返回稳定英文 code，同时返回中文 message，后续支持 message_key。
- 知识来源保留原始语言、中文译名、翻译审核状态。

新增任务：`I18N-002`、`I18N-003`、`I18N-004`。

当前覆盖：

新增任务：`DOC-006`。

## 5. 多角色评审

本次方案按以下角色完成交叉评审。结论不是“已实现”，而是“方案可执行，必须按任务台账落地并验收”。

| 角色 | 评审重点 | 结论 | 必须补充 |
|---|---|---|---|
| 产品负责人 | 9 项需求是否变成闭环能力 | 通过，有任务化要求 | `DOC-006` 精简产品事实源 |
| 临床专家 | CDSS 安全、医生确认、来源证据 | 条件通过 | `CDSS-001/003/004` |
| 医保/质控专家 | 智能评估、复核、报告 | 条件通过 | `EVAL-001/002/003` |
| 架构师 | Provider、DB-only、知识包同步 | 条件通过 | `AIK-001/003`、`PKG-AI-001` |
| 后端工程师 | 用户、SSO、知识任务、评估引擎 | 条件通过 | `SEC-006/007`、`AIK-*` |
| 前端工程师 | 交互、审核台、验收中心 | 条件通过 | `FE-011/012/013/014` |
| 数据库负责人 | 多库 DDL、备注、审计、知识包表 | 条件通过 | 每个新增表同步 Oracle/DM/PG/LOCAL_H2 |
| 测试负责人 | AI 自动验收、UAT 证据、回归矩阵 | 条件通过 | `TEST-004/005/006` |
| 安全运维 | SSO、脱敏、权限、日志、灾备 | 条件通过 | `SEC-009`、`OPS-006/007/008` |
| 实施交付 | 院内同步、知识包导入、用户培训 | 条件通过 | `FE-014`、`DOC-006` |

总体评审结论：

```text
APPROVED_WITH_TASKS
```

放行条件：

- 所有 P0 任务进入任务台账。
- 后续 AI 必须逐项认领，不允许只按文档口头承诺。
- 每个客户可见功能必须进入 `ai-dev-input/13_feature_acceptance`。
- AI 验收机器人和多角色评审必须作为正式验收证据。

## 6. 后续开发优先级

第一批 P0（扩展后）：

1. `DOC-006` 产品文档精简、冲突消解和最小阅读路径同步。
2. `COMP-001` 合规基线与证据包。
3. `DATA-001` 主数据、数据字典和数据质量治理。
4. `INTEROP-001` 院内互联互通标准适配矩阵。
5. `SEC-006` 院内用户体系同步。

## 7. 验收标准

本方案完成后，客户应能看到：

- 用户从院内系统同步，并能通过 SSO 登录。
- 医生、质控、医保、信息科看到各自工作台和待办。
- 用户可订阅医疗知识主题，手动或定时同步。
- AI 生成候选知识、规则、映射和图谱，但必须前台审核。
- 审核通过后生成知识包，并导入院内系统 dry-run、发布、回滚。
- AI 能自动打开界面、执行功能、截图、跑接口和生成验收证据。
- 所有面向用户的文案是标准中文，技术标识保持稳定英文。
- 系统具备临床决策支持底座和智能评估底座。
- 多语言能力有资源框架，不会被中文硬编码锁死。
- 产品文档精简后，后续 AI 读最少文档即可获取完整信息。
