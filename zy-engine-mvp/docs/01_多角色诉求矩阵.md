# 01 多角色诉求矩阵

## 0. 用途

本文把 **顶级医疗产品经理 / 顶级医疗产品设计师 / 顶级架构师 / 顶级后端工程师 / 顶级前端工程师 / 顶级测试工程师 / 信息科工程师 / 顶级临床医生 / 医院信息化院领导** 9 类角色的核心诉求结构化成一张矩阵。

任何 AI 在领到任务时，先回答下面的问题：

1. 本任务主要服务哪一个或哪几个角色？
2. 这些角色的"必看"是否在我的实现里可看到？
3. 这些角色的"必交付"是否在我的产出里出现？
4. 这些角色的"否决项"我是否触碰？

任一条不满足 → 视为 DoD 未达成。

## 1. 矩阵速览

| 角色 | 主要场景 | 主要 UI/接口入口 | 谁审核 |
|---|---|---|---|
| 产品经理 | 需求定义、客户演示、UAT 编排 | 工作台 / 演示与校验 / 院级驾驶舱 | 院方 |
| 产品设计师 | 信息架构、交互、可访问性、错误恢复 | 全部前端页面 | 产品经理 + 临床医生 |
| 架构师 | Provider 边界、配置生命周期、可演进性 | 后端代码评审 + Provider 状态页 | 内部 |
| 后端工程师 | 接口实现、契约测试、Oracle 持久化 | `src/main/java` + 契约测试 | 测试 + 架构师 |
| 前端工程师 | 配置/演示/规则校验/质控看板 | `frontend/`（FE-002 后建）| 产品设计师 + 产品经理 |
| 测试工程师 | 契约测试 / DB-only / 降级 / UAT 矩阵 | `src/test` + scripts + `07_tests/` | 内部 |
| 信息科 | 集成对接、字典治理、运维部署、数据安全 | 适配器中心 / 标准化中心 / 审计运维 | 院领导 + 安全 |
| 临床医生 | 路径推荐、医嘱安全、规则结果可解释 | 医生工作台（FE 后建）+ 规则结果卡片 | 自己 |
| 院领导 | 多组织运营指标、合规追责、变更治理 | 院级驾驶舱（QC 看板） | 自己 |

## 2. 详细诉求（每角色 4 段：必看 / 必交付 / 否决项 / 验收口径）

### 2.1 顶级医疗产品经理

**必看（Must Know）**

- 当前每个任务编号背后服务的业务场景：AMI 入径 / 病历质控 / 医保质控 / 医嘱安全 / 用药适应症 / 检查合理性。
- 每个场景的"客户验收剧本"——从导入配置 → review → publish → 演示 → 报告导出的完整闭环。
- 当前组织模型（集团/医院/院区/站点/科室）和配置继承顺序。
- 临床/医保/质控规则的来源完整性。
- 全功能蓝图中 P0–P3 当前进度。

**必交付（Must Ship）**

- 客户演示首屏是 **工作台**，不是宣传页（见 `前端配置平台规划与开发验证.md` §4.1）。
- 每个新功能都能映射到 9 角色诉求矩阵的至少 1 个角色。
- 每个临床或质控类规则结果显示：命中原因 + 患者事实 + 规则版本 + 包版本 + 来源摘要 + 审核人 + traceId。
- 客户验收报告模板（见 [04_客户验收剧本与报告模板.md](04_客户验收剧本与报告模板.md)）按角色填写完整。
- UAT 通过率 ≥ 95%。

**否决项（Must Not）**

- 把营销页或品牌展示页作为首屏。
- 把"接口能通"当作产品交付。
- 临床/医保规则结果只输出"命中/未命中"而没有依据和动作建议。
- 把单医院定制需求写进核心引擎（必须沉淀为配置包）。

**验收口径**

- 至少 5 类剧本（AMI 推荐 / 病历质控 / 医保质控 / 医嘱安全 / 院级驾驶舱）可在演示工作台一键 dry-run。
- 演示报告 PDF/HTML 可导出并能提交给客户签字。

### 2.2 顶级医疗产品设计师（含交互/视觉/可用性）

**必看**

- 医院后台是"内网生产工具"，不是 ToC 产品；密集表格、表单、状态标签优先于动效。
- NHS / GOV.UK / IBM Carbon / Ant Design Pro 等 4 套体系的取舍（见 `前端配置平台规划与开发验证.md` §2）。
- 9 类角色每天用得最多的 3 个页面分别是哪几个。
- 表单错误的服务端校验、字段定位、JSON 行号、画布节点跳转能力。
- WCAG AA 可访问性目标。

**必交付**

- 所有页面具备 **空态 / 加载态 / 错误态 / 无权限态** 4 种状态。
- 表单错误必须：顶部摘要 + 字段级 + JSON 行号 + 画布节点定位。
- 颜色 **不是** 唯一状态表达；状态标签同时有图标 + 文字 + 颜色。
- 危险操作（发布、回滚、同步、删除）必须二次确认 + 显示影响范围。
- AI 生成内容必须显式标注"AI 生成"+ 模板版本 + 调用时间。
- 来源卡片格式：标题 + 机构 + 版本 + 章节/页码 + 摘要 + 证据等级 + 推荐强度 + 审核人 + 适用组织。

**否决项**

- 用颜色单独表达"已发布/未发布"（弱视用户不可读）。
- 用 Modal 弹窗承载长表单（应使用抽屉或独立页）。
- 让用户在没看到 diff 的情况下点"发布"。
- AI 生成内容混入正式临床字段而不标注。

**验收口径**

- Playwright 截图 + 键盘焦点检查 + axe-core 自动可访问性扫描通过。
- 5 类剧本走查无明显交互断点。

### 2.3 顶级架构师

**必看**

- Oracle 主源 / Neo4j 投影 / Dify 执行目标 的分层。
- Provider 体系：DatabaseProvider / GraphProvider / WorkflowProvider / AdapterProvider / TerminologyProvider / SourceProvider / AuditProvider。
- 配置包生命周期 `DRAFT → REVIEWED → PUBLISHED → SYNCED → ACTIVE → RETIRED`。
- 组织上下文五段式（tenant / group / hospital / campus / site / department）。
- 来源追溯横切：SRC_DOCUMENT / SRC_CITATION / SRC_ASSET_BINDING / SRC_REVIEW_RECORD / SRC_RUNTIME_EVIDENCE。
- 五类底座：CFG_* / ORG_* / SRC_* / RUN_* / SEC_*。

**必交付**

- 任何新引入的外部依赖都必须是 Provider 接口实现，而不是服务内直接耦合。
- 任何新配置类必须接入配置包生命周期，不允许各模块独立发布。
- 任何新表必须：生产库 Oracle/达梦/PostgreSQL-Kingbase + 开发库 LOCAL_H2_FILE 四类结构同步，包含中文备注、索引设计、迁移脚本、开发库验证和生产库 smoke 计划。
- 同步任务（Oracle→Neo4j / Dify binding / 配置导入导出）必须走 OPS_SYNC_TASK 异步模型。
- 任何运行链路：组织上下文 → 标准化事实 → 选择 ACTIVE 包 → 执行 → 生成证据链 → 审计 + traceId。

**否决项**

- 让某模块独立保存 ACTIVE 版本指针（必须统一在配置包/资产注册表）。
- 在业务服务里写 `if (provider == "Dify")`（必须 Provider 注入 + 适配器）。
- 用单例缓存配置而不走版本化机制。
- 让外部系统传入任意 SQL / Cypher 到核心引擎。

**验收口径**

- 关闭 Neo4j 和 Dify 后，run-tests 仍 PASS、AMI 演示剧本仍可走通。
- 同一规则编码在两个医院范围下可独立存在并被分别命中。

### 2.4 顶级后端工程师

**必看**

- 当前模块边界（adapter / audit / common / config / dify / dto / graph / organization / pathway / persistence / quality / rule / system / terminology / util）。
- JDK 1.8 + Spring Boot 2.7.18 + Oracle 兼容性约束。
- `ApiResult` + `traceId` + 统一错误码。
- 已有契约测试模式（`EngineApiContractTests` 等）。
- Provider 实现位置、IoC 注入方式。

**必交付**

- 新接口必返回 `ApiResult` 且透出 `trace_id`。
- 新接口必有契约测试 + API 示例 + 样例 JSON。
- 新表 + 新字段必须同步更新 Oracle、达梦、PostgreSQL-Kingbase、LOCAL_H2_FILE DDL、迁移脚本、中文备注。
- ID 应用层生成，不依赖数据库自增。
- JSON 大字段可存 CLOB，但查询关键字段必须结构化（可索引）。
- 涉及组织上下文的接口必须支持 Header/Query/Body 三方合并。

**否决项**

- 在业务代码里硬编码任何医院编码、医嘱名称、规则字面值。
- 在 controller 里直接拼 SQL 字符串。
- 在日志里 `log.info("password={}", password)` 之类输出敏感信息。
- 跳过 review/publish 直接激活配置。

**验收口径**

- `run-tests.ps1` + `build.ps1` + `git diff --check` 全部 PASS。
- 涉及 Oracle 时 `run-oracle-org-smoke.ps1` PASS。

### 2.5 顶级前端工程师

**必看**

- React + TypeScript + Ant Design / ProComponents 选型。
- AntV X6（路径/图谱画布）、Monaco（JSON/DSL）、ECharts（看板）选型。
- 前端规划 8 大区域：工作台 / 演示与校验 / 组织权限 / 配置包中心 / 来源追溯 / 路径/规则/图谱/Dify/标准化/适配器 / 发布同步 / 质控运营 / 审计运维。
- 接口契约（`/api/system/providers`、`/api/rule-engine/*`、`/api/config-packages/*`、`/api/patient-pathways/*` 等）。
- 内网离线 npm 源构建要求。

**必交付**

- 工程脚手架：路由、Layout、API client、统一错误处理、Provider 状态页、mock 数据、基础测试。
- 任何列表页：标题 + 组织上下文 + 筛选 + 表格 + 批量 + 分页 + 详情抽屉。
- 任何详情页：基础信息 + 状态版本 + 配置内容 + 校验结果 + 影响范围 + 审计历史 + 操作区。
- 任何编辑器：表单 + JSON 双向同步 + 服务端校验 + 错误定位 + 草稿。
- 任何 review 页：diff + 校验问题 + 影响范围 + dry-run 结果 + 审核意见。
- 任何演示页：场景选择 + 上下文编辑 + 一键运行 + 命中结果 + 证据 + 标准化差异 + traceId + 报告导出。

**否决项**

- 把 token / API Key / 数据库密码放进前端代码或 localStorage。
- 用前端权限替代后端权限（前端只做体验，后端必须校验）。
- 让 dry-run 默认写正式数据。
- 同一规则结果展示组件在"演示"和"配置编辑"中实现两份（必须共用）。

**验收口径**

- Playwright E2E 跑通 AMI / EMR_QC / INSURANCE_QC / ORDER_SAFETY 4 类剧本。
- Lighthouse / axe 可访问性扫描无致命问题。

### 2.6 顶级测试工程师

**必看**

- 已有契约测试覆盖范围（health / rules / pathways / terminology / adapters / graph / dify / config-packages / org-context）。
- 已有 smoke 脚本（pathway / rule / graph-dify / terminology-adapter / config-import / oracle-org）。
- Oracle 真实验证模式。
- DB-only + Neo4j 降级 + Dify 降级三种验收模式。
- 测试矩阵口径：组织范围 / 来源状态 / 发布状态 / Provider 状态 / 数据库 / 安全。

**必交付**

- 每个新接口对应至少 1 条契约测试 + 1 条样例 JSON。
- 涉及多组织的接口必须有"科室 vs 院区 vs 集团"独立覆盖测试。
- 涉及来源追溯的发布接口必须有"缺来源 / 来源过期 / 未审核 / 已审核"4 状态测试。
- 涉及 Provider 的接口必须有"Provider 不可用→降级输出"测试。
- 涉及 Oracle 的任务必须新增/扩展 Oracle smoke。
- 每个新发现的回归 Bug 必须落地为契约测试。

**否决项**

- 用内存测试代替 Oracle 真实落库验证。
- 用 mock Provider 跑出"PASS" 就声称功能完成。
- 长期 disabled / @Ignore 测试不在最终回复中说明。

**验收口径**

- 测试矩阵（`07_tests/test_case_matrix.md`）覆盖率 ≥ 90% 的 P0/P1 任务。
- 关闭 Neo4j 和 Dify 后 run-tests PASS。
- Oracle smoke 在 CI 或 nightly 触发并通过。

---

### 2.7 信息科工程师（医院侧实施 / 集成 / 运维）

**必看**

- HIS / EMR / LIS / PACS / 医保 接口现状（每家医院不同）。
- 院内字典治理流程（医嘱、诊断、检查、检验、收费）。
- 离线部署、升级、回滚、灾备能力。
- 生产库 Oracle / 达梦 / PostgreSQL-Kingbase 与开发库 LOCAL_H2_FILE DDL 兼容性。
- 监控指标（接口错误率、规则命中率、Dify 降级率、Neo4j 同步失败率、同步任务积压）。
- 安全合规（脱敏日志、API Key 管理、IP 白名单、签名、防重放）。

**必交付**

- 适配器中心可按医院/院区/站点绑定不同适配器。
- 标准化中心未映射清单可由信息科治理（导入映射、确认建议、提交审核）。
- 离线部署包：JDK + Maven 仓库 + Oracle 客户端 + 启动脚本 + 升级脚本 + 回滚脚本。
- 初始化检查：JDK / sqlplus / Oracle 连通性 / 端口 / 编码 / DDL 版本 一键诊断。
- 配置导出/导入：测试环境导出，内网导入并 hash 校验。
- 审计与 traceId 链路：通过 traceId 可串联前端 → 后端 → Oracle → 第三方调用。

**否决项**

- 让信息科手改 Java 代码完成医院适配。
- 让信息科直接登录 Neo4j 改图谱主数据。
- 把数据库密码硬编码到 properties。
- 没有回滚方案就直接升级生产配置。

**验收口径**

- 一台干净 Windows Server + Oracle 11g/19c 上能按手册离线完成部署 + 初始化 + smoke。
- 同步失败任务可重试 + 可回滚。
- 任何接口异常都能通过 traceId 在 24 小时内查到链路。

### 2.8 顶级临床医生

**必看**

- 推荐路径的命中原因 + 患者事实 + 缺什么资料 + 下一步如何确认。
- 规则结果的：`clinical_severity / action_mode / applicable_population / exclusion_criteria / evidence_summary / evidence_level / recommendation_grade`。
- 来源卡片：标题 + 机构 + 版本 + 章节 + 摘要 + 证据等级 + 推荐强度 + 审核人。
- 医生覆盖入口（覆盖原因 / 暂缓原因）。

**必交付（系统对医生）**

- 推荐结果必须可解释、可覆盖、可记录原因。
- 医嘱安全规则区分 4 档：REMIND / STRONG_REMIND / BLOCK / REQUIRE_CONFIRM / RECORD_ONLY。
- 阻断类规则必须院内授权 + 医学审核记录 + 紧急豁免通道。
- 来源摘要必须可一眼看懂（不超过 80 字 + "查看完整来源"链接）。
- 提醒疲劳防护：低价值提醒可统计、合并、降级、按科室关闭。

**否决项**

- AI 给出未绑定来源的治疗建议。
- 把阻断类规则混在普通规则里 一起命中。
- 让医生在没看到来源的情况下被强制阻断。
- 提醒卡片堆 5 条以上而不分级折叠。

**验收口径**

- 临床走查：3 位主治医生在 30 秒内能看懂为什么推荐。
- 医生覆盖原因统计可被质控复盘。

---

### 2.9 医院信息化院领导

**必看**

- 院级运营驾驶舱：路径入径率 / 节点完成率 / 变异率 / 滞留时长 / 规则命中率 / 医保风险金额 / 病历质控缺陷率 / 提醒忽略率。
- 多组织下钻：集团 → 医院 → 院区 → 科室 → 医生。
- 配置发布责任链：起草人 / 医学审核 / 医保审核 / 信息科审核 / 上线批准 / 回滚人。
- 来源影响分析：某文献/指南更新时影响哪些规则、路径、图谱。
- 外部系统稳定性：适配器超时/错误统计、Provider 状态。

**必交付（系统对院领导）**

- 院级驾驶舱按"近 7 天/30 天/季度"切换。
- 风险榜单：高频被忽略规则 / 高金额医保风险 / 高滞留节点 / 高变异科室。
- 发布审批台账：每次发布的版本、hash、diff、审批人、影响范围、回滚目标。
- 上线前后对比：入径率、质控缺陷率、医保风险率、平均处理时长。

**否决项**

- 看板上有数字但点不进详情（必须可下钻）。
- 看板上指标定义不一致（必须有口径说明）。
- 发布操作只有时间没有责任人（必须 4 段责任链）。

**验收口径**
